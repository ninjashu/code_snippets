AUTOSAR CAN Driver (CanDrv) complete study guide — R24-11

What is the CAN Driver (CanDrv)?

CanDrv (Can module) is the lowest layer in the AUTOSAR CAN stack. It owns the hardware — directly reading/writing CAN controller registers, firing ISRs, and implementing all hardware-specific logic. Its ONLY upper-layer client is CanIf. Everything above CanIf never touches CanDrv directly.

Core concepts — know these cold

CAN Hardware Unit

One or more CAN controllers of the SAME type sharing HW resources (RAM, etc). Represented by ONE CanDrv instance. Multiple HW Units of different types → multiple CanDrv modules with unique prefixes.

HTH — Hardware Transmit Handle

Identifies a CAN Tx mailbox in HW RAM. Passed by CanIf to Can_Write(HTH,...). One HTH can map to one or multiple HW objects (multiplexed Tx). HTH numbers are unique within one HW unit, start at 0.

HRH — Hardware Receive Handle

Identifies a CAN Rx mailbox. CanDrv passes it back to CanIf via CanIf_RxIndication(Mailbox→HRH,...). CanIf uses HRH to apply software filtering. One HRH can cover a FIFO of multiple HW objects.

Priority Inversion

Inner: Low-priority msg in only Tx buffer blocks higher-priority msg from the same node. Fix: use multiple Tx objects (FullCAN or multiplexed). Outer: gap between consecutive Tx allows another node to insert lower-priority frame. Fix: very fast internal HW copy to fill gap.

Two driver state levels (important for exams)

Driver-level: CAN_UNINIT ↔ CAN_READY — one state for the whole module (all controllers in HW unit). Controlled by Can_Init() and Can_DeInit().

Controller-level: UNINIT → STOPPED → STARTED ↔ SLEEP — one state per CAN controller. Controlled by Can_SetControllerMode(). These are independent from each other.

Processing modes — interrupt vs polling

Interrupt mode

ISR fires on Tx done / Rx received / BusOff / Wakeup. ISR calls CanIf_TxConfirmation / CanIf_RxIndication directly. Fastest response.

Polling mode

SchM calls Can_MainFunction_Write/Read/BusOff/Wakeup periodically. These scan HW status bits and call the same callbacks as ISR mode. No interrupts used.

Mixed mode

Some HW objects use interrupt, others use polling. Configured per HW object via CanHardwareObjectUsesPolling. Most flexible but requires careful timing.

State transitions are asynchronous — Can_SetControllerMode() initiates the transition; CanIf_ControllerModeIndication() callback confirms it. Monitoring is the upper layer's job, not CanDrv's.

Key transition rules (exam traps)

CS_STARTED re-initializes

Every transition to STARTED calls Can_SetControllerMode(CS_STARTED) which re-initializes with the SAME baud rate config last used. Tx requests before fully operational may be lost — only Tx confirmation proves operability.

CS_STOPPED cancels pending Tx

Can_SetControllerMode(CS_STOPPED) shall cancel still-pending transmit requests. BusOff also cancels them. CanIf will call LSduR_CanIfTxConfirmation(E_NOT_OK) for each.

BusOff: no auto-recovery

CanDrv SHALL disable automatic BusOff recovery. Recovery is exclusively driven by CanSm via CanIf → Can_SetControllerMode(CS_STOPPED) then CS_STARTED.

SetBaudrate constraints

Can_SetBaudrate() requires CS_STOPPED if re-init is needed. It changes only the addressed controller's registers. Leaves controller in STOPPED after re-init.

Wakeup Concept

Interrupt wakeup: CAN bus activity → HW wakeup ISR → CanDrv ISR calls EcuM_CheckWakeup(WakeupSource). EcuM then calls Can_CheckWakeup() via CanIf. If confirmed, Can_CheckWakeup() calls EcuM_SetWakeupEvent(). EcuM then drives CS_STARTED and wakeup validation.

Polling wakeup: EcuM periodically calls Can_CheckWakeup() directly. Can_CheckWakeup() polls HW status bit. Same callback chain follows.

During sleep transition: If wakeup event occurs while transitioning to SLEEP → Can_SetControllerMode(CS_SLEEP) returns E_NOT_OK. Controller goes STOPPED instead.

CanDrv does NOT validate wakeup — that is EcuM + CanIf's job. CanDrv does not process the L-PDU that caused the wakeup (dropped).

Partial Networking — selective wakeup

When CanHwPnSupport=TRUE, controller can be woken by predefined frames only. Configured via CanPnFrameCanId, CanPnFrameCanIdMask, CanPnFrameDlc, and CanPnFrameDataMaskSpec. When CS_STARTED confirmed → CanDrv calls CanIf_ConfirmCtrlPnAvailability() to notify CanNm (via CanIf/CanSm) that selective wakeup is enabled.

Interrupt Enable/Disable — reentrancy mechanism

Can_DisableControllerInterrupts(): Uses an internal counter (incremental disable). Each call increments counter. Disables all HW interrupts for that controller IF not already disabled.

Can_EnableControllerInterrupts(): Decrements counter. Only re-enables interrupts when counter reaches 0. Does nothing if DisableInterrupts was never called. This prevents nested disable/enable from accidentally re-enabling mid-critical-section.

CRITICAL: Can_SetControllerMode() also enables/disables interrupts internally. The driver must track the "should-be-enabled" state via software flags so that when EnableInterrupts is called, the correct set is restored — not all interrupts blindly.

Data mapping rule (byte order)

CAN data byte sent first = array[0]. CAN data byte sent last = array[7] (CAN 2.0) or array[63] (CAN FD). If HW buffer layout differs from this, CanDrv provides an adapted SDU buffer — upper layers always see this standardized format.

Hardware Timestamping (CanGlobalTimeSupport)

Free-running counter, ≥1µs resolution, ≤1ns resolution. 32-bit counter recommended.
Can_GetCurrentTime(ControllerId, *ts) — read current HW time.
Can_EnableEgressTimeStamp(Hth) — arm timestamp capture on next Tx.
Can_GetEgressTimeStamp(TxPduId, Hth, *ts) — call inside TxConfirmation to retrieve Tx timestamp.
Can_GetIngressTimeStamp(Hrh, *ts) — call inside RxIndication to retrieve Rx timestamp.
Used for CanTSyn (global time sync over CAN). Enable via CanGlobalTimeSupport=TRUE.

Remote frames

CanDrv SHALL NOT transmit messages triggered by remote frames and SHALL initialize HW to ignore remote transmission requests. Remote frames are completely unsupported by AUTOSAR CAN stack.

Critical type definitions

Can_IdType (uint32)

Bit 31 (MSB) = Extended ID flag. Bit 30 = CAN FD flag.

00 = Standard CAN 2.0 (11-bit ID in bits 0-10)
01 = CAN FD with Standard ID
10 = Extended CAN 2.0 (29-bit ID in bits 0-28)
11 = CAN FD with Extended ID

CanDrv sets MSB when calling CanIf_RxIndication for extended frames.

Can_PduType

swPduHandle (PduIdType) — stored by CanDrv, returned in TxConfirmation
length (uint8) — SDU length in bytes
id (Can_IdType) — CAN ID with type bits
sdu (uint8*) — pointer to data (NULL for trigger Tx)

Can_HwType (Mailbox)

CanId — received CAN ID
Hoh — HRH or HTH handle
ControllerId — abstract CanIf controller ID

Passed to CanIf_RxIndication as *Mailbox. CanIf uses ControllerId to distinguish which CanDrv called (multi-driver support).

Can_ErrorStateType

CAN_ERRORSTATE_ACTIVE — full participation, sends active error frames
CAN_ERRORSTATE_PASSIVE — participates but sends passive error frames only (TEC or REC > 127)
CAN_ERRORSTATE_BUSOFF — off bus, no Tx or Rx (TEC > 255)

Overview — why scheduled functions exist

When events are polled (not interrupt-driven), CanDrv cannot call callbacks from an ISR. Instead, the BSW Scheduler (SchM) calls these functions periodically. They scan HW status registers and fire the same callbacks as the ISR would. Upper layers cannot distinguish between interrupt vs polling mode — the API is identical.

All scheduled functions (called by SchM)

FunctionSIDPurposeWhen active
Can_MainFunction_Write()0x01 Poll Tx confirmation flags. Calls CanIf_TxConfirmation() for confirmed frames. CanTxProcessing = POLLING or MIXED (only for HW objects with CanHardwareObjectUsesPolling=TRUE)
Can_MainFunction_Read()0x08 Poll Rx indication flags. Calls CanIf_RxIndication() for received frames. CanRxProcessing = POLLING or MIXED
Can_MainFunction_BusOff()0x09 Poll BusOff status bit. Calls CanIf_ControllerBusOff() if detected. CanBusOffProcessing = POLLING
Can_MainFunction_Wakeup()0x0a Poll wakeup status flags. Calls EcuM_CheckWakeup() if wakeup detected. Also confirms mode transitions after Can_SetControllerMode(). CanWakeupProcessing = POLLING or to confirm async mode transitions
Can_MainFunction_Mode()0x0c Poll CAN status register flags for mode transition completion. Calls CanIf_ControllerModeIndication() when transition confirmed. Always needed for async mode transitions (Can_SetControllerMode is async)

Multiple periods for Read/Write (exam detail)

If CanMainFunctionRWPeriods container defines multiple periods, the generated function names become:
Can_MainFunction_Write_<ShortName>() — one per period group, for HW objects referencing that period
Can_MainFunction_Read_<ShortName>() — same concept

This allows different HW objects (e.g. high-speed CAN vs low-speed diagnostic) to be polled at different rates without missing fast frames. SchM configures each function at its own cycle.

Critical rules for scheduled functions

• All MainFunction functions: void return, no parameters (BSW Scheduler requirement)
• All MainFunction functions: non-reentrant — the scheduler must not call the same function again before it returns
• No required execution order between MainFunction_Write, _Read, _BusOff, _Wakeup
• Can_MainFunction_Read must not be interrupted by itself (guarantees consistent HW buffer reading)
• MainFunction_Write and _Read may be implemented as empty macros/defines if no polling used (saves code size)
• Timeout for mode transitions: CanTimeoutDuration parameter. Uses OS GetCounterValue() — not a blocking wait.

Runtime Error (Det_ReportRuntimeError)

0x01CAN_E_DATALOSTHW overwrite (new frame overwrites unread buffer) or HW overrun (new frame dropped because buffer full). Configure larger HW FIFO (CanHwObjectCount) or increase polling/ISR frequency. Critical for real-time debugging — indicates receive processing too slow.

Can_Write return values (know all three)

E_OK (0x00)

HW object was free, data written to mailbox, transmission initiated. TxConfirmation will follow.

E_NOT_OK (0x01)

Development error (invalid params). Module not initialized. Trigger-transmit API returned error. CanIf should NOT retry.

CAN_BUSY (0x02)

HW mailbox occupied by ongoing Tx, OR preemptive call with same HTH. CanIf queues the PDU (if buffering enabled) and retries at TxConfirmation. Not an error — normal behavior under load.

Debugging checklist — interface testing

1. No TxConfirmation received — Is controller STARTED? Is PDU channel ONLINE (CanIf)? Is HTH correctly mapped (CanObjectId valid)? Is CanTxProcessing=INTERRUPT and ISR wired? Or is _MainFunction_Write() called in Polling mode? Check TEC counter — BusOff?

2. No RxIndication received — Is HRH correctly configured (CanObjectType=RECEIVE, CanIdType, CanHwFilterCode/Mask)? Is BasicCAN software filter in CanIf accepting the CanId? Is CanRxProcessing=POLLING and _MainFunction_Read() called? Check REC counter.

3. BusOff recovery not working — CanDrv must NOT auto-recover. CanSm must call SetControllerMode(STOPPED) then (STARTED). Is CanSM_ControllerBusOff() being called (check CanIf dispatch config)? Is BusOffProcessing=POLLING but MainFunction_BusOff not scheduled?

4. CAN_E_DATALOST runtime errors — Increase CanHwObjectCount (FIFO depth). Increase polling frequency. Move to interrupt mode for that HRH.

5. Baud rate mismatch — Verify CanControllerPropSeg + CanControllerSeg1 + CanControllerSeg2 sum gives correct sample point. Check CanControllerSyncJumpWidth. For FD: additionally check CanControllerSspOffset.

6. CAN FD frames not working — CanControllerFdBaudrateConfig must exist for the controller. CanControllerTxBitRateSwitch=TRUE for BRS (bit rate switch). Check Can_IdType MSBs (bit 30 must be set for FD). CanFdPaddingValue for DLC padding.

Step-by-step DaVinci configuration for a new CAN controller

Step 1: CanGeneral — Set CanDevErrorDetect=TRUE (during development). Set CanMultiplexedTransmission=TRUE if using multiple Tx objects per HTH. Set CanTimeoutDuration (e.g. 0.5ms, must be < MainFunction period). Set CanOsCounterRef to your OS counter for timeout.
Step 2: CanController — Add one per physical CAN peripheral. Assign CanControllerId=0 (must match CanIf CanIfCtrlCfg.CanIfCtrlId). Set CanRxProcessing=INTERRUPT or POLLING. Set CanWakeupSourceRef → EcuM wakeup source if wakeup needed.
Step 3: CanControllerBaudrateConfig — Add at least one. Set CanControllerBaudRate=500 (kbps). Calculate PropSeg/Seg1/Seg2/SJW from your oscillator and desired sample point (typically 75-80% sample point). Set CanControllerBaudRateConfigID=0.
Step 4: CanControllerFdBaudrateConfig — Add if CAN FD needed. Set FdBaudRate (e.g. 2000 kbps for 2 Mbit/s data phase). Set CanControllerTxBitRateSwitch=TRUE. Configure data phase timing segments separately from arbitration phase.
Step 5: CanHardwareObject (Tx) — Add one per HTH. CanObjectType=TRANSMIT. CanHandleType=BASIC (for multiplexed) or FULL. CanHwObjectCount=1 (Full) or N (Basic/mux). CanObjectPayloadLength=CAN_OBJECT_PL_8 (classic) or CAN_OBJECT_PL_64 (FD). Reference CanControllerRef → your controller. CanObjectId is auto-assigned or manually set starting from 0.
Step 6: CanHardwareObject (Rx) — CanObjectType=RECEIVE. Add CanHwFilter with CanHwFilterCode (base CanId to accept) + CanHwFilterMask (which bits must match, 0=don't care). Set CanHwObjectCount for FIFO depth. CanHardwareObjectUsesPolling=FALSE for interrupt.
Step 7: Generate code — DaVinci generates Can_PBcfg.c/h with HOH tables, baud rate tables, controller configs. Verify EcuM_AL_DriverInitZero() calls Can_Init(&Can_Config). Verify ISR vectors are connected (port driver may configure TX/RX pins separately).
Step 8: Connect to CanIf — In CanIf DaVinci, set CanIfHthCfg.CanIfHthIdSymRef → your HTH symbolic name. Set CanIfHrhCfg.CanIfHrhIdSymRef → your HRH symbolic name. Ensure ControllerId numbering is consistent.

Common DaVinci integration mistakes

• CanControllerId in CanDrv ≠ CanIfCtrlId in CanIf → CAN_E_PARAM_CONTROLLER at runtime
• HTH/HRH CanObjectId mismatch between CanDrv and CanIf → CAN_E_PARAM_HANDLE
• CanHwFilter mask wrong → receives wrong frames or misses expected frames
• CanControllerFdBaudrateConfig missing but CAN FD frames requested → length error (>8 bytes rejected)
• CanHwObjectCount=1 on BasicCAN Rx → overrun when back-to-back frames arrive (increase to 2-4)
• CanTxProcessing=POLLING but Can_MainFunction_Write not scheduled → Tx confirmations never fired
• Multiple CanDrv instances with same function names → linker error (use CanIndex + naming prefix)
• MCU driver not initialized before Can_Init() → SFR access fails silently or corrupts
• CanTimeoutDuration too large → Can_SetControllerMode blocks too long, starving other tasks
• CanControllerBaudRateConfigID collision → SetBaudrate picks wrong timing parameters
