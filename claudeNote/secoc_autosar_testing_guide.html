The V-Model maps left-side specification and design artifacts to right-side test levels. In SecOC testing, each level targets a specific concern: interface tests verify API contracts and PDU structure, integration tests verify BSW stack interactions (COM ↔ SecOC ↔ Crypto), and functional/system tests verify end-to-end secure communication on the real bus with real ECUs.
Interface Tests — Component Level (V-Model: Detailed Design ↔ Interface Test)
Interface tests validate the API contracts and PDU I/O boundaries of the SecOC module in isolation — i.e., the CDD-configured parameters, RTE port connections, and PDU header/footer fields visible at the module interface.
Integration Tests — BSW Stack Level (V-Model: Architecture ↔ Integration Test)
Integration tests verify that SecOC cooperates correctly with adjacent BSW modules: COM, PduR, Crypto, KeyM, and the OS/SchM scheduler. Both transmit and receive paths are exercised together.
Functional / System Tests — E2E on Bus (V-Model: System Requirements ↔ System Test)
Functional tests exercise the complete SecOC chain on real hardware with CANoe/CANalyzer observing the bus. These tests cover correct MAC generation, freshness counter behaviour, and security-relevant fault injection.
Tool Reference — SecOC Test Setup
CANoe / CANalyzer
Primary bus-level analysis and stimulation platform. Used for all functional and system tests.
  • CAPL scripting for PDU injection and MAC manipulation
  • Trace window — byte-level PDU inspection (SecOC header, FV, MAC)
  • Symbol editor — map CAN IDs to SecOC PDUs
  • Bus statistics — detect missing/repeated frames (freshness)
  • Diagnostic interface — trigger DTC checks on SecOC failures
  • Test module (XML/CAPL) — automated pass/fail logging
winIDEA iSystem Debugger
JTAG/SWD-connected ECU debugger. Essential for interface and integration-level tests where internal SecOC state must be observed.
  • Breakpoints on SecOC_MainFunctionTx / Rx
  • Memory watch — FreshnessValue array, MAC buffer, VerificationStatus
  • Variable monitoring — AttemptCounter, VerificationStatusPropagationMode
  • Stack trace — SecOC → CsmJob → Crypto interaction
  • Scripting (Python/isystem API) — automated register reads
CDD (Component Description Designer)
Configuration authoring tool. Used to verify correct parameter instantiation before flashing.
  • Check SecOCFreshnessCounterStartValue, SecOCFreshnessValueLength
  • Verify SecOCMsgLinkLen, SecOCAuthDataFreshnessLen
  • Confirm Crypto job linkage (CsmJobReference)
  • Review SecOCReceptionOverrideStatus default values
  • Cross-check PDU IDs vs ARXML Com signal mapping
CRO (CAN Remote Object / XCP on CAN)
Used to read/write ECU memory and calibration variables at runtime without halting the CPU — ideal for non-intrusive interface probing.
  • Live-read FreshnessValue counter during bus activity
  • Override SecOC verification status for negative testing
  • Inject MAC override to test VERIFICATION_FAILED path
  • DAQ mode — log SecOC internal buffers at high rate
  • Combine with CANoe XCP transport for synchronised capture
