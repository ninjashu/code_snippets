AUTOSAR CP OS — Complete Study Guide

R24-11 · Document ID 34 · Classic Platform · Based on OSEK/VDX (ISO 17356-3) · Exam, Integration, Debugging & DaVinci Config

What is the AUTOSAR OS?

The AUTOSAR OS is a statically-configured, priority-based, real-time operating system for automotive ECUs. It is a superset of the OSEK/VDX OS (ISO 17356-3) — all OSEK applications are backward compatible. It adds memory protection, timing protection, OS-Applications, multicore support, and the IOC.

Key properties

• Statically configured at build time
• Priority-based preemptive scheduling
• No dynamic memory allocation
• Real-time deterministic behaviour
• Scalable (4 Scalability Classes)

OSEK + AUTOSAR extensions

• OS-Applications (partitioning)
• Memory protection (MPU-based)
• Timing protection (execution budget, time frame, lock budget)
• Schedule Tables
• Multicore + IOC
• Spinlocks

Scalability Classes

SC1: Basic + alarm callbacks
SC2: SC1 + timing protection
SC3: SC1 + memory protection
SC4: SC2 + SC3 (full protection)
Most automotive ECUs use SC3 or SC4.

Key document rule

AUTOSAR OS provides an API backward compatible to OSEK [SWS_Os_00001]. No dynamic resource management. No MMU (only MPU). No POSIX. Designed for microcontrollers without external memory.

Tasks & Scheduling

Task types

Basic Task

Cannot block. No WaitEvent call. Runs to completion (or terminates/chains). Multiple activations can be queued. Lower overhead.

TASK(MyBasicTask) {
  /* do work */
  TerminateTask();
}
Use for: cyclic processing, event-driven non-blocking work, ISR bottom halves.

Extended Task

Can call WaitEvent() — suspends until Event is set. Must have at least one Event configured. Requires Extended Conformance Class (ECC). More RAM (stack persists during WAITING).

TASK(MyExtTask) {
  WaitEvent(MyEvent);
  ClearEvent(MyEvent);
  /* process event */
  TerminateTask();
}
Use for: state machines needing synchronization, C/S server tasks with WaitPoints.

Task state machine — the most examined diagram

Scheduling algorithm

AUTOSAR OS uses fixed-priority preemptive scheduling. Higher priority Tasks always preempt lower priority Tasks. Scheduling is per-core in multicore systems — cores are independent schedulers.

Interrupts & ISRs

Two ISR categories — the most critical distinction

Category 1 ISR

Fully managed by the hardware and the user. The OS is completely unaware of Category 1 ISRs. They bypass the OS wrapper entirely — no OS services can be called inside them. No timing protection. No memory protection. No stack monitoring.

/* Category 1: NO OS services allowed */
ISR(MyISR_Cat1) {
  /* Direct HW register access only */
  /* NO ActivateTask, NO SetEvent etc. */
}
⚠️ If OS-Applications and timing protection are used: Category 1 ISRs must belong to trusted OS-Applications. Category 1 ISRs with memory/timing protection create undefined behavior.

Category 2 ISR

Managed by the OS through an ISR wrapper. OS services are allowed (see allowed calling context table). Timing protection applies. Memory protection applies. Stack monitoring applies.

/* Category 2: OS services allowed */
ISR(CanRxISR) {
  /* Can call: ActivateTask, SetEvent,
     GetResource, IncrementCounter etc. */
  SetEvent(CanRxTask, CanRxEvent);
}
✅ Cat2 ISRs always have higher effective priority than any Task. The OS manages context switch in/out.

⚠️ Critical multicore rule: ALL interrupt disable/suspend calls affect ONLY the local core [SWS_Os_00590-595]. Interrupts on other cores continue unaffected. This is the most common integration mistake in multicore designs.

Resources & Spinlocks — Mutual Exclusion

Resources — same-core mutual exclusion (Priority Ceiling Protocol)

Resources use the Priority Ceiling Protocol (PCP). When a Task takes a Resource, its priority is raised to the ceiling priority of that Resource (the highest priority of all Tasks/ISRs that can use the Resource). This prevents priority inversion and guarantees bounded priority inversion duration.

/* Resource usage pattern */
GetResource(MyResource);    /* Priority raised to ceiling */
  /* Critical section */
ReleaseResource(MyResource); /* Priority restored */

/* STRICT LIFO ORDER REQUIRED for nested resources:
   GetResource(R1);
     GetResource(R2);    <- OK: R2 taken after R1
     ReleaseResource(R2); <- Release inner first
   ReleaseResource(R1);  <- Release outer last
*/
🔴 Resources CANNOT be shared across cores [SWS_Os_00662]. The OS generator MUST reject this configuration. PCP relies on local priority scheduling — it fails across independent cores.
Resource ruleRequirement
Resources and Spinlocks must be released in strict LIFO order[SWS_Os_00801]
Task terminating with held Resource → OS forcibly releases it[SWS_Os_00070]
ISR ending with held Resource → OS forcibly releases it[SWS_Os_00369]
Cannot call WaitEvent or Schedule while holding Spinlock[SWS_Os_00622], [SWS_Os_00624]

RES_SCHEDULER — special resource

When OsUseResScheduler=TRUE, RES_SCHEDULER is created per-core. It has the highest Task ceiling priority on that core — holding it prevents preemption by any other Task on the same core (but NOT on other cores).

Spinlocks — cross-core mutual exclusion

Spinlocks use busy-waiting to protect critical sections across cores. A Task/ISR spins (loops continuously) until the lock is available. No OS descheduling happens during spinning — this is intentional for predictability.

GetSpinlock (blocking)

GetSpinlock(MySpin);
  /* cross-core critical section */
ReleaseSpinlock(MySpin);

Blocks until spinlock is available. Keeps trying. May waste CPU cycles. Use when critical section is very short.

TryToGetSpinlock (non-blocking)

TryToGetSpinlockType result;
TryToGetSpinlock(MySpin, &result);
if(result == TRYTOGETSPINLOCK_SUCCESS) {
  /* critical section */
  ReleaseSpinlock(MySpin);
}

Returns immediately. Caller handles failure case. Better for polling patterns.

Spinlock ordering — deadlock prevention

To prevent deadlock from nested spinlocks, AUTOSAR OS enforces a configured ordering. Spinlocks can only be acquired in the configured order (skipping is allowed, but reversal is not). Cycles in the successor chain are rejected by the OS generator.

/* Configured order: S1 → S2 → S3 */
GetSpinlock(S1);   /* OK */
  GetSpinlock(S3); /* OK: S3 is after S1 in chain */
    /* Cannot GetSpinlock(S2) here — S2 < S3 in chain */
  ReleaseSpinlock(S3);
ReleaseSpinlock(S1);

/* DEADLOCK EXAMPLE (not allowed):
Core 0: GetSpinlock(S1) then tries GetSpinlock(S2)
Core 1: GetSpinlock(S2) then tries GetSpinlock(S1)
→ Circular wait → deadlock */

OsSpinlockLockMethod — how to protect during spinning

Events & Alarms

Events — intra-task synchronization (Extended Tasks only)

Events are bit flags associated with an Extended Task. They allow a Task to synchronize with ISRs, other Tasks, or Alarms without polling. An event is owned by exactly one Task.

/* Extended Task waiting for event */
TASK(MyExtTask) {
  while(1) {
    WaitEvent(DataReadyEvent | TimeoutEvent);
    EventMaskType events;
    GetEvent(MyExtTask, &events);
    ClearEvent(events);
    if(events & DataReadyEvent) {
      /* process data */
    }
    if(events & TimeoutEvent) {
      /* handle timeout */
    }
  }
}

/* From ISR or other Task: */
SetEvent(MyExtTask, DataReadyEvent);
💡 SetEvent is cross-core capable in multicore systems [SWS_Os_00602]. It can set events on Tasks running on a different core. This is how cross-core synchronization is achieved alongside IOC.

Alarms — counter-based actions

An Alarm fires when a Counter reaches a configured value, and performs one of these actions:

Activate Task

Calls ActivateTask() on the configured Task. Most common alarm action for cyclic tasks.

Set Event

Calls SetEvent() on the configured Task+Event. For synchronizing Extended Tasks.

Increment Counter

Calls IncrementCounter() on a software Counter. Allows cascaded counters.

Alarm Callback

Calls a configured callback function. Only allowed in SC1! [SWS_Os_00242]

Counter types

TypeHow it advancesAPIUse case
Hardware CounterHardware timer peripheral advances it automaticallyGetCounterValue, GetElapsedValueSystem tick, GPT-driven scheduling
Software CounterExplicit call to IncrementCounter() from ISR or Alarm actionIncrementCounter, GetCounterValueAngular position (crankshaft), event counting

Alarm API

/* Set relative alarm: fires after 'increment' ticks, then every 'cycle' ticks */
SetRelAlarm(MyAlarm, 100, 10);  /* fires at now+100, then every 10 ticks */

/* Set absolute alarm: fires when counter reaches 'start', then every 'cycle' */
SetAbsAlarm(MyAlarm, 1000, 10); /* fires at counter=1000, then every 10 */

/* Cancel alarm */
CancelAlarm(MyAlarm);

/* Get remaining time to alarm expiry */
TickType ticksLeft;
GetAlarm(MyAlarm, &ticksLeft);

/* Get counter properties */
AlarmBaseType base;
GetAlarmBase(MyAlarm, &base);  /* MaxAllowedValue, TicksPerBase, MinCycle */
⚠️ SetRelAlarm with increment=0 returns E_OS_VALUE [SWS_Os_00304]. This is an AUTOSAR extension that tightens undefined OSEK behavior. Always validate increment > 0 before calling.

Counter multicore rules

🔴 A Counter can only be incremented by the core on which its OS-Application resides [SWS_Os_00629]. Alarms and ScheduleTables must be bound to a Counter on the SAME core [SWS_Os_00631, 00630]. The OS generator checks and rejects cross-core counter/alarm configurations.

Schedule Tables — Deterministic Multi-Task Activation

What problem does a ScheduleTable solve?

Multiple alarms with interdependent timing are hard to synchronize. A ScheduleTable encapsulates a statically-defined sequence of expiry points — each specifying Tasks to activate and/or Events to set at exact counter offsets. All activations share a single Counter, guaranteeing consistent relative timing.

ScheduleTable state machine

STOPPED → StartScheduleTableRel/Abs() → RUNNING → NextScheduleTable() → NEXT → (current table finishes) → RUNNING

Additional states for explicitly synchronized ScheduleTables: WAITING (synchronous start, waiting for first sync count) and RUNNING_AND_SYNCHRONOUS (deviation ≤ precision threshold).

ScheduleTable configuration rules (exam critical)

RuleRequirement
Must have at least one expiry point[SWS_Os_00401]
Each expiry point must activate ≥1 Task OR set ≥1 Event[SWS_Os_00407]
Each expiry point offset must be unique on the table[SWS_Os_00442]
Initial offset: 0 OR in range [OsCounterMinCycle, MaxAllowedValue][SWS_Os_00443]
Delay between adjacent expiry points: in range [MinCycle, MaxAllowedValue][SWS_Os_00408]
At same expiry point: Task activation processed BEFORE Event setting for same Task[SWS_Os_00412]
Alarm Callbacks NOT allowed in SC2/SC3/SC4 (only SC1)[SWS_Os_00242]
Implicit sync table: Duration = OsCounterMaxAllowedValue + 1[SWS_Os_00429]

Synchronization modes

Implicit Synchronization

The Counter driving the ScheduleTable IS the synchronization Counter (e.g. FlexRay/TTP hardware timer). Duration must equal counter modulus. Start ONLY with StartScheduleTableAbs() — never relative.

/* FlexRay-driven: start at absolute value 0 */
StartScheduleTableAbs(FlexRaySchedTable, 0);
/* Processing stays aligned with FlexRay cycle */

Explicit Synchronization

Drive Counter ≠ sync Counter. OS adjusts expiry point delays to converge with sync counter. Configure OsScheduleTableMaxShorten/MaxLengthen on adjustable expiry points.

/* Provide current sync counter value */
SyncScheduleTable(MySchedTable, syncCounterValue);
/* OS calculates deviation and adjusts
   delays at adjustable expiry points */
/* Can cancel sync */
SetScheduleTableAsync(MySchedTable);

ScheduleTable API quick reference

StartScheduleTableRel(SchedTableID, Offset)Start ScheduleTable. Initial expiry point fires after Offset + InitialOffset ticks from now. Returns E_OS_STATE if already running.
StartScheduleTableAbs(SchedTableID, Start)Start ScheduleTable at absolute counter value. Initial expiry fires when Counter = Start + InitialOffset.
StopScheduleTable(SchedTableID)Cancel ScheduleTable immediately. State → STOPPED. Restarting begins from Initial Expiry Point.
NextScheduleTable(SchedTableID_From, SchedTableID_To)Switch from current table to next after Final Expiry Point + Final Delay. Avoids gap between tables.
StartScheduleTableSynchron(SchedTableID)Explicit sync: enter WAITING state. Table starts only after SyncScheduleTable provides first count.
SyncScheduleTable(SchedTableID, Value)Provide synchronization counter value. OS calculates deviation and begins adjustment at adjustable expiry points.
SetScheduleTableAsync(SchedTableID)Cancel synchronization. State → RUNNING (no longer RUNNING_AND_SYNCHRONOUS).
GetScheduleTableStatus(SchedTableID, &Status)Query current ScheduleTable state. Returns STOPPED, RUNNING, RUNNING_AND_SYNCHRONOUS, WAITING, or NEXT.

OS-Applications — Software Partitioning

What is an OS-Application?

An OS-Application is a named container of OS objects (Tasks, ISRs, Alarms, ScheduleTables, Counters, Trusted Functions, hooks). It is the unit of protection and fault isolation. All OS objects must belong to exactly one OS-Application if OS-Applications are configured.

Trusted vs Non-Trusted OS-Applications

Trusted OS-Application

• Runs in privileged mode (hardware-supported)
• Unrestricted memory and OS API access
• Can disable protection features (optional)
• Can provide Trusted Functions to others
• Can terminate other OS-Applications
• OS itself is always trusted
• BSW modules typically run as trusted

OsTrustedApplication = TRUE

Non-Trusted OS-Application

• Runs in non-privileged mode
• Restricted memory access (own data/stack/code only)
• Cannot call ShutdownOS [SWS_Os_00054]
• Cannot access other OS-Apps' data
• Subject to timing protection
• Application SWCs in safety partitioning

OsTrustedApplication = FALSE

OS-Application state machine

APPLICATION_ACCESSIBLE → TerminateApplication() or ProtectionHook → APPLICATION_TERMINATED

Initial state after StartOS is ACCESSIBLE [SWS_Os_00500]. Terminated state is final — it cannot transition back [SWS_Os_00502]. Objects of a terminated OS-App are inaccessible from other OS-Apps [SWS_Os_00504].

Access control between OS-Applications

By default, OS objects can only be accessed by Tasks/ISRs within the same OS-Application. Cross-OS-Application access must be explicitly granted via OsTaskAccessingApplication, OsAlarmAccessingApplication, etc. Violation returns E_OS_ACCESS [SWS_Os_00056].

TerminateApplication — the partition kill switch

/* Self-termination from within the faulty OS-App */
TerminateApplication(GetApplicationID(), NO_RESTART);

/* Terminating another OS-App (trusted only) */
TerminateApplication(FaultyAppId, NO_RESTART);

When an OS-Application is terminated [SWS_Os_00447]:

1
All running, ready, and waiting Tasks/ISRs of the OS-App are forcibly terminated
2
All interrupt sources of the OS-App are disabled
3
All active Alarms of the OS-App are stopped
4
All ScheduleTables of the OS-App are stopped
5
State changes to APPLICATION_TERMINATED
💡 CallTrustedFunction is cross-OS-Application but SAME core only [SWS_Os_00623]. It switches to privileged mode via a trap/software interrupt mechanism. This is the standard interface for non-trusted SWCs to access trusted BSW services.

Protection Facilities — The Safety Layer

Three protection dimensions

Memory Protection

Prevents unauthorized read/write/execute access to memory regions. Requires MPU hardware. Private data, stack, code sections are protected per OS-Application and per Task/ISR.

Timing Protection

Prevents Tasks/ISRs from exceeding execution budgets, violating inter-arrival time frames, or holding Resources/interrupt locks too long. Requires timer hardware.

Service Protection

Prevents API calls from wrong context, with invalid parameters, or on unauthorized objects. Implemented via extended status checks at OS service entry.

Hook Functions & Error Handling

Hook types — two scope levels

System-specific hooks (global)

Called in context of the entire OS. Applied to all OS-Applications. Implemented by the system integrator.

• StartupHook() — after OS init, before scheduling
• ShutdownHook(StatusType) — during ShutdownOS
• ErrorHook(StatusType) — on every OS service error
• PreTaskHook() — before Task enters RUNNING
• PostTaskHook() — after Task leaves RUNNING
• ProtectionHook(StatusType) → ProtectionReturnType

Application-specific hooks (per OS-App)

Called only in context of their OS-Application. Configured per OS-Application in OsApplicationHooks.

• StartupHook() — called after global StartupHook
• ShutdownHook(StatusType) — called before global ShutdownHook
• ErrorHook(StatusType) — for errors in this OS-App

App-specific hooks run only on the core to which the OS-Application is bound [SWS_Os_00582].

Startup sequence with hooks (multicore)

1
All cores synchronize inside StartOS (first sync point) [SWS_Os_00580]
2
Global StartupHook called on ALL cores simultaneously [SWS_Os_00581]
3
OS-Application-specific StartupHooks called on their respective bound cores [SWS_Os_00582]
4
All cores synchronize (second sync point) [SWS_Os_00579]
5
Auto-start Tasks, Alarms, ScheduleTables activated on each core
6
Scheduling begins on all cores

ShutdownHook sequence (ShutdownAllCores)

1
OS-Application-specific ShutdownHooks called on bound cores [SWS_Os_00586]
2
All cores synchronize [SWS_Os_00587]
3
Global ShutdownHook called on ALL cores [SWS_Os_00588]
4
OS shuts down — EcuM_Shutdown() called from ShutdownHook

ErrorHook — OS error reporting

void ErrorHook(StatusType Error) {
  /* Called on every OS service error in EXTENDED status */
  /* Use OSService(), OSErrorXxx() macros to get context */
  OSServiceIdType svc = OSErrorGetServiceId();
  if(svc == OSServiceId_ActivateTask) {
    TaskType task = OSError_ActivateTask_TaskID();
    /* log task ID that failed activation */
  }
  /* Do NOT call: ActivateTask, ChainTask, WaitEvent,
     GetResource, Schedule, ShutdownOS */
}
⚠️ ErrorHook is called for EVERY OS error when extended status is enabled. Keep it short. Avoid complex logic. Never call OS services that are restricted from ErrorHook context (see Table 7.1).

Stack monitoring

Even without MPU hardware, the OS can detect stack faults by monitoring stack usage at context switch time. A pattern (e.g. 0xAA) is written into the stack area during initialization and checked at switch time.

Stack fault detected + ProtectionHook configured → ProtectionHook(E_OS_STACKFAULT) [SWS_Os_00396].
Stack fault + no ProtectionHook → ShutdownOS(E_OS_STACKFAULT) [SWS_Os_00068].

Multicore OS & IOC

Multicore architecture principles

Single configuration

One OS configuration for all cores. All IDs (TaskID, ResourceID etc.) are unique across cores [SWS_Os_00567]. No per-core config files.

Independent schedulers

Each core schedules independently. A low-priority Task on Core 0 may run in parallel with a high-priority Task on Core 1. No global priority ordering.

Static assignment

OS-Applications (and their Tasks/ISRs) are statically assigned to cores at configuration time. No dynamic migration.

Master-slave startup

Master core calls StartCore() for each slave. Slaves call StartOS() after activation. All cores must reach synchronization points in StartOS.

Multicore startup sequence

/* Master core code (C startup / EcuM_Init): */
StatusType rv1, rv2;
StartCore(OS_CORE_ID_1, &rv1);  /* Activate slave 1 */
StartCore(OS_CORE_ID_2, &rv2);  /* Activate slave 2 */
if((rv1 != E_OK) || (rv2 != E_OK)) { /* handle error */ }

StartOS(OSDEFAULTAPPMODE);  /* Master calls StartOS */

/* Each slave core (after StartCore activates it): */
StartOS(DONOTCARE);  /* Use master's AppMode */
/* OR: StartOS(OSDEFAULTAPPMODE); — must match master */
⚠️ All cores calling StartOS with a real AppMode must use the SAME AppMode [SWS_Os_00608]. If they differ → OS enters endless loop on every core. DONOTCARE means "inherit from other cores."

IOC — Inter-OS-Application Communicator

The IOC provides communication between OS-Applications, especially across core and memory protection boundaries. The RTE uses IOC internally — it generates IOC calls for cross-partition RTE communication. Direct IOC use by CDDs requires hand-written IOC configuration.

IOC communication modes

• Unqueued (Last-is-Best): Sender always overwrites. Receiver reads latest value.
• Queued (FIFO): Sequential order preserved. Queue depth configurable.
• Group: Multiple data items transferred atomically in one operation.

IOC communication patterns

• 1:1 — one sender, one receiver (unqueued or queued)
• N:1 — multiple senders, one receiver (each sender has own API)
• N:M — multiple senders, multiple receivers (unqueued only)

DaVinci Configurator Pro — OS Configuration

Step-by-step OS configuration workflow

Step 1 — OsOS container (global OS settings)

1
Set OsScalabilityClass: SC1 (basic), SC2 (+ timing protection), SC3 (+ memory protection), SC4 (full). This determines which protection features are generated.
2
Set OsNumberOfCores (1 for single core). Each core needs its own startup entry point and OS object assignments.
3
Set OsUseGetServiceId, OsUseParameterAccess: enable to use OSErrorGetServiceId() and parameter macros in ErrorHook. Required for meaningful error diagnostics.
4
Configure OsHooks: enable StartupHook, ShutdownHook, ErrorHook, PreTaskHook, PostTaskHook as needed. Each hook requires an implementation in the integration code.
5
Set OsStatus: EXTENDED (enables parameter checking and ErrorHook calls) or STANDARD (minimal checking, faster). Always use EXTENDED in development.

Step 2 — OsTask containers

1
For each task: set OsTaskPriority, OsTaskSchedule (FULL/NON), OsTaskActivation (1 for Extended Tasks). Add OsTaskAutostart with AppMode reference for auto-activation.
2
Set OsTaskStackSize. Rule of thumb: function call depth × 40 bytes + ISR nesting overhead + 20% safety margin. DaVinci stack analyzer can help calculate WCET stack usage.
3
For SC2/SC4: add OsTaskTimingProtection sub-container. Set OsTaskExecutionBudget (nanoseconds WCET) and OsTaskTimeFrame (minimum inter-arrival time in nanoseconds).
4
For SC3/SC4: assign the Task to an OsApplication via the OsApplicationTaskRef. Memory sections are generated based on this assignment.

Step 3 — OsIsr containers

1
For each ISR: set OsIsrCategory (1 or 2), OsIsrPriority (hardware interrupt level). Category 2 ISRs get an OS-generated wrapper.
2
For SC2/SC4: add OsIsrTimingProtection. Set OsIsrExecutionBudget and OsIsrTimeFrame (babbling idiot protection for CAN/LIN ISRs).

Step 4 — OsCounter containers

1
Create a counter for each time base needed. Hardware counters: set OsCounterType=HARDWARE, link the GPT channel via OsSecondsPerTick/OsCounterMaxAllowedValue/OsCounterMinCycle.
2
In multicore: assign counter to the correct OS-Application (and hence core). Alarms and ScheduleTables must reference counters on the SAME core.

Step 5 — OsAlarm containers

1
Add OsAlarmActivateTask or OsAlarmSetEvent or OsAlarmIncrementCounter as sub-containers. Each alarm can have only ONE action.
2
For auto-start alarms: add OsAlarmAutostart sub-container. Set OsAlarmAutostartType (ABSOLUTE or RELATIVE), OsAlarmStartTime, OsAlarmCycleTime.

Step 6 — OsScheduleTable containers

1
Add OsScheduleTable container. Set OsScheduleTableDuration (ticks), OsScheduleTableRepeating (TRUE for periodic), OsScheduleTableCounterRef.
2
Add OsScheduleTableExpiryPoint sub-containers for each expiry point. Set OsScheduleTableEPOffset (ticks from start). Add OsScheduleTableTaskActivation or OsScheduleTableEventSetting sub-containers.
3
For explicit synchronization: add OsScheduleTableSync container. Set OsScheduleTblExplicitPrecision. For adjustable expiry points: add OsScheduleTblAdjustableExpPoint with MaxShorten and MaxLengthen.

Step 7 — OsApplication containers

1
Create one OsApplication per software partition. Set OsTrustedApplication (TRUE for BSW, FALSE for safety-partitioned SWCs). Add references to all owned Tasks, ISRs, Alarms, Counters, ScheduleTables.
2
Configure cross-application access rights: add OsAppTaskRef/OsAppAlarmRef etc. sub-containers in each OS-Application that needs to access objects from another OS-Application.
3
For multicore: set OsApplicationCoreRef to assign the OS-Application to a specific core. All objects in the OS-Application run on that core.
4
For Trusted Functions: add OsApplicationTrustedFunction sub-containers. Each has a function name that maps to a C function in the trusted OS-Application code.

Step 8 — OsSpinlock containers (multicore)

1
Create one OsSpinlock per cross-core critical section. Set OsSpinlockLockMethod (LOCK_NOTHING, LOCK_WITH_RES_SCHEDULER, LOCK_ALL_INTERRUPTS, LOCK_CAT2_INTERRUPTS).
2
Configure OsSpinlockSuccessor references to define the ordering chain. The OS generator checks for cycles and rejects invalid configurations [SWS_Os_00666].

Common DaVinci configuration mistakes

🔴 Resource assigned to Tasks on different cores: Generator must reject this [SWS_Os_00662]. Check: all Tasks referencing a Resource must have the same OsApplicationCoreRef.
🔴 Alarm/ScheduleTable bound to Counter on different core: Generator rejects [SWS_Os_00663, 00665]. Each alarm/table must reference a counter owned by the same core's OS-Application.
🔴 Extended Task with OsTaskActivation > 1: Extended Tasks can only have 1 activation. DaVinci may warn but this is a specification violation.
⚠️ Missing TerminateTask() at end of Basic Task body: OS calls ErrorHook(E_OS_MISSINGEND) then forcibly terminates [SWS_Os_00069]. Always end every Task entry function with TerminateTask() or ChainTask().
⚠️ ScheduleTable InitialOffset = 0 for implicit sync: With implicit sync, you must start with StartScheduleTableAbs(table, 0) — NOT StartScheduleTableRel [SWS_Os_00430].
⚠️ Spinlock cycle in successor chain: OS generator checks and rejects [SWS_Os_00666]. Plan spinlock ordering on paper before configuration.
💡 ARTI (AUTOSAR Run-time Interface) support: Enable in DaVinci for trace tool integration (Lauterbach Trace32, iSYSTEM, etc.). ARTI hook macros are inserted into OS code at key points (task switch, ISR entry/exit, alarm expiry) for runtime trace without code instrumentation.

Integration Testing & Debugging

Common integration bugs

Bug: Task never runs / stuck in READY

Root causes: (1) Higher priority Task never terminates (infinite loop, missing TerminateTask). (2) Resource held indefinitely by higher-priority code. (3) Task assigned to wrong core (activating core 0 task from core 1 code path).
Debug: ARTI trace shows task state transitions. Hook GetTaskState() in a periodic debug task to log all task states. Check PreTaskHook/PostTaskHook counters.

Bug: ProtectionHook called with E_OS_PROTECTION_TIME

Root cause: Task WCET exceeded OsTaskExecutionBudget. Often happens on first integration when budget is set too tight, or when cache is cold, or during burst workload.
Debug: Use ARTI timing trace to measure actual execution time per task invocation. Increase budget temporarily to identify true WCET. Check for infinite loops or unexpected blocking in the task.

Bug: E_OS_PROTECTION_ARRIVAL — timing frame violation

Root cause: Task/ISR triggered more frequently than OsTaskTimeFrame/OsIsrTimeFrame. Common with CAN interrupt flooding (babbling idiot), alarm configured with cycle time shorter than time frame, or erratic SetEvent calls.
Debug: Check actual inter-arrival time via ARTI. Verify alarm cycle time >= time frame. For ISRs: check hardware interrupt rate. ProtectionHook can log the violating Task/ISR ID.

Bug: E_OS_SPINLOCK returned from TerminateTask

Root cause: Task calling TerminateTask while holding a spinlock. This is a design error — spinlocks must be released before task termination. [SWS_Os_00612]
Debug: Use ARTI spinlock trace. Add assertion in ProtectionHook to log which spinlock was held. Review all code paths through the task for early returns that skip ReleaseSpinlock.

Bug: Deadlock in multicore — all cores stuck

Root cause: Circular spinlock acquisition (e.g., Core 0 holds S1, waits for S2; Core 1 holds S2, waits for S1). [See Fig 7.21 in spec]
Debug: OS generator should have caught this if spinlock ordering is properly configured. In runtime: ARTI spinlock trace shows which core holds which spinlock. Add GetSpinlock timeout using TryToGetSpinlock in a loop with counter.

Bug: IOC_E_LIMIT on IocSend

Root cause: Receiver is not consuming IOC queue fast enough — queue overflow. Sender production rate > receiver consumption rate.
Debug: Increase queue depth (OsIocQueueLength) temporarily to confirm this is the issue. Long-term: increase receiver task frequency, reduce sender rate, or switch to unqueued (last-is-best) communication if ordering doesn't matter.

ARTI — AUTOSAR Run-Time Interface for debugging

ARTI provides standardized hook macros inserted into OS code for tracing. The trace tool (Lauterbach, ETAS RTAOS, Vector CANoe etc.) provides Os_Arti.h with macro definitions. When ARTI is not configured, macros expand to empty (zero overhead).

ARTI classEvents captured
AR_CP_OS_TASKTask RUNNING/WAITING/READY/SUSPENDED state transitions with timestamps
AR_CP_OS_CAT2ISRCat2 ISR entry/exit with ISR ID and timestamps
AR_CP_OS_APPLICATIONOS-Application state changes (ACCESSIBLE → TERMINATED)
AR_CP_OS_SPINLOCKGetSpinlock/ReleaseSpinlock calls with spinlock ID and core
AR_CP_OS_SERVICECALLSAll OS API calls with parameters and return values
AR_CP_OS_HOOKHook function entry/exit (ErrorHook, ProtectionHook, PreTaskHook etc.)
💡 For automotive-grade integration testing: combine ARTI trace with ISO 26262 timing analysis. The ARTI timing data feeds directly into WCET analysis tools (TA Tool Suite, RapiTime) to verify execution budgets with measurement-based WCET instead of static analysis. This is the standard practice for ASIL B and above ECU development.
