AUTOSAR CSM Complete Study Guide

1. Architecture & Stack Overview

CSM = Crypto Service Manager. It sits in the Service Layer of AUTOSAR Classic Platform, acting as the single gateway for all crypto operations.

The 4-Layer Crypto Stack

Think of it as a vertical pipe. Your application never touches hardware directly — it calls CSM, which delegates downward.

Application / SW-C / BSW modules
Calls Csm_Hash(), Csm_Encrypt(), Csm_MacGenerate(), etc.
↕ (Job-based API + Direct API)
CSM — Crypto Service Manager (Service Layer)
Schedules jobs, manages queues, routes to CryIf. No crypto math here.
↕ (CryIf_ProcessJob, CryIf_KeyElementSet, …)
CryIf — Crypto Interface (HAL-style abstraction)
Dispatches to the correct Crypto Driver Object via channel routing.
↕
CRYPTO — Crypto Driver + Crypto Driver Object
HW (HSM, SHE) or SW library. Actually performs AES/SHA/RSA/ECC math. Holds the keys.

Key Concepts (Exam Definitions)

Job

A configured CsmJob instance. Links: primitive (algorithm), key, queue, priority, sync/async mode. The unit of work CSM processes.

Channel

Path from a CSM queue → CryIf → specific Crypto Driver Object. Each queue maps to exactly one channel.

Primitive

A configured algorithm instance. Specifies AlgorithmFamily (AES, RSA…), AlgorithmMode (CBC, GCM…), and service type (HASH, MACGENERATE…).

Crypto Driver Object

Independent crypto "device" (HW accelerator or SW lib). Has its own workspace — only ONE primitive at a time per object.

Key / Key Element

Key lives in Crypto Driver. CSM holds only an ID reference. Key has elements (material, IV, seed…) each with a standard ID.

Operation Modes

START, UPDATE, FINISH (bit flags). Can be ORed together for single-call. Only streaming services support SUF; others use SINGLECALL.

Two API Families

Job-Based API

All crypto services (Hash, MAC, Encrypt…). Uses Crypto_JobType structure. Supports sync AND async. Goes through queue/scheduler.

Examples: Csm_Hash, Csm_MacGenerate, Csm_Encrypt, Csm_SignatureVerify

Direct API

Key management functions only. Always synchronous. Bypasses queue. Passes params directly to CryIf function.

Examples: Csm_KeyElementSet, Csm_KeySetValid, Csm_KeyElementGet, Csm_KeyDerive

Exam trap: Keys are NEVER stored in CSM. CSM only holds numeric IDs. The physical key material lives inside the Crypto Driver, protected from the application.

2. Job State Machine

Every job has exactly two states. The state transitions happen in the Crypto Driver (where the actual workspace lives), but CSM reads the state to decide how to route requests.

CRYPTO_JOBSTATE_IDLE (0x00)

After Init or Finish
START/UPDATE ⇄ FINISH
CRYPTO_JOBSTATE_ACTIVE (0x01)
ACTIVE
After START or UPDATE

Streaming Operation Modes (Bit Flags)

ModeValueMeaningWhen Used
CRYPTO_OPERATIONMODE_START0x01Initialize the primitive, cancel previousFirst call of a streaming sequence
CRYPTO_OPERATIONMODE_UPDATE0x02Feed input data, may provide intermediate resultsCalled ≥1 times after START
CRYPTO_OPERATIONMODE_STREAMSTART0x03START | UPDATE combinedFirst call with data
CRYPTO_OPERATIONMODE_FINISH0x04All data fed, finalize and get resultLast call
CRYPTO_OPERATIONMODE_SINGLECALL0x07START | UPDATE | FINISH in one shotSmall data, single-call APIs
CRYPTO_OPERATIONMODE_SAVE_CONTEXT0x05Snapshot workspace to bufferCsm_SaveContextJob
CRYPTO_OPERATIONMODE_RESTORE_CONTEXT0x06Restore workspace from bufferCsm_RestoreContextJob
Exam trap: START + FINISH without UPDATE (mode 0x05 = 0x01|0x04) → E_NOT_OK. START+FINISH without UPDATE is explicitly forbidden. UPDATE must appear at least implicitly.

Which Services Support Streaming?

Only these 9 services support START / UPDATE / FINISH. All others use SINGLECALL internally regardless of what the caller passes:

HASH MACGENERATE MACVERIFY ENCRYPT DECRYPT AEAD_ENCRYPT AEAD_DECRYPT SIGNATUREGENERATE SIGNATUREVERIFY

Context Save/Restore

Allows pausing a streaming job, saving its internal workspace, and resuming later. Useful in multicore or time-sharing scenarios.

1
Job must be in ACTIVE state. Calling Save/Restore on IDLE job → returns E_NOT_OK.
2
Csm_SaveContextJob(jobId, contextBufferPtr, contextBufferLengthPtr) — CSM sets mode to SAVE_CONTEXT, calls CryIf_ProcessJob.
3
Crypto Driver writes workspace bytes into contextBufferPtr. Key elements are NOT part of context.
4
Csm_RestoreContextJob(jobId, contextBufferPtr, contextBufferLength) — CSM sets mode to RESTORE_CONTEXT, calls CryIf_ProcessJob.
5
Continue with UPDATE/FINISH normally.

3. Functional Specification

Queue & Priority System

CSM can have multiple queues (CsmQueues). Each queue maps to one channel → one Crypto Driver Object. Jobs sit in queues sorted by priority (higher numeric value = higher priority).

Priority rules: If sync job priority > highest queued priority → queue is "paused" until sync job completes. If sync job priority < highest queued → returns CRYPTO_E_BUSY. Queue can optionally have zero size (synchronous-only channels).

CSM Module Startup

Csm_Init(configPtr) — configPtr must ALWAYS be NULL_PTR. Returns void. CSM does NOT trigger CryIf init — that is BswM's responsibility.

If init fails → reports CSM_E_INIT_FAILED to DET (if CsmDevErrorDetect = true).

Partitioned ECUs: CSM may provide one init function per partition.

Multicore Support

CsmQueues are handled in the MainFunction context. Each CsmQueue references a CsmQueueMainFunctionRef to assign it to a partition. Csm_<Service> APIs can be called from partitions their jobs are assigned to, with appropriate inter-partition mechanisms.

I/O Redirection

Input or output of a job can be redirected to a key element (instead of a pointer buffer). This is static compile-time config — cannot change at runtime.

Crypto_JobRedirectionInfoType.redirectionConfig is a bit field:

Bit 0 (0x01)Primary input → inputKeyId/KeyElementIdCRYPTO_REDIRECT_CONFIG_PRIMARY_INPUT
Bit 1 (0x02)Secondary input redirectedCRYPTO_REDIRECT_CONFIG_SECONDARY_INPUT
Bit 2 (0x04)Tertiary input redirectedCRYPTO_REDIRECT_CONFIG_TERTIARY_INPUT

If redirection is configured for an I/O but the corresponding length parameter is NOT 0 → job not processed, returns E_NOT_OK.

General Interface

void Csm_Init(const Csm_ConfigType* configPtr)

configPtr MUST be NULL. Non-reentrant. Synchronous. Returns void. Reports CSM_E_INIT_FAILED on failure.

void Csm_GetVersionInfo(Std_VersionInfoType* versioninfo)

Reentrant. Returns version info. NULL ptr → CSM_E_PARAM_POINTER reported.

void Csm_MainFunction(void)

Called cyclically by BSW Scheduler. Processes queued async jobs — dispatches to CryIf_ProcessJob. Each CsmMainFunction container instance generates one Csm_MainFunction_<shortName>. Non-reentrant.

Callback

void Csm_CallbackNotification(Crypto_JobType* job, Crypto_ResultType result)

Called by CryIf when async job finishes. CSM then calls the application's configured callback:

void <CsmCallbackFunc>(uint32 jobId, Crypto_ResultType result)

Triggered only once per job call — when FINISH bit is set in mode. The app callback name is configured via CsmJobPrimitiveCallbackRef → CsmCallbackFunc.

Conditions for callback: CsmProcessingMode==ASYNC AND CsmJobInterfaceUsePort==CRYPTO_USE_FNC AND CsmJobPrimitiveCallbackRef != 0.

5. Sequence Diagrams

Select a scenario to see the full call sequence.

Synchronous Hash (Single-call) — SHA-256

1
App → CSM: Csm_Hash(jobId, SINGLECALL, dataPtr, dataLen, resultPtr, &resultLen)
2
CSM checks: initialized? jobId valid? pointers non-null? If any fail → DET error, return E_NOT_OK.
3
CSM checks job state = IDLE, packs Crypto_JobType (I/O pointers, mode=SINGLECALL, primitiveInfo with SHA-256 config).
4
CSM → CryIf: CryIf_ProcessJob(channelId, &job)
5
CryIf → Crypto Driver: Crypto_ProcessJob(objectId, &job) — Driver executes SHA-256 in-place.
6
Crypto Driver writes hash to resultPtr, updates resultLen, returns E_OK.
7
Return value bubbles back: CryIf → CSM → App. No callback (synchronous).

Asynchronous MAC-Generate (Streaming — AES-CMAC)

1
App: Key must be VALID before calling. Check with Csm_KeyGetStatus if needed.
2
App → CSM: Csm_MacGenerate(jobId, START, dataPtr1, len1, macPtr, &macLen) → returns E_OK (job queued or passed to CryIf immediately).
3
Csm_MainFunction() runs cyclically — picks job from queue, calls CryIf_ProcessJob. Job transitions to ACTIVE.
4
App → CSM: Csm_MacGenerate(jobId, UPDATE, dataPtr2, len2, macPtr, &macLen) → CSM sees job ACTIVE, calls CryIf immediately.
5
Repeat UPDATE as many times as needed for all data chunks.
6
App → CSM: Csm_MacGenerate(jobId, FINISH, NULL, 0, macPtr, &macLen)
7
Crypto Driver finalizes CMAC, writes to macPtr. Calls back via CryIf_CallbackNotification → Csm_CallbackNotification(job, E_OK).
8
CSM sees FINISH bit set → calls App's configured callback MyMacCallback(jobId, E_OK).
9
App reads MAC value from macPtr. Job state returns to IDLE.

Key Setup Sequence — Setting an AES-128 Key

1
App → CSM: Csm_KeyElementSet(keyId, CRYPTO_KE_CIPHER_KEY, aesKeyBytes, 16) — write 128-bit key material to element ID 1 of key "keyId".
2
CSM validates params (no DET errors), calls CryIf_KeyElementSet(cryIfKeyId, CRYPTO_KE_CIPHER_KEY, ...).
3
CryIf maps to Crypto Driver, calls Crypto_KeyElementSet(cryptoKeyId, ...). Key material written to Driver's secure storage.
4
(Optional) Set IV: Csm_KeyElementSet(keyId, CRYPTO_KE_CIPHER_IV, ivBytes, 16)
5
App → CSM: Csm_KeySetValid(keyId) — marks key as valid. Key cannot be used for crypto ops until this is called.
6
CryIf_KeySetValid → Crypto_KeySetValid. Key state = VALID.
7
Now Csm_Encrypt(jobId, SINGLECALL, ...) will succeed. If called before KeySetValid → CRYPTO_E_KEY_NOT_VALID.
Key management functions call Csm_DisableProcessing internally — CSM stops loading new jobs from queue until next MainFunction cycle. This prevents race conditions.

ECDH Key Exchange (Alice & Bob)

Both ECUs run same procedure. Alice initiates.

1
Alice — Setup: Set CRYPTO_KE_KEYEXCHANGE_BASE (generator point / modulus), CRYPTO_KE_KEYEXCHANGE_PRIVKEY (private key or let driver generate). Call Csm_KeySetValid.
2
Alice → CSM: Csm_KeyExchangeCalcPubVal(keyId, pubValPtr, &pubValLen) → Crypto Driver generates own public value using private key.
3
Alice sends pubValPtr data to Bob over network (may need signing/encryption depending on protocol).
4
Bob does same: sets up keys, calls CalcPubVal, sends his public value to Alice.
5
Alice → CSM: Csm_KeyExchangeCalcSecret(keyId, bobPubValPtr, bobPubValLen) → Crypto Driver computes shared secret, stores in CRYPTO_KE_KEYEXCHANGE_SHAREDVALUE.
6
Bob does same with Alice's public value. Both now have same shared secret in their SHAREDVALUE key element.
7
Copy shared secret to a session key using Csm_KeyElementCopy, call Csm_KeySetValid, then use for AES/CMAC.

Job Cancellation

1
App → CSM: Csm_CancelJob(jobId, mode) — mode parameter is ignored (compatibility only). Synchronous call.
2
If job is in CSM queue (not yet dispatched to driver): removed from queue. Returns E_OK. Callback is NOT called in this path.
3
If job is already in Crypto Driver: CSM calls CryIf_CancelJob.
4a
Instant cancel (CryIf returns E_OK): CSM calls job callback with CRYPTO_E_JOB_CANCELED. App must NOT read output buffers.
4b
Deferred cancel (CryIf returns CRYPTO_E_JOB_CANCELED): Driver is mid-operation. CSM waits — callback will be called with CRYPTO_E_JOB_CANCELED at next Csm_CallbackNotification. Buffers may still be written!
5
Cannot cancel sync jobs → CSM_E_PROCESSING_MODE reported to DET, returns E_NOT_OK.

7. Key Management — Complete Reference

Key Architecture

Keys live in the Crypto Driver. CSM references them by ID only. Applications never see symmetric or private key material — the Crypto Driver enforces access control.

Key (CsmKey)

Named ID in CSM config. Has 1+ elements. Can be marked valid/invalid. Config: CsmKeyUsePort (RTE port or function interface)

Key Element

Typed byte array within a key. Standard IDs defined by spec. Vendor-specific IDs start at 1000+. Access-controlled (read/write/exportable).

Key Formats

BIN_OCTET, BIN_SHEKEYS (M1-M5), BIN_IDENT_PRIVATEKEY_PKCS8, BIN_IDENT_PUBLICKEY, BIN_RSA_PRIVATEKEY, BIN_RSA_PUBLICKEY
