AUTOSAR COM R24-11 — Study Guide

Architecture Overview

AUTOSAR COM (Communication) is a BSW module that handles signal/signal-group packing into I-PDUs (Interaction Layer PDUs), transmission modes, deadline monitoring, filtering, and notification. It sits between the RTE/Application layer and the PDU Router (PduR).

Scheduled Functions

Com_MainFunctionTx()

Processes TX activities not handled in send-path. Triggers periodic/cyclic transmissions, repetitions (N-Times), MDT expiry, TX deadline monitoring. Called from OS task at configured period.

Critical

An I-PDU is sent at most once per Com_MainFunctionTx() call.

Com_MainFunctionRx()

Processes DEFERRED signal unpacking and indication callbacks. Also handles RX deadline monitoring timer counting. Called from OS task at configured period.

Com_MainFunctionRouteSignals()

Executes the Signal Gateway. Routes signals from source to destination I-PDUs. Latency directly determined by the period of this call. Called from OS task.

🎯 Exam Critical — COM's Role

COM does NOT know about physical bus protocols. It operates purely at the I-PDU level, packing/unpacking signal bits according to ComBitPosition, ComBitSize, ComSignalEndianness. PduR handles all routing. COM never accesses bus hardware directly.

Signal & PDU Model

Understanding the hierarchy of COM objects is the foundation of everything. A signal is the smallest unit of application data. Signals are packed into I-PDUs. I-PDUs are transported by PduR.

Object Hierarchy

I-PDU (I-Signal PDU) — the transport container; has a direction (TX or RX), I-PDU group membership, and transmission mode config
├──
Signal — a scalar value (uint8/16/32, boolean, float32, etc.) packed at a specific bit position within the I-PDU
├──
Signal Group — a consistent set of group signals sharing a shadow buffer; treated atomically
└──
Group Signal — individual field within a signal group; accessed via shadow buffer

Update-Bits

What are they?

A 1-bit flag inside the I-PDU (at ComUpdateBitPosition) that tells the receiver whether the sender has actually updated this signal before the I-PDU was sent. Invisible to SWCs — managed entirely by COM.

  • 1 = "set / data was updated by sender"
  • 0 = "cleared / not updated"
  • Cleared by Com_Init() and Com_IpduGroupStart()

Update-Bit Rules

EventUpdate-Bit
Com_SendSignal() calledSET (→1)
Com_SendSignalGroup() calledSET (→1)
ComTxIPduClearUpdateBit = TransmitCLEARED after PduR_ComTransmit returns E_OK
ComTxIPduClearUpdateBit = ConfirmationCLEARED after TX Confirmation received
ComTxIPduClearUpdateBit = TriggerTransmitCLEARED after Com_TriggerTransmit

Update-bits NOT allowed with DIRECT mode + ComTxModeNumberOfRepetitions ≥ 1.

Signal Group — Shadow Buffer Mechanism

Signal groups use a shadow buffer to guarantee atomic access (complex data type consistency). The shadow buffer is a separate RAM area that mirrors the group signal portions of the I-PDU buffer.

TX — Sending a Signal Group

1

Write group signals to shadow buffer

Call Com_SendSignal(groupSignalId, &val) for each group signal — only updates shadow buffer

2

Atomically copy shadow → I-PDU

Call Com_SendSignalGroup(signalGroupId) — atomically packs all group signals into I-PDU buffer. TMS re-evaluated.

3

COM schedules transmission

Based on ComTransferProperty and transmission mode, COM calls PduR_ComTransmit() in next MainFunctionTx

RX — Receiving a Signal Group

1

Copy I-PDU → shadow buffer

Call Com_ReceiveSignalGroup(signalGroupId) — atomically copies group signals from I-PDU buffer to shadow buffer

2

Read group signals from shadow buffer

Call Com_ReceiveSignal(groupSignalId, &val) for each group signal — reads from shadow buffer

Endianness Conversion

Supported

  • All integer types (uint8..uint64, sint8..sint64)
  • Signals within signal groups (group signals)
  • float32/float64 — endianness only (no complex conversion)
  • Converted before filtering/notification on RX side
  • Converted before I-PDU callout on TX side

Sign Extension (RX side only)

When a signed signal (e.g., 10-bit sint) is received and copied to a larger type (e.g., sint16), COM extends the sign bit.

Example: 10-bit signed 1111111101b → extended to 16-bit: 1111111111111101b (both = -3)

Not performed for float32/float64. Sender side does not need sign extension (values are already in correct format).

Signal Initialization

During Com_Init(), for each I-PDU:

  1. Fill entire I-PDU byte-by-byte with ComTxIPduUnusedAreasDefault (e.g., 0xFF)
  2. Overwrite each signal's bits with the lower N bits of ComSignalInitValue (bit-wise)
  3. Clear all update-bits to 0
🎯 Exam Critical

Filter's old_value is set to ComSignalInitValue at startup. Until first application send, new_value for filter evaluation also uses ComSignalInitValue.

Transmission Side

COM's TX processing determines when and how often an I-PDU is sent, based on the Transmission Mode (TM), Transfer Property of signals, and the Transmission Mode Selector (TMS).

4 Transmission Modes (ComTxModeMode)

PERIODIC

  • Cyclic: sends every ComTxModeTimePeriod
  • No send-path trigger from signal writes
  • Start delay: ComTxModeTimeOffset
  • MDT not applied by default (unless ComEnableMDTForCyclicTransmission=true)

DIRECT

  • Event-driven: triggered by signal with TRIGGERED transfer property
  • N+1 transmissions (N = ComTxModeNumberOfRepetitions)
  • Period between repetitions: ComTxModeRepetitionPeriod
  • New send request cancels outstanding repetitions

MIXED

  • Combination: cyclic baseline + event-driven bursts
  • TRIGGERED signal → N-Times burst overlaps with cyclic
  • Cyclic TX counts as one of the N-Times transmissions
  • Most complex mode — used for important data that needs both reliability and responsiveness

NONE

  • COM never initiates transmission autonomously
  • Only sent via Com_TriggerTransmit() from PduR (LIN/FlexRay slots)
  • I-PDU still accessible for reading latest data
  • Can be used to disable sending (when TMS evaluates to false)

Transmission Mode Selector (TMS)

Each TX I-PDU has two statically configured transmission mode sets: ComTxModeTrue and ComTxModeFalse. The TMS mechanism selects which one is active based on signal values.

TMS Logic Flow

  • Signals with a ComFilter container → "contributing signals"
  • Each contributing signal has a Transmission Mode Condition (TMC) = its filter evaluation result (true/false)
  • TMS = true if ANY contributing signal's TMC is true → use ComTxModeTrue
  • TMS = false if NO contributing signal's TMC is true → use ComTxModeFalse
  • If NO signals contribute to TMS → TMS is always true

On TMS Change

  • Mode switch takes effect immediately (within next MainFunctionTx)
  • PERIODIC/MIXED timer restarted on TMS change
  • Switch to DIRECT → immediate N-Times transmission initiated
  • Switch to PERIODIC/MIXED → first TX immediately (respecting MDT)
  • TMS re-evaluated after every Com_SendSignal() or Com_SendSignalGroup()
🎯 Exam Critical

A signal with ComFilterAlgorithm=ALWAYS always sets TMS to true. If you configure a PENDING signal with ALWAYS filter, the I-PDU will always be in the ComTxModeTrue transmission mode — regardless of other signals.

Minimum Delay Timer (MDT)

Purpose

Prevents sending an I-PDU more frequently than ComMinimumDelayTime. If a transmission is requested while MDT is running, it is postponed until MDT expires.

  • MDT starts when I-PDU is sent via PduR_ComTransmit
  • Reloaded on successful TX confirmation
  • Reset by I-PDU group start
  • If ComRetryFailedTransmitRequests=false: MDT starts on any transmit attempt
  • If ComRetryFailedTransmitRequests=true: MDT starts only if PduR returns E_OK

MDT and Cyclic Transmissions

By default, ComEnableMDTForCyclicTransmission=false means MDT does NOT apply to:

  • PERIODIC transmissions
  • Cyclic part of MIXED transmissions
  • N-Times repetitions

MDT still applies to event-driven (TRIGGERED) transmissions in DIRECT/MIXED mode, even when cyclic MDT is disabled.

N-Times Transmission (Repetitions)

When a TRIGGERED signal fires in DIRECT/MIXED mode with ComTxModeNumberOfRepetitions > 0:

  1. COM sends the I-PDU immediately (1st transmission)
  2. COM sends again every ComTxModeRepetitionPeriod
  3. This continues until ComTxModeNumberOfRepetitions + 1 successful confirmations received
  4. A new send request cancels remaining repetitions and starts fresh from step 1

If TX deadline monitoring expires before all repetitions complete → remaining repetitions are cancelled (not continued). Transmission deadline monitoring timer is NOT restarted after N-Times.

MIXED mode special case

If a cyclic TX in MIXED mode is pending confirmation when an N-Times request starts, that cyclic confirmation counts as the 1st N-Times confirmation. Only N repetitions of the new value are observed.

Retry Failed Transmit Requests

If ComRetryFailedTransmitRequests=true and PduR_ComTransmit() returns NOT E_OK:

  • COM retries in the next Com_MainFunctionTx() call
  • If I-PDU data was updated in between, the new data is sent
  • TX deadline monitoring starts immediately on first attempt (regardless of result)
  • If TX DM timer expires → pending retry is expired/cancelled

Complete TX Signal Flow

1

SWC calls Com_SendSignal(signalId, &value)

COM writes value to signal's position in I-PDU buffer (packing). Sets update-bit. Re-evaluates TMS. Determines if transmission should be triggered based on transfer property and TM.

2

I-PDU Callout (if configured: ComTxIPduCallout)

Optional user callout before sending. Returns true → send; false → discard. Endianness conversion already applied at this point.

3

Com_MainFunctionTx(): PduR_ComTransmit() called

COM passes I-PDU to PduR. TX DM timer started (if not already running). MDT started. At most once per MainFunctionTx call.

4

Com_TxConfirmation() (from PduR)

Successful: clear update-bits (if Confirmation mode). Cancel TX DM timers. MDT reloaded. Trigger CbkTxAck notifications. Count N-Times confirmation.

5

TX DM Timeout (if no confirmation in time)

Notify SWC via CbkTxTOut callbacks. Timer NOT restarted (TX DM unlike RX DM). Cancel any pending transmissions if CancellationSupport=true.

Reception Side

COM receives I-PDUs from PduR via Com_RxIndication, unpacks signals, applies filtering, handles invalid values, and notifies applications. Understanding IMMEDIATE vs DEFERRED unpacking is critical.

Two Unpacking Modes (ComIPduSignalProcessing)

IMMEDIATE

Signal processing and callbacks happen inside Com_RxIndication() — in the same execution context as the interrupt/task that received the PDU.

  • Lower latency — SWC notified in same call
  • Risk: callback runs in interrupt context → must be fast, no blocking
  • Use for time-critical signals

DEFERRED

Data copied into COM's internal buffer inside Com_RxIndication(), but actual signal unpacking + callbacks deferred to Com_MainFunctionRx().

  • Decoupled from interrupt context — safer
  • If Com_ReceiveSignal() called before next MainFunctionRx(), returns previous value
  • Latency = up to 1 MainFunctionRx period
  • Use for non-time-critical signals; standard approach

Complete RX Signal Flow

1

PduR calls Com_RxIndication(PduId, &PduInfo)

COM checks if I-PDU is active (started). If stopped → ignore. Checks received length against configured length. I-PDU Callout (if configured) → if returns false, discard.

2

Unpack signals from I-PDU bytes

Endianness conversion. Sign extension for signed types. Only completely received signals unpacked (partial signals skipped for length-checking safety). Update-bit check for signals with update-bits.

3

Reset RX Deadline Monitoring timer

Timer reset to ComTimeout (or ComFirstTimeout for first reception after start). Done inside Com_RxIndication regardless of IMMEDIATE/DEFERRED.

4

Check for Invalid Value

If received value = ComSignalDataInvalidValue: NOTIFY action → call CbkInv, don't store value. REPLACE action → replace with ComSignalInitValue, continue normal processing.

5

Apply Reception Filter

Evaluate ComFilterAlgorithm. If filter evaluates to false → discard signal, don't notify. old_value only updated if filter passes. (IMMEDIATE: happens in Com_RxIndication. DEFERRED: happens in Com_MainFunctionRx.)

6

Notify SWC via CbkRxAck

If filter passed and value was stored: call configured CbkRxAck callback. SWC can then call Com_ReceiveSignal() to read the new value.

🎯 Filter Key Rules

1. Filtering happens ONLY on RX side. TX side uses filter only for TMS evaluation (not to suppress sending).
2. If filter rejects → value NOT stored, NOT notified, old_value NOT updated.
3. After a RX DM timeout, first value received for MASKED_NEW_DIFFERS_MASKED_OLD always passes (treated as different from old_value).
4. Filter old_value = ComSignalInitValue at startup.

Signal Invalidation

TX: Sending Invalid Value

Sender indicates "I cannot provide valid data" (e.g., faulty sensor).

  • Call Com_InvalidateSignal(signalId) — internally performs Com_SendSignal(signalId, ComSignalDataInvalidValue)
  • Transfer property and TMS apply as for normal send
  • Invalid value used for TMS filter evaluation
  • For signal groups: Com_InvalidateSignalGroup() invalidates all group signals

RX: Receiving Invalid Value

Receiver handles based on ComDataInvalidAction:

ActionBehavior
NOTIFYCall CbkInv. Do NOT store invalid value. Do NOT run normal filter/notification. Next ReceiveSignal returns last valid (or init).
REPLACEReplace with ComSignalInitValue. Process normally (filter, notify CbkRxAck) as if init value was received.

RX DM timer is still restarted when invalid signal/signal group is received — it counts as "something arrived".

RX Behavior for Partial I-PDUs (length < configured)

  • Signals completely within received bytes → unpacked and notified normally
  • Signals partially received → NOT unpacked, NOT notified
  • Signal groups partially received → entire signal group discarded
  • A signal whose update-bit is outside the received bytes → treated as if update-bit was SET (signal is considered updated)

I-PDU Groups

I-PDU groups are logical collections of I-PDUs that can be started and stopped together. They are the mechanism for controlling which I-PDUs are actively transmitting or receiving.

I-PDU Group Rules

  • An I-PDU can belong to any number of I-PDU groups
  • An I-PDU group contains I-PDUs of one direction only (TX or RX — not both)
  • An I-PDU is active (started) if at least one of its groups is started
  • All groups must be stopped to fully deactivate an I-PDU
  • I-PDUs not assigned to any group are started automatically during Com_Init() and can never be stopped
  • Maximum number of I-PDU groups: ComSupportedIPduGroups
🎯 Critical

I-PDU group "nesting" is a conceptual tool only — COM does NOT handle nested groups. If group B contains group A, then I-PDUs of group A must also be explicitly added to group B in configuration. BswM must consistently manage all group states.

Com_IpduGroupStart(groupId, initialize)

Always performed (regardless of initialize flag):

  • Reset ComMinimumDelayTime timer for DIRECT/MIXED I-PDUs
  • Restart all RX DM timers for signals with non-zero ComFirstTimeout
  • Cancel TX DM timers; use ComFirstTimeout for first TX DM
  • Clear all update-bits
  • Reset ONE_EVERY_N filter OCCURRENCE counter
  • Start RX deadline monitoring (if I-PDU in DM-enabled group)
  • Determine transmission mode from current data content (TMS evaluation)

Additionally, if initialize=true:

  • Re-initialize I-PDU data as in Com_Init() (ComTxIPduUnusedAreasDefault + ComSignalInitValue)
  • Re-initialize shadow buffers of signal groups
  • Reset old_value of filters to ComSignalInitValue
  • Reset ComTxModeTimePeriod and ComTxModeTimeOffset for PERIODIC/MIXED I-PDUs

Com_IpduGroupStop(groupId) — What Gets Cancelled

For TX I-PDUsFor RX I-PDUs
  • Cancel all outstanding TX requests (N-Times, pending sends)
  • Cancel TX deadline monitoring timers
  • Invoke CbkTxErr for signals/signal groups with outstanding unconfirmed TX
  • Ignore any subsequent Com_TxConfirmation() for this I-PDU
  • Internal buffers still updated by Com_SendSignal() (but returns COM_SERVICE_NOT_AVAILABLE)
  • Com_TriggerTransmit() returns E_NOT_OK
  • If large I-PDU TX in progress: stop immediately
  • Disable RX processing (ignore incoming PDUs)
  • Cancel RX deadline monitoring
  • No callbacks fired on reception
  • Com_ReceiveSignal() returns last known value (or init) with COM_SERVICE_NOT_AVAILABLE

CbkTxErr (TX error callback) is triggered when Com_IpduGroupStop() is called while there are signals/signal groups with outstanding unconfirmed transmissions. Reception Deadline Monitoring Control

Com_EnableReceptionDM(ipduGroupId)

  • Sets RX DM timer to ComFirstTimeout for all signals in this RX I-PDU group
  • Silently ignored if DM already enabled for this group
  • Silently ignored if called on TX I-PDU group

Com_DisableReceptionDM(ipduGroupId)

  • Disables RX DM — timer expiry events are suppressed (no CbkRxTOut called)
  • Reception of I-PDU is still processed normally (not stopped)
  • Silently ignored if DM already disabled or called on TX group

Deadline Monitoring

COM monitors both reception (are messages arriving on time?) and transmission (are sent messages being confirmed on time?). These are separate, configurable mechanisms with different restart behaviors.

RX Deadline Monitoring — "Is my data arriving on time?"

Timer Configuration

ParameterMeaning
ComFirstTimeoutTimeout for first-ever reception after I-PDU group start. If 0 → no monitoring until first reception.
ComTimeoutTimeout for all subsequent receptions. If 0 → no monitoring at all.

RX DM timer is restarted after each timeout — periodic timeout notifications if reception fails for an extended period. (Unlike TX DM!)

🎯 Exam Critical

I-PDU-level DM does NOT check signal values. It only checks if the I-PDU arrived. Per-signal DM additionally checks if the update-bit was set, meaning the sender actually updated this particular signal.

TX Deadline Monitoring — "Was my sent message confirmed on time?"

Purpose

Supervises the lower layers and bus — not COM itself. Detects bus errors, CAN arbitration failures, missing confirmations from physical transmission.

Timer Start Rules

  • Started when I-PDU is sent to PduR via PduR_ComTransmit()
  • If timer already running → NOT restarted on new send request
  • If multiple signals in I-PDU have different ComTimeout → use smallest value
  • First timer start → uses ComFirstTimeout; subsequent → ComTimeout
  • Independent of transfer property (PENDING or TRIGGERED)

Timer Stop/Cancel Rules

  • Cancelled on successful Com_TxConfirmation()
  • NOT restarted after timeout (unlike RX DM)
  • Stopped when I-PDU group is stopped
  • N-Times: cancelled after all N+1 confirmations received
  • For DIRECT + N-Times: timer reset on new send request (SWS_Com_00739)

TX DM fires even if transmission was filtered out by I-PDU callout. Also fires even if confirmation returned E_NOT_OK. Only successful E_OK confirmation cancels the TX DM timer.

ComFirstTimeout vs ComTimeout

ComFirstTimeout: Applied the first time a deadline timer starts after I-PDU group activation. Allows for a longer grace period on startup (e.g., sensor needs time to warm up before first transmission).

ComTimeout: Applied for all subsequent timeouts. Usually shorter than ComFirstTimeout.

If ComFirstTimeout=0 → no monitoring until first reception (I-PDU monitoring is waived for first reception).

Callbacks & Notifications

COM uses configurable callback functions to notify applications of significant events. Understanding each callback's trigger condition, which side (TX/RX), and when it fires is essential for testing and debugging.

All 5 Callback Types

RX side onlyNotification Class 1

CbkRxAck — Reception Acknowledge

Called when a signal or signal group is successfully received and stored. Indicates new data is available.

  • IMMEDIATE: called inside Com_RxIndication()
  • DEFERRED: called inside Com_MainFunctionRx()
  • NOT called if filter rejects the value
  • NOT called if invalid value received (NOTIFY action)
  • Called after each successful periodic reception
void CbkRxAck(CbkHandleIdType ComUserCbkHandleId)
TX side onlyNotification Class 2

CbkTxAck — Transmission Acknowledge

Called immediately after successful transmission confirmation. For N-Times: called only after ALL N+1 confirmations received.

  • Triggered by Com_TxConfirmation() from PduR
  • For periodic: called after each successful periodic TX confirmation
  • For N-Times: only after final (Nth+1) confirmation
  • NOT called if I-PDU group is stopped when confirmation arrives
void CbkTxAck(CbkHandleIdType ComUserCbkHandleId)
RX side onlyNotification Class 3

CbkRxTOut — Reception Timeout

Called when RX deadline monitoring timer expires — signal/signal group has not been received within the configured timeout.

  • Called in Com_MainFunctionRx() context
  • Called periodically if signal continues to be missing (RX DM timer restarts)
  • Fires independently from ComRxDataTimeoutAction (replace/substitute/none)
  • Not fired if DM is disabled via Com_DisableReceptionDM
void CbkRxTOut(CbkHandleIdType ComUserCbkHandleId)
TX side onlyNotification Class 4

CbkTxTOut — Transmission Timeout

Called when TX deadline monitoring timer expires — sent I-PDU was not confirmed in time by the bus.

  • Called in Com_MainFunctionTx() context
  • Called only ONCE (TX DM timer not restarted)
  • Fires even if TX was filtered by I-PDU callout
  • Fires even if PduR returned E_NOT_OK
  • NOT called if I-PDU group stopped before timeout
void CbkTxTOut(CbkHandleIdType ComUserCbkHandleId)
RX side only

CbkInv — Invalid Value Received

Called when a signal with ComDataInvalidAction=NOTIFY receives its configured ComSignalDataInvalidValue.

  • Called instead of CbkRxAck (NOT both)
  • Received invalid value NOT stored — last valid value preserved
  • No filter evaluation when invalid action=NOTIFY
  • RX DM timer is still restarted on invalid value receipt
  • For signal groups: fires if ANY group signal = invalid value
void CbkInv(CbkHandleIdType ComUserCbkHandleId)
TX side onlyNotification Class 4

CbkTxErr — Transmission Error

Called when I-PDU group is stopped while signals/signal groups have outstanding unconfirmed transmissions.

  • Triggered only by Com_IpduGroupStop() with pending TX
  • NOT called for TX DM timeout (that's CbkTxTOut)
  • Allows SWC to know that requested TX will never complete
void CbkTxErr(CbkHandleIdType ComUserCbkHandleId)

I-PDU Callouts

Com_TxIpduCallout

User-defined TX callout. Called after endianness conversion, before sending to PduR.

  • Returns true → I-PDU transmitted
  • Returns false → I-PDU NOT transmitted (filtered)
  • TX DM timer still fires even if callout returns false
  • Can inspect/modify I-PDU data before transmission

Com_RxIpduCallout

User-defined RX callout. Called after I-PDU received from PduR, before signal unpacking.

  • Returns true → I-PDU processed normally
  • Returns false → I-PDU discarded, no signals processed
  • Receives complete I-PDU data for custom validation
  • Use for custom authentication, CRC verification, filtering

API Reference

COM has a rich API set. Every function has a fixed Service ID used in DET error reporting. Key principle: COM never rejects a signal write — even to a stopped I-PDU, the buffer is updated but returns COM_SERVICE_NOT_AVAILABLE.

Com_Init / Com_DeInitService ID: 0x01 / 0x02SyncNon-Reentrant
void Com_Init(const Com_ConfigType* config); void Com_DeInit(void);

Com_Init: Initializes all I-PDUs (unused areas + init values), clears update-bits, shadow buffers. All I-PDU groups STOPPED. I-PDUs without group are started implicitly.
Com_DeInit: Stops all I-PDU groups. No further communication.

Com_IpduGroupStart / Com_IpduGroupStop0x03 / 0x04Sync
void Com_IpduGroupStart(Com_IpduGroupIdType IpduGroupId, boolean initialize); void Com_IpduGroupStop(Com_IpduGroupIdType IpduGroupId);

Start: Starts all I-PDUs in group. If initialize=true: full data re-init. Resets timers, clears update-bits. Determines TMS from current data.
Stop: Stops all I-PDUs. Cancels TX, DM timers. Fires CbkTxErr for pending TX. Disables RX processing.

Com_EnableReceptionDM / Com_DisableReceptionDM0x06 / 0x05Sync
void Com_EnableReceptionDM(Com_IpduGroupIdType IpduGroupId); void Com_DisableReceptionDM(Com_IpduGroupIdType IpduGroupId);

Enable: Sets RX DM timer to ComFirstTimeout. Silently ignores if already enabled or if TX group. Disable: Suppresses timeout notifications but does NOT stop reception. Silently ignores if already disabled or TX group.

Com_SendSignal0x0ASyncReentrant (for different signals)
uint8 Com_SendSignal(Com_SignalIdType SignalId, const void* SignalDataPtr); // Returns: E_OK, COM_SERVICE_NOT_AVAILABLE (stopped), COM_BUSY (large I-PDU TX in progress)

Writes signal to I-PDU buffer. Sets update-bit. Re-evaluates TMS. If signal belongs to signal group: only updates shadow buffer. If I-PDU stopped: buffer still updated but returns COM_SERVICE_NOT_AVAILABLE.

Com_ReceiveSignal0x0BSyncReentrant
uint8 Com_ReceiveSignal(Com_SignalIdType SignalId, void* SignalDataPtr); // Returns: E_OK, COM_SERVICE_NOT_AVAILABLE (stopped)

Reads signal from I-PDU buffer (or shadow buffer for group signals). If stopped: returns last known value (or init) with COM_SERVICE_NOT_AVAILABLE. If DEFERRED and not yet processed: returns previous value.

Com_SendSignalGroup / Com_ReceiveSignalGroup0x0D / 0x0ESync
uint8 Com_SendSignalGroup(Com_SignalGroupIdType SignalGroupId); uint8 Com_ReceiveSignalGroup(Com_SignalGroupIdType SignalGroupId);

Send: Atomically copies shadow buffer → I-PDU buffer. TMS re-evaluated. Sets update-bit. Triggers transmission based on transfer property.
Receive: Atomically copies I-PDU buffer → shadow buffer. Then read individual group signals via Com_ReceiveSignal.

Com_InvalidateSignal / Com_InvalidateSignalGroup0x10 / 0x1BSync
uint8 Com_InvalidateSignal(Com_SignalIdType SignalId); uint8 Com_InvalidateSignalGroup(Com_SignalGroupIdType SignalGroupId);

Internally performs Com_SendSignal with ComSignalDataInvalidValue. Transfer property and TMS rules apply. For signal groups: invalidates all group signals.

Com_SendDynSignal / Com_ReceiveDynSignal0x20 / 0x21Sync
uint8 Com_SendDynSignal(Com_SignalIdType SignalId, const void* SignalDataPtr, uint16 Length); uint8 Com_ReceiveDynSignal(Com_SignalIdType SignalId, void* SignalDataPtr, uint16* LengthPtr);

For UINT8_DYN signals. Send: sets I-PDU length to smallest length containing the dynamic signal. Receive: returns actual received length in *LengthPtr. Formula: dynamic_len = received_PDU_len - (ComBitPosition/8).

Com_TriggerIPDUSend0x17Sync
uint8 Com_TriggerIPDUSend(PduIdType PduId);

Triggers transmission of the I-PDU regardless of its transmission mode. Used to force an immediate send. TX DM timer started if configured. MDT applies.

Com_TriggerTransmit0x41SyncReentrant
Std_ReturnType Com_TriggerTransmit(PduIdType TxPduId, PduInfoType* PduInfoPtr); // Returns E_NOT_OK if I-PDU is stopped

Called by PduR to request I-PDU data for bus-triggered transmission (LIN, FlexRay slots, NM). COM fills PduInfoPtr with current I-PDU data. Does NOT interfere with MDT. Returns E_NOT_OK for stopped I-PDUs.

Com_RxIndication / Com_TxConfirmation0x42 / 0x40Reentrant

RxIndication: Called by PduR on normal I-PDU reception. COM unpacks signals (IMMEDIATE) or copies data (DEFERRED). Resets RX DM timer.
TxConfirmation: Called by PduR on TX completion. result=E_OK → cancel TX DM, notify CbkTxAck, clear update-bits. result=E_NOT_OK → TX DM unaffected (timer not cancelled).

Signal Gateway

COM's integrated Signal Gateway routes signals and signal groups between I-PDUs in a 1:N fashion. A received signal can simultaneously be a local signal AND a source for gateway routing.

How Signal Gateway Works

Source Signal (received on RX I-PDU)
→
COM Signal Gateway (ComGwMapping)
→
Destination Signal(s) on TX I-PDU(s)
  • Gateway acts as a receiver for source signals, then immediately acts as sender for destination signals
  • Routing happens in Com_MainFunctionRouteSignals() — latency = one call period
  • Same signal processing rules apply: endianness, filters (if configured)
  • Optimizations allowed (e.g., skip unnecessary endianness conversion) as long as logical behavior is identical
  • 1:N mapping — one source signal can route to multiple destination signals
  • Signal groups can also be routed
  • Scales to zero if no gateway is configured (SWS_Com_00370)

Gateway Configuration

Configured via ComGwMapping containers. Each mapping has:

Source (ComGwSource)

  • Reference to source signal or signal group in RX I-PDU
  • Can be signal (ComGwSignal) or signal group
  • Source signal is still received locally AND routed

Destination (ComGwDestination)

  • Reference to destination signal or signal group in TX I-PDU
  • Multiple destinations per source allowed (1:N)
  • ComGwIPduRef: for I-PDU-to-I-PDU routing without signal decomposition

Signal Gateway Timeout Handling

🎯 Exam Critical

When a reception deadline monitoring timeout occurs for a source signal that is being gateway-routed, the Signal Gateway ignores the timeout — it does NOT propagate the timeout to the destination signal. The timeout notification goes to local RTE users only. (SWS_Com_00377, SWS_Com_00701, SRS_GTW_06089)

Large Data Types (TP I-PDUs)

TX Sequence (Large I-PDU)

  1. COM calls PduR_ComTransmit()
  2. PduR calls back Com_CopyTxData() one or more times
  3. PduR calls Com_TpTxConfirmation() when done

After PduR_ComTransmit succeeds, COM must NOT modify the TX buffer until TpTxConfirmation received.

RX Sequence (Large I-PDU)

  1. PduR calls Com_StartOfReception() — COM provides buffer size
  2. PduR calls Com_CopyRxData() one or more times
  3. PduR calls Com_TpRxIndication(result)

TpRxIndication(E_NOT_OK) → all signals treated as invalidated. CopyRxData/TxData may be called in interrupt context — data consistency must be ensured by implementation.

Dynamic Length Signals

  • Type: UINT8_DYN only
  • Must be the LAST signal in the I-PDU
  • Maximum length configured at configuration time
  • Sent via Com_SendDynSignal(signalId, ptr, length) — sets actual I-PDU length
  • Received via Com_ReceiveDynSignal(signalId, ptr, &length)
  • RX length formula: dyn_signal_len = received_PDU_len - (ComBitPosition / 8)
  • Max 1 dynamic length signal per I-PDU
  • Update-bit allowed (must be positioned before the dynamic signal)
CAN FD Note

For CAN FD dynamic length I-PDUs, using IpduM Multi PDU Mapping is recommended to avoid length misinterpretation due to CAN FD's discrete data lengths (8, 12, 16, 20, 24, 32, 48, 64 bytes).
