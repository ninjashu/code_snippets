What is the AUTOSAR Firewall module?

BSW module that inspects and filters ingress Ethernet traffic based on pre-defined FirewallRules. Key characteristics:

  • Ingress only — only incoming (RX) traffic filtered. Egress NOT filtered.
  • Sits between LSduR and the upper network stack
  • Supports stateless, stateful, and deep packet inspection
  • Integrates with IdsM (raises Security Events) and BswM (state control)
  • Supports deployment on smart switches (TCAM acceleration)

Core concepts — memorize these

FirewallRule

Pattern (expected header values) + Action (allow/block). Ordered list — first match wins. Firewall stops iterating on first match.

FirewallState

Vehicle state context (driving, parking, diagnostic session). BswM calls Fw_SetFirewallState(). Only rules in active FirewallStateDependentRules are checked.

FirewallDefaultAction

What happens when NO rule matches. Per-state setting. Set to block = allowlist mode. Set to allow = blocklist mode.

Pattern matching algorithm

Ordered iteration. Check ALL fields in a rule (AND logic — all must match). First rule that fully matches → carry out its action → stop.

Rule match condition

ALL of: DataLinkFilterConfig + NetworkLayerFilterConfig + TransportLayerFilterConfig + SOME/IP/DDS/DoIP/Payload configs must match for a rule to be a match.

Ingress-only limitation

Critical exam fact: Firewall only filters incoming (RX) traffic. Egress traffic passes unfiltered. Known limitation, explicitly stated in spec.

Firewall module state — simple, but important

Unlike EA, Firewall has no complex state machine with BUSY/IDLE. States are:

Fw_Init and Fw_GetVersionInfo: only APIs usable before initialization. All others require initialized state.

Packet flow through Firewall — complete sequence

1Network packet arrives at Ethernet layer → EthDrv/EthIf receives it
2LSduR calls Fw_RxIndication(RxPduId, PduInfoPtr) to deliver packet to Firewall
3Firewall checks active FirewallState → loads only rules in active FirewallStateDependentRules
4Firewall iterates rules in configured order. For each rule: checks DataLink → Network → Transport → App layer fields (ALL must match)
5aRule match found + Action = ALLOW: Firewall calls LsduR_FwRxIndication() — packet continues up the stack
5bRule match found + Action = BLOCK: packet dropped. If KeepLocalBuffer=TRUE also calls LSduR_FwRxReleaseBuffer(). SEv raised (future release).
5cNo rule matches + DefaultAction = BLOCK: drop packet. Firewall identifies which OSI layer caused mismatch → raises protocol-specific SEv to IdsM
5dNo rule matches + DefaultAction = ALLOW: packet passes. No SEv raised.
6Rate limiting check: if leaky bucket counter > FirewallBucketSize → block even if rule matched allow. Raises SEV_FW_PACKET_BLOCKED_RATELIMIT.
SEv raising currently only in no-match/default-block case. Match-based SEv raising planned for future release (R24-11 state).

FirewallState switching flow

1Vehicle state changes (e.g., enters diagnostic session)
2Application/mode manager informs BswM about new vehicle state
3BswM calls Fw_SetFirewallState(FirewallState)
4Firewall activates FirewallStateDependentRules referenced by that FirewallState
5If switch deployment: Firewall calls EthIf_SetStreamState() to sync switch TCAM rules with new active state
6Waits for Fw_StreamStateIndication() callback from EthIf confirming switch rules updated
7If timeout (FirewallSwitchRuleMgmtTimeout) or wrong status → retry → if still fails → FW_E_SWITCHRULEMGMT_FAILED
Leaky bucket counters PRESERVED across state switches — man-in-middle attacks cannot reset rate limiters by forcing state changes.

Three inspection types — know these cold

Stateless

Header field comparison against static configured values. OSI L2-L4. Supports exact values, ranges (min/max), subnets (address mask). Protocols: Ethernet, IPv4, IPv6, ICMP, TCP, UDP.

Stateful

TCP-specific. Tracks connection state, timeouts, parallel session count. Checks against TCP state machine (RFC 793). Only TCP — UDP is always stateless.

Deep Packet (DPI)

Application layer: SOME/IP, SOME/IP-SD, DDS-RTPS, DoIP, Generic byte-pattern. Allows access control to individual services (SOME/IP Service ID level).

Stateless inspection — field matching rules

Config elementWhat it checksMatch condition

Range and subnet matching

  • MAC/IP addresses: subnet match via FirewallFilterMACAddressMask / FirewallFilterIPAddressMask
  • IPv4 TTL: range via FirewallIpv4TtlMin / FirewallIpv4TtlMax (inclusive)
  • Port numbers: range via FirewallFilterPortLowerValue / FirewallFilterPortUpperValue (inclusive)
  • SOME/IP versions: range via Min/Max values
  • DoIP addresses: range via LowerValue/UpperValue

Checksum verification

If FirewallChecksumVerification = TRUE for IPv4/ICMP/TCP: Firewall checks if checksum field is available and valid. Config is per rule per protocol.

Not-modeled protocols — two mechanisms

  • EtherType blocking: configure FirewallFilterEtherType to block by EtherType (e.g., 0x0806=ARP, 0x22F0=AVTP, 0x88F7=PTP)
  • Generic byte pattern: FirewallPayloadBytePatternFilterConfig — expected bytes at specified offsets in packet

Deep Packet Inspection — per protocol

SOME/IP inspection

  • Inspects SOME/IP header fields: Service ID, Method ID, Length, Client ID, Session ID, Protocol Version, Interface Version, Message Type, Return Code
  • Multiple SOME/IP messages can be in one TCP/UDP frame → Firewall must find valid rule for EACH SOME/IP message
  • FirewallRule aggregates at most ONE SOME/IP message config
  • If any SOME/IP message has no matching rule → entire packet blocked (can't block individual SOME/IP messages within a packet)
  • Length verification: FirewallSomeipLengthVerification = TRUE → checks TCP/UDP payload = sum of all SOME/IP message lengths

SOME/IP-SD inspection

  • Same multi-message logic as SOME/IP
  • Validates SOME/IP-SD header fields against PRS_SOMEIPServiceDiscoveryProtocol default values
  • Supports version ranges (minor/major version min/max)

DDS inspection

  • Based on DDS-RTPS wire protocol
  • RTPS header + submessages. Multiple RTPS messages per TCP/UDP packet possible
  • Each FirewallRule: at most one RTPS header + submessage config
  • Same block-all-or-allow-all constraint as SOME/IP

DoIP inspection

  • Inspects DoIP header + source/dest address + UDS service codes
  • Does NOT inspect individual UDS DIDs/RIDs (use generic byte-pattern for that)
  • Supports address ranges (LowerValue/UpperValue)

Generic byte-pattern inspection

  • One FirewallRule can aggregate MULTIPLE FirewallPayloadBytePatternFilterConfig
  • Each specifies: offset into packet + expected byte value
  • All must match → rule is a match
  • Use for: unsupported protocols, payload inspection, UDS DID-level checks
Key exam fact: Firewall can only ALLOW or BLOCK entire network packets. If multiple SOME/IP messages in one packet and only one is bad → entire packet blocked.

Allowlist vs Blocklist — fundamental modes

Allowlist mode

FirewallDefaultAction = BLOCK. All patterns defined = what is ALLOWED. Anything without matching rule → blocked. Most secure — whitelist approach.

Industry standard for safety-critical ECUs: define exactly what traffic is expected, block everything else.

Blocklist mode

FirewallDefaultAction = ALLOW. All patterns defined = what is BLOCKED. Anything without matching rule → allowed through.

Used when only specific known-bad patterns must be blocked. Less restrictive.

Mixed mode: combine allow and block rules in same ruleset. Useful for IDS: define both allowed and explicitly-blocked patterns to get specific SEv mapping per pattern.

Rate limiting — leaky bucket algorithm

Detects cyclic messages sent at wrong frequency (man-in-middle, faulty ECU). Implemented as leaky bucket:

  • Counter: incremented by 1 on every rule pattern match
  • FirewallBucketSize: maximum counter value before blocking
  • FirewallRefillAmount: decrement rate (how fast counter drains, controlled by Fw_MainFunction / time)
  • If counter > FirewallBucketSize on match → block packet → raise SEV_FW_PACKET_BLOCKED_RATELIMIT
  • Counter persists across FirewallState switches — cannot be reset by state change
Both FirewallBucketSize AND FirewallRefillAmount must be configured for rate limiting to activate on a rule. If only one is set, rate limiting does not apply.
Rate limit SEv context data: FirewallRuleId (uint16) + MAC_Address (uint8[6]) — includes MAC so IdsM can identify attacker.

State-dependent filtering

  • FirewallRules grouped into FirewallStateDependentRules
  • Only ONE FirewallStateDependentRules active at a time
  • Only rules in active group are evaluated — other rules completely ignored
  • Each FirewallStateDependentRules has its own FirewallDefaultAction
  • Initial state: set by FirewallInitialStateRef — active before first Fw_SetFirewallState() call
  • BswM switches state by calling Fw_SetFirewallState(uint16)
Exam: "Which rules are active if Fw_SetFirewallState not yet called?" → Rules in FirewallStateDependentRules where referenced FirewallState = FirewallInitialStateRef.

SEv protocol identification algorithm (exam critical)

When no rule matches and default action = block, Firewall must identify WHICH protocol layer caused the mismatch to choose the right SEv:

1Iterate all FirewallRules, count how many consecutive OSI layers (starting from L2) match the packet
2Find rule(s) with most consecutive matching layers from lowest layer upward
3The FIRST layer that causes a non-match in those best-fitting rules = the protocol for the SEv
Example: Rules 1&2 match L2+L3+L4 but fail at SOME/IP → raise SEV_FW_PACKET_BLOCKED_SOMEIP_MISMATCH. Rule 3 matches only L3 → worse fit, ignored. This algorithm finds the "deepest" violation point.

If mismatch only in payload (generic byte-pattern) → SEV_FW_PACKET_BLOCKED_GENERIC_MISMATCH (no protocol header to blame).

SEv reporting requires FirwallEnableSecurityEventReporting = true in config AND IdsM integration. If not enabled → no SEv raised even if packet blocked.

Error handling conditions

  • DET reporting enabled by FirewallDevErrorDetect = TRUE
  • Errors reported via Det_ReportError()
  • FW_E_PARAM_UNINIT: all APIs must check init state first (except Fw_Init, Fw_GetVersionInfo)
  • FW_E_ALREADY_INITIALIZED: prevents double-init corruption
  • FW_E_SWITCHRULEMGMT_FAILED: raised only in switch deployment scenario when EthIf_SetStreamState fails or times out
Config variant: Supports VARIANT-PRE-COMPILE, VARIANT-LINK-TIME, and VARIANT-POST-BUILD — more flexible than EA which only supports VARIANT-PRE-COMPILE.

Smart switch deployment — why it exists

AUTOSAR stack deployed on a smart switch (CPU-equipped switch). Switch core has TCAM hardware — can do stateless filtering at wire speed. But TCAM can't do stateful or DPI. Firewall module on switch CPU handles the parts TCAM can't.

Switch core (TCAM) handles

  • Stateless L2-L4 filtering at hardware speed
  • Pre-filtering — reduces load on switch CPU
  • Rule identified by EthSwtStreamIdentification

Firewall (switch CPU) handles

  • Stateful TCP inspection (connection tracking)
  • Deep packet inspection (SOME/IP, DDS, DoIP)
  • Security event generation
  • Rate limiting (leaky bucket)

Three switch deployment scenarios

Scenario 1: Packet passes switch core, needs FW inspection

1Packet arrives → switch core partially inspects → passes (doesn't block)
2Switch core inserts FIREWALL_RULE_ID_16 into PDU metadata (identifies which TCAM rule passed it)
3Packet forwarded to switch CPU → LSduR → Fw_RxIndication with FIREWALL_RULE_ID_16 in metadata
4Firewall only checks rules COMPATIBLE with that FIREWALL_RULE_ID_16 → reduced rule set → faster
5Firewall performs stateful/DPI inspection on reduced rule set → allow/block → SEv if needed

Scenario 2: Packet blocked by switch core → SEv needed

1Packet blocked by switch core default rule → FIREWALL_RULE_ID_16 = default rule ID
2Packet forwarded to switch CPU for SEv identification
3Fw_RxIndication called → Firewall inspects to find which protocol caused mismatch → raises appropriate SEv
4Firewall blocks packet (already decided by switch core) → calls LSduR_FwRxReleaseBuffer

Scenario 3: Switch counter statistics polling

1Fw_MainFunction runs every FwSwitchBucketCounterPollingInterval seconds
2Calls EthIf_GetStreamStatistics() for all configured SwitchIdxs
3EthIf → EthSwtDrv retrieves bucket counters → calls Fw_StreamStatisticsIndication()
4Firewall compares new vs cached counters. Any increase → raises SEV_FW_PACKET_BLOCKED_BY_PERSTREAMFILTERING
5One SEv per increased bucket counter (not per individual packet)

Scenario 4: FirewallState change → sync switch rules

1BswM calls Fw_SetFirewallState()
2Firewall calls EthIf_SetStreamState() for each affected StreamHandleIdx to enable/disable TCAM rules
3EthIf confirms via Fw_StreamStateIndication(). Firewall checks StreamActivityStatus matches expected.
4If timeout (FirewallSwitchRuleMgmtTimeout) or mismatch → retry. Persistent failure → FW_E_SWITCHRULEMGMT_FAILED

Vector DaVinci Configurator — Firewall step by step

Step 1: Add Firewall module, set config variants

Firewall supports VARIANT-PRE-COMPILE, VARIANT-LINK-TIME, VARIANT-POST-BUILD. Choose based on project needs. Post-build allows rule updates without recompile.

Step 2: Configure FirewallGeneral container

FirewallGeneral parameters
FirewallDevErrorDetectboolTRUE during integration/test. FALSE production. Enables FW_E_PARAM_UNINIT, FW_E_ALREADY_INITIALIZED, FW_E_SWITCHRULEMGMT_FAILED reporting to DET.
FirewallVersionInfoApiboolTRUE to enable Fw_GetVersionInfo(). Usually FALSE in production.
FirewallMainFunctionPeriodfloatSeconds. Default 0.05 (50ms). Match to OS task period. Controls rate limit decrement cadence and switch polling interval.
FirwallEnableSecurityEventReportingboolTRUE to enable SEv reporting to IdsM. MUST be TRUE for IDS integration. Typo in spec: "Firwall" not "Firewall" — watch for this in DaVinci.
FirewallSwitchRuleMgmtTimeoutfloatSeconds. Only needed for switch deployment. Timeout for EthIf_SetStreamState confirmation. 0..INF, optional (0..1 multiplicity).

Step 3: Define FirewallState objects

Create one FirewallState for each vehicle state your project needs (e.g., DRIVING, PARKING, DIAGNOSTIC). Each gets a uint16 ID. Set one as FirewallInitialStateRef.

Step 4: Define FirewallStateDependentRules (one per state)

Each FirewallStateDependentRules: references a FirewallState, has its own FirewallDefaultAction (block or allow), and aggregates the FirewallRules active in that state.

Step 5: Configure FirewallRules (most work)

FirewallRule components
FirewallActionForMatchingRulesrefReferences FirewallActionForMatchingRules container. Set FirewallAction = allow or block. One per rule.
FirewallDataLinkFilterConfigoptL2: MAC src/dest (with optional mask), EtherType, VLAN tag. Leave unconfigured fields = don't care (wildcard).
FirewallNetworkLayerFilterConfigoptIPv4 or IPv6: src/dest IP (with mask), TTL range, protocol number, DSCP, checksum verification flag.
FirewallTransportLayerFilterConfigoptTCP or UDP: src/dest port ranges. TCP additionally: flags, stateful params (timeout, max connections, TCP state machine check).
FirewallSomeipProtocolFilterConfigoptSOME/IP: ServiceId, MethodId, MessageType, ReturnCode, ProtocolVersion, InterfaceVersion, length verification flag.
FirewallSomeipSdFilterConfigoptSOME/IP-SD: entry type, service ID, instance ID, major/minor version ranges.
FirewallDdsFilterConfigoptDDS-RTPS: GUID prefix, submessage kind, domain ID.
FirewallDoipFilterConfigoptDoIP: payload type, source/dest address ranges, UDS service IDs.
FirewallPayloadBytePatternFilterConfigoptGeneric: list of (offset, expected_byte_value) pairs. Multiple per rule, ALL must match. For custom protocols or UDS DIDs.
FirewallBucketSize / FirewallRefillAmountuintRate limiting. Both must be configured for rate limiting to activate. BucketSize = max matches before block. RefillAmount = decrement per MainFunction cycle.

Step 6: Configure PDU routing (FirewallPduRouting)

Maps LSduR PDUs to Firewall inspection paths. Each FirewallPduRoutingItem maps an incoming RxPduId to which FirewallRules apply. Pdu.KeepLocalBuffer: if TRUE, Firewall calls LSduR_FwRxReleaseBuffer on block.

Step 7: Configure BswM connection

In BswM config: add mode request for Firewall state. BswM action = call Fw_SetFirewallState(). Define mode conditions for each vehicle state that triggers a Firewall state change.

Step 8: Configure IdsM connection (if SEv enabled)

FirewallSecurityEventRefs container: reference each SEv defined in IdsM config. Map Firewall SEv IDs (51-64, 77, 83) to IdsM security event definitions. FirwallEnableSecurityEventReporting must be TRUE.

Step 9: Switch deployment (optional)

FirewallSwitchBucketCounterStatistics parameters
FwSwitchBucketCounterPollingIntervalfloatHow often Fw_MainFunction calls EthIf_GetStreamStatistics. In seconds.
FirewallSwitchIdxuintWhich switch to poll (EthSwt driver index)
FirewallWithPerStreamFilteringboolEnable FIREWALL_RULE_ID_16 metadata usage for reduced rule set inspection.

Step 10: OS/Scheduler setup

  • Add Fw_MainFunction to a periodic BSW task
  • Period must match FirewallMainFunctionPeriod
  • Task must run frequently enough for rate limiting accuracy
  • Fw_RxIndication is synchronous — called in LSduR task context
Common DaVinci mistakes: FirwallEnableSecurityEventReporting (note the typo — "Firwall") not enabled so no SEv raised. FirewallDefaultAction wrong (allow instead of block = blocklist instead of allowlist). Missing FirewallInitialStateRef. Rate limiting only configured BucketSize but not RefillAmount. FirewallMainFunctionPeriod not matching OS task.

Integration testing checklist

Init & state verification

  • Call Fw_RxIndication before Fw_Init → FW_E_PARAM_UNINIT in DET
  • Call Fw_Init twice → FW_E_ALREADY_INITIALIZED
  • Verify Fw_SetFirewallState(X) activates correct rule set
  • Verify packets allowed by state A are blocked in state B

Stateless inspection tests

  • Send exact-match packet → verify allowed (LsduR_FwRxIndication called)
  • Send packet with wrong src IP → verify blocked
  • Send packet in IP subnet range → verify match
  • Send packet with port outside configured range → blocked
  • Invalid checksum with FirewallChecksumVerification=TRUE → blocked

Stateful TCP tests

  • Open max TCP sessions → try new SYN → blocked + SEV_TCP_MAXCONNECTIONS
  • Send TCP packet after timeout → SEV_TCP_TIMEOUT
  • Send invalid TCP flag sequence → SEV_TCP_STATETRANSITION
  • Verify TCP state tracked per peer (not global)

Rate limiting tests

  • Send packets at configured rate → all pass
  • Burst packets exceeding BucketSize → overflow blocked + SEV_RATELIMIT
  • Wait for bucket refill (FirewallRefillAmount) → packets pass again
  • Change FirewallState → verify counter NOT reset

SOME/IP DPI tests

  • Send SOME/IP packet with allowed ServiceId → pass
  • Send SOME/IP packet with unlisted ServiceId → blocked + SEV_SOMEIP_MISMATCH
  • Send packet with 2 SOME/IP messages, one valid one invalid → entire packet blocked
  • FirewallSomeipLengthVerification=TRUE: send with wrong length → blocked

SEv verification tests

  • Send packet that fails at IPv4 layer → verify SEV_IPV4_MISMATCH raised with correct FirewallRuleId
  • Verify context data matches actual packet headers (byte-by-byte check)
  • Disable FirwallEnableSecurityEventReporting → verify no SEv raised
  • Verify SEv ID numbers match spec (51-64, 77, 83)

Exam traps — memorize these

  • INGRESS ONLY — egress traffic NOT filtered
  • First match wins — rules evaluated in ORDER, stop at first match
  • A rule is a match ONLY if ALL configured layer filters match
  • Whole packet blocked or allowed — can't block one SOME/IP message within a frame
  • SEv raised only on no-match+default-block case (not on explicit block rules yet)
  • Leaky bucket counter NOT reset on FirewallState change
  • FirewallInitialStateRef: rules active before first Fw_SetFirewallState call
