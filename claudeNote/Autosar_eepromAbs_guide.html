What is EA? — "The EEPROM translator"

EA (EEPROM Abstraction) sits between the NVRAM Manager (NvM) above and the Memory Access module (MemAcc) below. Its job: translate NvM's abstract block-based requests into real physical EEPROM addresses, hide device-specific quirks, and provide virtually unlimited erase cycles by spreading write wear across multiple physical locations.

Mental model: Think of EA as a librarian. NvM says "give me book #5." EA knows where book #5 is physically stored (which shelf, which row), fetches it, and if the shelf is wearing out, quietly moves the book to a fresher shelf — NvM never has to know.

Core Responsibilities

Virtual AddressingProvides 32-bit virtual address space to NvM: 16-bit block number + 16-bit block offset. Hides physical EEPROM layout completely.
Wear LevelingIf block requires more write cycles than EEPROM supports, EA spreads access across multiple physical areas. Configured via EaNumberOfWriteCycles vs EepAllowedWriteCycles.
Block ConsistencyEA marks each block "inconsistent" at write start, "consistent" at write end. This survives power-loss detection. Distinguishes inconsistent (corrupted) from invalid (deliberately erased).
Immediate DataHigh-priority blocks (EaImmediateData=TRUE) can be written without pre-erasing. Pre-erase area with Ea_EraseImmediateBlock() beforehand. No delay from ongoing internal ops.
Async Job ExecutionAll operations (Read/Write/Invalidate/Erase) are asynchronous — accepted synchronously, executed in Ea_MainFunction(). NvM polls or gets callback when done.
Single Job QueueEA handles only ONE job at a time. No internal queue. NvM is responsible for serializing requests. [SWS_Ea_00137]

Key Constraints (Exam favourites)

1. One job at a time — EA has no job queue. NvM serializes. [SWS_Ea_00137]

2. ConfigPtr always NULL_PTR — Ea_Init(NULL_PTR). Config is pre-compile only. [SWS_Ea_00191]

3. Block 0x0000 and 0xFFFF reserved — shall not be configured. [SWS_Ea_00006]

4. Write operates on WHOLE block — Ea_Write() always writes the full configured block size. No partial writes. [SWS_Ea_00151]

5. Read allows partial access — Ea_Read() allows any offset and length within block boundaries. [SWS_Ea_00021]

6. No job result from internal ops — only NvM-requested jobs affect Ea_GetJobResult(). Internal management ops (garbage collection) don't change it. [SWS_Ea_00174]

7. Ea_MainFunction cycle = EEPROM driver cycle — should be same timing for synchronisation. [SWS_Ea_00096 Note]

8. Post-build NOT supported — only VARIANT-PRE-COMPILE. [ECUC_Ea_00133]

9. Ea_Cancel is asynchronous w.r.t. HW — it calls MemAcc_Cancel immediately (sync to software), but hardware operation in progress cannot be stopped mid-way. [Sequence 9.3]

Addressing Scheme — How EA maps virtual to physical

Virtual 32-bit address structure

Bits 31:16
Block Number (16-bit)
0x0001 to 0xFFFE
+
Bits 15:0
Block Offset (16-bit)
0x0000 to BlockSize-1

The block number acts as a page selector. Each block occupies an integer multiple of EaVirtualPageSize virtual pages. [SWS_Ea_00005]

Virtual Page Size — The alignment unit

EaVirtualPageSize must be an integer multiple of the physical EEPROM page size. [SWS_Ea_00075]

Each configured block occupies an integer number of virtual pages — block size rounded UP to next virtual page boundary. [SWS_Ea_00005]

Block number calculation example (from spec):
Virtual page size = 8 bytes.
Block #1: 32 bytes → uses 4 virtual pages (block numbers 1, 2, 3, 4 consumed).
Block #2 (next available) = block number 5. Size = 100 bytes → 13 pages (block numbers 5–17 consumed).
Block #3 = block number 17. [Figure 7.1 in spec]
Critical exam point: Block numbers 0x0000 and 0xFFFF are reserved and must never be configured. [SWS_Ea_00006]

Dataset/Redundant Copy addressing

NvM uses LSBs of block number to encode dataset index (configured via NVM_DATASET_SELECTION_BITS). EA must strip these LSBs when calculating physical start address — only MSBs represent the actual block base. [SWS_Ea_00066]

Example: NVM_DATASET_SELECTION_BITS=4 → EA uses bits [15:4] for base address calculation, bits [3:0] = dataset index. Dataset 0 at base, dataset 1 at base + block_size, etc.

Block overlap rule

Logical blocks must NOT overlap and must NOT be contained within one another. [SWS_Ea_00068]

Config error: If two blocks share virtual address space (e.g., block #1 size=32 bytes with VPS=8 means block numbers 1-4 are occupied, so block #2 must be ≥5, not 2, 3, or 4). Tool should catch this, but verify manually.

Wear Leveling mechanism [SWS_Ea_00080]

If EaNumberOfWriteCycles > EepAllowedWriteCycles (from EEPROM driver published info): EA must spread write access across N physical areas where N = ceil(EaNumberOfWriteCycles / EepAllowedWriteCycles).

Example: Block configured for 500,000 writes. EEPROM rated for 100,000 cycles. EA needs 5 physical areas and rotates through them, so each physical cell is written max 100,000 times.

This is implementation-specific — the spec defines the requirement but not the exact algorithm. Vector's implementation details are in the MCAL documentation.

Buffer Alignment (R21-11)

EaBufferAlignmentValue: Reference to MemAccAddressAreaConfiguration. EA aligns internal buffers to this value. [SWS_Ea_00197]

EaMinimumReadPageSize: EA aligns read requests to this page size. Relevant for EEPROM HW that has minimum read granularity. [SWS_Ea_00198]

If caller's buffer is not aligned → EA may use an internal aligned buffer and copy. This adds latency. Ensure NvM buffer alignment matches EaBufferAlignmentValue.

Init sequence state flow

1. Power-on: Status = MEMIF_UNINIT. All APIs except GetVersionInfo rejected.
2. EcuM calls Ea_Init(NULL_PTR): Status → MEMIF_BUSY_INTERNAL [SWS_Ea_00017]
3a. Init completes inside Ea_Init(): Status → MEMIF_IDLE immediately [SWS_Ea_00128]
3b. Init completes in Ea_MainFunction(): Ea_MainFunction sets MEMIF_BUSY_INTERNAL → MEMIF_IDLE [SWS_Ea_00178]
Note: Do NOT call Ea_Init() while a job is running. Env must ensure no operation is ongoing. [SWS_Ea_00084 Note]

Block Consistency Management [SWS_Ea_00046, 00047]

EA maintains consistency information per block in NV memory (must survive reset):

Write started → EA marks block as INCONSISTENT
Write completed successfully → EA marks block as CONSISTENT
If power fails mid-write → block stays INCONSISTENT → next read returns MEMIF_BLOCK_INCONSISTENT
Ea_InvalidateBlock() called → EA marks block as INVALID (distinct from inconsistent)
How exactly consistency is stored (special page marker, separate table, etc.) is implementation-specific. EA must distinguish all three states: consistent, inconsistent, and invalid.

Sequence Diagrams — Step by Step

Sequence 1: Ea_Init
EcuM → Ea_Init(NULL_PTR)
EA: Status → MEMIF_BUSY_INTERNAL
EA performs internal initialization (scan EEPROM, rebuild block table, check consistency)
Case A: Init completes within Ea_Init() call:
Status → MEMIF_IDLE. Returns. [SWS_Ea_00128]
Case B: Init requires multiple MainFunction cycles (deferred):
Ea_Init() returns immediately with Status=MEMIF_BUSY_INTERNAL
Subsequent Ea_MainFunction() calls continue initialization
When done: Ea_MainFunction() sets Status → MEMIF_IDLE [SWS_Ea_00178]
NvM must NOT call Ea_Read/Write until Ea_GetStatus() returns MEMIF_IDLE. Calling before init is complete will get EA_E_UNINIT (if UNINIT) or be accepted as immediate data (if BUSY_INTERNAL).
Sequence 2: Ea_Write (full async flow through MemAcc)
MemIf → Ea_Write(BlockNumber, DataBufferPtr) [NvM request via MemIf]
EA: validate params → Status=IDLE or BUSY_INTERNAL → accept
EA: copy params to internal vars. Status → MEMIF_BUSY. JobResult → MEMIF_JOB_PENDING. Return E_OK.
Ea_MainFunction() cycle 1:
EA: calculate physical address from BlockNumber. Mark block INCONSISTENT.
EA → MemAcc_Write(AddressAreaId, TargetAddress, DataPtr, Length)
MemAcc returns E_OK (accepted). Job queued to MemAcc.
MemAcc_MainFunction() loop:
MemAcc → Mem_Write() → EEPROM HW write starts
MemAcc_MainFunction() polls Mem_GetJobResult() each cycle
When Mem job done → MemAcc_MainFunction() calls Ea_JobEndNotification()
Ea_JobEndNotification() [can be interrupt or task context]:
EA: mark block CONSISTENT. JobResult → MEMIF_JOB_OK. Status → MEMIF_IDLE.
EA → NvM_JobEndNotification()
NvM processes completion and queues next job.
Sequence 3: Ea_Cancel (with async hardware behavior)
Ea_Write() running (Status=MEMIF_BUSY). MemAcc_Write started, HW writing a page.
MemIf_Cancel() → Ea_Cancel()
EA: Status is MEMIF_BUSY → accept cancel.
EA → MemAcc_Cancel(AddressAreaId) [immediately, synchronous call]
MemAcc marks job as canceled. No more new write commands to HW.
EA: JobResult → MEMIF_JOB_CANCELED. Status → MEMIF_IDLE. [SWS_Ea_00078]
⚠ HW is still finishing current page write (hardware cannot abort). This is normal.
MemAcc_MainFunction() continues until HW idle:
Ea_JobEndNotification() may fire once HW finishes — EA handles gracefully (ignores or processes cleanup).
New job can now be issued:
Status=IDLE → NvM can call Ea_Write/Read for immediate data.
The cancel sequence shows TWO asynchronous boundaries: (1) Ea_Cancel is synchronous to software but asynchronous to hardware. (2) After cancel, a new job can start even before the hardware finishes the old page — MemAcc handles the hardware serialization.
Sequence 4: Ea_Read with BLOCK_INCONSISTENT result
NvM → Ea_Read(BlockNumber, 0, Buffer, FullLength)
EA: accept → Status=BUSY, JobResult=PENDING.
Ea_MainFunction():
EA → MemAcc_Read() to read block data + management info
After data read: EA checks consistency marker
Consistency marker shows "INCONSISTENT" (write was interrupted)
EA: JobResult → MEMIF_BLOCK_INCONSISTENT. Status → IDLE. [SWS_Ea_00104]
EA → NvM_JobErrorNotification()
⚠ NvM must NOT use DataBuffer contents — they may be corrupt.
NvM handling: NvM knows this block is inconsistent. It uses ROM default values instead (if configured) or reports error to application.
Sequence 5: Immediate Data Write (high-priority path)
Lower priority write in progress (Status=BUSY).
NvM detects immediate data must be written urgently:
NvM → Ea_Cancel() → Status=IDLE, JobResult=CANCELED
NvM → Ea_EraseImmediateBlock(ImmediateBlockNumber) [if not pre-erased]
Wait for NvM_JobEndNotification() — erase complete.
NvM → Ea_Write(ImmediateBlockNumber, ImmediateDataPtr)
EA: immediate block area is pre-erased → write proceeds without erase delay
✓ Immediate write completes quickly (only hardware page-write time, no erase overhead)
Note: HW may still be finishing old page write from canceled job — this is the minimum unavoidable delay. [SWS_Ea_00025 + Immediate data section 7.1.4]

Error Handling & Debugging Guide

Problem: Ea_Write() returns E_NOT_OK unexpectedly
Check: what is Ea_GetStatus() returning?
If MEMIF_UNINIT: Ea_Init() was not called, or init is still in progress. Poll Ea_GetStatus() before first job.
If MEMIF_BUSY: Another job is in progress. NvM serialization bug — NvM should never request a new job before previous one completes. Check NvM_JobEndNotification is being called.
Check DET: EA_E_UNINIT (0x01), EA_E_BUSY (0x06), EA_E_INVALID_BLOCK_NO (0x02), EA_E_PARAM_POINTER (0x04) — DET will tell you exactly why.
Check: is block number valid? Block numbers 0x0000 and 0xFFFF are forbidden. Block number must be configured in EaBlockConfiguration.
Problem: Ea_GetJobResult() returns MEMIF_BLOCK_INCONSISTENT after read
Block was written but power failed mid-write. This is expected automotive behavior — EA correctly detected it.
NvM will use ROM default values for the block (if NvMInitBlockCallback configured) or report NVM_REQ_INTEGRITY_FAILED to application.
NOT a bug unless it happens after every successful write. If consistent after first write and inconsistent after a reset, look at power failure patterns.
Also returned if "block cannot be found" — i.e., block number was changed/added to config but EEPROM still has old layout. Re-format/reinitialize EEPROM memory.
Debug: Read raw EEPROM memory to see the consistency marker byte. Compare to EA implementation's marker definition.
Problem: Ea_GetJobResult() returns MEMIF_JOB_FAILED
The MemAcc/hardware operation failed (write/erase error). Root cause is typically hardware.
Check if EEPROM device is responsive (power supply, SPI/I2C connection, CS line).
Check EaNumberOfWriteCycles vs EepAllowedWriteCycles — if write count exceeded, EA marks job as failed.
Check if Ea_GetJobResult() called while MEMIF_UNINIT — returns MEMIF_JOB_FAILED by design. [SWS_Ea_00134]
Check MemAcc_GetJobResult() return value at the MemAcc level for more detail.
Problem: NvM_JobEndNotification() never called (EA job hangs)
EA job accepted but completion callback never fires.
Check Ea_MainFunction() is being called by SchM at correct period. If main function not scheduled → job never processes.
Check MemAcc_MainFunction() is also being called — MemAcc drives the HW and must call Ea_JobEndNotification.
Check EaPollingMode config: if TRUE, EA polls Eep_GetStatus() instead of waiting for callback — ensure callback routing is correct for chosen mode.
Check EaNvmJobEndNotification is correctly configured to NvM_JobEndNotification (or your project's function name). Misconfigured = notification goes to wrong function or NULL.
Watch Ea_GetStatus() — if stuck at MEMIF_BUSY, job started but notification never came back.
Problem: EA_E_INVALID_CANCEL in DET repeatedly
Ea_Cancel() being called when no job is in progress (Status ≠ MEMIF_BUSY). [SWS_Ea_00173]
NvM is calling MemIf_Cancel() unnecessarily. Check NvM configuration: NvMDynamicConfiguration, NvMWriteVerification, NvMCrcNumOfBytes — these can trigger extra cancel calls.
Verify sequence: NvM should only cancel if it knows a job is pending (it accepted a job and hasn't received completion yet).

Block number calculation (exam-critical)

Step 1: Determine NVM_DATASET_SELECTION_BITS (from NvM config). E.g., 4 bits = 16 datasets max.
Step 2: First valid block number = 2^NVM_DATASET_SELECTION_BITS. E.g., = 16 (0x0010).
Step 3: Each block occupies ceil(EaBlockSize / EaVirtualPageSize) virtual pages.
Step 4: Next block starts at (previous_block_number + pages_occupied). All intermediate numbers are consumed.
Example: VPS=8, datasets=4 bits. Block A: number=16, size=32 bytes → uses 4 pages → numbers 16,17,18,19 consumed. Block B: number=20, size=100 bytes → uses ceil(100/8)=13 pages → numbers 20-32 consumed. Block C: number=33.

Step 3: EaPublishedInformation — Set by EA implementor

Path: Ea → EaPublishedInformation

ParameterWhat it isImpact
EaBlockOverheadManagement bytes added per logical blockUsed by configuration tool to calculate total EEPROM space needed. Include when calculating EaBlockSize.

These are published by the EA vendor/implementor. They determine how much physical EEPROM is consumed per block. If not set correctly, you may run out of EEPROM space despite configuration looking correct on paper.

☐ EaVirtualPageSize is integer multiple of physical EEPROM page size
☐ ALL EaBlockSize values are integer multiples of EaVirtualPageSize
☐ No two blocks overlap — check consumed page ranges manually
☐ EaBlockNumber 0x0000 and 0xFFFF not used
☐ EaNvmJobEndNotification and EaNvmJobErrorNotification set (not empty)
☐ EaMemAccAddressArea (not EaDeviceIndex) used for R21-11+
☐ EaMainFunctionPeriod matches SchM/OS task period
☐ Total physical EEPROM space = sum of (ceil(BlockSize/VPS) × VPS + BlockOverhead × numBlocks + PageOverhead × numPages)

Automotive Integration Checklist

NvM–EA Integration

  • EaBlockNumber must match NvMBlockDescriptor's NvMNvramBlockIdentifier (with dataset bits stripped)
  • EaBlockSize must match NvMNvBlockLength × datasets + any redundant copy overhead
  • EaNumberOfWriteCycles must be ≥ NvMMaxBlocksForWriteAll × vehicle_lifetime_cycles
  • Total EEPROM consumption = sum of all (ceil(EaBlockSize/EaVPS) × EaVPS + overhead)
  • Verify NvM_JobEndNotification function pointer is correctly set in EaNvmJobEndNotification

Debugging Tools

  • TRACE32: Break on Ea_JobErrorNotification → inspect Ea_JobResult variable
  • TRACE32: Watch Ea_ModuleStatus — stuck at BUSY = notification lost
  • CANoe/DaVinci Monitor: Live view of NvM block states
  • Read raw EEPROM via CAN diagnostics (0x23 ReadMemoryByAddress) to inspect consistency markers
  • Check that Ea_MainFunction is scheduled in OS and period matches EaMainFunctionPeriod

Common Integration Mistakes

1. EaBlockSize not multiple of EaVirtualPageSize — EA may not detect this at runtime (DET won't fire). Physical layout becomes corrupted. Always verify: EaBlockSize % EaVirtualPageSize == 0.
2. EaNvmJobEndNotification not configured — NvM writes but never gets acknowledgment. NvM state machine stalls forever. Job hangs.
3. Using EaDeviceIndex (obsolete) instead of EaMemAccAddressArea — R21-11 moved to MemAcc. Old EaDeviceIndex references EepGeneral which no longer exists in the interface.
4. EaBlockNumber overlaps — block numbers 1-4 consumed by 32-byte block (VPS=8), then setting next block to number 2. Data corruption — blocks write to same physical location.
5. DataBuffer freed before NvM_JobEndNotification — Ea_Write uses DataBufferPtr throughout async job. If NvM frees/reuses buffer before completion, corruption. Buffer must remain valid until callback.
