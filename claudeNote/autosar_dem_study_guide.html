What is DEM?

Diagnostic Event Manager manages ALL diagnostic faults in the ECU. Receives monitor results from SW-Cs (via RTE) and BSW modules (via C-API). Stores DTCs and freeze frames in event memory. Provides DTC status to Dcm (UDS tester). Coordinates aging, healing, displacement, and NvM storage.

Key DEM terminology

TermMeaning
EventIdUnique numeric ID for each diagnostic event. Symbolic name in config, number at runtime.
DTCDiagnostic Trouble Code — the fault code shown to the tester. One DTC can map to multiple events (combination).
MonitorSW-C or BSW routine that checks hardware/software functionality and reports FAILED/PASSED to Dem.
FDCFault Detection Counter. -128 (fully passed) to +127 (fully failed). Debouncing moves this value.
UDS status byte8-bit ISO14229 status byte per event — what the tester reads via $19 service.
Monitor statusInternal Dem status byte (2 bits relevant: TestFailed, TestNotCompleteThisOpCycle). Updated synchronously.
Event memoryRAM+NvM storage of confirmed/pending DTC entries including freeze frames and extended data.
AgingAutomatic removal of DTC from event memory after N passed operation cycles (no fault recurring).
HealingDeactivation of warning indicator (MIL) after N passed operation cycles.
DisplacementReplacement of lowest-priority DTC from full event memory by a higher-priority new entry.

DTC priority rule (exam trap!)

Priority 1 = HIGHEST. Larger number = LOWER priority. Used in displacement: lower priority DTC displaced by higher. If priorities equal: untested-in-cycle DTC displaced first. If still equal: oldest (least recent) displaced.

Component dependency (DemComponent)

Events assigned to DemComponents. If a higher-priority event on same component is FAILED → lower-priority events report is treated as CONSECUTIVE FAULT (not stored). If parent DemComponent is FAILED → child events also considered CONSECUTIVE. Causal fault delay (DemCausalityDelayTime) allows time window to reconsider.

Processing model: Monitor status updated SYNCHRONOUSLY in Dem_SetEventStatus context. UDS status updated ASYNCHRONOUSLY in Dem_MainFunction. Both are processed in sequence — monitor status first, UDS status queued for main function.

Debouncing Algorithms

Debounce = fault maturation. Prevents single glitch from causing DTC storage. FDC range: -128 (fully passed) to +127 (fully failed). Dem updates FDC, then qualifies FAILED or PASSED.
Counter-Based

Uses internal sint16 counter. Reports PREFAILED/PREPASSED to increment/decrement by step size. Instant thresholds on FAILED/PASSED reports.

Time-Based

Uses internal timer. PREFAILED starts timer. Timer reaches threshold → qualified. FAILED/PASSED instantly set timer to threshold. Timer can be frozen or reset.

Monitor-Internal

Monitor handles debouncing itself. Only reports FAILED/PASSED to Dem. PREFAILED/PREPASSED NOT allowed (DET error if sent). Dem retrieves FDC via callback.

Counter-Based in detail

Monitor reportsDem action on internal counter
DEM_EVENT_STATUS_PREFAILEDIncrement by DemDebounceCounterIncrementStepSize. Cap at FailedThreshold.
DEM_EVENT_STATUS_PREPASSEDDecrement by DemDebounceCounterDecrementStepSize. Cap at PassedThreshold.
DEM_EVENT_STATUS_FAILEDSet counter = DemDebounceCounterFailedThreshold immediately → qualified FAILED
DEM_EVENT_STATUS_PASSEDSet counter = DemDebounceCounterPassedThreshold immediately → qualified PASSED
Jump behavior:
JumpUp When direction changes from decrement→increment AND counter < JumpUpValue → snap to JumpUpValue first, then increment. Prevents slow response when fault reappears.
JumpDown When direction changes from increment→decrement AND counter > JumpDownValue → snap to JumpDownValue first, then decrement. Fast recovery.

Debounce behavior when enable condition not fulfilled

DEM_DEBOUNCE_STATUS_FREEZE

Counter/timer frozen at current value. Resumes when conditions fulfilled. Does NOT reset. Use: temporary monitoring interruptions (e.g. sensor offline briefly).

DEM_DEBOUNCE_STATUS_RESET

Counter/timer reset to 0 when enable condition fails or ControlDTCSetting disabled. Must re-debounce from scratch. Use: conditions are prerequisites for valid monitoring.

Debounce reset conditions

Counter/timer reset on: Dem_PreInit() / operation cycle restart (Dem_RestartOperationCycle) / Dem_ClearDTC / Dem_ResetEventDebounceStatus(DEM_DEBOUNCE_STATUS_RESET) / enable condition → not fulfilled (if DEM_DEBOUNCE_RESET configured)

NvM persistence of counter: If DemDebounceCounterStorage=TRUE → counter saved non-volatile. Restored in Dem_Init. BSW events with NvM debounce cannot report before Dem_Init (counter not yet restored)!

FDC retrieval — Dem_GetFaultDetectionCounter

Counter-based/time-based: Dem reads internal value directly.
Monitor-internal: Dem calls DemCallbackGetFDC callback to SW-C/BSW. If no callback configured → returns DEM_E_NO_FDC_AVAILABLE.

FDC_THRESHOLD_REACHED: Special event status — monitor reports when FDC hits configured FdcThresholdStorageValue. Triggers pre-storage of freeze frame before full qualification. Used for early capture of volatile data.

Event displacement — memory full handling

When memory full and new event needs storage → displacement algorithm:

1
Priority: Lower priority DTC displaced by higher priority new event. (Priority 1 = best, larger number = worse)
2
Active/Passive: (DEM_DISPLACEMENT_FULL only) Passive events (TF=0) displaced before active events (TF=1)
3
Occurrence/Age: Older stored events displaced before newer. If event re-fails → it moves to most recent position.

Displacement strategies (DemEventDisplacementStrategy)

DEM_DISPLACEMENT_NONE

No displacement. Memory full = new event discarded. Simple, but risks missing important new faults.

DEM_DISPLACEMENT_PRIO_OCC

Priority + occurrence only. No passive/active distinction. Simpler than FULL.

DEM_DISPLACEMENT_FULL

Full algorithm: Priority → Active/Passive → Occurrence. Most complete OBD-compliant strategy.

Post-displacement status reset (important for testing!)

Config paramWhen trueEffect
DemResetConfirmedBitOnOverflowTRUEBit3 (CDTC) reset to 0 when event displaced
DemResetPendingBitOnOverflowTRUEBit2 (PDTC) reset to 0 when event displaced

Memory overflow indication

Dem_GetEventMemoryOverflow() — returns if memory was full and a storage attempt was made. Reset only when ALL DTCs cleared via ClearDTC. NOT reset by aging or individual DTC delete. Used in workshop diagnostics.

Chronological reporting order

When GetNextFilteredDTC called with pending+confirmed bits: most recent event reported first. When event re-fails → it becomes most recent. New entry → also becomes most recent.

Storage condition vs enable condition: Enable condition blocks event report processing entirely (no debounce). Storage condition allows debounce but prevents memory entry creation and freeze frame storage.

Aging and Healing

Healing — deactivate warning indicator (MIL)

Event confirmed, WarningIndicator=1 (MIL on)
↓
Monitor reports PASSED in operation cycle (TFTOC=0)
Each completed cycle with TFTOC=0 and TNCTOC=0 → increment HealingCounter
↓
HealingCounter ≥ DemIndicatorHealingCycleCounterThreshold
↓
WarningIndicatorOffCriteriaFulfilled → Bit7 (WIR) = 0 → MIL off
SWS_Dem_00698: Aging counter starts ONLY AFTER healing complete!

Aging — remove event from memory

Healing complete (WIR=0), event passes
↓
Each completed aging cycle with event qualified PASSED → AgingCounter++
DemAgingRequiresTestedCycle=TRUE: only count if TNCTOC=0 (test ran in cycle)
DemAgingRequiresNotFailedCycle=TRUE: only count if no FAILED in that cycle
↓
AgingCounter ≥ DemAgingCycleCounterThreshold
↓
Event memory entry DELETED. CDTC (bit3) = 0. Freeze frames removed.

Aging configuration options

ConfigEffect
DemAgingCycleCounterThresholdN aging cycles needed to age out. Common values: 40 (EU OBD), varies by OEM.
DemAgingRequiresTestedCycle=TRUEAging cycle only counts if monitor ran (TNCTOC=0) during that cycle. Prevents aging on idle/untested cycles.
DemAgingRequiresNotFailedCycle=TRUEAging cycle not counted if event failed during that cycle. Strict requirement.
DemAgingCycleCounterThresholdForTFSLCSeparate threshold to also age-out bit5 (TFSLC). Optional.
DemAgingCycleRefWhich operation cycle drives aging counter. Default often = ignition cycle.
Aging can't start until healing complete! SWS_Dem_00698 explicit. Aging counter = 0 while WIR=1. Heal first, age second.

What gets reset on aging (SWS_Dem_00498, 01075)

Bit3 (CDTC) → reset to 0. Freeze frame data → deleted. Extended data records → deleted. Occurrence counter → stays (not reset). Bit5 (TFSLC) → reset ONLY if DemAgingCycleCounterThresholdForTFSLC configured and reached. Bit0 stays as-is. Other bits stay.

Reoccurrence after aging (SWS_Dem_00161)

If aged-out event occurs again → treated as a NEW event (fresh entry). AgingCounter reset to 0. Occurrence counter restarted from 1.

Enable Conditions & Storage Conditions

Enable ConditionsBlocks reports

Gate on Dem_SetEventStatus processing. If any assigned enable condition NOT fulfilled → event report IGNORED. Debounce counter frozen or reset (per DemDebounceBehavior). UDS status byte NOT changed.

Set via: Dem_SetEnableCondition(conditionId, TRUE/FALSE)

When re-enabled → callback InitMonitorForEvent fires with DEM_INIT_MONITOR_REENABLED.

Storage ConditionsBlocks storage

Gate on event memory storage. Debouncing proceeds normally. Event qualified FAILED/PASSED normally. But if storage condition NOT fulfilled: no memory entry created, no freeze frame stored, bits 2 (PDTC) and 3 (CDTC) NOT set from 0 to 1.

Set via: Dem_SetStorageCondition(conditionId, TRUE/FALSE)

Replacement event: if storage condition FALSE and original event FAILED → replacement event (configured via DemStorageConditionReplacementEventRef) gets set FAILED instead.

Enable condition group logic

Multiple conditions → grouped into DemEnableConditionGroup. Event references a group. All conditions in the group must be TRUE → group fulfilled. Event assigned to group → ALL conditions must be fulfilled for event to be processed.

ControlDTCSetting (UDS service $85)

Dcm calls Dem_DisableDTCSetting / Dem_EnableDTCSetting. Effect similar to enable condition:
• Dem_DisableDTCSetting → debounce frozen/reset (per DemDebounceBehavior) + InitMonitorForEvent NOT called
• Dem_EnableDTCSetting → debounce resumes + InitMonitorForEvent with DEM_INIT_MONITOR_REENABLED

ClearEventAllowed callback

Per-event callback. Called before clearing that event. If returns FALSE + DEM_NO_STATUS_BYTE_CHANGE → UDS byte unchanged, event not cleared. If returns FALSE + DEM_ONLY_THIS_CYCLE_AND_READINESS → bits 1, 4, 5, 6 reset but event not fully cleared. Designed for special modes (transport, assembly, flash mode).

Debug tip — common test: Disable enable condition → verify events not processed (FDC stays) → re-enable → verify InitMonitorForEvent callback fires → verify events process again. This tests the full enable condition loop.

Extended data records

Internal data elements (DemInternalDataElement)

Dem-managed counters automatically available per event:
DEM_OCCCTR — occurrence counter
DEM_FAILED_CYCLES — cycles where event failed
DEM_CYCLES_SINCE_LAST_FAILED — cycles since last failure
DEM_CYCLES_SINCE_FIRST_FAILED — cycles since first failure
DEM_MAX_FDC_DURING_CURRENT_CYCLE — peak FDC this cycle
DEM_MAX_FDC_SINCE_LAST_CLEAR — peak FDC ever
DEM_AGINGCTR_UPCNT — current aging counter value

External data elements

Collected from SW-C via ReadData operation or C-callback (DemCallbackGetFDC). Captured asynchronously — at latest in next Dem_MainFunction cycle.

If SW-C returns other than E_OK → Dem fills with 0xFF padding and reports runtime error DEM_E_NODATAAVAILABLE to DET.

Capture timing (critical!)

Freeze frames and extended data captured ASYNCHRONOUSLY — in the NEXT Dem_MainFunction cycle after the trigger. NOT in the context of Dem_SetEventStatus. This means there is a small window where data may not be from the exact fault moment.
Dem_SetEventStatusWithMonitorData() — extends Dem_SetEventStatus with monitorData0 and monitorData1 parameters. monitorData0 can be passed to ReadData operation to help align captured data with fault report context. Used for highly volatile data.

Operation Cycle Management

Common operation cycle types

Ignition Cycle

IGN ON/OFF. Most common basis for TFTOC, CDTC confirmation, aging, healing. Standard AUTOSAR cycle.

OBD Driving Cycle

Specific drive conditions met. Used for OBD readiness and MIL confirmation (DemOBDDrivingCycleRef). Qualification required (DemOBDDelayedDCYConfirmedAndMIL).

Warm-up Cycle

Engine reaches operating temp. Used by some OBD systems for catalyst, misfire monitors. OEM specific.

Operation cycle restart flow

BswM calls Dem_RestartOperationCycle(cycleId)
SWS_Dem_01318: allowed after Dem_PreInit. Before Dem_Init → queued.
↓
End of previous cycle processing:
• Increment Cycles-since-last-failed / Cycles-since-first-failed (if stored)
• Increment Failed-cycles counter (if TFTOC=1)
• Increment AgingCounter (if healing complete and pass conditions met)
• Update ConfirmedDTC (if failure counter ≥ threshold)
↓

Dependent (qualified) operation cycles

DemLeadingCycleRef → makes a cycle dependent on a "leading" cycle. Dependent cycle only restarts when leading cycle restarts AND dependent cycle is "qualified" (Dem_SetCycleQualified). Qualification state persisted over power cycles. OBD driving cycle qualification used to delay CDTC confirmation until valid drive conditions met.
