AUTOSAR CP Key Manager (KeyM) — Complete Study Guide

R24-11 · Document ID 907 · Classic Platform · For exam, integration, interface testing & DaVinci configuration

What is the AUTOSAR KeyM module?

The Key Manager (KeyM) is an AUTOSAR Classic Platform Basic Software module responsible for managing cryptographic key material and certificates on an ECU. It acts as a key client — receiving key material from a key master (either an in-vehicle ECU or a cloud backend) and provisioning it securely into the Crypto Service Manager (CSM) and optionally into NvM. It consists of two independent sub-modules.

Crypto key submodule

Introduces, updates, and maintains cryptographic key material. Receives key data from a key server (typically via diagnostic services), validates it, and stores it in CSM key slots. Used for SecOC session keys, SHE keys, derived keys, etc.

Enabled by: KeyMCryptoKeyManagerEnabled = TRUE

Certificate submodule

Manages X.509, CVC, and CRL certificates. Configures a PKI hierarchy (root → intermediate → working certificates), parses, validates, and stores certificates. Extracts public keys into CSM for use by crypto jobs.

Enabled by: KeyMCertificateManagerEnabled = TRUE

Why KeyM exists — the automotive use case

Modern automotive ECUs need cryptographic keys for SecOC (Secure On-Board Communication), TLS, diagnostic authentication, and OTA update signing. These keys cannot be hardcoded — they must be provisioned at the end of the production line or updated in the field. KeyM provides the standard AUTOSAR interface for this operation, abstracting away the OEM-specific key provisioning protocol through configurable key handlers (KeyM_KH_*).

💡 Key design principle: KeyM is a key client, not a key server. It does not generate keys independently — it receives them from an external key server (often via UDS diagnostic commands 0x27/0x2E or OEM-specific services) and provisions them into the CSM/crypto driver. The key server logic lives outside KeyM, optionally in a configurable Key Handler.

KeyM in the AUTOSAR architecture

Key architectural rules

Only one KeyElement per CsmKey

The spec mandates: only a single KeyElement with ID=1 per CsmKey is currently supported. This is a known limitation of R24-11.

Fully configurable scope

Either sub-module can be independently enabled/disabled. An ECU needing only certificate management can disable the crypto key sub-module entirely — saving RAM and code.

Custom handling via HSM

Certificate operations can be proxied to a Crypto Driver (HSM) via Csm_CustomService/Csm_CustomSync. KeyM then acts as a proxy with no local certificate processing.

RTE service interface

KeyM exposes all its functions both as direct C-API and as AUTOSAR service interfaces (via RTE) for access from application SWCs. Both interfaces have the same behaviour.

Crypto key submodule — session-based key update

The crypto key submodule uses a session-based protocol for key updates. A session is opened with KeyM_Start() and closed with KeyM_Finalize(). Between them, key material is loaded using KeyM_Update() and optionally prepared using KeyM_Prepare().

Key generation types — how keys are stored

KEYM_STORED_KEY

Raw key material is provided directly in the KeyM_Update() call. KeyM calls Csm_KeyElementSet() to push the data into the CSM key slot. The key element ID is always 1.

Typical use: SHE keys (AES-128), asymmetric public keys, pre-shared symmetric keys.

KeyM_Update("MyKey", ...) →
  Csm_KeyElementSet(CsmKeyRef, 1, data, len) →
  [if no session] Csm_KeySetValid(CsmKeyRef)

KEYM_DERIVE_KEY

A password/salt is provided; KeyM calls Csm_KeyDerive() to derive a new key from a source key. The source key is configured in KeyMCryptoKeyCsmKeySourceDeriveRef.

Typical use: Deriving session keys from a master key using KDF (HKDF, etc.)

KeyM_Update("DerivedKey", password_data) →
  Csm_KeyDerive(SourceKeyRef, TargetKeyRef) →
  [if no session] Csm_KeySetValid(TargetKeyRef)

SHE key identification (keyNameLength = 0)

When KeyNameLength = 0 and KeyMCryptoKeyCryptoProps is configured, KeyM interprets the data as SHE M1/M2/M3 values. It extracts bits 121-124 from the input data to identify the SHE key slot and triggers the key update operation. The result data contains M4/M5 values for proof-of-correctness.

💡 SHE (Secure Hardware Extension) is an automotive hardware security standard. M1/M2/M3 are inputs for key programming; M4/M5 are the MAC-based proof that the key was programmed correctly. This is the classic automotive HSM key provisioning protocol.

Verify function

KeyM_Verify() is independent of the session — it can be called at any time (not inside a Start/Finalize session). It runs a configured CSM job (referenced by KeyMCryptoKeyCsmVerifyJobRef) on the provided input data and returns the result — either synchronously or asynchronously. Used by the key server to confirm a key was programmed correctly.

Verification order — sequential steps (exam critical)

When verification is triggered, KeyM processes these checks in order. If any step fails, verification stops and the corresponding status is set:

1
Check all upper hierarchy certificates (KeyMCertUpperHierarchicalCertRef) are VALID. If NOT_PARSED or PARSED_NOT_VALIDATED, trigger their parse/verify first (top-down recursion).
2
Verify subject field of current certificate matches issuer field of the upper certificate (chain-of-trust linkage). Failure → INVALID_CHAIN_OF_TRUST
3
Verify the certificate signature using the CSM job referenced in KeyMCertCsmSignatureVerifyJobRef. For self-signed (root) certificates, the public key must be set first. Failure → SIGNATURE_FAIL
4
Check CRL if available — is the certificate revoked? Failure → REVOKED
5
Compare current time (from StbM) against certificate NotBefore and NotAfter fields. Failure → VALIDITY_PERIOD_FAIL
6
Check all configured certificate element rules (KeyMCertCertificateElementRuleRef). Failure → INVALID_CONTENT
7
All steps passed → Set public key in CSM (Csm_KeyElementSet + Csm_KeySetValid) → Status = KEYM_CERTIFICATE_VALID → Callback fired
⚠️ Exam trap: verification goes top-down (root first, then intermediates, then working cert). You cannot verify a certificate if its upper hierarchy is not yet VALID. The KeyM will recursively trigger verification of the entire chain before proceeding.

SetCertificate vs ServiceCertificate

KeyM_SetCertificate()

Temporary storage — typically in RAM. Used to provide a certificate for verification against the already-installed chain. The certificate is parsed automatically but not permanently stored. Call with length=0 to reset the slot back to NOT_AVAILABLE.

KeyM_SetCertificate(CertId, &certData)
/* Triggers parsing in background */
/* Then call KeyM_VerifyCertificate() */

KeyM_ServiceCertificate()

Permanent storage — in NvM or CSM key slot. Used for root, intermediate, or working certificates that survive ECU reset. Must be triggered by key master. Security events are reported to IdsM for root/intermediate cert changes.

KeyM_ServiceCertificate(
  KEYM_SERVICE_CERT_SET_ROOT,
  certName, nameLen,
  certData, certLen,
  resp, &respLen)

Callback notifications & key handlers

Callback notification functions (called by KeyM → application)

Since most KeyM operations are asynchronous, the application provides callback functions. These are called from within the background task or main function when the operation completes.

KeyM_CryptoKeyUpdateCallbackNotification(Result, ResultDataLength, ResultDataPtr)Called after KeyM_Update() completes asynchronously. Result indicates success or failure. ResultDataPtr points to the same buffer provided with the KeyM_Update call (e.g. M4/M5 for SHE).
KeyM_CryptoKeyFinalizeCallbackNotification(Result, ResultDataLength, ResultDataPtr)Called after KeyM_Finalize() completes. If at least one Csm_KeySetValid() failed, overall result is failure but all keys are still attempted. Only called if KeyMCryptoKeyStartFinalizeFunctionEnabled=TRUE.
KeyM_CryptoKeyVerifyCallbackNotification(Result, KeyId, ResultDataLength, ResultDataPtr)Called after KeyM_Verify() completes in async mode. Provides KeyId of the verified key and crypto operation result.
KeyM_ServiceCertificateCallbackNotification(CertId, Result)Called after KeyM_ServiceCertificate{ByCertId}() completes. Provides CertId and result. Function name is configured by KeyMServiceCertificateCallbackNotificationFunc.
KeyM_CertificateVerifyCallbackNotification(CertId, Result)Called after KeyM_VerifyCertificate(), VerifyCertificates(), or VerifyCertificateChain() completes. Result is of type KeyM_CertificateStatusType (the final status). Configured by KeyMCertificateVerifyCallbackNotificationFunc.

Key handler functions (called by KeyM → OEM handler)

Key handlers are OEM-specific callback functions that extend KeyM's behavior. They are enabled/disabled individually via configuration flags. All key handlers are called synchronously from within the KeyM background task.

KeyM_KH_Start(StartType, RequestData, RequestDataLength, ResponseData, ResponseDataLength)Called by KeyM_Start() if KeyMCryptoKeyHandlerStartFinalizeEnabled=TRUE. OEM can verify signatures, check authorization, or perform custom negotiation. Return E_OK to allow the session.
KeyM_KH_Prepare(RequestData, RequestDataLength, ResponseData, ResponseDataLength)Called by KeyM_Prepare() if KeyMCryptoKeyHandlerPrepareEnabled=TRUE. Used to exchange info with the key server (e.g. SHE challenge data, key exchange parameters).
KeyM_KH_Update(RequestData, RequestDataLength, KeyName, KeyNameLength, KeymId, ResultDataPtr, ResultDataMaxLength)Called by KeyM_Update() if KeyMCryptoKeyHandlerUpdateEnabled=TRUE. The handler can process OEM-specific key containers. Returns KEYM_KH_UPDATE_KEY_UPDATE_REPEAT to provide more data, or KEYM_KH_UPDATE_FINISH when done.
KeyM_KH_Finalize(RequestData, RequestDataLength, ResponseData, ResponseDataLength)Called by KeyM_Finalize() BEFORE Csm_KeySetValid(). If it returns E_NOT_OK, no key validation is performed. Used for final authorization checks.
KeyM_KH_Verify(KeyName, KeyNameLength, RequestData, RequestDataLength, ResponseData, ResponseDataLength)Called by KeyM_Verify() if KeyMCryptoKeyHandlerVerifyEnabled=TRUE. OEM handles the verify operation directly.
KeyM_KH_ServiceCertificate(Service, CertName, CertNameLength, RequestData, RequestDataLength, ResponseData, ResponseDataLength)Called by KeyM_ServiceCertificate() if KeyMCryptoKeyHandlerServiceCertificateEnabled=TRUE. If this handler is configured, it replaces KeyM's built-in certificate service logic entirely.

Error detection rules

If KeyMDevErrorDetect = TRUE, KeyM checks on every function call:

  • Module initialized (KEYM_E_UNINIT if not)
  • All pointer parameters are non-NULL (KEYM_E_PARAM_POINTER if NULL)
  • All output buffers are large enough (KEYM_E_SMALL_BUFFER if too small)
  • Configuration is consistent (KEYM_E_CONFIG_FAILURE on mismatch)

Step-by-step DaVinci Configurator Pro workflow

Step 1 — Enable submodules (KeyMGeneral)

1

ParameterEnable if

Step 4 — Configure KeyMCryptoKey containers

1
For each crypto key managed by KeyM, create a KeyMCryptoKey container. Set KeyMCryptoKeyName (UTF-8 string used to identify the key in KeyM_Update() calls).
2
Set KeyMCryptoKeyGenerationType: KEYM_STORED_KEY for direct key data, or KEYM_DERIVE_KEY for derived keys.
3
Set KeyMCryptoKeyCsmKeyTargetRef to the CSM key ID that will receive this key material.
4
For derived keys: set KeyMCryptoKeyCsmKeySourceDeriveRef (the master/source key) and optionally KeyMCryptoKeyGenerationInfo (salt).
5
Set KeyMCryptoKeyStorage: KEYM_STORAGE_IN_NVM or KEYM_STORAGE_IN_CSM. If NVM: add a KeyMNvmBlock container referencing the NvM block ID.
6
If KeyM_Verify() is needed: set KeyMCryptoKeyVerifyFunctionEnabled = TRUE in General and configure KeyMCryptoKeyCsmVerifyJobRef.

Step 5 — Configure KeyMCertificate containers

1
Create one KeyMCertificate container per certificate slot. Set KeyMCertificateId (uint16 handle), KeyMCertFormatType (X509/CVC/CRL), and KeyMCertEncodingType.
2
Configure the certificate hierarchy: set KeyMCertUpperHierarchicalCertRef. For root certificates, this reference points to itself (self-signed). For intermediate/working certs, it points to the issuing cert's container.
3
Set KeyMCertCsmSignatureVerifyJobRef — the CSM job to use for signature verification of this certificate's issuer's signature.
4
Set KeyMCertCsmSignatureVerifyKeyRef — the CSM key holding the issuer's public key. For self-signed root: this key must be populated by KeyM during init or verification.
5
Configure KeyMCertificateStorage: KEYM_STORAGE_IN_RAM (for temporary certs), KEYM_STORAGE_IN_NVM, or KEYM_STORAGE_IN_CSM.
6
If time-based validation needed: set KeyMCertTimebaseRef to a StbM time base ID.
7
Optionally add KeyMCertificateElement sub-containers for each certificate field to extract, and KeyMCertificateElementRule sub-containers for validation constraints.

Step 6 — OS / SchM scheduling

1
Map KeyM_MainFunction to a periodic OS task (e.g. 10ms). It handles all asynchronous state machine processing.
2
Map KeyM_MainBackgroundFunction to a background/idle-priority task. This is where the time-consuming synchronous key handler calls (KeyM_KH_Update()) are executed. This avoids blocking the main 10ms task.
🔴 Common DaVinci mistake: not mapping KeyM_MainBackgroundFunction to any task. Without this, asynchronous key operations (update, finalize) will never complete — they get accepted (E_OK returned) but the callback is never called. Symptoms: application waits forever for the callback notification.
🔴 Common DaVinci mistake: KeyMCertTimebaseRef not set when X.509 certs are used. Certificate validity period check will fail with VALIDITY_PERIOD_FAIL at runtime. Always configure StbM and set the timebase reference.
⚠️ KeyMCertUpperHierarchicalCertRef misconfiguration is the #1 source of chain-of-trust errors. Draw the PKI hierarchy on paper first, then set references from bottom to top. Always verify: the root cert's upper ref must point to itself.

Integration, interface testing & debugging

Common integration bugs

Bug: KeyM_VerifyCertificate returns E_OK but callback never fires

Root cause: KeyM_MainFunction or KeyM_MainBackgroundFunction is not scheduled, or the OS task mapping is incorrect in DaVinci.

Debug: Add a trace to KeyM_MainFunction entry. Verify it is called periodically. Check SchM task mapping for KeyM main functions.

Bug: Certificate always in VALIDITY_PERIOD_FAIL

Root causes: (1) StbM time base not synchronized — StbM returns invalid/zero time. (2) KeyMCertTimebaseRef not set or set to wrong time base. (3) Certificate has expired.

Debug: Call StbM_GetCurrentTime() manually and check the returned timestamp and status flags. Verify the status flag indicates SYNCED. Check NotBefore/NotAfter fields of the certificate against the current time.

Bug: INVALID_CHAIN_OF_TRUST on working certificate

Root cause: Root or intermediate certificate not in VALID state when working cert verification is triggered. Common if boot order causes KeyM_VerifyCertificate for root to fail before working cert verification.

Debug: Call KeyM_CertGetStatus() for every certificate in the chain from root to leaf. Fix the first non-VALID state you find. Remember: verification is top-down.

Bug: KeyM_Update returns KEYM_E_PARAMETER_MISMATCH

Root causes: (1) Key name string does not exactly match KeyMCryptoKeyName configured in DaVinci (case sensitive, UTF-8). (2) KeyNameLength is wrong. (3) SHE mode: bit 121-124 of the M1 data don't match any configured KeyMCryptoKeyCryptoProps value.

Debug: Add DET development error detection and log all DET calls. Print the key name bytes being passed and compare with configuration.

Bug: Callback fires with KEYM_RT_KEY_CERT_WRITE_FAIL

Root cause: NvM write failed — NvM block not configured, NvM busy, or NvM write-protection active. Or CSM key element set failed due to key size mismatch.

Debug: Trace NvM_WriteBlock return value. Check NvM block configuration (size must accommodate the full certificate or key element data). Check NvM DEM events.

Bug: SIGNATURE_FAIL during certificate verification

Root causes: (1) Wrong CSM job configured in KeyMCertCsmSignatureVerifyJobRef. (2) Issuer public key not loaded into KeyMCertCsmSignatureVerifyKeyRef. (3) Certificate data corrupted. (4) Algorithm mismatch (RSA vs ECC configured wrong in crypto driver).

Debug: Verify the CSM job is configured for the right algorithm (ECDSA/RSA-PSS etc.). Use XCP to read back the key bytes from the CSM key slot and verify against the expected issuer public key. Check crypto driver logs.

Key debugging tips for CAN/diagnostic-based key provisioning

💡 In production ECUs, KeyM_Start/Update/Finalize are typically triggered by UDS service 0x27 (Security Access) + 0x2E (Write Data By Identifier) sequences. Capture the CAN bus traffic and verify: (1) the UDS security access grants the correct privilege level that maps to KEYM_START_OEM_PRODUCTIONMODE or WORKSHOPMODE, (2) the key data in 0x2E request matches expected format, (3) the ECU response 0x6E confirms acceptance, (4) M4/M5 response is returned for SHE keys.
💡 Use the AUTOSAR XCP protocol with the generated A2L file to read back CSM key element values and certificate slot status during development — but note that symmetric key data should be protected and not accessible via XCP in production software (configure access protection on the measurement objects).
