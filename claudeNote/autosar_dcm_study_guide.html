AUTOSAR CP DCM — Diagnostic Communication Manager — Complete Study Guide

R24-11 · Doc ID 18 · ISO 14229-1 (UDS) / ISO 15031-5 (OBD) · OSI layers 5–7 · Exam + Integration + DaVinci Config

What is AUTOSAR DCM?

DCM (Diagnostic Communication Manager) is the AUTOSAR BSW module that implements UDS (ISO 14229-1) and OBD (ISO 15031-5) diagnostic services. It sits at OSI layers 5–7, sitting above the transport layer (CanTp/DoIP) and PDU Router. External testers (diagnostic tools, OBD scanners) interact with the ECU exclusively through DCM.

DCM handles

• UDS services: 0x10–0x86
• OBD services: $01–$0A
• Session management (default/extended/programming)
• Security access (seed/key)
• DID read/write
• Routine control
• DTC management (via DEM)
• Memory access (read/write)

Three internal submodules

DSL — Diagnostic Session Layer: manages sessions, security, timing, communication with PduR
DSD — Diagnostic Service Dispatcher: validates requests, routes to DSP
DSP — Diagnostic Service Processing: executes actual service logic

Key dependencies

PduR — Tx/Rx interface (mandatory)
DEM — DTC services (0x14, 0x19, 0x85)
ComM — Communication mode notification
BswM — Mode management
RTE — SWC interface for DIDs/routines
NvM — Non-volatile data access
CSM/KeyM — Service 0x29 authentication

Key limitations

• No SAE J1939 support
• No Service 0x83 (AccessTimingParameter)
• No Service 0x84 (SecuredDataTransmission)
• Not for bootloader use
• No multi-channel parallel processing
• Service 0x29 supports PKI only (no challenge-response)

DCM Architecture — Data Flow

Request flow — the complete path

1
Tester → CanTp/DoIP → PduR: Transport layer reassembles frames. PduR calls Dcm_StartOfReception() → Dcm_CopyRxData() → Dcm_TpRxIndication(E_OK).
2
PduR → DSL: DSL receives complete PDU. Blocks the DcmPduId (no further requests on this connection until response sent [SWS_Dcm_00241]). Forwards to DSD.
3
DSL → DSD: DSD checks in strict order: ① Manufacturer permission (Xxx_Indication) ② SID lookup ③ Auth state ④ Session ⑤ Security level ⑥ Supplier permission ⑦ Mode rules [SWS_Dcm_01535].
4
DSD → DSP: DSP executes the service. May call DEM, RTE SWC interface, NvM, IoHwAb. Returns synchronously or asynchronously via DCM_E_PENDING / DCM_E_FORCE_RCRRP.
5
DSP → DSD → DSL → PduR: Response assembled. DSL calls PduR_DcmTransmit(). PduR calls Dcm_CopyTxData(). After full transmission: Dcm_TpTxConfirmation() → DcmPduId released.

DSL — Diagnostic Session Layer

DSL is the heart of DCM. It manages sessions, security levels, timing, authentication, protocol priority, and communication modes. All inbound/outbound PDU flows go through DSL.

Key DSL responsibilities

Session management

Tracks current session. Resets to Default on S3Server timeout. Provides Dcm_GetSesCtrlType(). Default session = 0x01 at Dcm_Init [SWS_Dcm_00034].

Security level management

Tracks current security level. Resets to LOCKED (0x00) on session change [SWS_Dcm_00139]. Provides Dcm_GetSecurityLevel().

Timing management

Monitors P2ServerMax / P2*ServerMax. Sends NRC 0x78 (Response Pending) when time limit reached. S3Server = 5000ms default [SWS_Dcm_01679].

Protocol priority

Multiple protocols configured with priority. Higher priority can preempt running lower-priority protocol [SWS_Dcm_00015]. OBD must be highest [CONSTR_06103].

Communication mode

ComM informs DCM of No/Silent/Full comm mode. DCM calls ComM_DCM_ActiveDiagnostic() to keep ECU awake during diagnostics.

Authentication state

Per-connection authentication state: deauthenticated or authenticated. Fallback to deauthenticated on S3Server timeout or idle in default session [SWS_Dcm_01483].

Communication mode behavior

Full CommunicationAll Tx/Rx enabled. Normal, periodic, ROE transmissions allowed.Dcm_ComM_FullComModeEntered()

⚠️ DCM waits for Full Comm mode before transmitting response [SWS_Dcm_01142]. If DCM_E_FORCE_RCRRP is needed during Silent comm → DET error DCM_E_FORCE_RCRRP_IN_SILENT_COMM [SWS_Dcm_01143].

Protocol preemption rules

1
Higher priority protocol preempts running protocol [SWS_Dcm_00015].
2
DCM calls PduR_DcmCancelTransmit() on pending response of preempted protocol [SWS_Dcm_00079].
3
External pending operations cancelled with DCM_CANCEL opStatus [SWS_Dcm_01046].
4
If preempted protocol was in non-default session → forced back to default session [SWS_Dcm_01371].
5
If DcmDslDiagRespOnSecondDeclinedRequest=TRUE → NRC 0x21 (BusyRepeatRequest) sent [SWS_Dcm_00727]. If FALSE → no response [SWS_Dcm_01605].

Generic connection handling

Generic connections carry source/target address at runtime via MetaData items SOURCE_ADDRESS_16 and TARGET_ADDRESS_16. Used for DoIP, Ethernet diagnostics, and some CAN formats. DCM stores tester source address from StartOfReception and uses it for response TARGET_ADDRESS_16 [SWS_Dcm_00848, 00849].

Security Access (Service 0x27) — Complete Flow

Security level state machine

DCM_SEC_LEV_LOCKED (0x00)
→ GetSeed + CompareKey OK →
DCM_SEC_LEV_1..63
→ Session change →
LOCKED again

Security access two-step protocol

Step 1 — Request Seed (requestSeed = odd subfunction)

Tester → ECU: 27 01 (requestSeed, SL1)
ECU calls: Xxx_GetSeed(OpStatus=INITIAL, securityAccessDataRecord, seed, ErrorCode)
ECU → Tester: 67 01 [SeedBytes] (positive response)

Odd subfunctions = request seed. Each security level has odd (request) and even (send key) subfunctions.

Step 2 — Send Key (sendKey = even subfunction)

Tester calculates key from seed using OEM algorithm
Tester → ECU: 27 02 [KeyBytes]
ECU calls: Xxx_CompareKey(key, OpStatus=INITIAL)
If E_OK: DslInternal_SetSecurityLevel(SEC_LEV_1)
ECU → Tester: 67 02 (positive response)

Attempt counter & delay timer — startup sequence

1
At Dcm_Init: call Xxx_GetSecurityAttemptCounter() for each DcmDspSecurityRow with AttemptCounterEnabled=TRUE [SWS_Dcm_01154].
2
If returns DCM_E_PENDING → retry in next Dcm_MainFunction. If DcmDspSecurityMaxAttemptCounterReadoutTime expires → cancel all pending, set to max [SWS_Dcm_01352].
3
If any AttemptCounter ≥ DcmDspSecurityNumAttDelay → start SecurityDelayTimer with max(DcmDspSecurityDelayTimeOnBoot, DcmDspSecurityDelayTime) [SWS_Dcm_01355].
4
Any SecurityAccess request while delay timer running → NRC 0x37 (requiredTimeDelayNotExpired).
⚠️ Security level reset to LOCKED on EVERY non-default → any session transition [SWS_Dcm_00139]. This includes transitioning from extended → programming session. Testers must always re-do security access if they change sessions.

DSD — Diagnostic Service Dispatcher

DSD is the gatekeeper. It validates incoming requests before allowing DSP to execute them. The validation order is strictly defined.

DSD verification order [SWS_Dcm_01535] — EXAM CRITICAL

1
Manufacturer permission: Call Xxx_Indication() on all DcmDsdServiceRequestManufacturerNotification ports. If ANY returns E_REQUEST_NOT_ACCEPTED → no response. If ANY returns E_NOT_OK → NRC from ErrorCode.
2
SID lookup: Check if SID exists in active DcmDsdServiceTable. If not found → NRC 0x11 (serviceNotSupported) [SWS_Dcm_00197].
3
Authentication state check: Verify service allowed for current authentication state/role. If fails → NRC 0x34 (authenticationRequired) [SWS_Dcm_01544].
4
Session check: Verify service allowed in current session (DcmDsdSidTabSessionLevelRef). Fails → NRC 0x7F (serviceNotSupportedInActiveSession) [SWS_Dcm_00211]. Subfunction not in session → NRC 0x7E [SWS_Dcm_00616].
5
Security level check: Verify service allowed in current security level (DcmDsdSidTabSecurityLevelRef). Fails → NRC 0x33 (securityAccessDenied) [SWS_Dcm_00217].
6
Supplier permission: Call Xxx_Indication() on all DcmDsdServiceRequestSupplierNotification ports. Same rules as manufacturer.
7
Mode rule check: Evaluate DcmDsdSidTabModeRuleRef. Fails → NRC from DcmModeRuleNrcValue or 0x22 (conditionsNotCorrect) [SWS_Dcm_00773].
🔴 Only if ALL 7 checks pass does DSD forward request to DSP. If any check fails, only Xxx_Confirmation() is called (NOT DspInternal_DcmConfirmation) [SWS_Dcm_01474].

SuppressPosRspMsgIndicationBit (SPRMIB)

Bit 7 of subfunction byte. If set = tester does NOT want a positive response. DSD handles this completely internally [SWS_Dcm_00200]. DCM clears this bit before passing to DSP [SWS_Dcm_00201]. If RCRRP sent → SPRMIB is cleared [SWS_Dcm_00203].

Functional addressing — suppressed NRCs

When request is functionally addressed, these NRCs are automatically suppressed (no response at all) [SWS_Dcm_00001]:

NRC 0x11 — serviceNotSupported
NRC 0x12 — subFunctionNotSupported  
NRC 0x31 — requestOutOfRange
NRC 0x7E — subFunctionNotSupportedInActiveSession
NRC 0x7F — serviceNotSupportedInActiveSession
💡 This is ISO 14229-1 behavior. Functional requests (broadcast) don't get NRC responses for "not applicable" conditions — only physically addressed requests get these NRCs.

Response pending (NRC 0x78 / RCRRP) mechanism

If application needs more time: returns DCM_E_FORCE_RCRRP from service function → DSL immediately sends NRC 0x78 from a SEPARATE buffer [SWS_Dcm_00119] → waits for TxConfirmation → calls function again with OpStatus=DCM_FORCE_RCRRP_OK [SWS_Dcm_00529].

Automatic RCRRP: DSL monitors P2ServerMax / P2*ServerMax timers and sends NRC 0x78 before timeout [SWS_Dcm_00024].

Max RCRRPs: DcmDslDiagRespMaxNumRespPend (default = infinite). If exceeded → cancel, report DCM_E_INTERFACE_TIMEOUT, send NRC 0x10 [SWS_Dcm_00120].

DSP — Diagnostic Service Processing

DSP executes the actual diagnostic service logic. It interacts with DEM, RTE SWC interfaces, IoHwAb, and NvM depending on the service.

Paged-buffer mechanism (for large responses)

When DcmPagedBufferEnabled=TRUE and response > DcmDslProtocolMaximumResponseSize, DCM streams the response in pages. DSP fills a page → DsdInternal_ProcessPage() → PduR sends page → requests next page → repeat until done. Avoids needing a huge single buffer.

⚠️ If DSP cannot provide next page in time → Tx confirmation with error → DspInternal_CancelPagedBufferProcessing(). Response is terminated.

DSP mode manager role

DSP acts as mode manager for diagnostic state changes. After successful service execution, DSP calls SchM_Switch to update the relevant mode group. For example after DiagnosticSessionControl → SchM_Switch_Dcm_DcmDiagnosticSessionControl(new_session).

/* DCM adjusts effective P2ServerMax for internal overhead */
Effective_P2Max = DcmDspSessionP2ServerMax - DcmTimStrP2ServerAdjust

/* This adjustment accounts for DCM processing time so that

⚠️ New timing parameters from DiagnosticSessionControl (0x10) response take effect ONLY after the positive response transmission is confirmed (TxConfirmation received). Not immediately on service execution.

OBD timing requirement

OBD regulations require response within 50 ms. This is a SYSTEM-LEVEL requirement — the integrator must ensure the full stack (CanTp + DCM + Application) meets this budget. DCM alone cannot guarantee it.

Dcm_MainFunction period

Configured via DcmTaskTime. Drives all timer ticks (P2, P2*, S3, SecurityDelay, ValidationTimeout). All timeouts must be multiples of DcmTaskTime. Typical values: 5ms or 10ms.

OBD Support

DCM supports OBD services $01–$0A per SAE J1979 / ISO 15031-5. OBD is mandatory for emission-related vehicles (OBDII/EOBD). Key differences from UDS:

OBD vs UDS differences

• Always functional addressing (broadcast)
• No session management (always default equiv.)
• No security access needed
• 50ms response time requirement (legislated)
• Only ONE OBD protocol allowed [CONSTR_06102]
• OBD must have HIGHEST priority [CONSTR_06103]

OBD services

$01 — Current powertrain data (PIDs)
$02 — Freeze frame data
$03 — Emission-related DTCs
$04 — Clear emission DTCs
$05 — O2 sensor monitoring
$06 — On-board monitor test results
$07 — Pending DTCs
$09 — Vehicle information
$0A — Permanent DTCs

Parallel OBD + UDS

OBD and UDS can run in parallel in default session [SWS_Dcm_01367]. No protocol preemption for OBD. OBD uses separate DEM client ID [SWS_Dcm_01369] so DEM can distinguish parallel calls.

OBD DTC separation

SAE J1979-2 UDS communication reports different 3-byte DTC numbers than non-J1979-2 UDS communication (OBD UDS DTC separation feature). Configured via DcmDspOBDDTCSeparation.

OBD integration rules

⚠️ OBD 50ms response time is a SYSTEM requirement. The entire stack (CanTp segmentation + PduR routing + DCM processing + DEM query + CanTp response) must complete within 50ms. DCM alone cannot guarantee this — integration team must verify end-to-end timing.
💡 For WWH-OBD (World-Harmonized OBD), DCM currently only supports FunctionalGroupIdentifier 0x33 per R24-11.

Step-by-step DaVinci configuration

Step 1 — DcmGeneral

1
Set DcmTaskTime: period of Dcm_MainFunction in ms (e.g. 5ms or 10ms). All timeouts must be multiples of this.
2
Set DcmPagedBufferEnabled: TRUE if large responses (ReadDTCInformation) expected and RAM is limited.
3
Set DcmDevErrorDetect: TRUE for development builds — enables DET reporting.
4
Set DcmDslDiagRespOnSecondDeclinedRequest: TRUE → send NRC 0x21; FALSE → silent ignore.

Step 2 — Protocol rows (DcmDslProtocolRow)

1
Create one DcmDslProtocolRow per addressing type: UDS physical, UDS functional, OBD (if needed).
2
For each: set DcmDslProtocolType (DCM_UDS_ON_CAN, DCM_OBD_ON_CAN, etc.), DcmDslProtocolPriority (OBD = highest number = highest priority), DcmDslBuffer (size = max expected request/response).
3
Link DcmDslProtocolRxPduRef to the PduR PDU configured for reception. Link DcmDslProtocolTxPduRef to Tx PDU. Link DcmDslProtocolSIDTable to the service table for this protocol.
4
Configure DcmDslProtocolComMChannelRef to link this protocol's PDU to a ComM channel. Needed for ComM_DCM_ActiveDiagnostic calls.

Step 3 — Session parameters (DcmDspSession)

1
Create DcmDspSessionRow for each session: DCM_DEFAULT_SESSION (level=1), DCM_PROGRAMMING_SESSION (level=2), DCM_EXTENDED_DIAGNOSTIC_SESSION (level=3), plus any OEM sessions.
2
Per session: set DcmDspSessionP2ServerMax (ms) and DcmDspSessionP2StarServerMax (ms). These are the raw ISO values before adjustment.
3
Configure DcmS3ServerTimeoutOverwrite if 5000ms default is not appropriate.

Step 4 — Security rows (DcmDspSecurity)

1
Create DcmDspSecurityRow for each security level. Set DcmDspSecurityLevel (1, 3, 5... odd numbers for standard levels), DcmDspSecurityUsePort (USE_SYNCH_FNC / USE_ASYNCH_FNC).
2
Set DcmDspSecurityGetSeedFnc → your GetSeed() function. Set DcmDspSecurityCompareKeyFnc → your CompareKey() function.
3
Set DcmDspSecurityNumAttDelay → max failed attempts before lockout. Set DcmDspSecurityDelayTime → lockout duration. Set DcmDspSecurityDelayTimeOnBoot → first-boot lockout duration.
4
If non-volatile attempt counter needed: set DcmDspSecurityAttemptCounterEnabled=TRUE, provide DcmDspSecurityGetAttemptCounterFnc and DcmDspSecuritySetAttemptCounterFnc.

Step 5 — Service table (DcmDsdServiceTable)

1
Create DcmDsdService for each supported UDS service. Set DcmDsdSidTabServiceId (e.g. 0x22), DcmDsdSidTabSubfuncAvail (TRUE if service has subfunctions).
2
Add DcmDsdSidTabSessionLevelRef for each session where this service is allowed. If no ref added → service available in ALL sessions.
3
Add DcmDsdSidTabSecurityLevelRef for each security level required. If no ref → service allowed in all levels.
4
For subfunctions: add DcmDsdSubService entries. Each subservice has its own session/security refs.

Step 6 — DID configuration (DcmDspDid)

1
Create DcmDspDid for each DID. Set DcmDspDidIdentifier (2-byte DID value, e.g. 0xF190 for VIN).
2
Add DcmDspDidSignal sub-containers. Each signal references a DcmDspData container. Set DcmDspDidByteOffset for multi-signal DIDs.
3
In DcmDspData: set DcmDspDataUsePort (USE_DATA_SYNCH_FNC, USE_DATA_ASYNCH_FNC, USE_DATA_SENDER_RECEIVER, USE_ECU_SIGNAL), DcmDspDataType (UINT8, UINT16, UINT8_N...), DcmDspDataByteSize.
4
Set read function: DcmDspDataReadFnc (for synch) or DcmDspDataReadDataLengthFnc + DcmDspDataConditionCheckReadFnc (for asynch). Set write function for WDBI DIDs.

Step 7 — Mode rules (DcmProcessingConditions)

1
Create DcmModeCondition for each condition (e.g. vehicle speed < 5 km/h, ECU in NORMAL mode). Configure either DcmBswModeRef (BSW mode) or DcmSwcModeRef (SWC mode) or DcmSwcSRDataElementRef (S/R signal).
2
Create DcmModeRule combining conditions with AND/OR logic. Assign DcmModeRuleNrcValue (the NRC to return if rule fails).
3
Reference DcmModeRules from DcmDsdSidTabModeRuleRef (service level) or DcmDsdSubServiceModeRuleRef (subservice level).

Common DaVinci mistakes

🔴 DcmDslProtocolPriority clash: Two protocols with same priority → undefined preemption behavior. Always use unique priority values. OBD must be numerically highest (highest priority).
🔴 Buffer size too small: DcmDslBuffer must accommodate largest possible request AND response. If ReadDTCInformation with many DTCs expected → size accordingly. Under-sizing causes RCRRP or lost responses.
⚠️ Session/security not added to service: Forgetting to add DcmDsdSidTabSessionLevelRef means service available in ALL sessions (including default). Adds DcmDsdSidTabSecurityLevelRef only when needed — omitting it means no security restriction.
⚠️ DcmTaskTime too large: If DcmTaskTime = 20ms but P2ServerMax = 25ms → only ONE Dcm_MainFunction tick before RCRRP threshold. Application gets almost no time. Keep DcmTaskTime ≤ 5–10ms for typical projects.
⚠️ Missing ComM channel reference: Without DcmDslProtocolComMChannelRef, DCM cannot call ComM_DCM_ActiveDiagnostic/InactiveDiagnostic. ComM may put ECU to sleep during active diagnostic session.
💡 DcmDslDiagRespMaxNumRespPend: Set this to a finite value (e.g. 10) for production. Infinite RCRRPs can keep ECU awake indefinitely if application hangs. Set to 0 = immediate cancel with NRC 0x10.

Integration Testing & Debugging

Common integration bugs

Bug: NRC 0x7F on every request

Service not allowed in current session. Either: (1) DcmDsdSidTabSessionLevelRef not configured for the active session. (2) ECU still in DEFAULT session when extended-only service is requested. (3) Session change not processed yet. Fix: verify DSD service table session config. Add trace to Dcm_GetSesCtrlType() call.

Bug: NRC 0x33 on every request

Security level too low. Service has DcmDsdSidTabSecurityLevelRef configured but security not unlocked. Fix: verify SecurityAccess was performed. Check Dcm_GetSecurityLevel(). Verify security level value matches what DSD table expects (odd = locked, even = not meaningful for service access).

Bug: NRC 0x78 repeated endlessly

Application always returns DCM_E_PENDING. Root cause: application blocked on NvM/DEM call that never completes. Or DcmDslDiagRespMaxNumRespPend not configured (infinite by default). Fix: configure max RCRRP count. Verify NvM job completes. Add timeout in application. Check DEM busy flags.

Bug: Session drops back to default unexpectedly

S3Server timeout. Tester not sending TesterPresent (3E 80) functionally. Or sending it on wrong connection/PDU ID. Fix: Verify tester sends 3E 80 before S3Server (5000ms) expires. Verify TesterPresent uses FUNCTIONAL addressing (broadcast). Verify DcmDslConnection routing.

Bug: SecurityAccess always returns NRC 0x37

SecurityDelayTimer running. Either: (1) Previous session had failed attempts and timer carried across reset (NvM-backed counter). (2) DcmDspSecurityDelayTimeOnBoot very long. (3) AttemptCounter read from NvM returned max value. Fix: clear NvM-backed counter. Reduce DcmDspSecurityDelayTimeOnBoot for development.

Bug: Large DID read returns NRC 0x14 (responseTooLong)

Response length exceeds configured buffer. Fix: (1) Increase DcmDslBuffer size. OR (2) Enable paged-buffer (DcmPagedBufferEnabled=TRUE) and configure DcmDslProtocolMaximumResponseSize. For very large responses (full DTC list) paged buffer is recommended.

Bug: OBD response arrives after 50ms

System-level timing violation. Possible causes: (1) DcmTaskTime too large (10ms task gives 10ms slop per cycle). (2) Application function takes too long (GetPIDData blocking). (3) CanTp configuration (STmin, BS) causing delay. Fix: reduce DcmTaskTime, make OBD PID callbacks non-blocking, verify CanTp STmin=0.
