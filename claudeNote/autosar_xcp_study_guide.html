AUTOSAR XCP module comprehensive study guide covering state machine, functional specification, API, sequence diagrams, and DaVinci configuration

What is XCP?

XCP (Universal Calibration Protocol) is an ASAM standard protocol for communication between a master (calibration tool like CANalyzer/INCA) and a slave (ECU). In AUTOSAR CP, the XCP module lives above PduR and communicates via CAN, FlexRay, or Ethernet.

Standard
ASAM XCP v1.1 / v1.2 (CAN)

Architecture: where XCP sits in the AUTOSAR stack

Key R24-11 change: XCP now uses PduR_XcpTransmit instead of directly calling bus interfaces. The old direct bus-interface calls (SWS_Xcp_00712/00714/00720/00734) are now OBSOLETE.

Key terminology for exams
CTO — Command Transfer Object
Commands from master to slave (e.g. CONNECT, GET_STATUS, SET_DAQ_LIST_MODE). Max size = XcpMaxCto (8–255 bytes)
DTO — Data Transfer Object
DAQ/STIM data packets. Max size = XcpMaxDto (8–65535 bytes)
ODT — Object Descriptor Table
Describes which memory elements are measured. Each DAQ list has multiple ODTs
DAQ list
Collection of ODTs defining what to measure. Static (pre-configured) or Dynamic
Event channel
Sampling trigger for DAQ lists. Has time cycle, consistency (ODT/DAQ/EVENT), and priority
PDU-ID
Unique identifier for each XCP message direction. Configured via XcpRxPduId/XcpTxPduId

DAQ — synchronous data acquisition (measurement)

The slave periodically reads ECU RAM variables and sends them to the master tool. Key SWS item: SWS_Xcp_00705

Static DAQ (XcpDaqConfigType = DAQ_STATIC)

ODTs pre-configured at compile time. Addresses defined in XcpOdtEntryAddress. Fast but inflexible. MAX_DAQ = MIN_DAQ (XcpMinDaq). XcpMaxOdt and XcpMaxOdtEntries defined per list.

Dynamic DAQ (XcpDaqConfigType = DAQ_DYNAMIC)

Master allocates DAQ lists at runtime using ALLOC_DAQ / ALLOC_ODT / ALLOC_ODT_ENTRY commands. MAX_DAQ = MIN_DAQ + DAQ_COUNT. Requires XcpDaqCount, XcpOdtCount, XcpOdtEntriesCount in config.

DAQ setup sequence (master tool perspective)
1. CONNECT 2. GET_DAQ_PROCESSOR_INFO → check DAQ capabilities 3. GET_DAQ_RESOLUTION_INFO → timestamp resolution 4. FREE_DAQ → clear all dynamic DAQ lists 5. ALLOC_DAQ(count) → allocate DAQ lists 6. ALLOC_ODT(daq, count) → allocate ODTs per list 7. ALLOC_ODT_ENTRY(daq, odt, count) → allocate ODT entries 8. SET_DAQ_PTR(daq, odt, entry) 9. WRITE_DAQ(bit_offset, size, addr_ext, addr) → per entry 10. SET_DAQ_LIST_MODE(mode, daq, event, prescaler, priority) 11. START_STOP_SYNCH(START_SELECTED) → begin measurement
STIM — synchronous data stimulation

Inverse of DAQ. Master writes data into ECU RAM at the rate defined by the event channel. Used for bypassing (model-in-the-loop). SWS_Xcp_00707 SWS_Xcp_00761

Bypassing = running DAQ and STIM simultaneously on the same signal path. The master reads a signal (DAQ), processes it in a model, then writes the result back (STIM), overriding the ECU calculation.

XcpDaqListType = STIM
DAQ list is write-only from slave perspective (master sends data in)
XcpDaqListType = DAQ_STIM
Same list can be used for either direction (bypassing scenario)
Identification field types (XcpIdentificationFieldType)
ABSOLUTE
Absolute ODT number — each ODT globally unique. Simple but uses more bits.
RELATIVE_BYTE
Relative ODT number + absolute DAQ list number (BYTE). Saves 1 byte in header.
RELATIVE_WORD
Relative ODT + DAQ list number as WORD. Supports large DAQ lists (>255).
RELATIVE_WORD_ALIGNED
Same as WORD but aligned to 2-byte boundary. Required for some platforms.
XCP on CAN specifics Integration critical

• At least two CAN identifiers must be configured (one for CTO/response, one for DAQ) — SWS_Xcp_00715

• XCP on CAN uses ASAM v1.2 (not v1.1) — SWS_Xcp_00703

• CAN FD supported from AUTOSAR 4.4.0 onwards

• GET_SLAVE_ID command must be supported — SWS_Xcp_00718

• SET_DAQ_ID is NOT part of the AUTOSAR XCP module (known limitation)

• XCP on Ethernet: slave accepts only ONE connection at a time — SWS_Xcp_00736

• Socket close while connected → XCP disconnect, all DAQ stops — SWS_Xcp_00737

• XCP on FlexRay: all LPDUs are event-driven — SWS_Xcp_00721

• FlexRay buffers assigned EXCLUSIVELY to XCP — SWS_Xcp_00722

• FlexRay can pack multiple XCP messages into one frame — SWS_Xcp_00732

Timestamp configuration
XcpTimestampType
NO_TIME_STAMP, ONE_BYTE, TWO_BYTE, FOUR_BYTE
XcpTimestampUnit
1PS to 1S (13 options). ECU time from AUTOSAR OS counter (XcpCounterRef)
XcpTimestampTicks
Ticks per unit. 0–65535. Wraps on overflow.
API functions — complete reference
Transmission path: R24-11 (important for exams — this changed!)
XCP on CAN: XCP → PduR_XcpTransmit() → LSduR → CanIf → CAN driver XCP on FlexRay: XCP → PduR_XcpTransmit() → LSduR → FrIf → Fr driver XCP on Ethernet:XCP → PduR_XcpTransmit() → SoAd → TcpIp → LSduR → EthIf OLD (obsolete): XCP → CanIf_Transmit() [direct, bypassing PduR — SWS_Xcp_00714 OBSOLETE] NEW (R24-11): XCP → PduR_XcpTransmit() [always via PduR — SWS_Xcp_00864/865/866]

After calling PduR_XcpTransmit(), the module WAITS for Xcp_TxConfirmation() before sending the next PDU to avoid overwriting the transmit buffer. SWS_Xcp_00863

Key timing insight: TxConfirmation gating

XCP must NOT call PduR_XcpTransmit() again until Xcp_TxConfirmation() (positive or negative) has been received for the previous transmission. This prevents buffer overwrite. Both callbacks (TxConfirmation and TriggerTransmit) can be called from interrupt context or task level.

Xcp_MainFunction() scheduling rules

• Called cyclically by BSW Scheduler (SchM) — SWS_Xcp_00824
• Must NOT enter wait state — SRS_BSW_00424
• Only called from task bodies, never from ISR — SRS_BSW_00433
• Period configured via XcpMainFunctionPeriod (float, seconds, pre-compile)

Vector DaVinci Configurator — step by step for XCP

In DaVinci, XCP configuration is under the XCP module. Here is the complete workflow for a CAN-based project.

Step 1: XcpGeneral — basic settings
XcpGeneral ├── XcpOnCanEnabled = TRUE ├── XcpOnFlexRayEnabled = FALSE ├── XcpOnEthernetEnabled = FALSE ├── XcpDevErrorDetect = TRUE ← always TRUE for development ├── XcpVersionInfoApi = TRUE ├── XcpDaqConfigType = DAQ_STATIC or DAQ_DYNAMIC ├── XcpMaxCto = 8 ← CAN max frame = 8, CAN FD = 64 ├── XcpMaxDto = 8 ├── XcpMinDaq = 0 ← predefined DAQ lists ├── XcpDaqCount = 2 ← (if DYNAMIC) number of dynamic DAQ lists ├── XcpMaxEventChannel = 4 ├── XcpTimestampType = NO_TIME_STAMP (or ONE_BYTE/TWO_BYTE/FOUR_BYTE) ├── XcpTimestampUnit = TIMESTAMP_UNIT_1MS ├── XcpTimestampTicks = 1 ├── XcpMainFunctionPeriod = 0.01 ← 10ms ├── XcpSuppressTxSupport = FALSE ├── XcpIdentificationFieldType = ABSOLUTE └── XcpCounterRef → OsCounter (reference to OS counter for timestamps)
Step 2: XcpConfig → XcpPdu (RX and TX PDUs)
XcpConfig └── XcpPdu[0]: XcpRxPdu ← command PDU (master → slave) ├── XcpRxPduId = 0 ← must match CanIf Rx PDU configuration └── XcpRxPduRef → Pdu ← reference to global PDU in EcucValueCollection └── XcpPdu[1]: XcpTxPdu ← response/DAQ PDU (slave → master) ├── XcpTxPduId = 0 ← used in TxConfirmation/TriggerTransmit └── XcpTxPduRef → Pdu ← reference to global PDU NOTE: XcpRxPdu and XcpTxPdu must reference PDUs where KeepLocalPduBuffer = FALSE (SWS_Xcp_CONSTR_00861 — new constraint in R24-11)
Step 3: XcpConfig → XcpDaqList (static DAQ only)
XcpDaqList[0] ├── XcpDaqListNumber = 0 ├── XcpDaqListType = DAQ ← DAQ | STIM | DAQ_STIM ├── XcpMaxOdt = 1 ├── XcpMaxOdtEntries = 4 └── XcpOdt[0] ├── XcpOdtNumber = 0 ├── XcpOdtEntryMaxSize = 4 └── XcpOdtEntry[0] ├── XcpOdtEntryNumber = 0 ├── XcpOdtEntryAddress = &EngineSpeed_rpm ← linker symbol └── XcpOdtEntryLength = 2 ← uint16
Step 4: XcpConfig → XcpEventChannel
XcpEventChannel[0] ├── XcpEventChannelNumber = 0 ← referenced from DAQ list mode ├── XcpEventChannelType = DAQ ← DAQ | STIM | DAQ_STIM ├── XcpEventChannelConsistency = ODT ← ODT | DAQ | EVENT ├── XcpEventChannelTimeCycle = 10 ← 10 ticks (= 10ms if unit is 1ms) ├── XcpEventChannelTimeUnit = TIMESTAMP_UNIT_1MS ├── XcpEventChannelMaxDaqList = 1 ├── XcpEventChannelPriority = 0 └── XcpEventChannelTriggeredDaqListRef → XcpDaqList[0]
Step 5: XcpConfig → XcpCommunicationChannel (new in R24-11 era)
XcpCommunicationChannel[0] ├── XcpComMChannelRef → ComMChannel[0] ← which ComM channel this belongs to ├── XcpChannelTxPduRef → XcpTxPdu[0] ← mandatory └── XcpChannelRxPduRef → XcpRxPdu[0] ← optional (0..1) Used by Xcp_SetTransmissionMode(Channel, XCP_TX_OFF/ON) to suppress/enable TX per communication channel
Step 6: PduR routing configuration (critical!)
In PduR configuration: PduRRoutingTable └── PduRRoutingPath (XCP_RX) ├── Source: PduRSrcPdu → XcpRxPdu (from CanIf via LSduR) └── Destination: PduRDestPdu → Xcp_RxIndication callback PduRRoutingPath (XCP_TX) ├── Source: PduRSrcPdu → PduR_XcpTransmit (from XCP module) └── Destination: PduRDestPdu → CanIf (via LSduR) TIP in DaVinci: Use the "PDU" view to connect XCP PDUs. The tool usually auto-generates routing paths when you link XcpRxPduRef/XcpTxPduRef to corresponding CanIf PDUs.
Step 7: SchM integration for Xcp_MainFunction
In OS/SchM configuration: OsTask[BSW_10ms_Task] OsTaskActivation = 1 OsTaskPeriod = 0.01 SchMExclusiveArea: Xcp_MainFunction_ExclusiveArea In BSW_MainFunctions_10ms: Xcp_MainFunction() ← period must match XcpMainFunctionPeriod
Common DaVinci mistakes to avoid
Wrong PDU buffer config
XcpRxPdu/TxPdu must reference PDUs with KeepLocalPduBuffer=FALSE. DaVinci will not always warn you.
Missing PduR routing
XCP PDU IDs in XcpRxPduId must match what PduR passes to Xcp_RxIndication. Mismatch → XCP_E_INVALID_PDUID.
XcpCounterRef not set
If timestamp enabled but OsCounter not referenced, GetCounterValue() fails at runtime.
DAQ_DYNAMIC but no sizes
XcpDaqCount, XcpOdtCount, XcpOdtEntriesCount must be set when DAQ_DYNAMIC selected.
Wrong MainFunction period
XcpMainFunctionPeriod must exactly match OS task period. Inconsistency causes timestamp drift.
Old API in code
Do not call CanIf_Transmit() directly. Use PduR_XcpTransmit(). R24-11 made old approach obsolete.

Debugging scenarios — integration testing
No response to CONNECT
Check: PduR routing path for RX. Verify XcpRxPduId matches CanIf config. Check Xcp_Init() called before OS starts. XcpDevErrorDetect errors in DET log.
DAQ data not arriving
Check: XcpEventChannelTriggeredDaqListRef set correctly. Verify event channel triggers (called from correct OS task). XcpOdtEntryAddress points to valid RAM. Xcp_MainFunction() period matches event cycle.
Timestamp incorrect/drifting
Check: XcpCounterRef → correct OsCounter. XcpTimestampUnit and XcpTimestampTicks match actual OS counter resolution. GetCounterValue() calls from correct context.
XCP_E_INVALID_PDUID in DET
XcpRxPduId configured in XCP must match PduIdType passed by PduR to Xcp_RxIndication(). Cross-check PduR and XCP configuration tables in DaVinci.
Ethernet: master connects then immediately disconnects
Check: only ONE master can connect at a time (SWS_Xcp_00736). Second master attempt triggers disconnect. Verify SoAd socket configuration and TcpIp stack initialization order.
Flash programming fails
Check: XcpFlashProgrammingEnabled = TRUE. Seed&Key configured if protection active. PROGRAM_RESET command: slave goes disconnected but does NOT force device reset (SWS_Xcp_00856).
Interface testing checklist
☐ Verify Xcp_Init() called once before any bus activity
☐ Verify RxIndication called from correct interrupt context
☐ Verify TxConfirmation arrives before next Transmit call
☐ Test XCP_E_UNINIT: call Xcp_RxIndication before Xcp_Init → DET log should show error
☐ Test NULL pointer: call Xcp_GetVersionInfo(NULL) → XCP_E_PARAM_POINTER in DET
☐ Verify Xcp_MainFunction() called at correct period (measure with oscilloscope or trace tool)
☐ Test dynamic DAQ allocation via ALLOC_DAQ/ODT/ODT_ENTRY commands
☐ Verify bypassing: DAQ read and STIM write on same signal, check timing consistency
☐ Test RESUME mode: power cycle with DAQ running, verify DAQ restarts automatically
☐ Test Xcp_SetTransmissionMode(XCP_TX_OFF) → no more TX PDUs until XCP_TX_ON
☐ Verify Seed&Key rejection: wrong key → slave does not unlock protected resource
☐ For CAN FD: MaxCto/MaxDto must be set to CAN FD payload sizes (up to 64)

R24-11 changes — exam favorite topic
What changed in R24-11

1. SWS_Xcp_00860 (DRAFT replaces 00701): XCP now sits ABOVE PduR, not just above bus interfaces directly
2. SWS_Xcp_00862 (DRAFT replaces 00712): Calibration data sent/received via PduR APIs, not bus APIs directly
3. SWS_Xcp_00863 (DRAFT replaces 00859): Wait for Xcp_TxConfirmation after PduR_XcpTransmit (was Lo_Transmit)
4. SWS_Xcp_00864/865/866: Use PduR APIs for CAN/FlexRay/Ethernet (old direct bus calls obsolete)
5. SWS_Xcp_00867/868/869/870/871: Callback function names without prefix (no longer bus-specific)
6. SWS_Xcp_CONSTR_00861: XcpRxPdu and XcpTxPdu must have KeepLocalPduBuffer = FALSE (NEW constraint)

Limitations (commonly examined)

• SET_DAQ_ID command (from XCP CAN Transport Layer Spec) is NOT part of AUTOSAR XCP
• AUTOSAR RTE does not offer APIs for direct communication with XCP
• NAX (node address extension) only configurable through A2L file, not via AUTOSAR config
• LIN bus: no ASAM XCP specified (no XCP on LIN)
• Ethernet: only ONE connection accepted at a time (SWS_Xcp_00736)
• A2L IF_DATA section generated by code generator based on XCP configuration (SWS_Xcp_00853)

Code file structure (SWS_Xcp_00501)
Xcp.c ← general source code Xcp_Cfg.c ← pre-compile time configurable parameters Xcp_Lcfg.c ← link time configurable parameters Xcp_PBcfg.c ← post build time configurable parameters Xcp.h ← public API header (all functions declared here) SchM_Xcp.h ← Xcp_MainFunction declared here (BSW Scheduler interface)
