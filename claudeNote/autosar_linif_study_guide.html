AUTOSAR LIN Interface (LinIf + LinTp) complete study guide — R24-11

What is LinIf?

LinIf (LIN Interface) is the AUTOSAR BSW module that provides ISO 17987 (LIN 2.x) protocol behaviour for both MASTER and SLAVE nodes. It sits between upper communication layers (LSduR, PduR, LinSM) and the lower LIN Driver + LIN Transceiver Driver. It includes an integrated LIN Transport Protocol (LinTp) for diagnostic messaging. One LinIf instance handles ALL LIN channels of an ECU.

Master vs Slave — key conceptual difference (exam essential)

MASTER — schedule-driven

Controls all bus timing via schedule table. LinIf_MainFunction_<Ch> runs at fixed tick rate and decides which frame to transmit next. Calls Lin_SendFrame to send header. Uses Lin_GetStatus to poll completion. Initiates go-to-sleep by sending diagnostic MRF with NAD=0. Can request schedule table switches at runtime.

SLAVE — interrupt/event-driven

No scheduling knowledge. Reacts to headers received from master via LinIf_HeaderIndication callback. Evaluates PID to decide: transmit, receive, or ignore (irrelevant). Enters sleep on go-to-sleep command received (MRF NAD=0) or bus idle timeout (LinIfBusIdleTimeoutPeriod). MainFunction_<Ch> supervises timing only.

LIN physical frame anatomy

BREAK (13+ bits dominant) → SYNC (0x55) → PID (Protected ID) → DATA (1–8 bytes) → CHECKSUM

PID = Protected ID: 6-bit frame ID + 2 parity bits. MRF always has frame ID 0x3C (PID 0x3C). SRF always 0x3D (PID 0x7D). Classic (LIN 2.0) checksum covers data only. Enhanced checksum (LIN 2.x) covers PID+data. LinIf configures checksum type per frame via LinIfChecksumType.

Channel sub-state machine — SLAVE node

Entry: LinIf_Init() → channel starts in LINIF_CHANNEL_SLEEP
SLEEP→OPERATIONAL: LinIf_Wakeup() called (returns E_OK)

Two triggers to enter sleep (slave-specific):
1. Reception of go-to-sleep command: MRF with NAD byte = 0x00 → calls <User>_GotoSleepIndication() → upper layer calls LinIf_GotoSleep() → Lin_GoToSleepInternal()
2. Bus idle timeout (LinIfBusIdleTimeoutPeriod expires): same flow as above

Slave wakeup confirmation: Only confirmed (WakeupConfirmation(TRUE)) when the FIRST LinIf_HeaderIndication arrives after LinIf_Wakeup(). If master never schedules → no confirmation → LinSM timeout.

Wakeup during sleep transition (slave): If LinIf_Wakeup() called after Lin_GoToSleepInternal but before GotoSleepConfirmation → Lin_Wakeup() called, GotoSleepConfirmation NOT invoked.

Schedule tables — master node only

The schedule table defines the sequence and timing of all LIN frames. LinIf_MainFunction_<Ch> is the only function that processes the schedule table and initiates frame transfers. Each channel has its own independent set of schedule tables.

Schedule table types

RUN_CONTINUOUS

Loops indefinitely until a new schedule is requested. Normal operational schedule. Can be interrupted by a RUN_ONCE request. After RUN_ONCE completes, resumes from the position defined by LinIfResumePosition (from start or from the interrupted position).

RUN_ONCE

Executes from first to last entry, then switches back to the latest requested RUN_CONTINUOUS schedule. Use cases: diagnostic sessions, node configuration, collision resolving. Exception: if NULL_SCHEDULE requested → always switches immediately (overrides RUN_ONCE in progress).

NULL_SCHEDULE — special mandatory schedule

Every channel has a predefined NULL_SCHEDULE containing NO entries. Set automatically on: LinIf_Init(), entering LINIF_CHANNEL_SLEEP, go-to-sleep successful. NULL_SCHEDULE can be requested at any time and always applies immediately — even interrupting a running RUN_ONCE. If LinIf_Transmit() is called while NULL_SCHEDULE is active → returns E_NOT_OK.

Schedule table switching — LinIf_ScheduleRequest()

• LinSM calls LinIf_ScheduleRequest(Channel, ScheduleTableIdx) → ASYNC (returns E_OK if accepted)
• Actual switch timing: configurable via LinIfScheduleChangeNextTimeBase:
  – FALSE (default): switch after current entry finishes
  – TRUE: switch after current frame's Tx/Rx status is checked
• <User>_ScheduleRequestConfirmation(channel, ScheduleTableIdx) called when new schedule is active
• Requesting same schedule → restarts from beginning
• Sporadic frame pending flags NOT cleared on schedule switch
• LinIf_ScheduleRequest in SLEEP state → DET error LINIF_E_SCHEDULE_REQUEST_ERROR
• BswM calls LinIf via LinSM for TP diagnostic schedule changes (if LinTpScheduleChangeDiag=TRUE)

Frame slot timing

Slot delay: Number of MainFunction ticks between start of consecutive frames. Configured per LinIfEntry as LinIfDelay (in MainFunction periods).

Maximum frame length = TFRAME_MAX (nominal frame length × 140%). Lin_GetStatus must be called between TFRAME_MAX elapsed and next slot due. Too early → LIN_TX/RX_BUSY (frame still ongoing). Too late → next frame delayed.

Jitter: Difference between worst-case and best-case execution time. LinIfMainFunctionPeriod should be significantly smaller than frame slot time to keep jitter low.

Scheduled function — called by OS/SchM

LinIf_MainFunction_<LinIfChannel.ShortName>() — SID 0x80

One instance per configured LIN channel. Named after channel ShortName from DaVinci config (e.g. LinIf_MainFunction_LIN1).

MASTER: Polls schedule table manager → triggers frame TX/RX via Lin_SendFrame → calls Lin_GetStatus → invokes upper layer callbacks. Processes sleep/wakeup transitions.
SLAVE: Supervises bus idle timer, N_As/N_Cs/N_Cr TP timeouts. Frame handling is interrupt-driven via LinIf_HeaderIndication callback.

Non-reentrant. Period = LinIfMainFunctionPeriod (the "tick" for schedule table slot delays).

LIN Transport Protocol (LinTp) — part of LinIf

LinTp implements the ISO 17987-2 diagnostic transport layer. It is used to transport multi-byte diagnostic messages (UDS/KWP) between tester and slave nodes over the LIN bus using MRF (Master Request Frame, PID 0x3C) and SRF (Slave Response Frame, PID 0x3D). Enabled by LinIfTpSupported=TRUE. One TP connection per channel, but functional and physical requests can coexist.

TP frame structure (single-wire UART, max 8 bytes payload)

Frame typePCI byte (byte 0)DataMax payload
SF (Single Frame)0x0N (N=length, 1–6)NAD, PCI, SID, data bytesUp to 6 bytes (SID + 5 data)

NAD (Node Address) identifies the slave. Physical address = 0x01–0x7D. Broadcast = 0x7F. Functional = 0x7E (LinIf uses NAD 0x7E for functional requests from master). Go-to-sleep command: MRF with NAD=0x00 (data byte 0 = 0).

Debugging checklist — integration testing

1. Frames not transmitted (master) — Is LinIf_Init called? Is schedule active (not NULL_SCHEDULE)? Is LinIf_MainFunction_<Ch> scheduled at LinIfMainFunctionPeriod? Is LIN driver initialized? Check schedule entry delay vs MainFunction period.

2. Slave not responding — Check NAD matches (initial and configured). Check PID matches LinIfFrameId. Check checksum type (enhanced vs classic). Check baud rate. Oscilloscope the bus.

3. LINIF_E_RESPONSE constantly reported — Physical layer issue (termination missing, short circuit, grounding). Wrong baud rate. Master frame length too short for slave response time.

4. Sleep/wakeup not working — Check WakeupSourceRef configuration. Check LinIfBusIdleTimeoutPeriod (slave). Check go-to-sleep confirmation callback (LinIfGotoSleepConfirmationUL = LIN_SM). If using transceiver: check LinIfTrcvDriverSupported=TRUE and LinIfSetTrcvMode called after sleep.

5. TP session failing — Check schedule table contains MRF+SRF slots. Check LinTpScheduleChangeDiag=TRUE if dynamic schedule switch needed. Check N_As/N_Cs/N_Cr timing parameters match slave node timing. Check LinTpNasTimeout > max bus latency.

6. Schedule switch not happening — Check LinIf_ScheduleRequest return value. Check LINIF_E_SCHEDULE_TABLE_SWITCH_REQUEST_NOT_ACCEPTED in DEM. Is channel in SLEEP state?

7. response_error signal always set (slave) — Frequent LinIf_LinErrorIndication callbacks. Check bus quality. Check checksum type mismatch. Check LIN_ERR_RESP_STOPBIT = UART timing issue.

DaVinci Configurator — LinIf setup

Navigate: BSW → LinIf (and LinTp if TP needed). Main containers: LinIfGeneral → LinIfGlobalConfig → LinIfChannel → LinIfFrame + LinIfScheduleTable + LinIfMaster/LinIfSlave.

LinIfGeneral — global pre-compile flags

LinIfDevErrorDetectTRUE in development. Enables all DET error checks.
LinIfTpSupportedTRUE to enable LinTp module (adds LinTp_Init, LinTp_Transmit, LinTp_Shutdown APIs and PduR_LinTp* interfaces).
LinIfTrcvDriverSupportedTRUE if any channel has a LIN transceiver driver. Enables LinIf_SetTrcvMode, LinIf_GetTrcvMode, LinIf_GetTrcvWakeupReason, LinIf_SetTrcvWakeupMode APIs.
LinIfNcOptionalRequestSupportedTRUE to enable "Assign NAD" node configuration command (master: in schedule; slave: react to it). Optional ISO 17987-3 service.
LinIfMultipleDriversSupportedTRUE for ECUs with multiple LIN hardware types requiring separate Lin module instances (rare — one LinDrv per hardware type).
LinIfBusMirroringSupportedTRUE to enable Mirror_ReportLinFrame() calls for bus mirroring. Enables LinIf_EnableBusMirroring() API.
LinIfResponseErrorSignalChangedCalloutOptional function name for response_error signal change notification callback (slave only).
LinIfSaveConfigurationCalloutOptional function name for Save Configuration request callback (slave only).
LinIfPublicCddHeaderFileHeader file names for CDD callbacks (if upper layer is CDD, not LinSM). 1–32 chars per entry.

LinIfChannel — one per physical LIN channel

LinIfMainFunctionPeriodPeriod (seconds) of LinIf_MainFunction_<Ch> call. The tick base for schedule slot delays. All LinIfDelay values are multiples of this. Match OS task period.
LinIfNodeTypeChoice container: LinIfMaster (master node) or LinIfSlave (slave node). Determines which APIs are available on this channel.
LinIfComMNetworkHandleRefReference to ComM channel. This is how LinIf, LinSM, and ComM find each other. Must be consistent across all three modules.
LinIfScheduleRequestConfirmationULLIN_SM or CDD. Determines which callback module receives schedule request confirmation. Typically LIN_SM → LinSM_ScheduleRequestConfirmation().
LinIfGotoSleepConfirmationULLIN_SM or CDD → LinSM_GotoSleepConfirmation(). Called after go-to-sleep sent/failed.
LinIfWakeupConfirmationULLIN_SM or CDD → LinSM_WakeupConfirmation(). Called after wakeup success/failure.
LinIfGotoSleepIndicationULLIN_SM or CDD → LinSM_GotoSleepIndication(). Slave only: called on go-to-sleep received or bus idle timeout.
LinIfTransceiverDrvConfigLinks this channel to a LIN transceiver driver instance. Set TransceiverId to LinTrcv channel ID.

LinIfMaster — master-specific channel config

LinIfTimebaseSchedule timebase in seconds. Usually same as LinIfMainFunctionPeriod.
LinIfResumePositionWHERE to resume RUN_CONTINUOUS after RUN_ONCE: CONTINUE_AT_IT_POINT (continue from interruption) or RESTART (start from beginning).
LinIfScheduleChangeNextTimeBaseFALSE: switch after current entry. TRUE: switch after current frame status confirmed. Affects schedule switch latency.

LinIfSlave — slave-specific channel config

LinIfBusIdleTimeoutPeriodTime (seconds) of bus inactivity before slave auto-enters sleep. Per ISO 17987: external nodes enter sleep after 4–10s inactivity. Set accordingly.
LinIfInitialNADFactory NAD (Node Address). Used for "Assign NAD" service matching. Does not change at runtime.
LinIfConfiguredNADOperational NAD. Can be updated at runtime by "Assign NAD" command or by LinIf_SetConfiguredNAD(). Used for TP and most node config services.
LinIfSupplierId / LinIfFunctionId / LinIfVariantIdProduct identification used by "Read by Identifier" service. Must match the ECU's LDF file values.

LinIfFrame — one per LIN frame

LinIfFrameIdLIN frame ID (0x00–0x3F). This is the unprotected ID. LinIf computes the PID internally. MRF=0x3C, SRF=0x3D are reserved and cannot be reassigned.
LinIfFrameTypeUNCONDITIONAL, EVENT_TRIGGERED, SPORADIC, MRF, SRF, NODE_CONFIGURATION, ASSIGN_FRAMEID_RANGE, UNASSIGN_FRAMEID, SAVE_CONFIGURATION, FREE_FORMAT, ASSIGN_NAD.
LinIfChecksumTypeCLASSIC (LIN 2.0/1.x) or ENHANCED (LIN 2.1+/ISO 17987). Must match what slave expects. Wrong checksum type → checksum errors on every frame.
LinIfLengthFrame data length in bytes (1–8).
LinIfPduDirectionContainer: LinIfRxPdu (reception) or LinIfTxPdu (transmission) with LSduR PDU reference and PduId.
LinIfFramePrioritySPORADIC frames only: priority among sporadic frames in same slot (0=highest). Sets transmission order when multiple sporadic frames pending.
LinIfCollisionResolvingRefEVENT_TRIGGERED frames only: reference to the collision-resolving RUN_ONCE schedule table to switch to when collision detected.

LinIfScheduleTable — one per schedule table

LinIfScheduleTableTypeRUN_ONCE or RUN_CONTINUOUS.
LinIfEntrySub-container list (ordered). Each entry: LinIfDelay (slot duration in MainFunction ticks) + LinIfFrameRef (→ LinIfFrame). Order determines schedule execution sequence.

LinTpChannelConfig — per-channel TP timing

LinTpNas / LinTpNcs / LinTpNcrN_As, N_Cs, N_Cr timeouts (seconds). (N_Cs + N_As) must be < 0.9 × N_Cr. Default: consult ISO 17987-2 clause 7. Must be multiples of LinIfMainFunctionPeriod.
LinTpP2Timing / LinTpP2MaxP2 and P2* timeouts for diagnostic response (master only). P2Max used after receiving UDS 0x78 Response Pending. LinTpMaxNumberOfRespPendingFrames limits 0x78 count.
LinTpScheduleChangeDiagTRUE: LinTp requests BswM to switch schedule via BswM_LinTp_RequestMode() for diagnostic sessions. FALSE: upper layer manages schedule switching independently.
LinTpDropNotRequestedNadTRUE: drop TP response if NAD doesn't match requested NAD. FALSE: accept response regardless (useful for broadcast or flexible NAD scenarios).
LinTpMaxBufReqMax retries when PduR_LinTpCopyTxData returns BUFREQ_E_BUSY before aborting transmission.

Step-by-step DaVinci workflow

Step 1: LinIfGeneral — Set LinIfDevErrorDetect=TRUE, LinIfTpSupported, LinIfTrcvDriverSupported, LinIfNcOptionalRequestSupported, LinIfBusMirroringSupported.
Step 2: LinIfChannel — One per bus. Set LinIfMainFunctionPeriod (match OS task), LinIfComMNetworkHandleRef → ComM channel, callback UL references (all to LIN_SM for standard setup).
Step 3: Node type — Add LinIfMaster or LinIfSlave container. For master: set LinIfResumePosition, LinIfScheduleChangeNextTimeBase. For slave: LinIfBusIdleTimeoutPeriod, NADs, product IDs.
Step 4: LinIfFrame — One per frame in LDF. Set FrameId, FrameType, ChecksumType, Length. Add LinIfRxPdu/LinIfTxPdu with LSduR PDU references.
Step 5: LinIfScheduleTable — One per schedule in LDF. Set type. Add LinIfEntry sub-containers in order with LinIfDelay + LinIfFrameRef. Ensure NULL_SCHEDULE is present (auto-generated).
Step 6: LinTp config — If LinIfTpSupported: add LinTpChannelConfig per channel with N_As/N_Cs/N_Cr/P2 timings. Add LinTpRxNSdu and LinTpTxNSdu with NAD and PduR references.
Step 7: Transceiver link — If LinIfTrcvDriverSupported: add LinIfTransceiverDrvConfig and reference to LinTrcv channel.
Step 8: OS scheduling — Add LinIf_MainFunction_<ChannelShortName> to OS BSW task at LinIfMainFunctionPeriod. If LinTp: also call LinTp_Init in EcuM init sequence after LinIf_Init.

Common DaVinci integration mistakes

• LinIfComMNetworkHandleRef pointing to wrong ComM channel → LinSM_ScheduleRequest calls go to wrong LinIf channel → LINIF_E_NONEXISTENT_CHANNEL
• LinIfMainFunctionPeriod not matching OS task period → frame slot timing off, LIN bus timing errors
• LinIfFrameId 0x3C or 0x3D explicitly configured for user frames → conflict with MRF/SRF (reserved IDs)
• LinIfChecksumType CLASSIC on slave but master expects ENHANCED → checksum error on every frame → LINIF_E_RESPONSE constantly
• Missing SRF slot in schedule table after MRF in TP diagnostic schedule → N_Cr timer fires immediately → LINTP_E_LINTPNCR_TIMEOUT_OCCURRED
• LinTpNas / LinTpNcs / LinTpNcr not multiples of LinIfMainFunctionPeriod → timer granularity issues
• LinIfBusIdleTimeoutPeriod too short → slave enters sleep prematurely during low-traffic phases
• Callback UL set to CDD but LinIfPublicCddHeaderFile not configured → missing header include → compile error
