AUTOSAR CP CanIf — CAN Interface — Complete Study Guide

R24-11 · Doc ID 12 · Classic Platform · CAN 2.0 / CAN FD / CAN XL · Exam + Integration + DaVinci Config

What is AUTOSAR CanIf?

CanIf (CAN Interface) is the abstraction layer between the CAN Driver (hardware-specific) and all upper-layer communication modules (PduR, CanSM, CanNm, CanTp, J1939, DCM). It provides a single, uniform interface regardless of how many CAN controllers or transceivers are present in the ECU.

CanIf does

• Maps upper-layer PDU IDs to HW objects (HTH/HRH)
• Routes Tx requests to correct CAN Driver
• Routes Rx indications to correct upper layer
• Software receive filtering (BasicCAN)
• Data Length Check
• PDU channel mode control
• Transmit buffering
• Controller/Transceiver mode forwarding
• Wakeup check & validation
• Bus Mirroring support
• Timestamp support (hardware-based)

CanIf does NOT

• Execute network management logic (→ CanSM)
• Handle transport protocol segmentation (→ CanTp)
• Implement COM signals (→ COM)
• Decide if mode transitions are valid (→ CanSM)
• Generate keys (→ KeyM)
• Support J1939 (separate module)
• Support CAN XL Tx buffering (→ CAN Driver)

Dependencies

Upper: PduR (PDU routing), CanSM (mode ctrl), CanNm, CanTp, J1939
Lower: CAN Driver (Can_*), CAN Transceiver Driver (CanTrcv_*)
Peer: EcuM (init, wakeup), DET (errors), IdsM (security events), Mirror (bus mirror)

Key concepts

HOH: Hardware Object Handle — mailbox ID
HTH: HW Transmit Handle
HRH: HW Receive Handle
ControllerId: Abstracted controller ID (starts at 0)
TransceiverId: Abstracted transceiver ID (starts at 0)
L-PDU: CAN protocol data unit (header + data)
L-SDU: Payload of L-PDU (upper layer data)

Position in AUTOSAR stack

Architecture — Hardware Object Handles (HOH)

HOHs are the most important concept for understanding CanIf configuration. They bridge the software PDU world to the hardware mailbox world.

HTH — Hardware Transmit Handle

Identifies a CAN hardware transmit mailbox (or a set of mailboxes for BasicCAN). CanIf uses the HTH to call Can_Write(Hth, PduInfo). Multiple Tx PDUs can map to one HTH (BasicCAN) — only one frame sent at a time. Each HTH is linked to one CAN controller.

BasicCAN HTH: multiple PDUs can share → queue needed. FullCAN HTH: one PDU maps to one dedicated mailbox → no queue needed.

HRH — Hardware Receive Handle

Identifies a CAN hardware receive mailbox. CanDrv passes HRH in CanIf_RxIndication(Mailbox, PduInfo). CanIf uses HRH to look up which Rx PDUs are expected and trigger software filtering (BasicCAN only).

FullCAN HRH: exact single CAN ID match → no SW filtering needed. BasicCAN HRH: ID range → SW filtering mandatory to find exact Rx PDU.

ControllerId and TransceiverId abstraction

CanIf maintains its own IDs starting at 0, abstracting away the driver-specific numbering. This allows using multiple CAN Driver instances transparently.

/* Example: 3 controllers across 2 drivers */
CanIf ControllerId 0 → CanDrvA :: Controller 0
CanIf ControllerId 1 → CanDrvA :: Controller 1
CanIf ControllerId 2 → CanDrvB :: Controller 0   ← different driver, same ControllerId namespace

/* CanSM and all upper layers only use the CanIf ControllerId */
/* CanIf internally maps to the correct driver + local controller number */

CAN Controller Mode State Machine

The controller state machine is implemented in CanSM. CanIf simply forwards mode requests to CanDrv and forwards mode indications from CanDrv back to CanSM. CanIf does NOT validate transition correctness — that is CanSM's job.

PDU Channel Mode State Machine

While the controller mode controls the hardware, PDU channel modes control the software flow of PDUs. Changed via CanIf_SetPduMode() — only when controller is in CAN_CS_STARTED.

PDU mode effects — exam critical

💡 TX_OFFLINE_ACTIVE use case: Diagnostic passive mode (DCM 0x28 service). ECU continues processing as if transmitting (COM signals update, application gets TxConfirmation) but no actual frames are sent on the bus. Avoids any impact on network traffic during diagnostic testing.
⚠️ When OFFLINE or TX_OFFLINE is set: Tx buffers are CLEARED [SWS_CANIF_00073, 00489]. Any pending Tx frames are lost. Upper layers must handle lost pending transmissions.

Transmit Data Path — Complete Flow

Step-by-step transmit flow

1
Upper layer calls CanIf_Transmit(TxPduId, PduInfoPtr) — PduInfoPtr contains SDU data pointer, length, and optional MetaData.
2
CanIf checks PDU channel mode — OFFLINE → CANIF_E_STOPPED (DET runtime error) + return E_NOT_OK. TX_OFFLINE → return E_NOT_OK silently. TX_OFFLINE_ACTIVE → fire TxConfirmation immediately, return E_OK.
3
CanIf resolves HTH from TxPduId — maps to correct CAN Driver instance and hardware mailbox.
4
CanIf calls Can_Write(Hth, CanPdu) — CAN FD or CAN 2.0 based on CanIfTxPduCanIdType. For CAN XL: calls CanXL_Write() [SWS_CANIF_00939].
5a
Can_Write() returns E_OK → frame accepted by CAN HW. CanIf_Transmit() returns E_OK. Wait for TxConfirmation.
5b
Can_Write() returns CAN_BUSY (HW mailbox in use, BasicCAN) → check if Tx buffering enabled (CanIfPublicTxBuffering=TRUE) and buffer available. If yes → store in CanIfTxBuffer, return E_OK. If no → return E_NOT_OK [SWS_CANIF_00381].
5c
Can_Write() returns E_NOT_OK → CanIf_Transmit() returns E_NOT_OK. Upper layer must retry.
6
CanDrv calls CanIf_TxConfirmation(Hth, PduHandle) after frame successfully transmitted on bus.
7
CanIf checks Tx buffer for pending PDUs on same HTH [SWS_CANIF_00386]. If found → calls Can_Write() for highest priority buffered PDU [SWS_CANIF_00668, 00070].
8
CanIf calls LSduR_CanIfTxConfirmation(TxPduId, E_OK) → upper layer notified of successful transmission [SWS_CANIF_00383].

Transmit buffering rules

RuleRequirement
Only BasicCAN HTH buffers supported (not FullCAN)[CanIfBufferSize]
Only ONE instance of each Tx PDU can be buffered[SWS_CANIF_00836]
New write to already-buffered PDU → OVERWRITE (last-is-best)[SWS_CANIF_00068]
Buffer full + new PDU not already buffered → E_NOT_OK[SWS_CANIF_00837]
Dynamic Tx PDU buffered → also store CAN ID in buffer[SWS_CANIF_00849]
Buffered PDUs transmitted in CAN priority order (lower CAN ID = higher priority)[SWS_CANIF_00070]
CAN XL: Tx buffering in CanIf NOT supported[SWS_CANIF ch 7.11]
Init: all Tx buffers cleared[SWS_CANIF_00387]

Trigger Transmit (pull model)

Instead of pushing data at Transmit request time, CanIf can use a pull model where CanDrv fetches data just-in-time when hardware is free. CanIf calls back the upper layer via CanIf_TriggerTransmit(TxPduId, PduInfoPtr). Only a transmit request (no data) is stored — data is fetched when needed. Configured via CanIfTxPduTriggerTransmit=TRUE.

💡 Used for periodic COM signals where the latest value should always be sent. The PDU request queued, and data is fetched at the last moment — ensuring the most current signal value is transmitted.

Receive Data Path — Complete Flow

Step-by-step receive flow

1
CAN HW fires receive interrupt → CanDrv processes it, normalizes data into PduInfoType (SDU data, length) with Can_HwType (HRH, CAN ID).
2
CanDrv calls CanIf_RxIndication(const Can_HwType* Mailbox, const PduInfoType* PduInfoPtr). Mailbox contains HRH + CanId + ControllerId. PduInfo has data + length. Note: hardware mailbox is locked until CanIf_RxIndication() returns [SWS_CANIF_00342 context].
3
Software filtering (BasicCAN only) — CanIf checks if received CAN ID matches any configured Rx PDU for this HRH [SWS_CANIF_00389]. If not found → drop frame silently.
4
Data Length Check (if enabled) — received DLC must be ≥ configured CanIfRxPduDataLength [SWS_CANIF_00026]. If fails → DET runtime error CANIF_E_INVALID_DATA_LENGTH, frame dropped [SWS_CANIF_00168].
5
Optional: copy to Rx buffer — if CanIfRxPduReadData=TRUE, store L-SDU in internal buffer for polling via CanIf_ReadRxPduData() [SWS_CANIF_00297, 00198].
6
CanIf calls LSduR_CanIfRxIndication(RxPduId, PduInfoPtr) → upper layer (PduR/CanNm/CanTp) notified. Actual received DLC is passed even if larger than configured [SWS_CANIF_00829].

Receive path data flow summary

CanIf_RxIndication(HRH, CanId, DLC, DataPtr)
  │
  ├─ If BasicCAN HRH: run SW filter to find Rx PDU → drop if not found
  │
  ├─ If DLC check enabled: received DLC ≥ configured DLC? → drop + DET if not
  │
  ├─ If PDU channel mode = OFFLINE: drop (no upper layer callback)
  │
  ├─ If CanIfRxPduReadData: copy to internal Rx buffer
  │
  └─ Call LSduR_CanIfRxIndication(PduId, &PduInfo) → upper layer

Read notification status APIs (polling approach)

Alternative to callback approach. Upper layer polls notification status instead of waiting for callback. Configured with CanIfPublicReadRxPduNotifyStatusApi/CanIfPublicReadTxPduNotifyStatusApi.

/* Read Tx confirmation status */
CanIf_NotifStatusType status = CanIf_ReadTxNotifStatus(CanIfTxSduId);
/* CANIF_TX_RX_NOTIFICATION: confirmed since last read */
/* CANIF_NO_NOTIFICATION: not confirmed */
/* Note: read-and-consume — resets status after each read */

/* Read Rx indication status */
CanIf_NotifStatusType status = CanIf_ReadRxNotifStatus(CanIfRxSduId);
/* Same semantics: read-and-consume */

Software Receive Filter & Data Length Check

When is SW filtering needed?

HRH typeHW filterSW filter needed?Why
FullCANExact single CAN ID matchNoHW already guarantees one specific ID
BasicCANID range (passes many IDs)YES [SWS_CANIF_00469]HRH may pass IDs not configured as Rx PDUs — need SW check

SW filter algorithms (implementation choice)

Linear search

Iterate through list of Rx PDU CanIDs for this HRH. Simple. O(n). Good for small lists.

Binary (table) search

Pre-sorted list. O(log n). Better for medium-size lists. Default for most tools.

Hash search

O(1) average. Best for large ID lists. Memory overhead for hash table.

Choice is implementation-specific. DaVinci/Vector tools typically generate binary search. The algorithm is not standardized — only the requirement to filter exists.

Range filtering rules

Configured asMethodConfig params
Exact IDDirect matchCanIfRxPduCanId
Upper/lower rangeLowerCanId ≤ received CanId ≤ UpperCanIdCanIfHrhRangeRxPduUpperCanId, CanIfHrhRangeRxPduLowerCanId
Mask + base(received CanId & Mask) == (BaseId & Mask)CanIfHrhRangeBaseId, CanIfHrhRangeMask
⚠️ Priority rule when ranges overlap [SWS_CANIF_00852]: (1) Exact single CanId always wins over range. (2) Smaller range wins over larger range. This ensures specific PDUs take priority over catch-all ranges.

Data Length Check

/* Configured: CanIfRxPduDataLength = 8 bytes */
/* Received: DLC = 6 → REJECTED (6 < 8) */
/* Received: DLC = 8 → ACCEPTED */
/* Received: DLC = 12 → ACCEPTED (DLC ≥ configured) [SWS_CANIF_00026] */

/* Why ≥ and not ==?
   CAN standard allows padding. A PDU of 8 bytes configured may arrive
   in a 12-byte CAN FD frame. CanIf accepts and passes received DLC
   to upper layer. Upper layer reads only what it expects. */
💡 DLC check can be disabled globally (CanIfPrivateDataLengthCheck=FALSE) or per-PDU (CanIfRxPduDataLengthCheck=FALSE). Individual disable only possible if global enable is TRUE [SWS_CANIF_00902].

Wakeup Handling — Detection & Validation

Two wakeup paths

CAN Controller wakeup

Controller detects bus activity while in sleep mode. CanDrv fires ISR. ISR calls EcuM callback → EcuM calls CanIf_CheckWakeup(WakeupSource). CanIf calls Can_CheckWakeup(ControllerId) on matching controllers.

CAN Transceiver wakeup

Transceiver detects WUF (Wake-Up Flag) via dedicated wakeup line or SPI. EcuM calls CanIf_CheckWakeup(WakeupSource). CanIf calls CanTrcv_CheckWakeup(TransceiverId) on matching transceivers.

Wakeup detection sequence

1
Wakeup event occurs (CAN bus activity or transceiver WUF pin)
2
EcuM calls CanIf_CheckWakeup(WakeupSource) [SWS_CANIF_00395]
3
CanIf queries Can_CheckWakeup() and/or CanTrcv_CheckWakeup() on mapped devices [SWS_CANIF_00395]
4
At least one returns E_OK → CanIf_CheckWakeup() returns E_OK [SWS_CANIF_00720]. All return E_NOT_OK → return E_NOT_OK [SWS_CANIF_00678].
5
EcuM sets wakeup event as PENDING → starts validation

Wakeup validation sequence

1
EcuM switches controller ON: CanIf_SetControllerMode(STARTED) + CanIf_SetTrcvMode(NORMAL)
2
EcuM calls CanIf_CheckValidation(WakeupSource) [SWS_CANIF_00179]
3
CanIf checks if first valid CanIf_RxIndication() was stored since controller restart [SWS_CANIF_00286]:
  • If CanIfPublicWakeupCheckValidByNM=TRUE: only NM messages count as valid first Rx
  • If FALSE: any configured Rx PDU counts as valid first Rx
4
If stored → User_ValidateWakeupEvent(WakeupSource) called → EcuM validates wakeup
5
If not stored → wakeup not yet validated → EcuM may put ECU back to sleep
🔴 Critical: PDU channel mode must NOT be ONLINE or TX_OFFLINE during wakeup validation [SWS_CANIF_00286 note]. Otherwise CanIf_RxIndication notifications to upper layers could be incorrectly triggered during validation phase. Keep mode in OFFLINE until validation complete.
⚠️ When controller transitions to CAN_CS_SLEEP, the stored first-Rx event is CLEARED [SWS_CANIF_00756]. Always fresh after each sleep-wakeup cycle.

CAN FD & CAN XL Support

CAN FD

Frame type indicator

CAN ID type encoding (2 MSBs of Can_IdType):

Bit 31 (MSB): IDE (Extended frame)
  0 = Standard CAN ID (11-bit)
  1 = Extended CAN ID (29-bit)
Bit 30: CAN FD flag
  0 = Classic CAN 2.0 frame
  1 = CAN FD frame

/* Examples: */
0x00000200 = Standard CAN 2.0, CAN ID 0x200
0x40000200 = Standard CAN FD, CAN ID 0x200
0x80001234 = Extended CAN 2.0, CAN ID 0x1234
0xC0001234 = Extended CAN FD, CAN ID 0x1234

CAN FD max payloads

Truncation: if CanIfTxPduTruncation=TRUE, CanIf silently truncates to PDU size instead of rejecting [SWS_CANIF_00894].

CAN XL filtering rules

1
Non-03h SDU type: match Acceptance Field, Priority ID, SDU Type, VCID in that order of specificity [SWS_CANIF_00951]
2
SDU Type 03h (tunneled): match Priority ID and VCID, then CAN ID filter (normal 2.0/FD filter applies to inner frame) [SWS_CANIF_00952]
3
Multiple matching configs: configured parameter wins over unconfigured. Match priority: SDU Type > Acceptance Field > VCID > Priority ID [SWS_CANIF_00956]
4
No matching config → frame dropped [SWS_CANIF_00955]
⚠️ CAN XL constraints: All nodes on same bus must support at least CAN FD [CAN XL spec]. Tx buffering in CanIf is NOT available for CAN XL [SWS_CANIF ch 7.11].

Partial Networking (PN)

Partial Networking allows ECUs to go to sleep while keeping a minimal receive path active for selective wakeup. CanIf implements the PnTxFilter to prevent all non-PN PDUs from being transmitted during the wakeup phase.

PnTxFilter state machine

ENABLED (blocks non-PN Tx)
→ TxConfirmation or RxIndication →
DISABLED (normal PDU mode applies)

PnTxFilter rules

EventPnTxFilter effectRequirement
CanIf_Init()ENABLED[SWS_CANIF_00863]
SetControllerMode(SLEEP)ENABLED[SWS_CANIF_00749]
SetPduMode(TX_OFFLINE) with PN enabledENABLED[SWS_CANIF_00878]
CanIf_TxConfirmation() calledDISABLED[SWS_CANIF_00751]
CanIf_RxIndication() called while filter ENABLEDDISABLED[SWS_CANIF_00896]
PnTxFilter ENABLED + Transmit request for non-PN PDUE_NOT_OK returned[SWS_CANIF_00750]
PnTxFilter ENABLED + Transmit request for PN PDU (CanIfTxPduPnFilterPdu=TRUE)PASSED to CAN Driver[SWS_CANIF_00750]
💡 PnTxFilter ensures that during the wakeup phase only PN-relevant messages (e.g., NM messages, wakeup confirmation PDUs) are sent. This prevents flooding the bus with non-essential traffic before the system is fully awake.
⚠️ PnTxFilter only has effect if at least one Tx PDU is configured as CanIfTxPduPnFilterPdu=TRUE [SWS_CANIF_00748]. If no PDU is configured, the filter has no effect regardless of enable/disable state.

Step-by-step configuration

Step 1 — CanIfPublicCfg (global feature flags)

1
Set CanIfPublicTxBuffering=TRUE if Tx buffering needed (BasicCAN with potential CAN_BUSY). Set CanIfPublicDevErrorDetect=TRUE for development builds.
2
Set CanIfPublicReadRxPduDataApi=TRUE if upper layers need CanIf_ReadRxPduData() polling access. Set CanIfPublicReadTxPduNotifyStatusApi and CanIfPublicReadRxPduNotifyStatusApi if polling notification status instead of callbacks.
3
Set CanIfPublicWakeupCheckValidSupport=TRUE for wakeup validation support. Set CanIfPublicWakeupCheckValidByNM=TRUE if only NM messages validate wakeup.
4
Set CanIfPublicPnSupport=TRUE and CanIfPublicTrcvPnEnable=TRUE for Partial Networking support.
5
Set CanIfSetBaudrateApi=TRUE if runtime baud rate switching needed. Set CanIfPublicMultipleDrvSupport=TRUE if multiple CAN Driver instances used.

Step 2 — CanIfPrivateCfg (internal settings)

1
Set CanIfPrivateDataLengthCheck=TRUE to enable global DLC check. Then per-PDU enable/disable via CanIfRxPduDataLengthCheck.
2
Set CanIfPrivateSoftwareFilterType: LINEAR, BINARY, TABLE, or INDEX. Determines SW receive filter algorithm.

Step 3 — CanIfCtrlDrvCfg (CAN Driver instance)

1
Create one CanIfCtrlDrvCfg per CAN Driver instance. Set CanIfCtrlDrvNameRef → link to CanDrv module (e.g., Can_99).

Step 4 — CanIfCtrlCfg (per CAN controller)

1
Create one CanIfCtrlCfg per physical CAN controller. Set CanIfCtrlId (0, 1, 2... unique CanIf ID). Set CanIfCtrlCanCtrlRef → reference to specific CanController in CanDrv configuration.
2
Set CanIfCtrlWakeupSupport=TRUE if this controller supports wakeup. Link to EcuM wakeup source via CanIfCtrlCanCtrlRef.

Step 5 — CanIfInitHohCfg (HOH definitions)

1
Create CanIfHthCfg for each TX mailbox. Set CanIfHthType: FULL_CAN (single ID) or BASIC_CAN (range). Set CanIfHthIdSymRef → reference to the HOH in CanDrv CanHardwareObject config.
2
Create CanIfHrhCfg for each RX mailbox. Set CanIfHrhType: FULL_CAN or BASIC_CAN. Set CanIfHrhSoftwareFilter=TRUE for BasicCAN HRH.
3
For BasicCAN HRH: add CanIfHrhRangeCfg sub-containers. Set CanIfHrhRangeRxPduLowerCanId and CanIfHrhRangeRxPduUpperCanId (or BaseId + Mask). Set CanIfHrhRangeRxPduRangeCanIdType: STANDARD or EXTENDED.

Step 6 — CanIfTxPduCfg (Tx PDU configuration)

1
Create one CanIfTxPduCfg per Tx PDU. Set CanIfTxPduId (unique CanIf handle). Set CanIfTxPduCanId (the CAN ID to use). Set CanIfTxPduCanIdType: STANDARD_CAN (11-bit), EXTENDED_CAN (29-bit), STANDARD_FD_CAN, EXTENDED_FD_CAN.
2
Set CanIfTxPduBufferRef → reference to a CanIfBufferCfg for Tx buffering. Set buffer size = 0 if no buffering needed (still required as HTH reference).
3
Set CanIfTxPduUserTxConfirmationUL: CAN_NM, CAN_TP, PDUR, J1939TP etc. + set CanIfTxPduUserTxConfirmationName to the callback function name of the upper layer.
4
Set CanIfTxPduTriggerTransmit=TRUE and configure callback name if trigger-transmit model is used.
5
For dynamic Tx PDU: set CanIfTxPduType=DYNAMIC. For Partial Networking PDUs: set CanIfTxPduPnFilterPdu=TRUE.

Step 7 — CanIfRxPduCfg (Rx PDU configuration)

1
Create one CanIfRxPduCfg per Rx PDU. Set CanIfRxPduCanId, CanIfRxPduCanIdType, CanIfRxPduDataLength (expected DLC). Set CanIfRxPduHrhIdRef → link to the CanIfHrhCfg that receives this frame.
2
Set CanIfRxPduUserRxIndicationUL: CAN_NM, CAN_TP, PDUR, etc. + set CanIfRxPduUserRxIndicationName to the upper layer callback.
3
Set CanIfRxPduReadData=TRUE if this PDU should have an internal Rx buffer for CanIf_ReadRxPduData(). Set CanIfRxPduReadNotifyStatus=TRUE for polling notification support.

Step 8 — CanIfDispatchCfg (callback routing)

1
Set CanIfBusOffNotification name → typically CanSM_ControllerBusOff. Set CanIfWakeUpValidation name → EcuM_ValidateWakeupEvent. Set CanIfCtrlModeIndication → CanSM_ControllerModeIndication. Set CanIfTrcvModeIndication → CanSM_TransceiverModeIndication.

Common DaVinci configuration mistakes

🔴 Missing CanIfHrhRangeCfg for BasicCAN HRH: BasicCAN HRH must have SW filter ranges configured. Without ranges, CanIf has no way to determine which Rx PDU received. All frames on that HRH will be silently dropped.
🔴 Wrong CanIfTxPduCanIdType: Sending a 29-bit extended ID with type=STANDARD_CAN → CanIf sets wrong IDE bit → CAN ID on bus will be incorrect. Always match CanIdType to the actual ID format used in the network description.
🔴 CanIfBufferCfg missing with BasicCAN HTH: All Tx PDUs must reference a CanIfBufferCfg (even if buffer size=0). Missing reference → DET CANIF_E_PARAM_HOH errors at runtime.
⚠️ CanIfRxPduDataLength mismatch: If configured DLC > actual received DLC → DLC check fails → frame dropped silently. If upper layer never receives frames, check this first: compare CanIfRxPduDataLength with what the network description says the sender transmits.
⚠️ CanIf_SetPduMode() before CanIf_SetControllerMode(STARTED): Returns E_NOT_OK [SWS_CANIF_00874]. Always ensure controller is STARTED before setting PDU mode to ONLINE. CanSM manages this order correctly — if calling manually, respect the sequence.
💡 Verify ControllerId mapping: DaVinci assigns CanIf ControllerId 0..n automatically based on the order of CanIfCtrlCfg containers. Make sure the CanIfCtrlId matches what CanSM expects. A misalignment means CanSM sends mode changes to the wrong controller.

Integration Testing & Debugging

Common integration bugs

Bug: No Tx frames on bus — E_OK returned but nothing sent

Root causes: (1) PDU channel mode is not ONLINE — call CanIf_GetPduMode(). (2) Controller not in STARTED — call CanIf_GetControllerMode(). (3) Transceiver in Standby/Sleep — call CanIf_GetTrcvMode(). (4) PN filter enabled and PDU is not PN PDU — check CanIf_GetTxConfirmationState(). Debug: Enable DET and watch for CANIF_E_STOPPED runtime error on CanIf_Transmit.

Bug: Tx returns E_NOT_OK persistently

Root causes: (1) Controller in STOPPED — mode not changed correctly. (2) Tx PDU ID not configured (CANIF_E_INVALID_TXPDUID DET). (3) BasicCAN HW busy + no Tx buffer configured. (4) DLC exceeds max for frame type (CANIF_E_DATA_LENGTH_MISMATCH). Debug: Check DET error buffer. Use CanIf_GetControllerErrorState() to check controller health.

Bug: Rx frames arrive (visible on oscilloscope/analyzer) but no callback fires

Root causes: (1) PDU channel mode OFFLINE — RxIndication blocked. (2) BasicCAN SW filter mismatch — CanId not in HrhRangeCfg range OR no range configured. (3) DLC check failure — received DLC less than configured. (4) Wrong CanIfRxPduCanIdType (extended vs standard). Debug: Enable DET, check CANIF_E_INVALID_DATA_LENGTH. Verify CAN ID type bits (MSB of received CanId).

Bug: Incorrect CAN ID on bus

Root causes: (1) CanIfTxPduCanIdType wrong (standard vs extended). (2) IDE bit not set correctly → CanId MSBs wrong. (3) Dynamic Tx PDU: CanIf_SetDynamicTxId() not called before Transmit. (4) For CAN XL tunneled frame: CAN ID should be in Acceptance Field, not Priority ID. Debug: Capture CAN traffic with analyzer. Compare raw CAN frame ID with configured CanIfTxPduCanId.

Bug: BusOff occurs, controller never recovers

Root causes: (1) CanSM not notified — check CanIfDispatchCfg CanIfBusOffNotification function name matches CanSM callback exactly. (2) CanSM recovery timing too short — TxConfirmation never received before re-entering BusOff. (3) CanIf_GetTxConfirmationState() not configured (CanIfPublicTxConfirmPollingSupport=FALSE) — CanSM cannot verify bus is active. Debug: Check CanIf_GetControllerErrorState() during/after BusOff. Monitor TxErrorCounter with CanIf_GetControllerTxErrorCounter().

Bug: Wakeup validation always fails (ECU keeps sleeping)

Root causes: (1) PDU channel mode set to ONLINE before calling CheckValidation — upper layers receive indications, but CanIf validation window already used. MUST keep PDU channel OFFLINE during validation. (2) CanIfPublicWakeupCheckValidByNM=TRUE but no NM frames sent from other nodes. (3) CanIf_CheckValidation() called before controller reaches STARTED + first RxIndication. Debug: Check sequence: SLEEP → STARTED → first RxIndication → CheckValidation() → User_ValidateWakeupEvent.
