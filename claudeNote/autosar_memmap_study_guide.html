AUTOSAR Memory Mapping — Complete Study Guide

R24-11 · SWS Document ID 128 · Automotive Integration, Testing & DaVinci Configuration

What is AUTOSAR Memory Mapping?

Memory Mapping is NOT a runtime module — it has no API, no state machine, no ISR, no runtime behaviour. It is a compile-time and link-time mechanism that uses standardised preprocessor macros called Memory Allocation Keywords (MAKWs) to assign code, variables, and constants to specific memory sections via compiler #pragma directives — in a compiler-independent, portable way. Every AUTOSAR BSW module and SWC must use this mechanism for all file-scope symbols.

MemMap controls only symbol → section assignment. The linker script controls section → address assignment. MemMap does NOT set any memory addresses.

Why is it needed? — 6 core use cases

1. Eliminate RAM alignment waste

Linkers insert padding between differently-aligned objects. By grouping uint8 in _8 sections, uint16 in _16 sections, etc., waste is eliminated. Especially important on small MCUs with limited RAM.

2. POWER_ON_CLEARED (NvM RAM mirrors)

NvM RAM mirrors must NOT be zero-cleared on warm reset (watchdog, software reset, external reset). Only zeroed on cold power-on. This is physically impossible without a dedicated memory section with special startup code.

3. Internal vs External Flash

High-frequency code → internal flash (fast, low latency). Rarely-called code → external flash (slower, but larger). Memory mapping allows this per-module assignment without touching source code.

4. Bootloader + Application reuse

Same C source compiled for both bootloader and application. Different MemMap header per build scenario → same code linked to different addresses. Eliminates code duplication and #ifdef spaghetti.

5. Memory Protection (MPU / ASIL)

Safety systems need separate MPU-protected address partitions per ASIL level. ASIL_B, ASIL_D qualifiers in MAKW allow sections to be placed in separate linker regions → hardware MPU can enforce read/write/execute access control.

6. Multi-core LOCAL allocation

LOCAL coreScope restricts a symbol to one core's local TCRAM/DTCM. Avoids cross-core memory access penalties (NUMA). Any other core accessing LOCAL memory → MPU fault. GLOBAL (default) = accessible from any core.

The three actors

BSW / SWC Developer

Wraps ALL file-scope variables, constants, functions with START/STOP MAKW pairs. Defines MemorySection entries in BSW Module Description (ARXML). Does NOT decide which physical address.

Integrator (DaVinci)

Configures MemMap module. Maps SwAddrMethod → MemMapAddressingModeSet → actual #pragma strings. Generates {Mip}_MemMap.h. Decides physical linker section for each MAKW.

Linker Script

Assigns named linker sections (created by #pragmas) to physical address ranges. MemMap does NOT control addresses — the linker script does.

The 5-step usage mechanism (SWS_MemMap_00003)

Every file-scope symbol wrapped by Memory Mapping MUST follow exactly this pattern. No preprocessor code between steps 1–5. No nesting. No MemMap include inside a function body (SWS_MemMap_00023).

/* Step 1: Define the START keyword (preprocessor symbol) */
#define EEP_START_SEC_VAR_INIT_16
/* Step 2: Include the MemMap header — processes the define, emits pragmas, undefines it */
#include "Eep_MemMap.h"
/* Step 3: Declare or define symbols belonging to this section */
static uint16 EepTimer = 100;
static uint16 EepRemainingBytes = 16;
/* Step 4: Define the STOP keyword */
#define EEP_STOP_SEC_VAR_INIT_16
/* Step 5: Include again — processes STOP, restores default pragmas, undefines STOP keyword */
#include "Eep_MemMap.h"

How the MemMap header file works internally (Example 7.4 from spec)

/* Eep_MemMap.h — simplified internal structure */
#define MEMMAP_ERROR   /* error flag: will fire #error if no MAKW matched */
 
#if defined EEP_START_SEC_VAR_INIT_16
  #undef MEMMAP_ERROR
  #undef EEP_START_SEC_VAR_INIT_16        /* SWS_MemMap_00011: always undef MAKW */
  #ifndef MEMMAP_SEQUENCE_OPEN
    #pragma section neardata "eep_var_16"  /* compiler-specific pragma — integrator fills this */
    #pragma section nearbss  "eep_var_16"
    #define MEMMAP_SEQUENCE_OPEN
    #define MEMMAP_SEQUENCE_OPEN_EEP_SEC_VAR_INIT_16
  #else
    #error "START must not follow START — nesting not allowed!"
  #endif
 
#elif defined EEP_STOP_SEC_VAR_INIT_16
  #undef MEMMAP_ERROR
  #undef EEP_STOP_SEC_VAR_INIT_16
  #ifdef MEMMAP_SEQUENCE_OPEN
    #ifdef MEMMAP_SEQUENCE_OPEN_EEP_SEC_VAR_INIT_16
      #pragma section neardata "unhandled"   /* restore: symbols here → linker error (catch-all) */
      #pragma section nearbss  "unhandled"
      #undef MEMMAP_SEQUENCE_OPEN
      #undef MEMMAP_SEQUENCE_OPEN_EEP_SEC_VAR_INIT_16
    #else
      #error "STOP doesn't match the open START!"
    #endif
  #else
    #error "STOP without matching START!"
  #endif
#endif
 
/* ... more #elif for other MAKWs ... */
 
#ifdef MEMMAP_ERROR
  #undef MEMMAP_ERROR
  #error "Undefined MAKW! Check source or regenerate MemMap header file!"
#endif

Rules enforced at compile time

SWS_MemMap_00007 — Sequence validation

START→START = #error (nesting). STOP without START = #error. Wrong STOP name (mismatched pair) = #error. Unknown MAKW = #error (MEMMAP_ERROR). All enforced by preprocessor — zero runtime cost.

SWS_MemMap_00011 — Auto-undef

Header always #undefines the MAKW after processing. After #include "Eep_MemMap.h", the EEP_START_SEC_VAR_INIT_16 symbol is guaranteed gone — no double-processing risk.

SWS_MemMap_00016 — No nesting, one type

Cannot open a VAR section then open a CODE section without closing VAR first. Only one section active at a time. Only one of code/var/const type at a time.

SWS_MemMap_00023 — No include in function body

MemMap headers MUST NOT be #included inside a function body {}. Some compilers don't support #pragma inside functions. Move function-local statics to file scope if they need mapping.

DaVinci Memory Mapping configuration

Navigate: BSW → MemMap. Configuration is pre-compile only (no post-build variant). DaVinci generates the MemMap header files containing the MAKWs and their #pragma implementations.

Container hierarchy overview

ContainerMultiplicityPurpose
MemMap0..1Root module configuration container
MemMapAddressingModeSet0..*Defines a set of #pragma instructions for one addressing strategy (e.g. "near-addressed initialized variables"). Groups related AddressingModes together.
MemMapAddressingMode1..* (per ModeSet)The actual #pragma strings for START and STOP, plus alignment selector. One entry per alignment value (8, 16, 32 etc.).
MemMapAllocation0..*Contains the mapping rules that connect MemorySections to AddressingModeSets.
MemMapGenericMapping0..* (in Allocation)Maps ALL MemorySections sharing a given SwAddrMethod to one AddressingModeSet. The bulk configuration approach. Can be filtered by MemMapMappingSelector regex.
MemMapSectionSpecificMapping0..* (in Allocation)Maps ONE specific MemorySection to an AddressingModeSet. OVERRIDES GenericMapping for that section. Used for special cases.
MemMapMappingSelector0..* (in Allocation)Optional: regex filter on PREFIX. Restricts a GenericMapping to specific module prefixes (e.g. only ADC_AUTOSCANKERNEL_*).

MemMapAddressingModeSet — key parameters

MemMapSupportedSectionType0..* enumeration: MEMMAP_SECTION_TYPE_VAR, _CODE, _CONST, _CONFIG_DATA, _CAL_PRM, _EXCLUDE_FROM_FLASH. Constrains which SwAddrMethod.sectionType values can use this ModeSet in GenericMapping. Used as validation/filter.
MemMapSupportedSectionInitializationPolicy0..* string: INIT, CLEARED, POWER-ON-CLEARED. Constrains which SwAddrMethod.sectionInitializationPolicy values can map to this ModeSet.
MemMapSupportedAddressingMethodOption0..* string: e.g. "safetyAsilB", "coreLocal", "configClassPostBuild". Must match SwAddrMethod.option attribute. Ensures only correctly-qualified SwAddrMethods map here.
MemMapSupportedMemoryAllocationKeywordPolicyADDR_METHOD_SHORT_NAME: MAKW uses SwAddrMethod shortName only (no alignment suffix). ADDR_METHOD_SHORT_NAME_AND_ALIGNMENT: MAKW appends alignment suffix (_8, _16, _32 etc.) → separate sections per alignment. Required for efficient RAM packing.

MemMapAddressingMode — the actual #pragma strings

MemMapAddressingModeStartMulti-line string: exact #pragma sequence for section start. Example (Altium CTC compiler): #pragma section fardata "ram.partition_asil_b.32"\n#pragma section farbss "ram.partition_asil_b.32"\n#pragma clear
MemMapAddressingModeStopMulti-line string: #pragmas to restore default sections after STOP. Best practice: redirect to "unhandled" or "illegal" named section → linker error if any symbol accidentally placed here.
MemMapAlignmentSelector1..* string matching AlignmentType (8, 16, 32, 64, BOOLEAN, PTR, UNSPECIFIED). This AddressingMode entry applies to MemorySections with matching alignment attribute. One AddressingMode can match multiple alignments (e.g. "8" and "UNSPECIFIED").

GenericMapping vs SectionSpecificMapping

MemMapGenericMapping — bulk rule

Links SwAddrMethod → AddressingModeSet. Applies to ALL MemorySections in the project that reference this SwAddrMethod. Most sections handled here. Key params: MemMapSwAddressMethodRef (→ SwAddrMethod), MemMapAddressingModeSetRef (→ ModeSet), optional MemMapMappingSelectorRef (regex filter on PREFIX).

MemMapSectionSpecificMapping — individual override

Links one specific MemorySection → AddressingModeSet. OVERRIDES any GenericMapping for this section. Key params: MemMapMemorySectionRef (→ exact MemorySection from ARXML), MemMapAddressingModeSetRef (→ ModeSet). Use for: ASIL-D critical sections, specific callouts, boot ROM sections.

Step-by-step DaVinci workflow

Step 1: Understand your target memory layout
Which compiler and what #pragma syntax? Which linker sections exist? Which physical memories (SRAM0, SRAM1, PFLASH, DFLASH, ITCM, DTCM)? Match compiler pragma naming to linker script section names before starting.

Step 2: Create MemMapAddressingModeSets
One set per addressing strategy. E.g.: VAR_FAST_INIT (near-addressed init vars), CODE_INTERNAL (fast internal flash), CODE_EXTERNAL (slow external flash), VAR_ASIL_D_INIT (ASIL-D variables), VAR_POWER_ON_CLEARED (NvM mirrors). Set MemMapSupportedSectionType and MemMapSupportedSectionInitializationPolicy as filters.

Step 3: Add MemMapAddressingMode entries per ModeSet
One per alignment (8, 16, 32 etc.). Fill MemMapAddressingModeStart with the compiler's #pragma to open the linker section. Fill MemMapAddressingModeStop with pragmas that redirect to "unhandled" section. Set MemMapAlignmentSelector.

Step 4: Create MemMapAllocation → MemMapGenericMapping entries
Link each SwAddrMethod (from BSW ARXML descriptions) to the correct AddressingModeSet. Covers the bulk of all MAKWs in the project.

Step 5: Add MemMapSectionSpecificMapping for special cases
Override GenericMapping for specific MemorySections that need non-standard placement (ASIL-D isolated sections, boot ROM, specific callouts).

Step 6: Use MemMapMappingSelector if needed
Add regex filter on PREFIX to restrict a GenericMapping. Example: ADC_AUTOSCANKERNEL_.* regex ensures only the ADC kernel allocatable part uses the LOCAL coreScope ModeSet.

Step 7: Generate and verify
DaVinci generates {Mip}_MemMap.h. Compile all modules. Examine linker .map file. Verify every symbol appears in the expected section. Check no symbols in "unhandled" sections (those should cause linker errors).

Common DaVinci configuration mistakes

Most common integration errors:
• MemMapSupportedSectionType mismatch: CODE ModeSet linked to VAR SwAddrMethod → no matching ModeSet found → MAKW not generated
• MemMapAlignmentSelector missing values: ModeSet only has selector "32" but module has _16 sections → MAKW not generated for _16
• MemMapSectionSpecificMapping not configured for a safety-critical section → it inherits the generic (possibly QM) mapping → ASIL partition violation
• Forgetting to add post-build config section to a separately flashable linker block → OTA update reflashes entire application instead of just config
• MemMapAddressingModeStop points to "unhandled" but linker script doesn't define that section → linker error (this is actually CORRECT behaviour — intended catch-all)

Linker .map file verification

1. Section assignment check — Open the linker .map file. Search for your module's function/variable names. Verify they appear in the expected linker section name. For example, a function wrapped with EEP_START_SEC_CODE should appear under section eep_code (or whatever you configured in MemMapAddressingModeStart).

2. Catch-all "unhandled" section — Any symbols in your "unhandled" or "illegal" section indicate a missing MAKW wrapper. The linker should ideally error on these sections being non-empty. If not, add them to a MEMORY region with no space in the linker script.

3. ASIL partition separation — Verify QM and ASIL_D symbols are in completely separate address ranges. The MPU configuration should make ASIL_D ranges inaccessible from QM OS Applications. Test by deliberate cross-partition access (expect MPU exception).

4. POWER_ON_CLEARED verification — Set a known pattern (0xDEADBEEF) in a POWER_ON_CLEARED variable using a debugger. Trigger a watchdog reset (warm reset). Connect debugger again — pattern must still be there. Then trigger a power-cycle (cold reset) — pattern should now be 0x00000000.

5. LOCAL coreScope verification — From a task running on core 1, attempt to access a LOCAL variable assigned to core 0's DTCM. This should trigger a memory protection exception. If it doesn't, the MPU is not configured correctly for LOCAL sections.

6. POST_BUILD config block separation — Verify CONFIG_DATA_POSTBUILD sections are in a separately flashable Flash block. Reflash only that block and verify the application runs with the new configuration (no need to reflash the entire application). Critical for OTA update strategies.

Function-level tracing (R24-11 new — SWS_MemMap_00047)

For function-level performance tracing, the MemMap header can append the symbol name and/or object filename to the linker section name. This groups all relevant symbols (functions, runnables, tasks) into a contiguous traceable memory block. The locator file (linker script) can then identify this block precisely for instrumentation tools. Warning: causes significant build time increase (unique section per symbol). Use only when tracing is active.
