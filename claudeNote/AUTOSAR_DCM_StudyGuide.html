What is DCM?

The Diagnostic Communication Manager (Dcm) is an AUTOSAR Basic Software (BSW) module in the Communication Services layer. It acts as the bridge between an external diagnostic tool (OBD scanner, HiL tester, ODX-based tool) and the ECU's internal application software.

DCM handles OSI layers 5–7: session layer (L5), presentation layer (L6), and application layer (L7). Network-specific concerns (CAN TP, DoIP, LIN) live below DCM in the stack and are handled by the PduR / CanTp / DoIP modules.

Key insight: DCM is network-independent. The same DCM code handles requests whether they arrive over CAN, Ethernet, or FlexRay. The transport adaptation happens in lower layers.

Why does DCM exist?

During development, manufacturing, and service, diagnostic tools need to:

Read

DTCs, DID values, sensor data

Write

Calibration data, NVM config

Control

ECU reset, I/O overrides

Update

Flash firmware (via programming session)

DCM Internal Architecture

Three submodules: DSL · DSD · DSP

Submodule Overview

DCM is internally divided into three submodules. Each has a specific responsibility and they communicate with each other in a defined pipeline:

Submodule Responsibilities

SubmoduleFull NameResponsibility
DSL Diagnostic Session Layer Manages sessions, security levels, timers (P2/S3), ComM mode, authentication state. Routes Rx from PduR to DSD; Tx from DSD to PduR.
DSD Diagnostic Service Dispatcher Validates the incoming SID. Checks session/security access conditions. Dispatches to the correct DSP handler. Notifies manufacturer/supplier.
DSP Diagnostic Service Processing Implements all UDS/OBD service processors. Interfaces to DEM (DTC), App SWCs (DID data), BswM (mode switches), NvM. Builds the positive response.

Important Rules (SWS Requirements)

[SWS_Dcm_00034] At initialization, session = DefaultSession (0x01)
[SWS_Dcm_00311] Session switch happens on Tx confirmation (after positive response sent), NOT on request received
[SWS_Dcm_00139] Security level resets to LOCKED (0x00) on any transition from non-default → any session
[SWS_Dcm_01667] Concurrent TesterPresent (3E 80 functional) only resets S3 timer on the connection that entered non-default session

Session-Specific Access Control

Every UDS service and sub-function can be restricted to specific sessions in the DCM configuration. This is configured in DcmDsdService → DcmDsdSubService → DcmDsdSubServiceSessionModeRule or DcmDsdServiceSessionModeRule.

Example: ReadMemoryByAddress (0x23) restricted to Extended + Programming: DcmDsdService: DcmDsdServiceId: 0x23 DcmDsdSubServiceSessionModeRule → DcmModeRule: conditions: ExtendedSession OR ProgrammingSession

Security Level State Machine

UDS Service 0x27 SecurityAccess — unlock gated services

State Diagram

Security Access Flow (0x27)

  1. Tester sends Request Seed: 27 01 (or 03, 05… odd sub-functions)
  2. DCM calls application: Xxx_GetSeed(securityAccessType, seedLen, seed) and returns seed in positive response: 67 01 [seed bytes]
  3. Tester computes key using OEM algorithm and sends Send Key: 27 02 [key bytes] (even sub-function = requestSeed + 1)
  4. DCM calls: Xxx_CompareKey(securityAccessType, key). If OK → unlock. If fail → attempt counter incremented.
  5. On success: 67 02 positive response. Security level set via DslInternal_SetSecurityLevel()

Security Attempt Counter

NRC 0x36 (exceededNumberOfAttempts): If attempts ≥ DcmDspSecurityNumAttDelay, DCM locks out for DcmDspSecurityDelayTime milliseconds and responds with NRC 0x37 (requiredTimeDelayNotExpired) until timer expires.
Config paramPurpose
DcmDspSecurityNumAttDelayMax failed key attempts before lockout
DcmDspSecurityDelayTimeLockout duration in ms after max attempts
DcmDspSecurityDelayTimeOnBootLockout duration on boot if counter was at max (NVM-backed counter)
DcmDspSecurityAttemptCounterEnabledEnable NVM-backed attempt counter persistence

Key Implementation Rules

[SWS_Dcm_00033] Security level is initialized to 0x00 (LOCKED) at Dcm_Init
[SWS_Dcm_00139] Security level resets to LOCKED on any non-default → any session transition
[SWS_Dcm_01329] Every security level change triggers mode switch of DcmSecurityAccess mode declaration group

Persistent Authentication (Production Use Case)

In production, ECUs power-cycle frequently. DcmDspAuthenticationPersistStateModeRuleRef allows the authenticated state to persist across power cycles — useful for end-of-line test where re-authentication every cycle is too slow.

If persistence is enabled and the mode rule evaluates TRUE, DCM restores the authentication state including role and whitelist from NVM at boot.

API: Dcm_SetDeauthenticatedRole

Allows an application SW-Component to overwrite the deauthenticated role without going through certificate-based service 0x29. Only active in deauthenticated state. Not persisted. Overwritten when connection becomes authenticated.

DSL — Diagnostic Session Layer

The gateway: routes messages, manages state, controls timing

DSL Responsibilities

Request Handling

Forward PduR → DSD; block during processing

Response Handling

Forward DSD → PduR; periodic Tx; ROE

Timing

P2/P2* timer; NRC 0x78 if exceeded

State

Session + Security + Auth per connection

ComM

Active/Inactive diagnostic signaling

Multi-client

Protocol resources per DcmDslProtocolRow

Rx Processing Pipeline

  1. PduR calls Dcm_StartOfReception(RxPduId, info, TpSduLength, bufferSizePtr) — DCM checks buffer space, rejects if overflow
  2. PduR calls Dcm_CopyRxData(RxPduId, info, bufferSizePtr) — DCM copies data into Rx buffer
  3. PduR calls Dcm_TpRxIndication(RxPduId, result) — only if result=E_OK does DSL forward to DSD [SWS_Dcm_00111]
  4. DSL blocks the DcmPduId until Tx confirmation arrives [SWS_Dcm_00241]

Tx Response Pipeline

  1. DSD signals DSL: response ready
  2. DSL calls PduR_DcmTransmit(TxPduId, PduInfoPtr)
  3. PduR calls Dcm_CopyTxData() to retrieve response bytes
  4. PduR calls Dcm_TpTxConfirmation(TxPduId, result) — DSL unlocks buffer, stops timers

Config: DcmDslDiagRespMaxNumRespPend limits total NRC 0x78 count. If not configured → infinite (application deadlock risk!)

Concurrent TesterPresent (Keep-Alive)

A functionally-addressed 3E 80 resets the S3 timer without affecting ongoing request processing. DCM tracks which connection entered non-default session and only accepts keep-alive from that same connection [SWS_Dcm_01667].

Periodic Transmission (0x2A)

ReadDataByPeriodicIdentifier (0x2A) sets up the ECU to periodically send DID values. Key rules:

[SWS_Dcm_00122] Periodic Tx uses a separate protocol + separate buffer
[SWS_Dcm_00123] Periodic Tx only in Full Communication Mode
[SWS_Dcm_00125] Events while not in FullCom are discarded — NOT queued

Generic Connection Handling

For DoIP and CAN 29-bit addressing, PDUs carry MetaData items SOURCE_ADDRESS_16 and TARGET_ADDRESS_16. DCM stores the source address and uses it as target address in the response [SWS_Dcm_00848, SWS_Dcm_00849].

DSD — Diagnostic Service Dispatcher

Validates, authorizes, and routes each diagnostic service request

DSD Processing Flow

  1. Receive service request from DSL
  2. SID lookup: Search configured DcmDsdServiceTable for matching SID. If not found → NRC 0x11 (serviceNotSupported)
  3. Session check: Is the service allowed in current session? If not → NRC 0x7F (serviceNotSupportedInActiveSession)
  4. Security check: Is required security level active? If not → NRC 0x33 (securityAccessDenied)
  5. Manufacturer notification: Call Xxx__Indication (if configured). Can override NRC.
  6. Dispatch to DSP: Call the appropriate DSP handler
  7. Supplier notification: After DSP returns, notify supplier (if configured)
  8. Response assembly: Forward to DSL for Tx

Configuration Structure

DcmDsd └── DcmDsdServiceTable (multiple tables supported) └── DcmDsdService (one per SID) ├── DcmDsdServiceId: 0x22 ├── DcmDsdSidTabServiceIdModeRule (session restrictions) ├── DcmDsdSidTabSecurityLevelRef (security restrictions) └── DcmDsdSubService (sub-functions per service) ├── DcmDsdSubServiceId: 0x01 ├── DcmDsdSubServiceSessionModeRule └── DcmDsdSubServiceSecurityLevelRef

Manufacturer / Supplier Notification

These are callouts giving OEMs/Tier-1s a chance to intercept processing. Configured via DcmDsdServiceRequestManufacturerNotification and DcmDsdServiceRequestSupplierNotification.

Use manufacturer notification to implement custom authorization logic (e.g., allow service 0x2E only if a specific DID value is set). The callout can return a custom NRC to abort processing.

Sub-function Validation

For services with sub-functions, DSD checks if the subfunction byte is configured. If not → NRC 0x12 (subFunctionNotSupported). The suppressPosRspMsgIndicationBit (bit 7 of subfn) is handled here — if set, positive response is suppressed but negative responses are still sent.

DID Access via RTE — USE_DATA_SYNCH_CLIENT_SERVER

For ReadDataByIdentifier (0x22) reading DID 0x1234: - DSP calls: Rte_Call_DataServices_DID0x1234_ReadData(Data, DataLength) - Application fills the buffer and returns E_OK - DSP assembles: 62 12 34 [data bytes] Port interface types (DcmDspDataUsePort): USE_DATA_SYNCH_FNC → Xxx_ReadData(Data, DataLength) — synchronous USE_DATA_ASYNCH_FNC → Xxx_ReadData(OpStatus, Data, DataLength) — async via OpStatus USE_DATA_ASYNCH_CLIENT_SERVER → via RTE Client-Server call USE_DATA_SENDER_RECEIVER → S/R port, direct signal read USE_ECU_SIGNAL → direct IoHwAb signal

DEM Integration for Service 0x14/0x19

  1. Call Dem_SelectDTC(ClientId, DTC, DTCFormat, DTCOrigin) to set filter
  2. Call Dem_SetDTCFilter() with mask parameters
  3. Call Dem_GetNumberOfFilteredDTC() or Dem_GetNextFilteredDTC() to read results
  4. For extended data: call Dem_DisableDTCRecordUpdate() before reading, Dem_EnableDTCRecordUpdate() after
  5. DEM may return DEM_PENDING → DSP re-tries on next Dcm_MainFunction() cycle

Key Services — Deep Dive

0x10 — DiagnosticSessionControl

Switches the ECU diagnostic session. Every session change resets security to LOCKED.

Request: 10 03 (switch to Extended Session) Response: 50 03 [P2ServerMax_high] [P2ServerMax_low] [P2StarServerMax_high] [P2StarServerMax_low] NRC 0x12: subfunction not configured (DcmDspSessionLevel missing)
Session switch is committed AFTER positive response Tx confirmation, not on request receipt.

0x11 — ECUReset

Sub-fnValueReset TypeMode Switch
hardReset0x01Power cycle equivalentDcmEcuReset → HARD, then EXECUTE
keyOffOnReset0x02Key cycleDcmEcuReset → KEYONOFF, then EXECUTE
softReset0x03Software restartDcmEcuReset → SOFT, then EXECUTE
enableRapidPowerShutDown0x04Short power-down timeDcmRapidPowerShutDown → ENABLE
BswM carries out the actual reset via action list. DCM ignores all further requests after sending 0x11 positive response [SWS_Dcm_00834].

0x19 — ReadDTCInformation

The most complex service — 20+ sub-functions for various DTC query modes. Key sub-functions:

Sub-fnNameReturns
0x01reportNumberOfDTCByStatusMaskCount of DTCs matching mask
0x02reportDTCByStatusMaskDTC list + status bytes
0x03reportDTCSnapshotIdentificationSnapshot/freeze frame IDs
0x04reportDTCSnapshotRecordByDTCNumberFreeze frame data
0x06reportDTCExtendedDataRecordByDTCNumberExtended data (counters, timestamps)
0x09reportDTCWithPermanentStatusPermanent DTCs
0x14reportDTCFaultDetectionCounterDebounce counter per DTC

0x22 — ReadDataByIdentifier (RDBI)

Request: 22 F1 90 (read DID 0xF190 = VIN) Response: 62 F1 90 [17 VIN bytes] Multi-DID: 22 F1 90 F1 86 (read VIN + ActiveSession in one request) Response: 62 F1 90 [17 bytes] F1 86 [1 byte]
DCM supports reading multiple DIDs in one request. The response is assembled in order. If any DID fails: NRC for that individual DID in AUTOSAR extended mode, or fail entire request in strict mode.

0x2E — WriteDataByIdentifier (WDBI)

Request: 2E F1 90 [17 VIN bytes] Response: 6E F1 90 NRC 0x31: DID not found or not writable NRC 0x33: Security level too low NRC 0x22: Conditions not correct (app returned E_NOT_OK)

0x31 — RoutineControl

Start: 31 01 [RID_H] [RID_L] [optional params] Stop: 31 02 [RID_H] [RID_L] [optional params] Result: 31 03 [RID_H] [RID_L] Positive: 71 [subfn] [RID_H] [RID_L] [result bytes]

0x85 — ControlDTCSetting

Enable DTC storage: 85 01 Disable DTC storage: 85 02 Response: C5 [subfn] NRC 0x22: App conditions not correct
On session return to Default, DCM must re-enable DTC setting. Integrate with BswM to handle this automatically.

When processing takes longer than P2ServerMax allows

Vector DaVinci Configuration Guide

Step-by-step setup of AUTOSAR DCM in Vector DaVinci Developer / Configurator Pro

Overall Configuration Workflow

  1. Add DCM module to your project in DaVinci. Go to Modules → Add → Dcm
  2. Configure DcmConfigSet — top-level container for all DCM config
  3. Configure DcmDsl — buffers, protocols, connections, PDU references
  4. Configure DcmDsd — service tables, session/security restrictions
  5. Configure DcmDsp — DIDs, routines, security levels, sessions, DTC parameters
  6. Link to PduR — configure DcmDslProtocolRxPduRef and TxPduRef matching PduR Rx/Tx PDUs
  7. Generate code — DaVinci generates Dcm_Cfg.h, Dcm_Lcfg.c, and callout stubs
  8. Implement callouts — fill in the generated stub functions for DID read/write, security access, routines

Linking PDUs (Critical Step)

The PDU links between DCM and PduR are the most common integration error. Each DcmDslProtocolRxPduRef must point to a matching PduR Rx PDU, and the DcmDslProtocolTxPduRef to the PduR Tx PDU. These are also linked in the CanTp/DoIP config.
Correct PDU chain: CanTp Rx PDU → PduR DcmRxPdu → DCM StartOfReception DCM TxPdu → PduR → CanTp Tx PDU In DaVinci: DcmDslProtocolRxPduRef = /ActiveEcuC/PduR/DcmRxPdu_UDS_Phys DcmDslProtocolTxPduRef = /ActiveEcuC/PduR/DcmTxPdu_UDS

Generated Callout Stubs

After generation, DaVinci creates stub files. You must implement these:

/* DID Read callout */ Std_ReturnType App_Read_VIN(uint8 *Data, uint16 DataLength) { /* Copy 17-byte VIN into Data buffer */ memcpy(Data, &g_VIN[0], 17); return E_OK; } /* Security seed callout */ Std_ReturnType App_GetSeed( Dcm_OpStatusType OpStatus, uint8 *Seed, uint16 *SeedLen, Dcm_NegativeResponseCodeType *ErrorCode) { if (OpStatus == DCM_INITIAL) { /* generate or retrieve seed */ Seed[0] = 0xAB; Seed[1] = 0xCD; Seed[2] = 0xEF; Seed[3] = 0x01; *SeedLen = 4; return E_OK; } return DCM_E_PENDING; /* if async NVM read needed */ } /* Security key compare callout */ Std_ReturnType App_CompareKey( Dcm_OpStatusType OpStatus, const uint8 *Key, uint16 KeyLen, Dcm_NegativeResponseCodeType *ErrorCode) { /* Compute expected key from stored seed using OEM algo */ uint8 expectedKey[4]; ComputeKey(g_Seed, expectedKey); if (memcmp(Key, expectedKey, 4) == 0) return E_OK; *ErrorCode = DCM_E_INVALIDKEY; return E_NOT_OK; }

Timing Verification

Use CANalyzer/CANoe with a CAPL script to measure response times. P2Server is typically 50ms. Measure from last bit of request to first bit of response. Any response within P2Server is compliant.

Addressing Verification

Physical (1:1): TX → 0x7DF (OBD) or configured phys ID (e.g. 0x7E0) RX ← 0x7E8 (or configured response ID) Functional (broadcast): TX → 0x7DF RX ← 0x7E8 (ECU responds if service supported) For DoIP: Source address in UDP/TCP header acts as generic connection MetaData

CANalyzer CAPL Snippet — Verify P2 Timing

/* CAPL: measure response time after request */ variables { dword tRequest; dword tResponse; } on message 0x7E0 { /* Tester request */ tRequest = timeNow(); } on message 0x7E8 { /* ECU response */ tResponse = timeNow(); float p2ms = (tResponse - tRequest) / 1000.0; write("Response time: %.1f ms", p2ms); if (p2ms > 50.0 && this.byte(1) != 0x78) write("WARNING: P2Server violation! %.1f ms > 50 ms", p2ms); }

Key Exam / Interview Points Summary

1. Session change timing: Committed at TpTxConfirmation, not at request receipt

2. Security reset: Always on any non-default → any session transition

3. NRC 0x78: Sent from a separate buffer; main buffer untouched

4. Functional addressing: NRC 0x11, 0x12, 0x7E, 0x7F are suppressed

5. S3 timer: Only reset by TesterPresent from the same connection that entered the session

6. DCM_E_PENDING: Application returns this; DCM retries on next MainFunction cycle

7. Generic connections: SOURCE_ADDRESS_16 MetaData used for response routing in DoIP/29-bit CAN

8. DID 0xF186: ActiveDiagnosticSession — managed internally by DCM, no callout needed

9. Paged-buffer: Allows responses larger than buffer; Tx only, not Rx

10. OSI layers: DCM handles L5 (session), L6 (presentation), L7 (application)
