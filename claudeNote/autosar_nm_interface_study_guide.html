AUTOSAR NM Interface comprehensive study guide covering state machine, functional spec, API, sequences, and DaVinci configuration

What NM Interface is

NM Interface (Nm) is a bus-independent abstraction layer between ComM (Communication Manager) and bus-specific NM modules (CanNm, FrNm, UdpNm, J1939Nm). It hides bus differences, coordinates multi-bus shutdown, and handles Partial Network Cluster (PNC) logic.

Three core functionalities
Basic
Pass-through between ComM and <Bus>Nm. Forwards requests & callbacks. Bus-independent API.
Coordinator
Coordinates shutdown of multiple buses in a cluster. Prevents early sleep if any bus is still active.
PNC Handling
PNC bit vector filter, EIRA/ERA aggregation, PNC timer management, synchronized PNC shutdown.

State transition diagram (CanNm-style, representative for all buses)
Repeat Message state: all nodes transmit NM PDUs rapidly. Duration = NmRepeatMessageTime (CanNm param). Purpose: node discovery, keeping bus awake during wakeup negotiation.
State rules you must know for testing
PassiveStartUp vs NetworkRequest
PassiveStartUp → enters RepeatMessage but does NOT actively request network (no NM PDU tx in active mode unless CanNm active). NetworkRequest → actively requests and transmits.
NmPassiveModeEnabled=TRUE
Node goes to OFFLINE state. Cannot NetworkRequest or NetworkRelease. Acts as listener only. Nm_NetworkRequest returns E_NOT_OK.
Ready Sleep exit
If ANY remote node sends NM PDU (not ready to sleep), Ready Sleep → Repeat Message (Nm_RemoteSleepCancellation fires). Local stays in Ready Sleep until all remote nodes agree.
Coordinator clusters
If PassiveStartUp received for a coordinated channel → treated as NetworkRequest (SWS_Nm_00178). This prevents early sleep on one channel while others are still negotiating.

Callback reentrancy rules — critical for ISR/task context

Per SWS_Nm_00028: all callbacks must assume they can run in task or interrupt context. They are all declared Reentrant (different channels). Nm_CoordReadyToSleepIndication is explicitly Non-Reentrant (critical section for coordinator state).

In multicore setups: Nm kernel lives in same partition as ComM kernel. Satellite Nm instances allowed for inter-partition API calls (SWS_Nm_00484–486).

1
Both buses: applications release → Nm_NetworkRelease() on each channel
2

3
ECU stays on bus, receives diag requests. NM PDU transmission disabled.
4
DCM ends session → Nm_EnableCommunication(handle) → NM PDU tx resumes
5
IMPORTANT: if Nm_NetworkRelease() was called during disabled state, ECU WILL sleep after step 4.

Partial Network Cluster (PNC) — concept

PNC lets groups of ECUs stay awake for specific functions (a PNC) while the rest of the network sleeps. Each PNC is represented by a bit in the NM PDU PNC bit vector. Nm manages PNC timers and aggregates requests.

EIRA
External/Internal Request Array. Combined view of all PNC requests (from bus + local app). Stored per PNC bit.
ERA
External Request Array. Only external (bus-received) PNC requests. Used by gateway ECUs to forward PNCs to other buses.
IRA
Internal Request Array. Local application PNC requests. ComM updates via Nm_UpdateIRA().
PNC filter algorithm

Each byte of received PNC bit vector is ANDed with NmPnFilterMaskByte. Only bits where both are '1' are "relevant PNC requests." Relevant requests restart PNC timer and update EIRA/ERA.

Example: received=0x86, mask=0xC6 → relevant=0x86 AND 0xC6 = 0x86. If received=0x05, mask=0xC6 → relevant=0x04. Non-masked bits ignored.

Nm_PncBitVectorRxIndication sets RelevantPncRequestDetectedPtr=TRUE if at least one relevant PNC bit detected (when NmPnEiraCalcEnabled or NmPnEraCalcEnabled).

PNC roles in a network topology
Top-level PNC coordinator
Has PNC gateway. Triggers synchronized PNC shutdown when PNC unrequested. Transmits PN shutdown message with CBIT set.
Intermediate PNC coordinator
Has PNC gateway. Forwards PN shutdown message on actively coordinated channels for passively coordinated PNCs.
PNC leaf node
No PNC gateway. Processes PN requests locally. Responds to shutdown messages by releasing PNCs.
Synchronized PNC shutdown sequence
1
ComM calls Nm_RequestSynchronizedPncShutdown(handle, PncId) on top-level coordinator
2
Nm stores request. Nm_MainFunction picks up pending shutdown requests.
3
Nm_PncBitVectorTxIndication called → Nm fills PN shutdown message with CBIT set + PNC bits for shutting PNCs
4
PN shutdown message transmitted on network. Intermediate coordinators forward it.
5
Leaf nodes receive via Nm_ForwardSynchronizedPncShutdown(handle, *vec) → ERA bits cleared → ComM_Nm_ForwardSynchronizedPncShutdown()
6
If tx fails: NM_E_TRANSMISSION_OF_PN_SHUTDOWN_MESSAGE_FAILED raised. If NmPnShutdownMessageRetransmissionDuration configured, retried.
If new PNC request arrives (external or internal) while shutdown pending → Nm cancels that PNC's pending shutdown (SWS_Nm_00529). Prevents race condition where PNC is released while still needed.

NM Coordinator algorithm overview

Coordinator ensures that when multiple buses are in one NM coordination cluster, they ALL sleep together. No bus goes to sleep while any other in the cluster is still active. Each channel belongs to 0 or 1 cluster (NmCoordClusterIndex).

Key rule: as long as ANY bus in a cluster is not ready to sleep, the coordinator keeps ALL buses in the cluster awake — even if the local ECU wants to sleep on all of them.

Nested sub-bus coordination

Nested coordinators build a hierarchy. Top coordinator has only actively coordinated channels. Sub-coordinators have a mix. Sub-bus shutdown time must be included in parent's NmGlobalCoordinatorTime.

SleepReadyBit: coordinator calls <Bus>Nm_SetSleepReadyBit() on actively coordinated channels to signal passively coordinated sub-coordinators that shutdown is imminent.

If a cyclic network is in the cluster and NmSynchronizingNetwork=FALSE, coordinator does not wait for SynchronizationPoint — it treats cyclic remote sleep as immediate. Set NmSynchronizingNetwork=TRUE on the cyclic network to enable cycle-boundary sync.
Abort conditions — must memorize

Coordinated shutdown is IMMEDIATELY ABORTED if:

!
Nm_RemoteSleepCancellation() received — a remote node woke up
!
Nm_NetworkMode() received — a network became active again
!
ComM calls Nm_NetworkRequest() or Nm_PassiveStartUp() on any cluster channel
!
<Bus>Nm_NetworkRelease() returns E_NOT_OK for any channel

On abort: coordinator calls CanNm_NetworkRequest() on ALL cluster buses to keep them awake. All conditions must be re-evaluated.

Nm_NetworkStartIndication is NOT an abort trigger. Only the above four conditions abort coordinated shutdown.
DaVinci Developer / Configurator — Nm module configuration path

In DaVinci Developer: BSW Modules → Nm → NmConfig → NmChannelConfig[0..N]. Each channel maps to one network handle and one bus NM type.

Step-by-step: configure Nm for a new CAN channel
1
Add NmChannelConfig: Right-click NmConfig → Add NmChannelConfig. Set NmChannelId = unique uint.
2
Set bus type: NmBusType = NM_BUSNM_CANNM. This tells Nm which function table to use.
3
Link to ComM channel: NmComMChannelRef → select matching ComMChannel. Handle mapping comes from here.
4
Passive mode: NmPassiveModeEnabled = FALSE for active nodes. TRUE for listen-only nodes (no tx, no NetworkRequest allowed).
5
Enable features: NmComControlEnabled (for DCM), NmUserDataEnabled, NmRemoteSleepIndEnabled, NmPduRxIndicationEnabled as needed.
6
Coordinator: NmCoordClusterIndex (if in a cluster), NmActiveCoordinator, NmChannelSleepMaster, NmSynchronizingNetwork.
7
PNC: NmPnEnabled, NmPnFilterMaskByte[0..N] (one per byte of PNC vector), NmPncBitVectorLength. Set NmPnEiraCalcEnabled, NmPnEraCalcEnabled per role.
8
State reporting: NmStateReportEnabled + NmStateReportSignalRef → select a Com signal (must be ≥6 bits).
9
Generate: DaVinci generates Nm_Cfg.c/h. Check NmChannelCount matches channel count. Verify function call tables in generated code.
10
Init sequence: EcuM calls CanNm_Init() FIRST, THEN Nm_Init(NULL_PTR). Order is mandatory.

Common config mistakes
NmGlobalCoordinatorTime too small → coordinator releases buses before bus-specific timers expire → bus can't complete sleep sequence → wakeup-sleep loop
NmRemoteSleepIndEnabled=FALSE with coordinator enabled → coordinator never gets RemoteSleepIndication → never initiates shutdown → network never sleeps
NmSynchronizingNetwork=TRUE on multiple channels in same cluster → coordinator gets confused by multiple sync points, may delay shutdown unpredictably
NmPassiveModeEnabled=TRUE on a coordinator channel → coordinator tries to NetworkRequest but passive mode blocks it → cluster coordination broken
