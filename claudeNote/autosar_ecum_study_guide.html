AUTOSAR CP ECU State Manager (EcuM) — Complete Study Guide

R24-11 · Document ID 78 · Classic Platform · Flexible EcuM variant · Exam, integration, testing & DaVinci config

What is the AUTOSAR ECU State Manager?

The ECU Manager (EcuM) is a Basic Software module that manages the lifecycle of the entire ECU. It takes control of the ECU during phases where generic mode management is unavailable — early startup, late shutdown, and sleep. During the UP phase, control is delegated to the BSW Mode Manager (BswM).

Primary responsibilities

• Initialize and de-initialize OS, SchM, BswM
• Manage wakeup events and validation
• Configure ECU for SLEEP and SHUTDOWN
• Coordinate multi-core lifecycle
• Arbitrate RUN/POST_RUN state requests

Flexible EcuM concept

The "Flexible EcuM" delegates ECU state management to BswM during the UP phase. There is no fixed state machine inside EcuM — integrators configure BswM rules to implement their ECU state machine.

Key relationship

EcuM owns the ECU during STARTUP, SHUTDOWN, and SLEEP. BswM owns it during UP. They hand control back and forth via mode switches and the EcuM_GoDownHaltPoll API.

Key design decisions (exam critical)

💡 EcuM does NOT have its own state machine in the Flexible version. All ECU modes (RUN, POST_RUN, SLEEP, SHUTDOWN) are AUTOSAR modes managed by BswM. EcuM only arbitrates RUN/POST_RUN requests and notifies BswM — BswM decides when transitions happen.
💡 When a wakeup event occurs during SHUTDOWN, EcuM completes the shutdown and restarts immediately [SWS_EcuM_02756]. It does not abort shutdown. The pending wakeup event causes the target to switch to RESET so the ECU restarts.

STARTUP Phase — detailed

Activities before EcuM_Init()

Before EcuM_Init is called from the C startup code (after reset vector), only minimal MCU initialization has taken place: stack is set up, C init of variables performed. No BSW is initialized yet.

StartPreOS sequence (inside EcuM_Init)

1
Callout EcuM_AL_SetProgrammableInterrupts() (optional) — Set interrupt priorities before OS starts. Only category I interrupts allowed here.
2
Callout EcuM_AL_DriverInitZero() (optional, Init Block 0) — Initialize drivers that don't need post-build config. Typically: DET, DEM pre-init, and any drivers needed to read post-build config. Interrupts must NOT be used here.
3
Callout EcuM_DeterminePbConfiguration() (mandatory) — Returns pointer to the fully initialized EcuM_ConfigType structure containing all post-build config for EcuM and all BSW modules.
4
Check configuration consistency — Compares hash in EcuM_ConfigType against compiled ECUM_CONFIGCONSISTENCY_HASH. Mismatch → calls EcuM_ErrorHook(ECUM_E_CONFIGURATION_DATA_INCONSISTENT) [SWS_EcuM_02796, 02798].
5
Callout EcuM_AL_DriverInitOne() (optional, Init Block I) — Initialize MCU Driver, Port, GPT, WDG (internal), WdgM, ADC, ICU, PWM, OCU. See recommended order in Table 7.2.
6
Get reset reason — Calls Mcu_GetResetReason() and maps reset reason to a wakeup source via EcuMWakeupSource config. Remembered for later validation [SWS_EcuM_02623].
7
Select default shutdown target — Calls EcuM_SelectShutdownTarget() with the configured EcuMDefaultShutdownTarget value [SWS_EcuM_02181].
8
Callout EcuM_LoopDetection() (optional) — If reset loop detection is enabled, this callout checks for rapid successive resets and takes preventive action.
9
StartOS(ECUM_DEFAULT_APP_MODE) — EcuM_Init() never returns. OS takes over. This is the last action in StartPreOS.

StartPostOS sequence (inside EcuM_StartupTwo)

EcuM_StartupTwo() must be called from an OS autostart task as its very first action [SWS_EcuM_02806]. This implements the "STARTUP II" state.

1
SchM_Start() — Start BSW Scheduler.
2
BswM_Init() — Initialize BSW Mode Manager. From this point, BswM can start arbitrating modes.
3
SchM_Init() — Initialize semaphores for BSW module critical sections.
4
SchM_StartTiming() — Start periodic events for BSW and SWC scheduling. Control now passes to BswM → UP phase begins.

Driver initialization order (recommended)

Init Block 0 (EcuMDriverInitListZero)

• Default Error Tracer (DET) — always first
• DEM pre-initialization
• Drivers needed to access post-build config

Init Block I (EcuMDriverInitListOne)

• MCU Driver
• Port Driver
• GPT Driver
• Watchdog Driver (internal only)
• WdgM, ADC, ICU, PWM, OCU

⚠️ EcuM MUST check configuration consistency of ALL BSW modules, not just its own, before initializing the first BSW module [SWS_EcuM_02796]. This is the only place this check is done ECU-wide.

Configuration consistency hash

At compile time, the EcuM configuration tool computes a hash over all pre-compile and link-time parameters of all BSW modules. This hash is stored in ECUM_CONFIGCONSISTENCY_HASH (link-time constant) AND in the EcuM_ConfigType structure (post-build). At runtime EcuM_Init compares them — they must match. A mismatch means post-build config was compiled with different pre-compile settings.

SHUTDOWN Phase — detailed

Initiated by BswM calling EcuM_GoDownHaltPoll(UserID) with shutdown target set to RESET or OFF. This call does not return.

OffPreOS sequence (inside EcuM_GoDownHaltPoll)

1
Callout EcuM_OnGoOffOne() — OEM-specific shutdown preparation. Called before BswM de-init.
2
BswM_Deinit() — De-initialize BSW Mode Manager.
3
SchM_Deinit() — De-initialize BSW Scheduler. After this, no more scheduled functions run.
4
Check for wakeup events — Depending on EcuMIgnoreWakeupEvValOffPreOS: if TRUE → check only validated events; if FALSE → check all pending events too. If wakeup events found → change shutdown target to RESET (using default reset mode) so ECU restarts immediately [SWS_EcuM_04151, 04152].
5
ShutdownOS() — Last operation. OS calls ShutdownHook, which must call EcuM_Shutdown() [SWS_EcuM_02952, 02953].

OffPostOS sequence (inside EcuM_Shutdown, called from ShutdownHook)

1
Callout EcuM_OnGoOffTwo() — Final OEM-specific actions after OS shutdown.
2
Branch on shutdown target:
• RESET → Callout EcuM_AL_Reset(EcuM_ResetType)
• OFF → Callout EcuM_AL_SwitchOff()
Neither callout returns [SWS_EcuM_04074, 04075].
⚠️ Critical: EcuM_Shutdown() is called from the OS ShutdownHook, not directly by EcuM. The integrator MUST configure the ShutdownHook to call EcuM_Shutdown. Without this, the shutdown sequence is never completed.

SLEEP Phase — detailed

Initiated by BswM calling EcuM_GoDownHaltPoll(UserID) with shutdown target = SLEEP. Unlike SHUTDOWN, this function returns when a valid wakeup occurs. The OS is NOT shut down during SLEEP [SWS_EcuM_02951].

GoSleep sequence

1
EcuM notifies BswM: BswM_EcuM_CurrentWakeup(sources, ECUM_WKSTATUS_NONE) — clearing all wakeup status.
2
Callout EcuM_EnableWakeupSources(EcuM_WakeupSourceType) — arm all wakeup sources configured for the target sleep mode (from EcuMWakeupSourceMask) [SWS_EcuM_02389].
3
On multicore: GetResource(RES_AUTOSAR_ECUM) — reserve core resource [SWS_EcuM_03010].
4
Branch to Halt or Poll based on EcuMSleepModeSuspend configuration. If a wakeup event already occurred during GoSleep, NEITHER Halt nor Poll is entered [SWS_EcuM_02188].

Halt sequence (EcuMSleepModeSuspend = HALT)

1
Disable interrupt sources for non-wakeup ISRs.
2
Callout EcuM_GenerateRamHash() — compute RAM checksum before halting [SWS_EcuM_02863]. On multicore: only master core.
3
Mcu_SetMode() — put MCU into halt/sleep mode. CPU stops here.
4
Wakeup interrupt fires → CPU resumes from Mcu_SetMode. EcuM_CheckWakeup() called → EcuM_CheckWakeupHook() → driver's CheckWakeup → if wakeup detected: EcuM_SetWakeupEvent().
5
Callout EcuM_CheckRamHash() — verify RAM integrity after wakeup [SWS_EcuM_02863]. Failure → EcuM_ErrorHook(ECUM_E_RAM_CHECK_FAILED).
6
→ WakeupRestart sequence.

Poll sequence (EcuMSleepModeSuspend = POLL)

Used for sleep modes where the CPU continues running at reduced speed. EcuM loops until a pending/validated wakeup event is reported.

1
DisableAllInterrupts → Mcu_SetMode(reduced clock) → EnableAllInterrupts.
2
Loop: Callout EcuM_SleepActivity() — OEM code runs here during poll (e.g. update EcuM clock).
3
Loop: For each polling wakeup source: EcuM_CheckWakeupHook() → driver's CheckWakeup → if detected: EcuM_SetWakeupEvent().
4
Check for pending events. If validation needed → BswM_EcuM_CurrentWakeup(PENDING), else → BswM_EcuM_CurrentWakeup(VALIDATED).
5
Exit loop when pending OR validated wakeup event found → WakeupRestart sequence [SWS_EcuM_03020].

WakeupRestart sequence

1
DisableAllInterrupts → Mcu_SetMode() restore normal MCU mode → EnableAllInterrupts.
2
EcuM_GetPendingWakeupEvents() — get list of wakeup candidates.
3
Callout EcuM_DisableWakeupSources() — disable currently pending sources but keep others armed [SWS_EcuM_02963].
4
Callout EcuM_AL_DriverRestart() — re-initialize drivers that need restarting. Drivers in EcuMDriverRestartList are re-initialized in same order as combined Init Block 0+I [SWS_EcuM_02561]. Drivers re-arm their wakeup callbacks here.
5
ReleaseResource(RES_AUTOSAR_ECUM) — unlock scheduler on multicore. All tasks can run again. EcuM_GoDownHaltPoll returns control to BswM → UP phase resumes. Wakeup validation proceeds in EcuM_MainFunction.
⚠️ If all CheckWakeupTimers expire without a validated wakeup event, EcuM re-enters GoSleep [SWS_EcuM_04149]. If a wakeup was signaled but no event set before timer expiry, runtime error ECUM_E_WAKEUP_TIMEOUT is reported [SWS_EcuM_04150].

Wakeup validation protocol

Wakeup events can be spurious (EMV spikes, CAN line glitches). The validation protocol distinguishes real events from noise:

1
Wakeup detected: Driver calls EcuM_SetWakeupEvent(sources) — sets source to PENDING. If no validation needed, immediately calls EcuM_ValidateWakeupEvent() implicitly [SWS_EcuM_02566].
2
EcuM notifies BswM: BswM_EcuM_CurrentWakeup(sources, ECUM_WKSTATUS_PENDING) and calls EcuM_StartWakeupSources().
3
Validation timeout starts [SWS_EcuM_02565] (per-source or global timer). Timeout value from EcuMValidationTimeout.
4
EcuM_MainFunction calls EcuM_CheckValidation() → driver's CheckValidation → if validated: driver calls EcuM_ValidateWakeupEvent(sources).
5
If validated: ComM_EcuM_WakeUpIndication() + ComM_EcuM_PNCWakeUpIndication() + BswM_EcuM_CurrentWakeup(VALIDATED). BswM returns ECU to RUN state.
6
If timeout: Source moves to EXPIRED. BswM_EcuM_CurrentWakeup(EXPIRED) + EcuM_StopWakeupSources(). BswM may put ECU back to sleep.

Wakeup sources with integrated power control

Some CAN transceivers have integrated power supplies — they switch ECU power ON when CAN activity is detected. From EcuM's perspective, this wakeup looks like a power-on reset (STARTUP, not SLEEP wakeup). The CAN transceiver driver must check during its initialization if it caused the ECU startup and call EcuM_SetWakeupEvent() accordingly. The integrator must configure EcuM_StartCheckWakeup() to prevent shutdown before this check completes.

UP Phase — EcuM_MainFunction & RUN Protocol

EcuM_MainFunction responsibilities

The EcuM_MainFunction() runs periodically (scheduled by SchM) and has exactly three responsibilities during the UP phase:

1. Wakeup validation

For each pending wakeup source: call EcuM_CheckValidation(), handle timeout, update BswM via mode switch requests.

2. Alarm clock update

Increment EcuM clock using OS counter/alarm mechanism. Time in seconds since battery connect [SWS_EcuM_04002].

3. RUN/POST_RUN arbitration

Track all user RequestRUN/ReleaseRUN/RequestPOST_RUN/ReleasePOST_RUN calls and notify BswM of state changes.

RUN/POST_RUN request protocol

SW-Cs and BSW modules indicate they need the ECU to stay operational by requesting the RUN state. EcuM arbitrates all requests and tells BswM the aggregate result:

SWC calls EcuM_RequestRUN(user)
→
EcuM tracks per-user
→
BswM_EcuM_RequestedState(RUN, REQUESTED)
All SWCs release RUN
→
Last release detected
→
BswM_EcuM_RequestedState(RUN, RELEASED)

RUN Request Protocol rules

Rules for RequestRUN

• Each user can only have ONE active RUN request at a time
• Multiple requests from same user → DET error ECUM_E_MULTIPLE_RUN_REQUESTS
• Returns E_OK if accepted, E_NOT_OK if not

Rules for ReleaseRUN

• Release without prior request → DET error ECUM_E_MISMATCHED_RUN_RELEASE
• When last RUN release: EcuM notifies BswM → BswM may switch to POST_RUN
• SWC needing POST_RUN must request it BEFORE releasing RUN

Alarm clock

Optional feature (enabled by EcuMAlarmClockPresent). Maintains an EcuM clock (seconds since battery connect) that persists across resets. Each user can have one alarm. EcuM tracks the earliest user alarm as the master alarm — wakeup triggered at master alarm time.

Callouts & callbacks

Callouts are function stubs the system integrator fills with OEM-specific or hardware-specific code. They are part of EcuM_Callout_Stubs.c.

STARTUP phase callouts

EcuM_AL_SetProgrammableInterrupts()Set interrupt priorities before OS starts. Optional. Category I interrupts only allowed after this.
EcuM_AL_DriverInitZero()Init Block 0. Initialize DET, DEM pre-init, and drivers needed for post-build config access. No OS, no interrupts.
EcuM_DeterminePbConfiguration(const EcuM_ConfigType*)Return pointer to fully initialized post-build config. Must select the correct PB config variant based on some criteria (e.g. DIO pin, NvM value). MANDATORY.
EcuM_AL_DriverInitOne()Init Block I. Initialize MCU, Port, GPT, WDG, WdgM, ADC, ICU, PWM, OCU etc. Typical main hardware init.
EcuM_LoopDetection()Called on every startup if loop detection enabled. Integrator checks reset counter and may trigger safe state.

SHUTDOWN phase callouts

EcuM_OnGoOffOne()Called at start of OffPreOS sequence. OEM-specific pre-shutdown actions (e.g. flush buffers, signal peripherals).
EcuM_OnGoOffTwo()Called at start of OffPostOS sequence, after OS shutdown. Last chance for OEM-specific actions before power/reset.
EcuM_AL_SwitchOff()Called when shutdown target is OFF. Integrator pulls the power hold pin low (or calls DIO/SPI to external power controller). Must not return.
EcuM_AL_Reset(EcuM_ResetType)Called when shutdown target is RESET. Calls Mcu_PerformReset or WdgM_PerformReset or toggles IO pin depending on reset type parameter. Must not return.

SLEEP phase callouts

EcuM_EnableWakeupSources(EcuM_WakeupSourceType)Called in GoSleep. Arm all wakeup sources for the target sleep mode. Enable interrupt/transceiver wakeup mechanisms.
EcuM_DisableWakeupSources(EcuM_WakeupSourceType)Called in WakeupRestart. Disable wakeup sources that caused wakeup (others remain armed for next wakeup).
EcuM_GenerateRamHash()Called before Halt. Compute and store a RAM checksum for integrity verification after wakeup.
EcuM_CheckRamHash()Called after Halt wakeup. Verify RAM against stored checksum. Failure → EcuM_ErrorHook(ECUM_E_RAM_CHECK_FAILED).
EcuM_SleepActivity()Called in each Poll loop iteration. Integrator can update EcuM clock, check hardware status, or perform minimal processing during reduced-clock sleep.
EcuM_AL_DriverRestart()Called in WakeupRestart. Re-initialize drivers in EcuMDriverRestartList. These drivers detect if they caused the wakeup and call EcuM_SetWakeupEvent() accordingly.

SLEEP phase callouts (wakeup checking)

EcuM_StartCheckWakeup(EcuM_WakeupSourceType)Called to start asynchronous wakeup checking for sources that cannot be synchronously polled (e.g. CAN transceiver via SPI).
EcuM_CheckWakeupHook(EcuM_WakeupSourceType)Called for each wakeup source that needs polling during Halt or Poll. Integrator calls driver's CheckWakeup function.

UP phase callouts

EcuM_StartWakeupSources(EcuM_WakeupSourceType)Called when wakeup source moves to PENDING in the UP phase. Start the network stack/transceiver to allow validation (e.g. start CAN bus to wait for frame).
EcuM_CheckValidation(EcuM_WakeupSourceType)Called periodically in EcuM_MainFunction to check if a pending wakeup source has been validated by its driver.
EcuM_StopWakeupSources(EcuM_WakeupSourceType)Called when wakeup source moves to EXPIRED. Stop the network stack/transceiver — validation failed.

Generic callout

EcuM_ErrorHook(uint16 reason)Called for unrecoverable errors. Integrator must NOT return from this. Handles: ECUM_E_RAM_CHECK_FAILED, ECUM_E_CONFIGURATION_DATA_INCONSISTENT, ECUM_E_OS_CALL_FAILED. Typically triggers safe state or reset.

Callbacks (called BY EcuM → other modules)

EcuM_SetWakeupEvent(EcuM_WakeupSourceType)CALLBACK — called by drivers when they detect a wakeup event. Must be callable from ISR context. Sets source to PENDING. Starts validation timeout.
EcuM_ValidateWakeupEvent(EcuM_WakeupSourceType)CALLBACK — called by drivers/stacks when they confirm a valid wakeup. Sets source to VALIDATED. Stops timeout. Triggers ComM notification.

Multicore ECU — EcuM coordination

In multicore ECUs, one core is designated the master core; all others are slave cores. EcuM runs on every core. All cores must be synchronized for STARTUP, SHUTDOWN, and SLEEP.

Master vs slave responsibilities

Master core

• Runs the main EcuM_Init() sequence
• Initiates ShutdownAllCores (not ShutdownOS individually)
• Computes RAM hash for halt/sleep
• Verifies RAM hash before releasing slaves from halt
• Signals all slaves to continue after wakeup
• Waits for all slave signals before calling ShutdownAllCores

Slave cores

• Runs its own EcuM_Init() (per-core MCAL init)
• Runs its own StartOS → EcuM_StartupTwo
• Initializes only MCAL modules on its core
• Sends ready signal to master before sleep/shutdown
• Waits for master signal before continuing after halt
• Has its own BswM instance

Core synchronization mechanism

Cores signal each other using OS Events (SetEvent/WaitEvent) or dedicated shared memory flags. The mechanism is implementation-specific but must guarantee:

1
SLEEP: All slave cores signal "ready to halt" → master waits for all signals → master generates RAM hash → master halts → wakeup → master verifies hash → master signals slaves to continue → slaves exit halt.
2
SHUTDOWN: BswM on each core de-initializes its BSW → slave cores signal "ready for shutdown" → master waits for all signals → master calls ShutdownAllCores → global shutdown hook → all cores stop.

Multicore OS resources

For SLEEP, EcuM reserves a dedicated OS resource RES_AUTOSAR_ECUM_<core#> for each core during GoSleep. This resource is released during WakeupRestart [SWS_EcuM_03010]. This prevents tasks from running on a core while it is in the sleep preparation/wakeup phase.

⚠️ Known limitation from spec: EcuM module interfaces must be specified as reentrant in the multicore context. Individual MCAL module re-initialization in EcuM_AL_DriverInitZero/One must only initialize modules assigned to the currently active core [SWS_EcuM_04145].

Error handling during startup and shutdown

💡 EcuM deliberately ignores return values from BSW init functions [SWS_EcuM_02980]. BSW modules are responsible for reporting their own init errors to DET/DEM. EcuM's philosophy: "keep going during startup, individual modules handle their own errors."

Step-by-step DaVinci configuration

Step 1 — EcuMGeneral settings

1
Set EcuMDevErrorDetect = TRUE for development builds (enables DET reporting for all development errors).
2
Set EcuMModeHandling = TRUE if using RUN/POST_RUN request protocol (recommended for all projects).
3
Set EcuMAlarmClockPresent if the project needs timed wakeup from sleep across resets.

Step 2 — Driver init lists

1
Add entries to EcuMDriverInitListZero for DET and DEM pre-init. Each entry: module name, init function pointer, config pointer. Order matters — DET must be first.
2
Add entries to EcuMDriverInitListOne for MCU, Port, GPT, WDG, etc. These run after EcuM_DeterminePbConfiguration returns post-build config.
3
Add entries to EcuMDriverRestartList for drivers that need re-initialization after sleep wakeup. Typically: CAN transceiver, LIN transceiver, GPT (if used for sleep timing).
4
Add entries to EcuMDriverInitListBswM for BSW modules initialized by BswM during UP phase: COM, CanSM, NvM, DEM, FIM, DCM, etc.

Step 3 — Wakeup sources

1
For each wakeup source, create an EcuMWakeupSource container. Set EcuMWakeupSourceId (0..31, bits 0-4 reserved for predefined sources → use 5-31 for custom). Must be unique.
2
Set EcuMValidationTimeout in seconds (0 = no validation needed, auto-validated). For CAN: typically 100ms-1s. For GPT timer alarm: 0 (no validation).
3
Set EcuMWakeupSourcePolling = TRUE if this source requires polling during the Poll sleep sequence.
4
Map Mcu reset reasons to wakeup sources via EcuMWakeupSourceMcuResetReason reference in the container.

Step 4 — Sleep modes

1
Create EcuMSleepMode containers. Set EcuMSleepModeSuspend: HALT (MCU halts, lowest power) or POLL (MCU keeps running at reduced speed).
2
Set EcuMNormalMcuModeRef — MCU mode to restore after wakeup restart.
3
Set EcuMWakeupSourceMask — bitmask of wakeup sources that should be armed for this sleep mode. Must include at least one wakeup source, otherwise ECU can never wake up.

Step 5 — Shutdown targets and causes

1
Configure EcuMDefaultShutdownTarget — what EcuM selects during StartPreOS. Typically RESET or OFF. References a specific reset or sleep mode.
2
Configure EcuMResetMode containers — define reset types (MCU, WDG, IO). Set EcuMDefaultResetModeRef for the mode used when wakeup-during-shutdown forces a restart.
3
Configure EcuMShutdownCause values — at least: ECU state machine shutdown, WdgM failure, DCM request.

Step 6 — GoDownAllowedUsers

1
Add BswM as an allowed user in EcuMGoDownAllowedUsers. Only configured users can call EcuM_GoDownHaltPoll. Returns E_NOT_OK for non-configured callers.

Common DaVinci configuration mistakes

🔴 EcuMWakeupSourceMask = 0 for a sleep mode: ECU enters sleep but can never wake up from that mode. Always verify at least one wakeup source is armed per sleep mode.
🔴 Missing ShutdownHook implementation: If EcuM_Shutdown() is not called from ShutdownHook, OffPostOS never executes — ECU does not actually power off or reset after ShutdownOS.
🔴 EcuMDriverRestartList order: Must maintain same relative order as the combined Init Block 0+I list [SWS_EcuM_02561]. Incorrect order can cause hardware to be in wrong state during wakeup.
⚠️ EcuMValidationTimeout too short: If shorter than the time it takes for the CAN stack to validate a frame, wakeups will constantly expire. Common debug issue: ECU wakes up, immediately goes back to sleep.
⚠️ EcuMIgnoreWakeupEvValOffPreOS setting: TRUE means only validated events are checked during shutdown — unvalidated (spurious) wakeup events are ignored. FALSE means even pending (unvalidated) events trigger a RESET instead of OFF. Set based on project safety requirements.
💡 Config consistency hash: After any pre-compile parameter change, the EcuMConfigConsistencyHash in EcuMCommonConfiguration MUST be regenerated by running DaVinci Configurator. Otherwise EcuM_Init fails the hash check at next startup.

Integration testing & debugging

Common integration bugs

Bug: ECU does not start (hangs in StartPreOS)

Root cause: Configuration hash mismatch triggers EcuM_ErrorHook with ECUM_E_CONFIGURATION_DATA_INCONSISTENT. Or driver in Init Block 0/I crashes. Debug: check if EcuM_ErrorHook is reached → if yes, reason = hash mismatch or driver crash. Add GPIO toggle to EcuM_AL_DriverInitZero entry/exit to locate hanging driver.

Bug: EcuM_StartupTwo never called

Root cause: No autostart OS task configured to call EcuM_StartupTwo, or autostart task has wrong priority. The OS started successfully (no hang in StartOS) but StartupTwo task never activates. Debug: check OS configuration — verify autostart task exists in default application mode.

Bug: ECU wakes from sleep but immediately sleeps again

Root causes: (1) EcuMValidationTimeout too short — wakeup expires before stack validates. (2) EcuM_ValidateWakeupEvent never called by driver. (3) BswM rules not configured to hold ECU awake after validation. Debug: trace EcuM_SetWakeupEvent → EcuM_ValidateWakeupEvent → BswM_EcuM_CurrentWakeup calls using DET/trace hooks.

Bug: ECU doesn't power off (stays in OffPostOS)

Root cause: EcuM_AL_SwitchOff() implementation doesn't actually cut power (power hold circuit not correctly controlled), or ShutdownHook is not configured to call EcuM_Shutdown(). Debug: verify ShutdownHook calls EcuM_Shutdown. Check power hold GPIO in EcuM_AL_SwitchOff callout.

Bug: ECUM_E_MULTIPLE_RUN_REQUESTS on startup

Root cause: A SWC is calling EcuM_RequestRUN twice for the same user ID. Usually happens if a BSW mode switch action list triggers RequestRUN multiple times, or if the user ID is shared between multiple callers. Debug: enable DET and check which user ID triggers the error. Verify EcuMGoDownAllowedUsers and user ID assignments.

Bug: RAM check fails after sleep (ECUM_E_RAM_CHECK_FAILED)

Root cause: RAM corruption during halt (power droop, EMC), or EcuM_GenerateRamHash algorithm covers a RAM region that is legitimately modified by interrupt handlers between hash generation and halt. Debug: exclude interrupt-modified regions from hash. Check power supply stability during sleep. Verify RAM retention voltage meets spec.
