AUTOSAR CP FOTA study guide covering architecture, state machine, sequences, Dcm interaction, and DaVinci configuration

What FOTA is — the core problem it solves

Traditional OTA: ECU goes offline, tester reprograms it, vehicle is down the entire time. Bricked ECU risk is high. FOTA solves this by installing new SW into an inactive partition in the background while the ECU keeps running, then switching partitions at a safe vehicle state.

Old OTA problem
ECU offline during update. Vehicle downtime = update time. Rollback = re-flash from scratch (more downtime). Brick risk.
FOTA solution
Install into inactive partition while active runs. Interruptible over multiple driving cycles. Switch partition at vehicle safe state. Rollback = re-activate previous partition.
Key constraint
Activation (partition switch) must NEVER happen while driving. Requires vehicle safe state: standstill, engine off, parking brake applied.
System actors — three entities
FOTA Target internal architecture
FOTA Handler is a CDD (Complex Device Driver) — not a full AUTOSAR BSW module yet. This is intentional: the read-while-write flash approach and program flash access at the MCAL level were still undecided when this was specified. CDD leaves architectural flexibility to connect through all BSW layers.
Why FOTA Master lives in Adaptive AUTOSAR

Classic Platform ECUs have limited memory, storage, and compute power. The FOTA Master must cache entire images for all target ECUs — potentially gigabytes. Adaptive ECUs (domain controllers, central compute platforms) have the resources. The FOTA Master = UCM-Master (Update and Configuration Management) on the Adaptive Platform. It speaks UDS to all FOTA Target ECUs on the vehicle bus.

The UCM-Master concept was still under development when this doc was written. Examiners expect you to know: FOTA Master = UCM-Master realization = Adaptive AUTOSAR. It orchestrates ALL targets, manages dependencies between multi-ECU updates, and controls activation timing.
System component terms
FOTA TargetAny ECU capable of being updated via FOTA. Runs FOTA Handler (CDD) + Dcm + Memory Stack. Classic AUTOSAR.
FOTA MasterInstance that orchestrates all software updates. Stores images, sends UDS chunks, controls state transitions, manages dependencies. = UCM-Master on Adaptive AUTOSAR.
FOTA ImageFull downloadable ECU software collection. Can be full, partial, or incremental update. Covers AppSW, BSW, or calibration data. After applying: ECU must be fully functional and system-compatible.
BackendRemote cloud/server that provides images to FOTA Master. Notifies vehicle of available updates (push) or waits to be polled (pull). Also receives progress/status reporting.
Data ChunkUDS TransferData (0x36) payload packet. Variable size. Image is split into chunks because transferring entire image at once is impractical. Chunk size bounded by DcmDslBufferSize.
Process terms — know these precisely
UpdateEntire FOTA procedure: download + installation + verification + activation + (optionally rollback). Whole campaign across one or more ECUs.
DownloadTransfer of image from Backend → FOTA Master (not to the target). Over-the-air link (LTE/5G/WiFi). FOTA Master caches it.
InstallationTransfer of chunks from FOTA Master → FOTA Target via UDS (0x34/0x36/0x37). Target writes chunks to inactive partition via memory stack. Can span multiple driving cycles. Finished when all chunks written successfully.
VerificationImplementation specific. Master verifies authenticity/integrity of image package. Target verifies semantic/functional integrity of flashed SW. Second verification after copy from external to internal flash is recommended. Not specified in detail by AUTOSAR.
ActivationActual partition switch (or copy from temp→runtime + reboot). Must be in vehicle safe state. Integrator responsibility to ensure safe state. Previous SW kept until activation is confirmed. Done in bootloader context — out of AUTOSAR scope.
RollbackRestore previous SW and user data. After rollback: ECU must be indistinguishable from pre-update state. Triggered by FOTA Master (master-triggered) or autonomously by ECU that cannot boot (self-triggered). All dependent ECUs must roll back too.
Exam trap: Download ≠ Installation
Download
Backend → FOTA Master. OTA radio link. FOTA Master stores the image. The target ECU is not involved yet.
Installation
FOTA Master → FOTA Target. Over vehicle bus (CAN/Ethernet). Chunks via UDS 0x36. Target writes to inactive flash partition.
FOTA Handler state machine — all 7 states
IDLEInitial state after ECU startup. No FOTA activity. FOTA Master reads this to know ECU is ready for a new FOTA campaign.
INITFOTA Handler initialized. Dcm is in FOTA diagnostic session AND security access granted. Transition: RC_InitializeFOTA() called by FOTA Master. Handler checks availability, preempted procedures, preconditions.
WAITAfter RequestDownload(0x34). Waiting for next TransferData(0x36) chunk. Handler successfully processed previous chunk and returned Dcm callout. Ready for more data.
PROCESSINGTransferData chunk received in Dcm RAM buffer. FOTAHandlerMain() actively writing chunk to flash via memory stack. Dcm callout returns DCM_WRITE_PENDING. FOTA Master kept alive by pending responses.
READYALL chunks transferred and written successfully. RequestTransferExit(OpStatus=READY) received. Activation can be triggered. FOTA Master may now call RC_TriggerActivation().
VERIFYOptional, implementer specific. RC_VerifyInstallation() routine. AUTOSAR does not define verification details.
ACTIVATERC_TriggerActivation() received. Handler sets partition switch flag accessible by bootloader. ECU resets. Bootloader executes actual partition switch. DcmResponseToEcuReset=AFTER_RESET sends positive response after reset.
ERROROptional, implementer specific. Can be entered from ANY state. OEM/implementer-specific recovery. FOTA Master controls recovery from here.
State transition diagram
Critical state transition rules — exam targets
IDLE vs READY distinction: both reached via RequestTransferExit(). Distinguished by OpStatus parameter. OpStatus=IDLE → cancel, go back to IDLE. OpStatus=READY → all chunks done, ready for activation.
FOTA Master rollback = transition from ANY state to ACTIVATE. Not a separate rollback state — rollback means activating the previous (now inactive) partition, which uses the same ACTIVATE state.
State must be persisted in NvM. If driving cycle ends during PROCESSING, NvM stores last successfully written address + chunk ID. On next startup: IDLE → INIT → WAIT and resume from last good position.
ERROR is optional and implementer-specific. The doc does not define how to recover from ERROR. OEM/supplier defines recovery. FOTA Master controls this via UDS services.
Two functions inside FOTA Handler (CDD)
FOTA_ProcessTransferDataWrite(...)

Called by Dcm each time a chunk is received (0x36 TransferData callout). Sets OpStatus = DCM_PENDING until done. Returns DCM_WRITE_PENDING while chunk being processed. On finish: returns DCM_WRITE_OK → Dcm sends positive response → Master sends next chunk. Signals FOTAHandlerMain via InterRunnable Variable.

FOTAHandlerMain(...)

Scheduled cyclically by OS. Independent of Dcm and comm stack. Does the actual memory write to flash via memory stack. On completion: signals ProcessTransferDataWrite via InterRunnable Variable. This decoupling allows flash writes to happen in a separate task without blocking the Dcm.

The asynchronous design is mandatory. Flash write operations can take milliseconds to seconds. Blocking the Dcm task would freeze the entire diagnostic stack. The callout returns PENDING repeatedly, keeping the FOTA Master's UDS timeout from expiring.
Buffer flow — how a chunk moves through the system
1
FOTA Master sends UDS 0x36 TransferData with chunk payload over vehicle bus
2
Comm stack delivers it to Dcm. Dcm stores chunk in its RAM buffer (DcmDslBufferSize controls max chunk size)
3
Dcm triggers FOTA_ProcessTransferDataWrite() callout. Callout signals FOTAHandlerMain via InterRunnableVariable. Returns DCM_WRITE_PENDING immediately.
4
Dcm polls callout cyclically → keeps getting DCM_WRITE_PENDING → keeps sending pending NRC to FOTA Master (prevents timeout)
5
FOTAHandlerMain() runs in OS task → takes chunk pointer from Dcm buffer → calls memory stack write API → writes to inactive flash partition
6
FOTAHandlerMain signals completion via InterRunnableVariable. Buffer is exclusively owned by FOTA Handler until this point — no other module touches it.
7
FOTA_ProcessTransferDataWrite() sees completion → returns DCM_WRITE_OK → Dcm sends positive response to FOTA Master → Master sends next chunk
Dcm buffer is BLOCKED (exclusively owned by FOTA Handler) until chunk processing completes. Buffer payload is consistent even across diagnostic protocol changes during this window. Best practice: FOTA buffer size should not exceed the max size of other UDS protocol buffers in Dcm.
NvM usage by FOTA Handler

FOTA Handler must persist its state through power cycles. Uses NvM (never direct flash access). Stored data includes: current FOTA state, last successfully written flash address, chunk ID. On next startup: Handler reads NvM → knows where to resume. Neither FOTA Master nor FOTA Target must access data flash directly — always via NvM module.

Direct data flash access bypassing NvM is forbidden. It can cause interference, blocking, or data corruption with normal NvData operation. Data model migrations (NvData changes between SW versions) must be handled by FOTA Handler or FOTA Master, never by directly touching flash.

FOTA diagnostic session — why it needs its own

FOTA runs at lowest Dcm priority (DcmDslProtocolPriority = lowest). It gets its own DcmDslProtocolRow entry with its own RAM buffer. This enables the Dcm protocol preemption feature — a higher priority diagnostic request (e.g. normal UDS tester) can preempt FOTA without corrupting it. FOTA session secured by SecurityAccess to prevent unauthorized flash.

Protocol preemption — critical concept

When a higher-priority diagnostic request arrives, FOTA is immediately stopped. The preemption callback (DcmDslCallbackDCMRequestService) fires — FOTA Handler must save consistent state here.

From FOTA Master's perspective: no notification is sent. FOTA Master sees a UDS timeout. After timeout, Master must reinitialize and resume. Master reads DIDs to find last successfully written address and chunk ID.

The implementer can optionally continue processing the already-received chunk in the background during preemption. But: the last successfully written address MUST always be reflected in a DID — this is mandatory for resume capability.

Sequence 1: FOTA initialization (first start or resume)
1
FOTA Master → UDS 0x22 GetFOTAStatus() → Target Dcm → Response: FOTAStatus=IDLE
2
FOTA Master → 0x10 DiagnosticSessionControl(FOTA_SESSION) → Positive response
3
FOTA Master → 0x27 SecurityAccess(FOTA) → seed/key challenge → Positive response (access granted)
4
FOTA Master → 0x31 RC_InitializeFOTA() → Target: checks availability, preempted state, preconditions → Response: FOTAStatus=INIT
5
FOTA Target waits in FOTA session for 0x34 RequestDownload from FOTA Master. If interrupted/preempted: this whole sequence repeats (reinit = same as first init).
If FOTA was previously preempted: RC_InitializeFOTA detects preempted state → FOTA Master reads DIDs for last written address/chunk → resume at that position after reinitialization.
Sequence 2: Data chunk transfer (installation loop)
1
FOTA Master → 0x34 RequestDownload(address, size) → Target enters WAIT state → Positive response
L
Loop: for each chunk
2
FOTA Master → 0x36 TransferData(chunk_N) → Dcm stores in RAM buffer → State → PROCESSING
3
Dcm calls FOTA_ProcessTransferDataWrite() → signals FOTAHandlerMain → returns DCM_WRITE_PENDING
4
Dcm polls callout in main function → each poll → DCM_WRITE_PENDING → NRC 0x78 (ResponsePending) to FOTA Master
5
FOTAHandlerMain() (OS task) → reads chunk from Dcm buffer → calls memory stack write → writes to inactive flash partition
6
FOTAHandlerMain signals completion → FOTA_ProcessTransferDataWrite() returns DCM_WRITE_OK → Dcm → positive response 0x76 to FOTA Master
7
State → WAIT. FOTA Master sends next chunk (goto step 2) OR sends 0x37 RequestTransferExit when done
Sequence 3: Activation
1
State = READY (or VERIFY after optional verification). Vehicle must be in safe state (integrator responsibility).
2
FOTA Master → 0x31 RC_TriggerActivation() → Target: state → ACTIVATE
3
FOTA Handler sets partition switch flag in memory area accessible by BOTH bootloader AND application. This is the key handoff point.
4
FOTA Master → 0x11 ECU Reset → ECU reboots. DcmSendRespPendOnRestart=TRUE informs Master about restart.
5
Bootloader reads partition switch flag → executes partition switch (A/B swap or copy from external to internal). Bootloader context — out of AUTOSAR scope.
6
ECU boots new SW. DcmResponseToEcuReset=AFTER_RESET → sends positive response to FOTA Master after boot to confirm ECU alive.
7
FOTA Master → reads SW version DID → confirms new version running. If multiple ECUs: checks all before considering campaign complete.
The partition switch flag memory location MUST be accessible by both bootloader AND application. If only accessible in application RAM (lost on reset) the bootloader cannot find it. This is a common integration mistake.
Sequence 4: Preemption + Resume
1
During chunk PROCESSING: higher-priority diagnostic tester connects on same or higher-priority protocol
2
Dcm preempts FOTA protocol. DcmDslCallbackDCMRequestService fires → FOTA Handler saves state to NvM
3
No notification to FOTA Master. Master experiences UDS timeout on its pending TransferData response.
4
After timeout: FOTA Master reinitializes (Sequence 1 again). Reads DIDs: last written address, chunk ID, FOTA status.
5
FOTA Master resumes from last successfully written position. Re-sends missed/incomplete chunk. Procedure continues as Sequence 2.

Memory stack prerequisites — what FOTA requires from MCAL
Program flash accessAUTOSAR previously only allowed data flash access at runtime. FOTA requires READ and WRITE access to program flash sections at runtime. MCAL extension required (planned R20-11).
Read-while-writeActive partition is read-executed. Inactive partition is written simultaneously. HW vendor must provide this in MCAL. Not all MCUs support this — it depends on flash architecture (A/B banks).
Memory solution agnosticInternal or external flash (e.g. SPI-connected). MCAL differences must be transparent to upper layers. All memory architectures supported without changing upper BSW.
NvData handlingUser data changes during FOTA must go through NvM module. Data model migration must happen before partition switch. Direct flash access forbidden.
External flash via SPI: if new SW is stored externally and needs copying to internal flash during activation, a SECOND verification step after copy is recommended to detect SPI transfer corruption.
Partition invalidation — security concept

After successful activation and boot confirmation: FOTA Master may trigger invalidation of old (now inactive) partition. Purpose: prevent unauthorized rollback to vulnerable previous SW. Implementer decides how: flag in flash, delete partition, invalidate checksum, etc.

Important: FOTA Master should NOT invalidate immediately after activation — it must wait until all ECUs in a multi-ECU campaign have confirmed successful update. A rollback may still be needed if another ECU fails.

Two rollback triggers
Master-triggered rollback

FOTA Master detects error (version incompatibility, backend request, system check fail). Master sends RC_Rollback() to FOTA Target → Target → ACTIVATE state with previous partition flag → ECU resets → Bootloader activates old partition.

Autonomous rollback

ECU cannot boot new SW successfully → bootloader detects boot failure → autonomously switches back to previous partition → ECU comes back up on old SW. FOTA Master then reads SW version via DID to confirm rollback occurred.

Multi-ECU rollback rule

If any ECU in a multi-ECU update campaign requires rollback, ALL dependent ECUs must be rolled back. Dependency management is FOTA Master's responsibility. FOTA Target ECU has no visibility into other ECUs' update status.

DcmResponseToEcuReset=AFTER_RESET: set this to TRUE on all FOTA Target ECUs. Without it, FOTA Master has no way to know when each ECU has completed its reset after activation. Setting DcmSendRespPendOnRestart=TRUE additionally informs Master about restart-in-progress during activation.
Security features
Encrypted transferFOTA Master → FOTA Target transfer can be encrypted. Target decrypts before writing to flash.
Encrypted at restNew SW in flash can be encrypted. HSM (Hardware Security Module) can handle crypto operations.
Dcm→Memory StackTransfer from Dcm buffer to memory stack can also be encrypted.
E2E protectionAUTOSAR end-to-end protection on FOTA Master ↔ FOTA Target communication to detect communication errors.
Verification strategy is explicitly out of scope of this document. Initial image authenticity/integrity check = FOTA Master's job. Semantic/functional integrity check on Target = implementer-specific. AUTOSAR only says it should happen.
Migration scenario — NvData model changes

When new SW ships with a changed NvData model (e.g. new DID layout, renamed blocks): old data model must be migrated to new model BEFORE partition switch. Migration uses NvM module features (create, delete, move NvDataBlocks). This migration must happen while old SW is still running — implemented by FOTA Handler or FOTA Master. Do NOT extend NvM module itself.

Important context before configuring
FOTA Handler is a CDD — it is NOT a standard AUTOSAR BSW module with a generated configurator. You configure the surrounding modules (Dcm, NvM, OS) to support FOTA, and implement the CDD callouts yourself. There is no "FOTA" module dropdown in DaVinci like there is for CanNm.
Dcm configuration for FOTA
1
Add FOTA protocol row: DcmDslProtocolRow for FOTA. Set DcmDslProtocolPriority = lowest (0 or lowest value). This ensures any other diagnostic request preempts FOTA.
2
FOTA buffer: DcmDslBufferSize on the FOTA protocol row. This controls max chunk size. Best practice: do not exceed other protocol buffer sizes. Separate buffer from normal UDS buffer.
3
FOTA diagnostic session: Add custom user session for FOTA (DcmDspSessionRow). Link SecurityAccess level to this session. FOTA can only run inside this session.
4
TransferData callout: Configure 0x36 service with callout to FOTA_ProcessTransferDataWrite. This is the entry point from Dcm to FOTA Handler.
5
Preemption callback: DcmDslCallbackDCMRequestService → point to FOTA Handler's preemption handler function. This fires when higher-priority protocol preempts FOTA.
6
RoutineControl services: Add RC entries for RC_InitializeFOTA, RC_TriggerActivation, RC_Rollback, RC_VerifyInstallation (opt), RC_Cancel. Each with callouts into FOTA Handler functions.
7
DIDs: Add DID entries for: FOTA status, last written address, SW version, chunk ID. Read callbacks → FOTA Handler getter functions. Write callbacks → FOTA Handler setter functions.
8
ECU Reset behavior: DcmResponseToEcuReset = AFTER_RESET (send positive response after boot). DcmSendRespPendOnRestart = TRUE (inform Master ECU about in-progress restart).
OS configuration for FOTA
1
Add FOTA main task: OS task (OsTask) for FOTAHandlerMain. Set priority LOWER than application tasks. FOTA must not interfere with normal operation. Cyclic scheduling (e.g. every 10ms).
2
InterRunnableVariable: Shared variable between FOTA_ProcessTransferDataWrite (Dcm task context) and FOTAHandlerMain (FOTA task context). Needs proper access protection / OS spinlock if on multicore.
NvM configuration for FOTA
1
FOTA state block: NvMBlockDescriptor for FOTA process state. Stores: current state enum, last written address, last chunk ID. Must survive power cycles.
2
Priority: Give FOTA NvM block appropriate write priority. State must be written before next OS cycle on preemption (use NvM immediate write if available).
Partition switch flag location: configure in a memory section accessible by bootloader (linker script + bootloader memory map). Not in RAM. Not in NvM-managed data flash that the bootloader cannot read. Often a dedicated flash sector shared between boot and app.
Common DaVinci config mistakes
FOTA buffer same as normal UDS buffer → normal UDS sessions starved during FOTA → always give FOTA its own DcmDslProtocolRow with own buffer
DcmDslProtocolPriority not set to lowest → FOTA cannot be preempted by real diagnostic requests → violates FOTA non-interference requirement
DcmResponseToEcuReset not set to AFTER_RESET → FOTA Master never gets boot confirmation after activation → Master assumes ECU bricked
Partition flag in application RAM → bootloader cannot see it → partition switch never happens → ECU boots old SW → FOTA Master confused by mismatched version
Interface testing checklist
✓
Verify GetFOTAStatus DID returns IDLE at ECU startup. State must be persisted and reset to IDLE after successful activation + boot.
✓
Verify DiagnosticSessionControl(FOTA_SESSION) is rejected without prior SecurityAccess. Unauthorized flashing prevention.
✓
Verify DCM_WRITE_PENDING responses during chunk processing. Monitor CANoe trace: ResponsePending (0x78 NRC) must appear repeatedly while flash write is in progress.
✓
Simulate driving cycle end during PROCESSING state. On next startup: verify FOTA status DID shows correct interrupted state and last written address is valid.
✓
Test preemption: send normal UDS request during active chunk transfer. Verify: FOTA stops, normal UDS is served, FOTA Master sees timeout, resume works correctly after reinitialization.
✓
Verify partition flag location: after RC_TriggerActivation, read the flag location directly in flash memory (via memory tool). Must be readable by bootloader before ECU reset.
✓
Test autonomous rollback: corrupt activation (e.g. bad checksum in new partition). Verify ECU boots old SW. FOTA Master reads SW version DID → must show previous version.
✓
Test RequestTransferExit with OpStatus=IDLE: verify state returns to IDLE, inactive partition is invalidated. Test with OpStatus=READY: verify state goes to READY, activation can be triggered.
✓
Verify DcmResponseToEcuReset=AFTER_RESET: after activation ECU reset, FOTA Master must receive positive response after new SW boots (not before reset). Check timing in CANoe.
✓
NvData migration: if new image changes NvM block layout, verify migration runs before partition switch. Read NvM blocks after activation — must reflect new data model without data loss.

Key facts to memorize for industry exam
FOTA Handler = CDD, not BSW module
Not enough features defined to warrant a full BSW module. CDD allows flexible connection through all BSW layers. May become BSW module in future releases.
Activation requires vehicle safe state
Integrator's responsibility. Standstill + engine off + parking brake. AUTOSAR does not enforce this — it is a system-level requirement.
Activation + Rollback sequences = bootloader
Out of AUTOSAR scope. No sequence charts in AUTOSAR spec for these. Done in bootloader context. AUTOSAR only defines up to triggering the reset.
FOTA Master = UCM-Master = Adaptive AUTOSAR
Classic ECUs too resource-constrained to act as master. Adaptive platform ECU (domain controller) runs UCM-Master. Speaks UDS over vehicle bus to Classic targets.
Installation spans multiple driving cycles
No requirement to complete in one drive. NvM persists state. Resume is transparent — same init sequence regardless of first start or resume.
Preemption = FOTA Master timeout
No notification to FOTA Master on preemption. Master detects via UDS timeout. After timeout: reinitialize, read DIDs, resume. This is by design — notification not in scope.
