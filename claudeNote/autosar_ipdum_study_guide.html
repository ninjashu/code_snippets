Concept & Architecture

The IpduM module is the communication "translator" that allows multiple logical messages (COM I-PDUs) to share a single physical bus frame ID, or to pack many small PDUs into one large container frame.

🧠 The Big Picture — Why IpduM Exists

On CAN, every frame has a fixed 11-bit or 29-bit identifier. In classic AUTOSAR, each COM I-PDU gets its own bus ID — but this wastes bandwidth. IpduM solves two problems:

Problem 1 — Multiplexing

Multiple signal layouts need to share one CAN ID. Think of a sensor that alternately reports temperature, pressure, or status — same ID, different payload layout each time. The selector field says which layout is active.

CANFlexRayLIN

Problem 2 — Container PDUs

Ethernet and CAN-FD support large frames (up to 4096+ bytes). Sending many tiny 8-byte PDUs wastes headers. IpduM packs multiple small PDUs into one large "container" frame — like packing boxes into a shipping container.

EthernetCAN-FDFlexRay

📐 IpduM Position in AUTOSAR Stack

IpduM lives beside the PDU Router, not strictly above or below it. It sits between COM and the Communication Hardware Abstraction layer.

COM Module — creates/interprets signals and signal groups
⇅
PDU Router (PduR) — routes PDUs between all BSW modules
⇅   ⇅
★ IpduM — I-PDU Multiplexer (sits BESIDE PduR)
⇅
Communication Hardware Abstraction (ComHAL)
⇅

⇅
Physical Bus (CAN, CAN-FD, FlexRay, LIN, Ethernet)
📌 Key Architectural Requirement
SWS_IpduM_00097: IpduM shall be implemented so that no other modules depend on it. A system can be built completely without IpduM if it's not needed — it is always optional.

⚠️ Exam Trap
IpduM does NOT set the selector field (SWS_IpduM_00098). The COM module must already have the correct selector field value embedded in the dynamic part I-PDU before passing it to IpduM. This is configured as a signal with an init value that is never updated at runtime.

PDU Anatomy & Layout

Understanding how bits are laid out inside a multiplexed I-PDU is the foundation of everything. Get this wrong in a project and you'll have garbled data on the bus.

🔬 Multiplexed I-PDU Structure

A multiplexed I-PDU = Static Part + Dynamic Part. The dynamic part contains the selector field.

📏 Key Configuration Parameters for Layout

ParameterWhereValue RangeMeaning
IpduMSelectorFieldLengthIpduMSelectorField1–16 bitsHow many bits is the selector field?
IpduMSelectorFieldPositionIpduMSelectorField0–2031 bitsWhere in the PDU does the selector start? (CAN: 0–63, CAN-FD: 0–511, FlexRay: 0–2031)
IpduMByteOrderIpduMTxRequest / IpduMRxIndicationBIG_ENDIAN / LITTLE_ENDIANDetermines if SegmentPosition = MSB (big-endian) or LSB (little-endian) bit position
IpduMSegmentPositionIpduMTxDynamic/StaticSegment0–2031 bitsBit position of this segment in the multiplexed PDU
IpduMSegmentLengthIpduMTxDynamic/StaticSegment1–2032 bitsLength of this segment in bits
📐 Segment Placement Rule — SWS_IpduM_00224
All static and dynamic parts shall be configured to allocate exactly the same bits in both the multiplexed and de-multiplexed I-PDUs. IpduM does only masking and copying — no bit shifting, no endianness conversion of data bytes.

📦 Container PDU Layout

Container PDUs come in two flavors: Dynamic (with headers) and Static (fixed offsets, no headers).

Dynamic Container (SHORT header)

/* IPDUM_HEADERTYPE_SHORT */
| Header (32-bit)    | Payload  |
| ID[23:0] | DLC[7:0] | data...  |
/* then next contained PDU: */
| Header (32-bit)    | Payload  |
| ...

Dynamic Container (LONG header)

/* IPDUM_HEADERTYPE_LONG */
| Header (64-bit)         | Payload |
| ID[31:0] | DLC[31:0]   | data... |
/* Supports very large PDUs */
/* ID=0 means padding, stop */

Static Container (IPDUM_HEADERTYPE_NONE)

/* Fixed layout — positions configured per contained PDU */
| ContainedPDU_A (offset=0)  | update_bit_A |
| ContainedPDU_B (offset=N)  | update_bit_B |
| ...
/* Update bits show if PDU was written since last send */
/* Only COLLECT_LAST_IS_BEST semantics allowed here */
⚠️ Critical Rule — Header byte order
IpduMHeaderByteOrder (IPDUM_BIG_ENDIAN or IPDUM_LITTLE_ENDIAN) applies to the container header bytes only, not to the contained PDU payloads. Set this to match how the ECU interprets the header ID and DLC fields.

Initialization

Correct initialization is critical. Call order, buffer pre-loading, and the init value chain are all exam-tested topics.

⚙️ Initialization Sequence

1
EcuM calls IpduM_Init(config) — before ANY other IpduM function. SWS_IpduM_00084: behavior is unspecified until IpduM_Init completes.
↓
2
All global variables initialized — SWS_IpduM_00033. Internal state goes from UNINIT to INIT.
↓
3
Tx buffer filled with IpduMIPduUnusedAreasDefault — SWS_IpduM_00067. The pattern covers areas that are neither static nor dynamic part bits.
↓
4
Initial dynamic part copied from COM I-PDU — SWS_IpduM_00068. The reference is IpduMInitialDynamicPart → IpduMTxDynamicPduRef. The selector field inside is also initialized here implicitly.
↓
5
Static part initial values copied from COM I-PDU — SWS_IpduM_00143. Reference: IpduMTxStaticPart → IpduMTxStaticPduRef.

📌 Initialization Function API

void IpduM_Init( const IpduM_ConfigType* config );
/* Service ID: 0x00
   Sync/Async: Synchronous
   Reentrancy: Non-Reentrant
   Must be called ONCE before any other IpduM function */
⚠️ UNINIT Error — SWS_IpduM_00084 + E_UNINIT (0x20)
If DET is enabled (IpduMDevErrorDetect=TRUE), calling any API except IpduM_MainFunctionTx, IpduM_MainFunctionRx, and IpduM_GetVersionInfo before IpduM_Init shall report IPDUM_E_UNINIT (0x20) to DET.

🏗️ Initialization Buffer Strategy

For performance, the initial bit pattern for the Tx buffer can be pre-calculated at configuration time and copied as a single block at runtime — no signal-by-signal initialization loop needed. This is an important optimization for embedded systems.

Multi-Core / Multi-Partition Init

For multi-partition systems, IpduM may provide one IpduM_Init per partition. Each partition initializes only its own IpduM resources.

Transmission

Transmission has four sub-topics that are individually examined: the transmit request, the trigger conditions, JIT update, and transmit confirmation.

📤 Transmission Request — IpduM_Transmit

COM sends its I-PDUs (static part, dynamic part) via PduR → IpduM_Transmit. IpduM assembles the multiplexed PDU and sends it out.

SWS_IpduM_00015 — Merge Rule
IpduM merges the corresponding two COM I-PDUs (static part + last received dynamic part) into one single IpduM I-PDU with a new unique I-PDU ID, then sends it to PduR.
🔴 SWS_IpduM_00152 — Pending Confirmation Blocks New Transmit
As long as no TxConfirmation is received for an IpduM I-PDU, IpduM_Transmit shall return E_NOT_OK for any new Tx request for the same IpduM I-PDU. This prevents overlapping transmissions.

🎛️ Transmission Trigger Modes — SWS_IpduM_00021

Configured via IpduMTxTriggerMode — determines which event triggers the merged PDU to be sent:

ModeEnum ValueTriggers WhenUse Case
STATIC_PART_TRIGGER STATIC_PART_TRIGGER Static COM I-PDU arrives at IpduM Static part drives timing (e.g., periodic static + event-driven dynamic)
DYNAMIC_PART_TRIGGER DYNAMIC_PART_TRIGGER Dynamic COM I-PDU arrives at IpduM Dynamic part drives timing
STATIC_OR_DYNAMIC STATIC_OR_DYNAMIC_PART_TRIGGER Either part arrives Either part can trigger — most flexible
NONE NONE Never from COM update — only via TriggerTransmit from lower layer FlexRay static segment, LIN Master schedule
⚠️ MDT Limitation
Not all trigger modes guarantee Minimum Delay Time (MDT). If triggered by static-or-dynamic or dynamic-only, COM treats parts as independent PDUs with their own timing — the IpduM assembled PDU can exceed MDT constraints.

⚡ JIT (Just-In-Time) Update — SWS_IpduM_00168

Sometimes a part might have stale data (e.g., update-bits already consumed). JIT update solves this by fetching the latest data immediately before transmission.

JIT Mechanism

If IpduMJitUpdate=TRUE for a part, and the other part triggered the Tx, IpduM calls PduR_IpduMTriggerTransmit to fetch the JIT part's latest data before calling PduR_IpduMTransmit.

// Dynamic part triggers → static part has JitUpdate=TRUE:
1. IpduM_Transmit(dynamicHandleId, ...)  // dynamic arrives
2. IpduM calls PduR_IpduMTriggerTransmit  // fetch fresh static
   → if E_NOT_OK: DO NOT SEND (SWS_IpduM_00171)
3. IpduM assembles and calls PduR_IpduMTransmit
🔴 JIT Failure Rule — SWS_IpduM_00171 & 00172
If JIT update returns E_NOT_OK: for triggered Tx → do NOT send the PDU. For IpduM_TriggerTransmit → return E_NOT_OK upward.

✅ Transmission Confirmation — SWS_IpduM_00022

When PduR calls IpduM_TxConfirmation, IpduM translates it into confirmations for the COM I-PDUs that were inside the last sent multiplexed PDU.

IpduMTxStaticConfirmationIpduMTxDynamicConfirmationCOM confirmations generated
FALSEFALSE0 — none forwarded
TRUEFALSE1 — static part only
FALSETRUE1 — dynamic part only
TRUETRUE2 — both, but both are derived from same confirmation event

Reception

On the receive path, IpduM reads the selector field and splits the incoming multiplexed PDU into static and dynamic parts, routing each to the correct COM I-PDU destination.

📥 Reception Flow

1
Bus frame arrives at interface (CanIf/FrIf/LinIf)
↓
2
PduR calls IpduM_RxIndication(RxPduId, PduInfoPtr)
↓
3
IpduM checks if a static part is configured for this PDU ID → if yes, extracts static segments and forwards via PduR_IpduMRxIndication (SWS_IpduM_00041)
↓
4
IpduM reads the selector field value from the received payload at the configured bit position
↓
5
IpduM looks up the matching IpduMRxDynamicPart with matching IpduMRxSelectorValue → extracts dynamic segments → forwards via PduR_IpduMRxIndication (SWS_IpduM_00042)
Silent Ignore Rules (no error)
  • Received I-PDU has segments not configured for this ECU → ignored silently
  • I-PDU has PduLength=0 → ignored silently
  • Selector value not in configured IpduMRxDynamicPart list → dynamic part ignored silently

🌐 Metadata Forwarding — SWS_IpduM_00227

If IpduMMetaDataSupport=TRUE, IpduM shall forward the received MetaData (e.g., CAN address information) along with all demultiplexed parts to the upper layer.

⚡ Direct COM Invocation Optimization — SWS_IpduM_00140

If IpduMRxDirectComInvocation=TRUE, IpduM can call COM API directly (bypassing PduR) for Rx/Tx confirmation paths. This is a performance optimization — IpduM.c must include Com.h in this case.

Container PDU Handling

Container PDUs are the "multiple-PDU-to-one-frame" feature. Master the collection semantics, trigger conditions, and queue behavior.

📊 Collection Semantics — The Most Tested Topic

IPDUM_COLLECT_QUEUED

Every instance is stored immediately in IpduM's buffer when IpduM_Transmit is called. FIFO-like ordering. Multiple instances of the same contained PDU CAN appear in one Container PDU.

Guarantees: every value appears on the wire.

ReliabilityE2E-safe

IPDUM_COLLECT_LAST_IS_BEST

Only the latest data is sent. Transmission request is stored; data is fetched via PduR_IpduMTriggerTransmit immediately before sending. Only ONE instance per contained PDU per Container PDU.

Trade-off: values can be lost between transmissions.

May lose dataE2E counter risk

🔴 Critical Mixing Rule — SWS_IpduM_00219
A Container PDU shall contain either solely QUEUED or solely LAST_IS_BEST contained PDUs. Mixed semantics are rejected at configuration time. Also: LAST_IS_BEST containers cannot have a queue (SWS_IpduM_00260).

⏱️ Transmission Trigger Conditions (Containers)

TriggerConditionNotes
IpduMContainedTxPduTrigger=ALWAYSContainer triggers immediately when this contained PDU is addedMost aggressive
IpduMContainedTxPduTrigger=NEVERThis PDU alone never triggers the containerRelies on other triggers
IpduMContainerTxFirstContainedPduTrigger=TRUEContainer queued for Tx when the first contained PDU is addedUsed for time-triggered buses
IpduMContainerTxSizeThreshold exceededContainer triggers when payload crosses the threshold bytesBefore max PDU size is reached

📬 Queue Behavior (IPDUM_COLLECT_QUEUED)

Configured via IpduMContainerQueueSize (1–255 instances).

Queue Overflow — SWS_IpduM_00199

If creating a new Container PDU instance would exceed IpduMContainerQueueSize, the oldest instance is discarded and IPDUM_E_QUEUEOVFL (0x31) is reported via Det_ReportRuntimeError.

Transmit Retry — SWS_IpduM_00195

If PduR_IpduMTransmit returns E_NOT_OK, the same transmit request is retried during the next IpduM_MainFunctionTx call, up to IpduMMaxTransmitRetries times (default=10).

🔀 TriggerMode: DIRECT vs TRIGGERTRANSMIT

IpduMContainerTxTriggerModeAssembly TimingFlow
IPDUM_DIRECT Container assembled when trigger fires, then PduR_IpduMTransmit called with data pointer and length IpduM pushes data
IPDUM_TRIGGERTRANSMIT PduR_IpduMTransmit called with NULL SduDataPtr first (reserves slot); container assembled only in IpduM_TriggerTransmit context Lower layer pulls data

📥 Container Reception

IPDUM_PROCESSING_IMMEDIATE

Container PDU is processed in the context of IpduM_RxIndication — all contained PDUs extracted and forwarded immediately. High-priority, no delay.

IPDUM_PROCESSING_DEFERRED

Container PDU is queued. Processing happens in the next IpduM_MainFunctionRx call. Deferred PDUs processed in reception order. Reduces interrupt context work.

ACCEPT_CONFIGURED vs ACCEPT_ALL (Rx)
IPDUM_ACCEPT_CONFIGURED: Only contained PDUs whose IpduMContainedRxInContainerPduRef points to this container are processed.
IPDUM_ACCEPT_ALL: Any contained PDU matching an IpduMContainedRxPdu without an IpduMContainedRxInContainerPduRef is accepted. Unmatched PDUs silently discarded.

⚡ Priority Handling (LAST_IS_BEST)

When IpduMContainedTxPduPriorityHandling=TRUE in IpduMGeneral:

  • Contained PDUs are placed in the Container in priority order (0=highest, 255=lowest default)
  • Same-priority PDUs: ordered by first occurrence of their transmission request
  • If priority handling is FALSE: order of first Tx request arrival is preserved

Vector DaVinci Configuration

Step-by-step walkthrough of configuring IpduM in Vector DaVinci Developer / DaVinci Configurator Pro for a typical CAN project.

🛠️ Step 1: Enable IpduM Module

1
In DaVinci Developer, go to BSW Modules → search for "IpduM" → Add to project
↓
2
Navigate to IpduM → IpduMGeneral. Set:
  • IpduMDevErrorDetect = TRUE (for dev/test builds)
  • IpduMStaticPartExists = TRUE (if using mux PDUs with static parts)
  • IpduMVersionInfoApi = TRUE (optional)
  • IpduMHeaderByteOrder = IPDUM_LITTLE_ENDIAN (for CAN usually)
  • IpduMMaxTransmitRetries = 3 (tune as needed)

🛠️ Step 2: Configure a Multiplexed TX Pathway

1
Under IpduMConfig → IpduMTxPathway: Add a new IpduMTxPathway instance
↓
2
Under IpduMTxRequest:
  • IpduMOutgoingPduRef → select the outgoing PDU (IpduM PDU defined in PduR)
  • IpduMTxTriggerMode → STATIC_PART_TRIGGER (or as needed)
  • IpduMIPduUnusedAreasDefault → 0x00 (fill unused bits with zeros)
  • IpduMByteOrder → LITTLE_ENDIAN
  • IpduMInitialDynamicPart → reference to the initial dynamic part container
↓
3
Configure IpduMSelectorField:
  • IpduMSelectorFieldPosition → bit position in the outgoing PDU (e.g., 0 for first byte)
  • IpduMSelectorFieldLength → 8 (for 1-byte selector)
↓
4
Add IpduMTxStaticPart (if needed):
  • IpduMTxStaticPduRef → COM I-PDU representing static signals
  • IpduMTxStaticConfirmation = TRUE/FALSE
  • IpduMJitUpdate = FALSE (typical)
↓
5
Add IpduMTxDynamicPart (one per selector value):
  • IpduMTxDynamicHandleId → auto-generated or manual ID
  • IpduMTxDynamicPduRef → COM I-PDU for this dynamic layout
  • IpduMTxDynamicConfirmation = TRUE/FALSE
↓
6
Add IpduMTxDynamicSegment(s) — define which bits of the COM I-PDU map to which bits in the outgoing Mux PDU:
  • IpduMSegmentPosition → start bit in the Mux PDU
  • IpduMSegmentLength → how many bits

🛠️ Step 3: Configure a Multiplexed RX Pathway

1
IpduMConfig → IpduMRxPathway → IpduMRxIndication:
  • IpduMRxHandleId → auto or manual, must match PduR routing table
  • IpduMRxIndicationPduRef → incoming multiplexed PDU from PduR
  • IpduMByteOrder → LITTLE_ENDIAN (for CAN)
↓
2
IpduMSelectorField (under RxIndication):
  • IpduMSelectorFieldPosition / IpduMSelectorFieldLength → same as Tx side for matching PDU
↓
3
IpduMRxStaticPart: IpduMOutgoingStaticPduRef → COM I-PDU for static signals destination
↓
4
IpduMRxDynamicPart (one per selector value):
  • IpduMRxSelectorValue → the selector field value this entry handles
  • IpduMOutgoingDynamicPduRef → COM I-PDU destination

🛠️ Step 4: Container PDU Configuration (Ethernet/CAN-FD)

IpduMContainerTxPdu

  • IpduMContainerTxPduRef → the large PDU (Ethernet frame) in PduR
  • IpduMContainerHeaderSize → IPDUM_HEADERTYPE_SHORT (32-bit) or LONG (64-bit)
  • IpduMContainerTxTriggerMode → IPDUM_DIRECT or IPDUM_TRIGGERTRANSMIT
  • IpduMContainerTxSendTimeout → e.g., 0.005 (5 ms) — max wait time before forced Tx
  • IpduMContainerTxSizeThreshold → e.g., 1400 bytes — trigger before full
  • IpduMContainerQueueSize → e.g., 3 (queue depth)
  • IpduMContainerTxFirstContainedPduTrigger → TRUE for time-triggered buses

IpduMContainedTxPdu (for each contained PDU)

  • IpduMContainedTxPduRef → small PDU
  • IpduMContainedTxInContainerPduRef → link to its container
  • IpduMContainedPduHeaderId → unique ID used in the container header (≥1)
  • IpduMContainedTxPduCollectionSemantics → IPDUM_COLLECT_QUEUED or IPDUM_COLLECT_LAST_IS_BEST
  • IpduMContainedTxPduTrigger → IPDUM_TRIGGER_ALWAYS or IPDUM_TRIGGER_NEVER
  • IpduMContainedTxPduSendTimeout → PDU-specific deadline override
  • IpduMContainedTxPduConfirmation → TRUE if upper layer needs TxConfirmation
  • IpduMContainedTxPduPriority → 0 (highest) to 255 (lowest)

🛠️ Step 5: PduR Routing Table Setup

Critical PduR Configuration
  • COM → PduR → IpduM: route the static and dynamic COM I-PDUs to IpduM's Transmit
  • CanIf → PduR → IpduM: route the incoming multiplexed frame ID to IpduM's RxIndication
  • IpduM → PduR → CanIf: route the assembled IpduM PDU to CanIf for Tx
  • IpduM → PduR → COM: route demultiplexed parts back to COM's RxIndication
  • The handle IDs in IpduM config must match PduR's look-up table entries

🛠️ Step 6: Scheduler / BSW Task Configuration

In DaVinci: BSW Task Manager or OS Task configuration:

  • Add IpduM_MainFunctionTx to a cyclic task — period must match IpduMMainTxTimeBase
  • Add IpduM_MainFunctionRx to a cyclic task — period must match IpduMMainRxTimeBase
  • For multi-core: assign each IpduMMainFunctionTx/Rx instance to its correct partition via IpduMMainTxPartitionRef / IpduMMainRxPartitionRef

Debugging & Interface Testing

Real-world scenarios, what to check, and how IpduM problems typically manifest in automotive integration.

🔍 Common Integration Issues & Fixes

🔴 PDU not transmitted / IpduM_Transmit always returns E_NOT_OK▼

Likely causes:

  • TxConfirmation not being received → check lower layer CAN/FlexRay configuration
  • IpduMTxConfirmationPduId not configured or wrong → PduR can't route confirmation back
  • IpduM_Init not called before IpduM_Transmit → IPDUM_E_UNINIT in DET log
  • Wrong TxPduId → check PduR routing table matches IpduMTxStaticHandleId/IpduMTxDynamicHandleId

Debug steps: Check DET log for errors. Set breakpoint in IpduM_TxConfirmation — is it being called? Verify PduR routing for the outgoing PDU.

🟡 Wrong data on bus / selector field value incorrect▼

Root cause: IpduM does NOT set the selector field. It's already in the COM I-PDU. Check:

  • Is the COM I-PDU representing the dynamic part initialized with the correct selector value?
  • Is IpduMSelectorFieldPosition/Length correct for both Tx and Rx sides?
  • Is IpduMByteOrder consistent between sender and receiver ECUs?
  • Are segment positions (IpduMSegmentPosition) correct for the bus byte order?

Debug: Capture raw bus frame with Vector CANalyzer/CANoe. Manually decode the selector field bits and compare to what COM initializes.

🟡 Container PDU received but contained PDUs not forwarded to upper layer▼
  • Check IpduMContainerRxAcceptContainedPdu — ACCEPT_CONFIGURED vs ACCEPT_ALL
  • Verify IpduMContainedRxPduShortHeaderId / LongHeaderId matches what's in the received frame
  • Check IpduMContainedRxInContainerPduRef — is it pointing to the right container?
  • Ensure IpduMContainerPduProcessing — if DEFERRED, the MainFunctionRx must be running
  • Check for IPDUM_E_HEADER in DET — header might be corrupt
  • Verify IpduMContainerHeaderSize matches the actual header in the frame (SHORT vs LONG)
🟡 IPDUM_E_QUEUEOVFL reported — contained PDUs being dropped▼
  • IpduMContainerQueueSize too small for the application's burst rate
  • IpduM_MainFunctionTx period too long — not processing queue fast enough
  • Lower layer (CanIf) queue full — PduR_IpduMTransmit returning E_NOT_OK, causing retries to pile up
  • Solution: increase IpduMContainerQueueSize, reduce MainFunctionTx period, or tune IpduMContainerTxSendTimeout
🔴 Static container: receiver gets garbage in "not updated" PDU slots▼

Transmitter sends Static Container before all contained PDUs are updated:

  • Configure IpduMUnusedAreasDefault to a recognizable pattern (e.g., 0xFF) so receiver can detect stale slots
  • Configure IpduMPduUpdateBitPosition per contained PDU — receiver checks update bit before processing
  • Ensure the unused areas default aligns with bus padding value (CAN-FD padding = 0xCC typically)
  • SWS_IpduM_00233: IpduM fills non-updated areas before sending
  • SWS_IpduM_00236: Receiver ignores contained PDUs whose update bit is NOT set

Mux PDU TX Tests

  • Trigger from static part only
  • Trigger from dynamic part only
  • Both parts trigger → MDT check
  • Multiple dynamic layouts
  • JIT update enabled path
  • JIT failure → no Tx verify

Mux PDU RX Tests

  • Static part extracted correctly
  • Each selector value → correct dynamic

  • No static part configured
  • MetaData forwarded

  • Timeout trigger fires
  • Size threshold trigger

  • Create a CAPL script that decodes the selector field and logs which dynamic layout is active per frame

📌 Selector Field Constraints

  • Size: 1–16 bits (configurable)
  • Position: 0–2031 bits (CAN: 0–63, CAN-FD: 0–511, FlexRay: 0–2031)
  • Position means: MSB position (BIG_ENDIAN) or LSB position (LITTLE_ENDIAN)
  • Only one selector field per multiplexed I-PDU
  • Not all selector values need to be used — SWS_IpduM_00011
  • IpduM reads but never writes the selector field

📌 Configuration Rules Summary

Forbidden Combinations

  • QUEUED + LAST_IS_BEST in same container
  • LAST_IS_BEST + IpduMContainerQueueSize
  • HEADERTYPE_NONE + COLLECT_QUEUED
  • Different MetaDataTypes in same container
  • IpduMPduUpdateBitPosition on dynamic container

Valid Combinations

  • LAST_IS_BEST + IPDUM_DIRECT ✓
  • LAST_IS_BEST + IPDUM_TRIGGERTRANSMIT ✓
  • QUEUED + IPDUM_DIRECT ✓
  • QUEUED + IPDUM_TRIGGERTRANSMIT ✓
  • Static container + update bits ✓
