AUTOSAR CAN Interface (CanIf) complete study guide — R24-11

What is CanIf?

CanIf (CAN Interface) is the AUTOSAR BSW module that sits between the low-level CAN device drivers and upper communication service layers. It provides one unified abstraction so upper layers (PduR, CanNm, CanTp, CanSm) never need to know which physical CAN hardware is underneath.

Key Responsibilities of CanIf

Tx Path

Accepts CanIf_Transmit() from upper layers → maps PduId to HTH → calls Can_Write() → handles buffering if CAN_BUSY

Rx Path

Receives CanIf_RxIndication() from CanDrv → software filtering → DLC check → dispatch to upper layer via LSduR_CanIfRxIndication()

Controller Mode

CanIf_SetControllerMode() → forwards to CanDrv, receives CanIf_ControllerModeIndication() callback → notifies CanSm

Transceiver Mode

CanIf_SetTrcvMode() → forwarded to CanTrcv. Wake-up detection, wake-up validation, partial networking filter

L-PDU vs L-SDU — critical distinction for exams

CAN L-PDU: What the CAN Driver sees — CanId + DLC + data (raw frame).
CAN L-SDU: What upper layers see — the data payload only (stripped of CAN framing). CanIf bridges the two views. When you call CanIf_Transmit() you pass an L-SDU; CanIf builds the full L-PDU and hands it to CanDrv.

HOH — Hardware Object Handles (exam favourite)

HTH — Hardware Transmit Handle

References a CAN TX mailbox. One or more Tx L-PDUs map to one HTH. CanIf uses it to call Can_Write(HTH, ...). Associated with CanIfBufferCfg for software buffering.

HRH — Hardware Receive Handle

References a CAN RX mailbox. Passed back in CanIf_RxIndication(). If HRH is BasicCAN → software filter runs. If FullCAN → exact match, no software filter needed.

CAN Controller Mode State Machine (CanIf view)

State transitions are asynchronous — CanIf_SetControllerMode() returns E_OK immediately; the actual transition is confirmed later via CanIf_ControllerModeIndication() callback from CanDrv.

PDU Channel Mode State Machine

PDU modes control Tx/Rx independently per CAN channel. Can only be changed when controller is in CAN_CS_STARTED.

CANIF_OFFLINE (0x00)
No Tx, No Rx ⇄ CANIF_TX_OFFLINE (0x01)
No Tx, Rx enabled ⇄ CANIF_TX_OFFLINE_ACTIVE (0x02)
Simulated Tx, Rx enabled ⇄ CANIF_ONLINE (0x03)
Full Tx + Rx

Key rules (examinable):
• OFFLINE → cleared by CanIf_Init() and when CS_SLEEP is set
• TX_OFFLINE → set automatically on CS_STOPPED or BusOff
• TX_OFFLINE_ACTIVE → diagnostic passive mode: upper layer gets TxConfirmation immediately (no bus traffic)
• Setting CS_SLEEP → forces PDU mode to CANIF_OFFLINE

Transceiver Mode State Machine

CANTRCV_TRCVMODE_NORMAL ⇄ CANTRCV_TRCVMODE_STANDBY ⇄ CANTRCV_TRCVMODE_SLEEP

Important: NORMAL→SLEEP is not direct; must go NORMAL→STANDBY→SLEEP. CanIf does NOT enforce this — it is up to CanTrcv. CanIf just passes the request through via CanTrcv_SetOpMode().

Software Receive Filter — BasicCAN vs FullCAN

FullCAN reception

One mailbox = exactly one CanId. Hardware filter is exact match. No software filter needed. Direct HRH→L-SDU mapping. Fast, but uses more HW mailboxes.

BasicCAN reception

One mailbox = range of CanIds. Hardware filter is mask/range based. Software filter is mandatory. CanIf searches the HRH's Rx L-PDU list using linear/table/hash algorithm (implementation choice).

Software filter algorithms (implementation choice)

Linear search — simple, slow for many IDs. Table/binary search — faster. Hash search — O(1) but complex. Filter can use: exact CanId, range (upper+lower limit), or base+mask. Priority: exact CanId > smaller range > larger range.

Transmit Buffering (BasicCAN only — NOT for CAN XL)

If Can_Write() returns CAN_BUSY (HW mailbox occupied), CanIf can buffer the L-PDU in software CanIfTxBuffer — but only if:
1. CanIfPublicTxBuffering = TRUE
2. CanIfBufferSize > 0 for that HTH
3. Not already buffered (CanIf overwrites if same PduId)
4. Buffer not full (if full and new PduId → return E_NOT_OK)

On next TxConfirmation callback, CanIf drains the buffer in CAN priority order (ISO 11898-1), not FIFO. Data integrity protected via SchM exclusive areas (SchM_Enter/Exit_CanIf).

DLC (Data Length Check)

CanIf accepts Rx L-PDUs with received length ≥ configured CanIfRxPduDataLength.
If check fails → report CANIF_E_INVALID_DATA_LENGTH to DET as runtime error.
DLC check enabled globally by CanIfPrivateDataLengthCheck; can be disabled per-PDU via CanIfRxPduDataLengthCheck.

Tx size limits: Classic CAN max 8 bytes, CAN FD max 64 bytes, CAN XL max 2048 bytes.

Wake-up sequence

1. Sleep → CAN bus activity detected → EcuM wakes up
2. EcuM calls CanIf_CheckWakeup(WakeupSource) → CanIf queries CanDrv/CanTrcv
3. If confirmed → CanIf_SetControllerMode(CS_STARTED)
4. CanIf_SetPduMode(CANIF_TX_OFFLINE) — listen only for validation
5. First valid Rx → stored internally. CanIf_CheckValidation() → calls User_ValidateWakeupEvent()
6. On validation success → PDU mode set to CANIF_ONLINE

Validation options: CanIfPublicWakeupCheckValidByNM=TRUE → only NM PDUs count; FALSE → any Rx PDU validates.

Partial Networking (PN)

PnTxFilter: enabled at init and on CS_SLEEP. Blocks ALL Tx requests except the configured CanIfTxPduPnFilterPdu (NM wakeup frame). Disabled when TxConfirmation of that frame arrives, or on any RxIndication while PN active. This ensures the first transmitted frame is always the NM wakeup frame — critical for PN-capable nodes to wake their peers.

CAN FD & CAN XL support

CAN FD

Up to 64 bytes. Indicated by 2nd MSB of Can_IdType. Configured via CanIfTxPduCanIdType. Falls back to classic if no FD baud rate configured. Same CanId space — each ID used only once.

CAN XL

Up to 2048 bytes. Uses CanXL_Write() instead of Can_Write(). New attributes: PriorityID (not linked to content), SDU Type, VCID (virtual network), Acceptance Field. No CanIf Tx buffering for XL frames.

Type definitions (know these cold)

CanIf_PduModeType

CANIF_OFFLINE=0x00, CANIF_TX_OFFLINE=0x01,
CANIF_TX_OFFLINE_ACTIVE=0x02, CANIF_ONLINE=0x03

CanIf_NotifStatusType

CANIF_NO_NOTIFICATION=0x00,
CANIF_TX_RX_NOTIFICATION — returned by ReadTxNotifStatus / ReadRxNotifStatus (read-and-clear)

CanIf_ConfigType

Post-build configuration structure. Passed to CanIf_Init(). Contains all L-PDU, controller, transceiver, and HOH config. Implementation-specific internals.

Can_IdType (from CanDrv)

MSB=Extended ID flag, 2nd MSB=CAN FD flag. CanIf sets these bits before calling Can_Write() based on CanIfTxPduCanIdType.

Debugging checklist — interface testing

1. Tx side — Is controller in CS_STARTED? Is PDU channel CANIF_ONLINE? Is TxPduId valid? Is CanIf_Init() called? Is PnTxFilter active (blocking all but NM frame)?

2. Rx side — Is HRH BasicCAN? If so, is software filter enabled and CanId in HRH range? Does received DLC ≥ configured length? Is PDU channel mode allowing Rx?

3. BusOff recovery — CanSm calls CanIf_SetControllerMode(CS_STOPPED) then (CS_STARTED). Check PDU mode after: should be TX_OFFLINE until CanSm explicitly sets ONLINE.

4. Wakeup not working — Check CanIfPublicWakeupCheckValidSupport=TRUE. Check EcuMWakeupSourceRef in CanIfCtrlCfg. Ensure CS_STARTED before CheckValidation. PDU mode must NOT be ONLINE during validation.

5. Multiple drivers — Each CanDrv has own HOH numbering from 0. CanIf maps them via CanIfCtrlDrvCfg. Ensure ControllerId abstraction is consistent. Callbacks identify driver via Mailbox.ControllerId.

Step-by-step DaVinci configuration for a new CAN network

Step 1: Add CanIfCtrlDrvCfg — one per CanDrv module. Set reference to your CanDrv instance.
Step 2: Add CanIfCtrlCfg — assign CanIfCtrlId=0 (or sequential). Reference the CanDrv controller object. Set wake-up support if needed.
Step 3: Configure HOHs — add CanIfHthCfg and CanIfHrhCfg. Reference the HTH/HRH objects from CanDrv. Set BasicCAN/FullCAN type. For BasicCAN HRH, add CanIfHrhRangeCfg with CanId range and enable CanIfHrhSoftwareFilter.
Step 4: Add CanIfBufferCfg — one per HTH (minimum). Set size=0 if no buffering, or >0 for software buffering of BasicCAN.
Step 5: Configure Tx PDUs (CanIfTxPduCfg) — one per signal/PDU to transmit. Set CanId, type, buffer reference. Set callback user (PduR, CanNm, etc.). Generate symbolic name matching PduRouter config.
Step 6: Configure Rx PDUs (CanIfRxPduCfg) — one per signal/PDU to receive. Set CanId, DLC, HRH reference, upper layer callback. Set CanIfRxPduDataLength carefully (≤ actual frame data length).
Step 7: CanIfPublicCfg — enable/disable features. Match with other module configs (PduR, CanSm, CanNm).
Step 8: CanIfDispatchCfg — connect BusOff, mode indication callbacks to CanSm functions.
Step 9: Generate code — DaVinci generates CanIf_PBcfg.c/h with all static tables. Verify generated CanIf_Init(ConfigPtr) call in EcuM_AL_DriverInitOne().

Common DaVinci integration mistakes

• Tx PDU not linked to correct CanIfBufferCfg → transmit silently fails
• HRH BasicCAN range too wide → unrelated frames passed to upper layer (false Rx)
• CanIfRxPduDataLength > actual frame size → every Rx triggers DLC error
• Missing CanIfDispatchCfg callback → BusOff not forwarded to CanSm → network stuck
• CanIfCtrlId mismatch between CanIf and CanSm configs → CANIF_E_PARAM_CONTROLLERID at runtime
• CanIfTxPduCanIdType EXTENDED but CanDrv HTH configured for STANDARD → wrong MSBs on bus
• PnTxFilter not considered → first Tx frame is not NM frame → PN peers stay asleep
