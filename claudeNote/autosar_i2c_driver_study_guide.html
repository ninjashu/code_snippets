I2C vs PORT
Brand new in R24-11 (initial release). I2C Driver is a MCAL module providing hardware-independent API for I2C bus communication to upper layers. Both Controller (Master) and Target (Slave) modes supported.

What I2C driver does

  • Reads from and writes to devices on I2C bus (EEPROMs, sensors, transceivers etc.)
  • Supports synchronous (blocking) and asynchronous (interrupt-driven) data transfers
  • Implements Controller mode (initiates transfers, generates SCL) and Target mode (responds to Controller)
  • Queues multiple Sequences on same channel (FIFO-based, async mode)
  • Reports bus-level errors: NACK, arbitration loss, bus failure, FIFO overflow
  • Provides callback I2C_SeqEndNotification on sequence completion (async)
  • Permanent listening mode for Target — I2C_MainFunction drives polling

What I2C driver does NOT do

  • Does NOT configure clock, prescaler(s), PLL — MCU driver handles that
  • Does NOT have mandatory interfaces from other modules (no DET mandatory, only optional)
  • Does NOT produce Production Errors or Extended Production Errors
  • No callback functions to lower layers (it IS the lowest MCAL layer)

I2C bus basics (exam context)

  • 2-wire: SDA (Serial Data) + SCL (Serial Clock)
  • Controller = old "Master" — initiates, generates clock, terminates
  • Target = old "Slave" — responds to Controller address
  • 7-bit addressing (≤0x7F) or 10-bit extended addressing (>0x7F)
  • Frame: START → TARGET_ADDR + R/W̄ → ACK → DATA → ACK/NACK → STOP
  • I2C speeds: 100 kbit/s (Standard), 400 kbit/s (Fast), 1 Mbit/s (Fast-Plus), 3.4 Mbit/s (High-Speed)

Architecture position

I2C Driver sits in MCAL layer (same as PORT, DIO, SPI). Upper layers (e.g. complex drivers, IoHwAb) call I2C APIs.

MCU Driver must be initialized first — MCU sets up clock/PLL that I2C peripheral depends on.

Standard: NXP UM10204 (I2C-bus specification) is the reference norm for bus behavior.

Header file: I2c.h (all types and functions)

Callback file: I2c_Externals.h (SeqEndNotification)

Core concept: Sequence → Jobs → I2C frames on bus. This hierarchy is the heart of the I2C driver. Understand it deeply.

Job explained

A Job is the smallest configurable unit of I2C transfer. It defines:

  • I2cJobId — numeric ID (0..255)
  • I2cDeviceAddress — target address (0..1023). Values >0x7F → extended 10-bit addressing.
  • Data pointers and length are set at runtime via I2C_SetupEB()
  • Direction (Read or Write) determined by which buffer pointer is NULL in SetupEB

Controller mode buffer rules (exam critical):

  • RxDataBufferPtr = NULL → WRITE operation (Tx data → bus)
  • TxDataBufferPtr = NULL → READ operation (bus data → Rx buffer)
  • Both NULL or both non-NULL → DET error I2C_E_PARAM_POINTER

Target mode buffer rules:

  • TxDataBufferPtr = NULL → READ (Target receives from Controller → into RxBuffer)
  • RxDataBufferPtr = NULL → WRITE (Target sends to Controller ← from TxBuffer)

Sequence explained

A Sequence groups multiple Jobs that execute in order on one I2cChannel. Key properties:

  • I2cSequenceId — numeric ID (0..254)
  • I2cAssignedChannel — reference to one I2cChannel (which HW peripheral)
  • I2cAssignedJob — ordered list of Job references (1 to many, requiresIndex=true)
  • I2cEndNotification — name of callback function to invoke when sequence completes

Execution order: Job[0] → Job[1] → Job[2] → ... → Job[N] → callback fires

Direction can change between jobs in a sequence (first job write, second job read — typical for register read pattern).

SetupEB — the link between config and runtime

I2C_SetupEB(JobId, NodeAddress, TxPtr, RxPtr, Length) must be called before transmit. It:

  • Links the configured Job to runtime buffers
  • Sets the data length for this transfer
  • Optionally overrides target address (NodeAddress≠0 → use it; NodeAddress=0 → use configured I2cDeviceAddress)
  • Determines direction (read vs write) by which pointer is NULL

Important: SetupEB does NOT need to be called every time before AsyncTransmit/SyncTransmit. If parameters haven't changed, one prior SetupEB is sufficient. The spec explicitly states this.

I2C frame structure (what hits the bus)

[S] [TARGET_ADDR + R/W̄] [A] [DATA byte 0] [A/Ā] [DATA byte 1] [A/Ā] ... [DATA byte N] [A/Ā] [P]

S=START, P=STOP, A=ACK, Ā=NACK. One Job = one such frame. Multiple Jobs in a Sequence can use Repeated START (Sr) instead of STOP+START between jobs.

Async queuing mechanism (FIFO)

When I2C_AsyncTransmit() called while another Sequence already PENDING on same channel:

  • New request queued in FIFO → state becomes I2C_SEQ_QUEUED → returns E_OK
  • After current sequence completes → driver auto-starts next queued element
  • If SAME sequence already PENDING → rejected → returns E_NOT_OK

SyncTransmit does NOT support queuing — if async pending on same channel → returns E_NOT_OK immediately.

Critical module-state rules

  • Re-initialization requires DeInit first. Calling I2C_Init again without DeInit is NOT allowed. This differs from PORT driver (which allows re-init directly).
  • I2C_DeInit in UNINIT state → DET error I2C_E_UNINIT (if dev error detect enabled)
  • I2C_DeInit puts peripheral back to Power-On-Reset state
  • All other APIs called in UNINIT → I2C_E_UNINIT DET error
Sequence-level result states (I2C_SequenceResultType)

All 5 sequence result values

ValueHexMeaning
I2C_SEQ_OK0x00Sequence completed successfully. Initial value set by I2C_Init for all sequences.
I2C_SEQ_PENDING0x01Sequence currently executing on bus. Also = I2C_BUSY.
I2C_SEQ_QUEUED0x02Async only — queued in FIFO, waiting for current sequence to finish.
I2C_SEQ_NACK0x03Target responded with NACK — address not acknowledged or data rejected.
I2C_SEQ_FAILED0x04Bus failure, arbitration loss, or other fatal error during sequence.

Sequence state transitions

1
After I2C_Init: all sequences → I2C_SEQ_OK
2

3
AsyncTransmit called, channel busy → I2C_SEQ_QUEUED, return E_OK
4
AsyncTransmit same seq already PENDING → stays PENDING, return E_NOT_OK
5

6
After last job done + NACK → I2C_SEQ_NACK + callback fires
7
After last job done + any error → I2C_SEQ_FAILED + callback fires
8
QUEUED sequence becomes PENDING after preceding sequence finishes

Target mode state (I2C_StartListening)

  • StartListening on idle channel → I2C_SEQ_PENDING, return E_OK
  • StartListening while already listening → return E_NOT_OK (cannot double-start)
  • After last message or error received → sequence state set (OK/NACK/FAILED) + callback
  • I2cTargetListening=TRUE → permanent listening — I2C_MainFunction keeps re-arming after each message, callback fires per message
  • StartListening while I2cTargetListening=TRUE set → runtime error I2C_E_WRONG_MODE (can't manually start when always-on)

I2C_Init (ConfigPtr) — Service ID: 0x00

Syntax: void I2C_Init(const I2C_ConfigType* ConfigPtr)

Reentrancy: Non-reentrant | Return: void | Sync

Does: Initializes all I2C HW channels (I2cHwUnitBaseAddress + I2cBaudRate per channel). Sets ALL sequence results to I2C_SEQ_OK.

Critical rule: Re-initialization requires DeInit first. Cannot call Init again without DeInit.

DET errors: None explicitly listed for Init itself (I2C_E_UNINIT via DeInit if Init not called).

I2C_DeInit (void) — Service ID: 0x01

Syntax: void I2C_DeInit(void)

Reentrancy: Non-reentrant | Return: void | Sync

Does: De-initializes all I2C peripherals to Power-On-Reset state.

DET errors: I2C_E_UNINIT if called before Init (when I2cDevErrorDetect=TRUE).

I2C_SetupEB (JobId, NodeAddress, TxPtr, RxPtr, Length) — Service ID: 0x02

Syntax: Std_ReturnType I2C_SetupEB(I2C_JobType, I2C_AddressType, I2C_DataConstPtrType*, I2C_DataPtrType*, I2C_NumberOfDataType)

Reentrancy: Non-reentrant | Return: E_OK / E_NOT_OK | Sync

Does: Configures the runtime buffer setup for a Job. Determines direction by NULL pointer. NodeAddress=0 → use configured address; nonzero → override address.

DET errors: I2C_E_PARAM_POINTER (both NULL or both non-NULL), I2C_E_PARAM_JOB (invalid JobId)

Call frequency: Does NOT need calling every time — only when parameters change.

I2C_AsyncTransmit (SequenceId) — Service ID: 0x03

Syntax: Std_ReturnType I2C_AsyncTransmit(I2C_SequenceType SequenceId)

Reentrancy: Non-reentrant | Return: E_OK / E_NOT_OK | Async (interrupt-driven)

Does: Starts async transfer. If channel idle → PENDING, E_OK. If other seq PENDING → queue → QUEUED, E_OK. If same seq PENDING → E_NOT_OK. On completion → I2C_SeqEndNotification callback.

DET errors: I2C_E_PARAM_SEQUENCE (invalid SequenceId or SetupEB not called for all Jobs in sequence)

Callback note: I2C_SeqEndNotification may be called at interrupt level.

I2C_SyncTransmit (SequenceId) — Service ID: 0x04

Syntax: Std_ReturnType I2C_SyncTransmit(I2C_SequenceType SequenceId)

Reentrancy: Non-reentrant | Return: E_OK / E_NOT_OK | Sync (blocking)

Does: Blocking transfer — function does not return until all Jobs in sequence complete. If async PENDING on same channel → E_NOT_OK (no queuing). No callback fired.

DET errors: I2C_E_PARAM_SEQUENCE (invalid SequenceId or SetupEB not called)

Use case: Simple scenarios where task can afford to block. Not suitable for time-critical tasks.

I2C_GetVersionInfo (VersionInfo) — Service ID: 0x07

Syntax: void I2C_GetVersionInfo(Std_VersionInfoType* VersionInfo)

Reentrancy: Reentrant | Return: void | Sync

Does: Returns vendor ID, module ID, SW version. Enabled by I2cVersionInfoApi=TRUE.

DET errors: I2C_E_PARAM_POINTER if NULL pointer passed.

I2C_GetSequenceResult (SequenceId) — Service ID: 0x09

Syntax: I2C_SequenceResultType I2C_GetSequenceResult(I2C_SequenceType SequenceId)

Reentrancy: Reentrant | Return: I2C_SequenceResultType | Sync

Does: Returns current state: OK, PENDING, QUEUED, NACK, or FAILED. Key function for polling in SyncTransmit scenario or checking async result after callback.

DET errors: I2C_E_PARAM_SEQUENCE if invalid SequenceId.

I2C_StartListening (SequenceId) — Service ID: 0x0A

Syntax: Std_ReturnType I2C_StartListening(I2C_SequenceType SequenceId)

Reentrancy: Non-reentrant | Return: E_OK / E_NOT_OK | Sync

Does: Target mode only. Makes channel available for incoming transfers from a Controller. Channel → PENDING. On message receive → copy data per direction → callback.

DET errors: I2C_E_PARAM_SEQUENCE (SetupEB not called), I2C_E_WRONG_CONDITION (channel not in Target mode / I2cHwUnitMode ≠ TARGET)

Runtime error: I2C_E_WRONG_MODE if I2cTargetListening=TRUE (always-on mode conflicts with on-demand StartListening)

I2C_MainFunction (void) — Service ID: 0x10 [Scheduled]

Syntax: void I2C_MainFunction(void)

Does: Used for permanent Target listening mode (I2cTargetListening=TRUE). Called cyclically by BSW scheduler. After each received message, fires SeqEndNotification and re-arms listening automatically.

When to use: Target device that must always be responsive — e.g. ECU acting as I2C slave to external Controller.

Callback: I2C_SeqEndNotification — Configurable interface

Syntax: void (*I2C_SeqEndNotification)(I2C_SequenceType SequenceId, I2C_SequenceResultType Result)

User-provided function configured per Sequence via I2cEndNotification in I2cSequence container. Passes which sequence finished and what result (OK/NACK/FAILED). May execute at interrupt level for async. Located in I2c_Externals.h.

3 key sequence diagrams. All start with Init and end with DeInit. SetupEB always comes before Async/Sync/StartListening.

Seq 9.1 — SetupEB + AsyncTransmit (3-Job Sequence)

1
User calls I2C_Init(ConfigPtr) → I2C module initializes all channels, all seq results = I2C_SEQ_OK
2
Loop Job:=0..2 — For each of 3 Jobs: User calls I2C_SetupEB(JobId, NodeAddr, TxPtr, RxPtr, Length) → returns E_OK, parameters saved internally
3
User calls I2C_AsyncTransmit(SequenceId) → driver sets Seq PENDING, returns E_OK immediately (non-blocking!)
4
Driver executes: Seq0.Job0() → Seq0.Job1() → Seq0.Job2() asynchronously (interrupt-driven)
5
After Job2 completes: driver sets state (OK/NACK/FAILED) and fires <I2C_Seq0EndNotification>() callback
6
Driver becomes IDLE. User may call I2C_DeInit() to shut down.

Key point: between step 3 and 5, user's task continues running — async means non-blocking.

Seq 9.2 — SetupEB + SyncTransmit (2 Sequences of 2 Jobs each)

1
User calls I2C_Init(ConfigPtr)
2
Loop Job:=0..3 — SetupEB called for all 4 Jobs (2 per sequence)
3
User calls I2C_SyncTransmit(Sequence0) → driver starts Seq0.Job0(), then Seq0.Job1() — ALL BLOCKING
4
I2C_SyncTransmit returns E_OK only after both jobs done. NO callback. User checks return value.
5
User calls I2C_SyncTransmit(Sequence1) → Seq1.Job0(), Seq1.Job1() blocking
6
Returns. Driver IDLE. User calls I2C_DeInit()

Key difference from async: no callback, task blocks at SyncTransmit call. Second SyncTransmit only starts after first returns.

Seq 9.3 — SetupEB + StartListening (Target mode, 3-Job Sequence)

1
User calls I2C_Init(ConfigPtr) — channel configured as I2C_HW_UNIT_MODE_TARGET
2
Loop Job:=0..2 — SetupEB called for each Job (set RxPtr for read jobs, TxPtr for write jobs)
3
User calls I2C_StartListening(SequenceId) → channel → PENDING, waits for Controller to address it
4
External Controller sends messages. Driver handles: Seq0.Job0(), Seq0.Job1(), Seq0.Job2() — ASYNCHRONOUSLY as messages arrive
5
After last Job received / error: state set → <I2C_Seq0EndNotification>() callback fires
6
Driver becomes IDLE. If I2cTargetListening=TRUE, MainFunction re-arms automatically for next message.

Read message received → driver copies data into RxDataBufferPtr. Write message → driver sends from TxDataBufferPtr.

Async queuing sequence (FIFO case)

1
SetupEB for Seq0 and Seq1 Jobs
2
AsyncTransmit(Seq0) → Seq0 PENDING, bus busy, E_OK returned
3
AsyncTransmit(Seq1) → Seq0 still PENDING → Seq1 queued → I2C_SEQ_QUEUED, E_OK returned
4
Seq0 finishes → callback(Seq0, OK) fires → driver checks FIFO → Seq1 found
5
Seq1 dequeued → PENDING → executes → callback(Seq1, OK) fires

AsyncTransmit(Seq0) again while Seq0 PENDING → E_NOT_OK (same seq can't be double-queued).

2
I2cGeneral: Set I2cDevErrorDetect=TRUE for dev phase. Set I2cVersionInfoApi=TRUE for testing.
3
I2cConfigSet: Create one I2cConfigSet (or multiple for variant support).

Integration testing, interface testing, debugging — industry-level checklist for I2C driver.

Integration testing — boot sequence verification

  • Verify MCU_Init called before I2C_Init — I2C clock comes from MCU PLL configuration
  • Verify I2C_Init called before any SetupEB/AsyncTransmit/SyncTransmit
  • Verify DeInit called before re-Init (unlike PORT driver which allows direct re-init)
  • Check I2C_GetSequenceResult returns I2C_SEQ_OK for all sequences immediately after Init
  • Monitor DET error log — no errors should appear during normal startup

Bus-level testing (logic analyzer / oscilloscope)

  • Verify START condition: SDA falling while SCL high
  • Verify correct target address transmitted (7-bit or 10-bit per I2cDeviceAddress config)
  • Verify correct R/W̄ bit: 0=write, 1=read
  • Verify ACK from target: SDA pulled low by target on 9th clock
  • Verify data bytes count matches I2C_NumberOfDataType Length from SetupEB
  • Verify STOP condition: SDA rising while SCL high
  • Test NACK scenario: disconnect target → verify I2C_E_NACK_RECEIVED reported, sequence result = I2C_SEQ_NACK
  • Multi-job sequence: verify Repeated START (Sr) between consecutive jobs (no STOP+START)
  • Verify baud rate matches I2cBaudRate config (measure SCL period with scope)

DET testing (development error coverage)

  • Call any API before I2C_Init → expect I2C_E_UNINIT (0x03)
  • Call I2C_SetupEB with both TxPtr and RxPtr non-NULL → expect I2C_E_PARAM_POINTER (0x02)
  • Call I2C_SetupEB with both TxPtr and RxPtr NULL → expect I2C_E_PARAM_POINTER (0x02)
  • Call I2C_SetupEB with invalid JobId → expect I2C_E_PARAM_JOB (0x00)
  • Call I2C_AsyncTransmit without prior SetupEB for all Jobs → expect I2C_E_PARAM_SEQUENCE (0x01)
  • Call I2C_AsyncTransmit with invalid SequenceId → expect I2C_E_PARAM_SEQUENCE (0x01)
  • Call I2C_StartListening on CONTROLLER-mode channel → expect I2C_E_WRONG_CONDITION (0x04)
  • Call I2C_StartListening when I2cTargetListening=TRUE → expect runtime I2C_E_WRONG_MODE (0x04)
  • Call I2C_GetVersionInfo with NULL → expect I2C_E_PARAM_POINTER (0x02)
  • Call I2C_DeInit without prior Init → expect I2C_E_UNINIT (0x03)
  • For each: verify function returns WITHOUT executing (no side effects)

Runtime error testing

  • NACK test: Configure wrong target address → AsyncTransmit → verify I2C_E_NACK_RECEIVED via DEM, sequence result = I2C_SEQ_NACK, callback fires with Result=NACK
  • Bus lockup test: Pull SCL low externally → verify I2C_E_BUS_FAILURE reported → implement bus recovery (9 clocks + STOP sequence)
  • Arbitration test (multi-master): If using multi-master setup, drive conflicting SDA → verify I2C_E_ARBITRATION_FAILURE reported
  • FIFO overflow: If HW has FIFO, overflow condition → I2C_E_FIFO_HANDLING reported

Async vs Sync behavioral testing

  • Async: verify I2C_AsyncTransmit returns immediately (before bus transfer completes)
  • Async: verify callback fires asynchronously after bus transfer done
  • Async: verify I2C_GetSequenceResult returns PENDING during transfer
  • Async FIFO: call AsyncTransmit twice for different sequences → verify QUEUED state → both callbacks fire in order
  • Sync: verify I2C_SyncTransmit blocks until complete, returns E_OK
  • Sync while async pending: verify E_NOT_OK returned (no bus access attempted)
  • Callback interrupt level: verify callback executes at ISR priority, no task-level blocking

Target mode testing

  • Verify I2cHwUnitMode = TARGET configured for the channel
  • Verify SetupEB called before StartListening (I2C_E_PARAM_SEQUENCE if not)
  • StartListening: external Controller sends message → verify data copied to RxDataBufferPtr
  • StartListening: external Controller reads → verify data sent from TxDataBufferPtr
  • Permanent listening (I2cTargetListening=TRUE): verify I2C_MainFunction called cyclically, verify callback fires per message, verify StartListening cannot be manually called

What both share

  • Both MCAL layer drivers in AUTOSAR Classic Platform
  • Both UNINIT state on reset, INIT after initialization
  • Both have I2C_E_UNINIT / PORT_E_UNINIT for calling APIs without init
  • Both have GetVersionInfo (conditional on config flag)
  • Both support pre-compile and post-build config variants
  • Both declared in their respective .h files (Port.h, I2c.h)
  • Both require DET enabled to report errors (I2cDevErrorDetect / PortDevErrorDetect)
  • Neither has mandatory interfaces from other modules
