RTE — What, Why, How

RTE = physical realization of VFB (Virtual Function Bus) per ECU. Generated C code. Glues SWCs to OS, COM, BSW. Only static comms — all connections known at codegen time [SWS_Rte_06026].

RTE does

• Route S/R, C/S, mode comms
• Schedule runnables via OS tasks
• Abstract SWC from OS/COM/HW
• Manage exclusive areas (critical sections)
• Handle implicit buffering
• Init/finalize data & mode machines

RTE does NOT

• Dynamic communication
• Dynamic task creation
• Direct BSW access for SWC
• Inter-ECU mode switch
• Generate keys or dynamic routing
• Allocate memory at runtime

RTE Generation — Two-Phase Model

Phase 1 — Contract Phase

Input: SWC description (ARXML) only — port interfaces, runnables, data accesses. No full system info needed.

Output: Rte_<SWCName>.h — SWC can compile against this header without knowing other SWCs.

SWC code compiled against contract header. No full system needed yet.

Phase 2 — Generation Phase

Input: Full ECU extract — ALL SWCs, port connections, task mapping, exclusive areas, OS config.

Output: Rte.c, Rte.h, SchM.c, SchM.h — complete RTE implementation.

Generated once per ECU. Different ECUs → different RTE even for same SWC.

Sender-Receiver Communication

Signal passing. One SWC writes, one or more read. Port interface = SenderReceiverInterface with VariableDataPrototypes.

Explicit vs Implicit — the key exam distinction

Explicit (dataReceivePoint / dataSendPoint)

Runnable calls API explicitly. Non-blocking read/write. Blocking read needs WaitPoint (cat 2 only). Data can change between calls during runnable execution.

/* Non-blocking non-queued */
Rte_Write_Port_DE(val);
Rte_Read_Port_DE(&val);

/* Queued (event semantics) */
Rte_Send_Port_DE(&val);
Rte_Receive_Port_DE(&val);

/* Direct read (no status) */
Rte_DRead_Port_DE();

Implicit (dataReadAccess / dataWriteAccess)

RTE copies data IN before runnable starts, copies OUT after runnable ends. Data frozen for duration of runnable. Last-is-best semantics for write. Cat 1A/1B/2 all supported.

/* RTE fills before runnable entry */
val = Rte_IRead_RunName_Port_DE();

/* RTE flushes after runnable exit */
Rte_IWrite_RunName_Port_DE(val);

/* Zero-copy write (returns ptr) */
ptr = Rte_IWriteRef_RunName_Port_DE();

/* Mark implicit write as invalid */
Rte_IInvalidate_RunName_Port_DE();
💡 Implicit write: multiple Rte_IWrite calls inside one runnable → only LAST one counts [SWS_Rte_03573]. Last-is-best. Data sent to others only after runnable terminates [SWS_Rte_03571].

InitValue priority rules

Receiver initValue overrides sender initValue [SWS_Rte_04502]. If only sender has it — applied to both [SWS_Rte_04501]. InitValue prevents undefined data before first receive (non-queued only — queued has no initValue [SWS_Rte_04500]).

⚠️ No initValue configured + no data received yet → Rte_Read returns undefined data. Always check return value for RTE_E_NEVER_RECEIVED on startup. This is #1 integration bug.

Client-Server Communication

Function invocation. Client calls server operation. Port interface = ClientServerInterface with ClientServerOperation elements. n:1 only — many clients, ONE server. 1:n NOT supported.

Synchronous C/S

Client blocks until server returns. Uses OS Event (WaitPoint) if cross-task. Cat 2 runnable needed if blocks. Direct C call if same task — RTE optimizes this away.

Std_ReturnType ret;
ret = Rte_Call_Port_Operation(
        inArg, &outArg);
/* returns after server done */
Cannot use from cat 1A runnable. Same task → direct call. Different task → WaitPoint (OS Event). Cross-partition → IOC or trusted function.

Asynchronous C/S

Client fires, returns immediately. Result collected by ASCR runnable triggered by AsynchronousServerCallReturnsEvent.

/* Fire */
Rte_Call_Port_Op(args);

/* In ASCR runnable: */
Rte_Result_Port_Op(&outArg);
Use async when server might be slow (e.g., NvM read). Avoids blocking client task.

Port-Defined Argument Values (PDAV)

RTE can supply constant argument values from port config — SWC doesn't pass them. Allows port-level specialization of generic operations.

Direct function call optimization

Client and server in same OS task + same partition → RTE generates direct C function call. No task activation, no OS event, no overhead. Visible in generated Rte.c as inline call. Key performance optimization.

Synchronous mode switch — full step sequence [SWS_Rte_02665]

1
Rte_Switch() called → mode machine enters transition. Activate mode disablings of NEXT mode [SWS_Rte_02661].
2
Wait until newly disabled runnables terminate [SWS_Rte_07152].
3
Execute on-exit runnables of PREVIOUS mode [SWS_Rte_02562].
4
Wait until on-exit runnables terminate [SWS_Rte_07153].
5
Execute on-transition runnables (A→B specific) [SWS_Rte_02707].
6
Wait until on-transition runnables terminate [SWS_Rte_02708].
7
Execute on-entry runnables of NEXT mode [SWS_Rte_02564].
8
Wait until on-entry runnables terminate [SWS_Rte_07154].
9
Deactivate previous mode disablings [SWS_Rte_02563]. Transition complete.
10
Fire ModeSwitchedAckEvent to mode manager [SWS_Rte_02587].
⚠️ All runnables triggered by mode switch = cat 1 (no WaitPoint) [SWS_Rte_02500]. Must guarantee finite execution time. Otherwise synchronous mode switch deadlocks.

Async mode switch — when valid

All mode users set supportsAsynchronousModeSwitch=TRUE → RTE uses async procedure. Steps 2/4/6/8 (waits) omitted. On-entry/exit/transition runnables called as direct function inside Rte_Switch. Faster but weaker ordering guarantees.

Rte_Mode() return during transition

/* During transition from OLD_MODE to NEW_MODE: */
mode = Rte_Mode_Port_ModeGroup();
/* Returns: RTE_TRANSITION_OLD_MODE_TO_NEW_MODE */

/* Stable in NEW_MODE: */
/* Returns: RTE_MODE_ModeGroup_NEW_MODE */

Mode switch queue

If mode machine already in transition when Rte_Switch called → new switch queued (if queue not full). Rte_Switch returns error if queue full [SWS_Rte_02667]. Queue processes sequentially. Fast double-switch: RTE may skip intermediate mode reporting in Rte_Mode.

Activation (ACT) vs WakeUp (WUP)

ACT — Activate

OS ActivateTask(). Runnable scheduled from SUSPENDED. Most events use ACT. Cat 1 runnables only ACT.

WUP — Wake WaitPoint

OS SetEvent(). Cat 2 runnable waiting at WaitPoint woken. Only subset of events can WUP (DataReceivedEvent, ASCRE, ModeSwitchedAckEvent, OperationInvokedEvent).

Inter-Runnable Variables (IRV)

/* Private to SWC instance — not visible outside */
val = Rte_IrvRead_RunName_IrvName();
Rte_IrvWrite_RunName_IrvName(val);

/* In PerInstanceMemory (pointer access) */
*Rte_Pim_MyPimName() = value;
IRVs live in PerInstanceMemory. Multiple SWC instances → separate blocks. Protect shared IRVs accessed from different tasks with ExclusiveArea.

RTE & SchM Lifecycle

Startup sequence — strict order

1
SchM_Init(config*) — init BSW scheduler. Before any BSW _Init. Initializes mode machine instances assigned to SchM. Starts BswBackgroundEvents [SWS_Rte_07584].
2
SchM_StartTiming() — start BswTimingEvents [SWS_Rte_07574].
3
BSW module _Init() functions called (COM, NvM, DEM, etc.) by EcuM action lists.
4
Rte_Init() — OPTIONAL. Init buffers to initValues. Called if InitEvent runnables mapped to RteInitializationRunnableBatch. Executes mapped init runnables.
5
Rte_Start() — init RTE mode machines. Start TimingEvent activations (if no Rte_StartTiming). Start BackgroundEvents. From here runnables can run [SWS_Rte_07575]. Called per core [SWS_Rte_07615].
6
Optional: Rte_StartTiming() — separate timing start for fine-grained control. If exists, Rte_Start does NOT start TimingEvents [SWS_Rte_06759].

Shutdown sequence

1
Rte_Stop() — stop runnable activations on calling core [SWS_Rte_02538]. Running runnables finish. Incoming comms ignored [SWS_Rte_02535, 02536].
2
SchM_Deinit() — stop BSW scheduler. No more BSW schedulable entity activations [SWS_Rte_07586].

Key rules

RuleRequirement
SchM_Init BEFORE Rte_Start [SWS_Rte_CONSTR_09036]RTE nested inside SchM lifecycle
Rte_Stop BEFORE SchM_Deinit [SWS_Rte_CONSTR_09056]Reverse of init order
Before Rte_Start: only SchM_Enter/Exit available[SWS_Rte_07578]
RunnableEntity not activated before RTE init [SWS_Rte_07580]SchM must block runnable activation until Rte_Start
After Rte_Stop: incoming data/C/S ignored [SWS_Rte_02535]Graceful shutdown — no late data processing

SWC initialization — two mechanisms

InitEvent approach

Map runnable to InitEvent. Fires during Rte_Init (if mapped to InitializationRunnableBatch) or Rte_Start. Simpler — no modes needed.

Mode-based approach

On-entry runnable for initial mode fires during Rte_Start. Allows multi-step init via successive mode transitions. More flexible.

Exclusive Areas — Critical Sections

Protect shared data between runnables of different tasks. Configured in SwcInternalBehavior. Generated as Rte_Enter/Rte_Exit macros.

Rte_Enter_MyArea();
  /* Critical section — shared data access */
  *Rte_Pim_SharedData() = newValue;
Rte_Exit_MyArea();

/* SchM equivalent for BSW modules: */
SchM_Enter_ModName_AreaName();
  /* BSW critical section */
SchM_Exit_ModName_AreaName();

Implementation mechanisms

Mechanism (OsExclusiveAreaMethod)Generated asUse when
INTERRUPT_BLOCKING_ALLDisableAllInterrupts() / EnableAllInterrupts()Short sections, Cat1 ISR access shared data
INTERRUPT_BLOCKING_OSSuspendOSInterrupts() / ResumeOSInterrupts()Cat2 ISR involved, Cat1 must not be blocked
OS_RESOURCEGetResource() / ReleaseResource()Task-level protection, no ISR involvement
NONEEmpty macrosSame preemption area — no real protection needed
🔴 DET error RTE_E_DET_WAIT_IN_EXCLUSIVE_AREA (WIEA): calling WaitPoint inside Rte_Enter → deadlock. RTE detects and reports [SWS_Rte_06637]. Fatal design error.
🔴 DET error RTE_E_DET_ILLEGAL_NESTED_EXCLUSIVE_AREA (INEA): nested Rte_Enter without Rte_Exit violating configured nesting rules [SWS_Rte_07675].
⚠️ Exclusive areas must NOT surround cross-partition or cross-ECU calls. Scope = intra-partition only. For cross-partition: use IOC or trusted functions with spinlocks.

NvBlock SWC — Persistent Data Management

NvBlockSwComponentType wraps NvM block. RTE provides dedicated API for NvM interaction. Persistent data across ECU resets.

Writing strategies (RteNvRamBlockDescription)

StrategyWhen write triggersUse case
WRITE_ON_FINALSWC explicitly calls NvM write via RTE service portData changes infrequently; explicit control needed

WRITE_ON_SHUTDOWNDuring EcuM shutdown sequenceCalibration data, odometer — write once per ignition cycle

NvBlock API

/* Trigger NvM read of block */
Rte_Call_NvMService_ReadBlock(&blockData);

/* Trigger NvM write */
Rte_Call_NvMService_WriteBlock(&blockData);

/* Callbacks from NvM → NvBlock SWC: */
Rte_NvMNotifyJobFinished_BlockName(SvcId, JobResult);
Rte_NvMNotifyInitBlock_BlockName(); /* on first read */

/* Direct RAM mirror access (R19-11+): */
/* Read from mirror without NvM call */
Rte_Call_NvMService_GetErrorStatus(&status);
💡 NvBlock SWC is a special SWC type — it has both application ports (for other SWCs to read/write its data) and NvM service ports. The RTE handles the synchronization between application data and NvM mirror.

💡 Instance ID for DET = OS Application ID [SWS_Rte_06631]. Unique across multi-core. Lets you trace error to specific partition and core in DET log. Service ID tells you which API was called.

Return value architecture

RTE API returns Std_ReturnType (uint8). High bit (0x80) set = overlaid error. Multiple errors OR'd together possible.

/* Check pattern: */
Std_ReturnType ret = Rte_Read_Port_DE(&val);

if(ret == RTE_E_OK) { /* use val */ }
else if(RTE_E_UNCONNECTED & ret) { /* port not connected — use initValue */ }
else if(RTE_E_INVALID & ret) { /* sender marked data invalid */ }
else if(RTE_E_MAX_AGE_EXCEEDED & ret) { /* inter-ECU timeout */ }
else if(RTE_E_NEVER_RECEIVED & ret) { /* startup — no data yet */ }

/* RULE: Always check return value of Rte_Read/Rte_Receive.
   Never use value blindly without checking status. */

VFB Tracing — debug support

RTE hooks into every API call. Sends trace events to ARTI-compatible trace tool (CANalyzer AUTOSAR, Lauterbach Trace32, ETAS INCA). Captures: API name, direction, value, timestamp, return code, caller runnable.

Configure: RteVfbTrace/RteVfbTracingEnabled=TRUE in DaVinci. Generates Rte_Trace_* call hooks around every Rte_* API. No VFB tracing for production builds — compile-time switch.

DaVinci Configuration — Step by Step

Step 1 — SWC design (DaVinci Developer)

1
Create ApplicationSwComponentType. Add P-Ports (provide) and R-Ports (require). Type each with SenderReceiverInterface or ClientServerInterface.
2
Define SwcInternalBehavior: add RunnableEntity for each function. For each runnable: set RTEEvent (TimingEvent with period, DataReceivedEvent, InitEvent, OperationInvokedEvent, etc.).
3
Configure data access: explicit = add DataReceivePointByValue or DataSendPoint. Implicit = add DataReadAccess (Rte_IRead) or DataWriteAccess (Rte_IWrite).
4
For C/S client: add ServerCallPoint (synchronous or asynchronous). For server: OperationInvokedEvent triggers server runnable.
5
Configure ComSpec per port: set InitValue, QueueLength (for queued), AliveTimeout (receiver inter-ECU), HandleInvalidOption (keep/replace/default).
6
Add ExclusiveArea elements. Assign runnables that share data to the same ExclusiveArea. Set OsExclusiveAreaMethod.
7
Add PerInstanceMemory and IRV elements for SWC-private data.
8
Export ARXML. Run contract phase: DaVinci → RTE → Contract Phase → generates Rte_SWCName.h.

Step 2 — System design

1
Place SWC instances in CompositionSwComponentType. Connect ports with AssemblySwConnector (SWC-to-SWC) or DelegationSwConnector (composition boundary).
2
Map SWC instances to ECUs. Generate ECU extract ARXML per ECU.

Step 3 — ECU configuration (DaVinci Configurator Pro)

1
Import ECU extract. All SWC descriptions, port connections, signal mappings loaded.
2
Configure OS tasks. Create RteEventToTaskMapping: map each RTEEvent to an OS task. E.g., 10ms TimingEvent → Task_10ms.
3
Configure ExclusiveArea implementation: OS_RESOURCE (cross-task), INTERRUPT_BLOCKING_ALL, INTERRUPT_BLOCKING_OS, or NONE (same preemption area).
4
Configure COM mapping: map SWC data elements to COM signals. Set PDU layout. For inter-ECU only.
5
Configure implicit communication: RteImplicitCommunication containers if coherent data access needed across runnables.
6
Run RTE Generation Phase: DaVinci → RTE → Generate All. Produces Rte.c, Rte.h, SchM.c, SchM.h. Check generation report for errors.

Common DaVinci mistakes

🔴 RteEventToTaskMapping missing: every RTEEvent needs mapping or generation fails. Check: all runnables in SwcInternalBehavior have matching RteEventToTaskMapping.
🔴 Task not in OS config: referenced OS tasks must exist in OS module. Generator validates cross-module consistency.
⚠️ ExclusiveArea mechanism mismatch: runnables in different tasks sharing data → use OS_RESOURCE (not NONE). NONE only valid when runnables cannot preempt each other.
⚠️ QueueLength too small: if production rate > consumption rate → RTE_E_LOST_DATA. Calculate worst-case burst and set queue accordingly.
⚠️ Missing InitValue on receiver: startup → Rte_Read returns undefined + RTE_E_NEVER_RECEIVED. Always set ComSpec initValue for inter-ECU and inter-partition connections.
💡 Contract phase first: run contract phase, compile SWC code against generated header, fix API mismatches BEFORE attempting full generation. Saves hours of debugging.

Integration Testing & Debugging

Integration bugs & fixes

Bug: Rte_Read returns RTE_E_NEVER_RECEIVED

Sender SWC not yet run, or signal not received from network, or port not connected (missing AssemblySwConnector). Check: (1) sender task scheduled and running. (2) For inter-ECU: COM signal received and mapped. (3) ARXML port connections complete in DaVinci. Fix: verify COM signal trace on bus; check ARTI runnable activation trace.

Bug: Rte_Read returns RTE_E_MAX_AGE_EXCEEDED

AliveTimeout configured on receiver ComSpec. Sender period longer than timeout. Or sender not running during startup. Fix: increase AliveTimeout to 2× sender period. Handle gracefully in receiver — use initValue as fallback.

Bug: Mode switch hangs — never completes

On-exit runnable doesn't terminate (infinite loop, missing TerminateTask/return, or waiting at WaitPoint). Cat 2 runnable as mode-triggered runnable [SWS_Rte_02500] violated. Fix: ensure all mode-triggered runnables are cat 1 (guaranteed finite). Check ARTI trace for runnable state.

Bug: DET error ILLEGAL_INVOCATION (II, 0x03)

Runnable calls Rte_Write on port not in its contract. Common if ARXML updated but contract phase not re-run. Fix: re-run contract phase. Recompile SWC against new Rte_SWCName.h. Verify DataSendPoint configured in SwcInternalBehavior for that runnable.

Bug: DET error UNINIT (0x07)

API called before Rte_Start or after Rte_Stop. Common in initialization code before EcuM calls Rte_Start, or in background tasks. Fix: enable RteDevErrorDetectUninit. Verify startup sequence order. Check that init tasks don't fire until RTE ready.

Bug: Rte_Receive returns RTE_E_LOST_DATA

Queue overflowed. Sender produces faster than receiver consumes. Fix: increase QueueLength in ComSpec. Or increase receiver task frequency. Or switch to non-queued if ordering not required.

Bug: C/S call hangs (blocking)

Server task has lower priority than client. When client blocks at WaitPoint, server can't run (same core, higher prio blocked). Fix: server task must have higher priority than client, or use asynchronous C/S. Check task priority assignment in OS config.
