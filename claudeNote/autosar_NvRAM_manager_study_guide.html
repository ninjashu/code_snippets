AUTOSAR NvM R24-11 — Study Guide

Architecture Overview

The NvM (Non-Volatile Memory) Manager is an AUTOSAR Basic Software (BSW) module that provides hardware-independent services to store, read, and maintain non-volatile data. It sits between application SW-Cs and the underlying memory abstraction stack.

🎯 Exam Critical

NvM is hardware-independent. It never accesses hardware directly — always via MemIf → EA/FEE → Driver. It uses a virtual 32-bit address space: upper 16 bits = block number, lower 16 bits = offset within block. This is configured per block via NvMNvBlockBaseNumber and NvMDatasetSelectionBits.

Key External Interactions

BswM BSW Mode Manager

Controls NvM lifecycle

  • Calls NvM_Init() exclusively at startup
  • Calls NvM_ReadAll() exclusively at startup
  • Calls NvM_WriteAll() exclusively at shutdown
  • Uses NvM_CancelWriteAll() to abort WriteAll
  • Receives job status via BswM_NvM_CurrentBlockMode() and BswM_NvM_CurrentJobMode()
DEM Diagnostic Event Manager

Receives production errors

  • NVM_E_HARDWARE — hardware read/write failure
  • NVM_E_INTEGRITY_FAILED — CRC mismatch on read
  • NVM_E_REQ_FAILED — lower-layer job failed
  • NVM_E_WRONG_BLOCK_ID — Static Block ID mismatch
  • NVM_E_VERIFY_FAILED — write verification failed
  • NVM_E_LOSS_OF_REDUNDANCY — redundant block recovery failed
DET Default Error Tracer

Receives development errors

  • Wrong block ID, wrong pointer, NvM not initialized
  • Block pending (already queued or in progress)
  • Block write-protected, WriteBlockOnce status unknown
  • Queue full (NVM_E_QUEUE_FULL runtime error)
  • Only active when NvMDevErrorDetect = TRUE
CRC Module Integrity Check

Provides CRC routines

  • 8-bit, 16-bit, or 32-bit CRC generation
  • Used optionally per block via NvMBlockUseCrc
  • Also calculated over RAM block if NvMCalcRamBlockCrc = TRUE
  • For ciphered blocks: CRC is over encrypted data, not plaintext

Virtual Address Scheme

NvM uses a virtual 16-bit block number split into two fields to address MemIf blocks:

FEE/EA_BLOCK_NUMBER = (NvMNvBlockBaseNumber << NvMDatasetSelectionBits) + DataIndex
FieldBitsRange (if DatasetSelectionBits=2)Purpose
NvMNvBlockBaseNumber16 − DatasetBits0x1 .. 0x3FFEUnique block identifier configured at tool time
DataIndexDatasetSelectionBits0x0 .. 0x3 (4 datasets max)Selects which NV/ROM copy within a Dataset/Redundant block
🎯 Exam Critical — Address Formula Example

If NvMDatasetSelectionBits=4: block range 0x1..0xFFE, data index 0x0..0xF. Native block BaseNum=2 → EA_BLOCK_NUMBER = 2<<4 = 32. Redundant block BaseNum=3 → copy0=48, copy1=49.

  • Upper layers include only: NvM.h (not MemIf or Dem directly)

Storage Objects & Block Types

Every NvM block is composed of Basic Storage Objects. Understanding each type and how they combine into block management types is fundamental to NvM design and testing.

4 Basic Storage Objects

MandatoryNV Memory

NV Block

Resides in non-volatile memory (EEPROM/Flash). Contains: User Data + optional CRC + optional NV Block Header (if Static Block ID enabled). This is the actual persistent storage.

Requirement

SWS_NvM_00125 — mandatory part of every NVRAM block.

MandatoryRAM

RAM Block

Resides in RAM. The application reads/writes this block directly. Contains: User Data + optional CRC. The address may be permanently assigned (config time) or temporarily passed at runtime.

  • Permanent RAM: address known at configuration time
  • Temporary RAM: address passed via API parameter at runtime
  • CRC type on RAM must match the corresponding NV block
OptionalROM/Flash

ROM Block

Resides in ROM (Flash). Provides default data for initialization or recovery when NV block is damaged/empty. Only readable — NvM treats a write to a ROM block like a write to a write-protected NV block.

If no ROM block and no NvMInitBlockCallback, the application must provide default data. Use NvM_GetErrorStatus() to distinguish first-init from corruption.

MandatoryRAM (internal)

Administrative Block

Resides in RAM. Invisible to the application — used exclusively by NvM. Contains:

  • Block Index — for Dataset block dataset selection
  • Error/Status field — result of last NvM request
  • Attribute field — write protection flag, validity flag
Testing Tip

Application must NOT call NvM_SetDataIndex, NvM_SetBlockProtection, or NvM_SetRamBlockStatus while a pending job exists for that block.

NVRAM Block Descriptor Table

Each configured NVRAM block has a descriptor entry in ROM (generated at configuration time). It contains:

  • Block ID (unique, assigned sequentially)
  • Block management type (Native/Redundant/Dataset)
  • Block size in bytes
  • Device ID (which MemIf device)
  • NvMNvBlockBaseNumber
  • Number of NV blocks (NvMNvBlockNum)
  • Number of ROM blocks (NvMRomBlockNum)
  • RAM block start address (if permanent)
  • ROM block start address (if present)
  • CRC configuration (NvMBlockUseCrc, type)
  • Single/multi block callback pointers
  • Write protection, WriteOnce, priority settings
🎯 Exam Critical

Block ID 0 is reserved — used with NvM_GetErrorStatus(0) to query global multi-block operation status (ReadAll/WriteAll). Application blocks start from ID 1.

RAM Block State Machine

Each RAM block has an internal state tracked by NvM in the Administrative block. Understanding state transitions is critical for debugging NvM behavior and writing correct application code.

The 3 States

VALID / UNCHANGED

RAM matches NV (or untouched since last successful R/W). Last operation was successful. Block not invalidated.

  • WriteAll is NOT triggered for this block
  • Left when SetRamBlockStatus(TRUE) called
VALID / CHANGED

RAM potentially differs from NV. Block data is valid for use by application but NvM knows it may need writing.

  • WriteAll WILL write this block to NV
  • Left when write completes successfully
INVALID / UNCHANGED

Block data is invalid. Could be: post-init, corrupted NV, failed read, after erase/invalidate.

  • WriteAll does NOT write this block
  • Application must provide valid data before next write

INVALID/CHANGED state does NOT exist — the combination is logically unreachable. NvM uses only the 3 states above.

🎯 Exam Critical — SetRamBlockStatus disabled case

If NvMSetRamBlockStatusApi = FALSE or NvMBlockUseSetRamBlockStatus = FALSE, NvM treats all permanent RAM blocks as VALID/CHANGED when writing → NvM_WriteAll() always writes every permanent block. And as INVALID when reading → NvM_ReadAll() always reads every permanent block from NV.

After Successful Write — Key Rule

SWS_NvM_00472: After a RAM block is successfully copied to NV memory (via WriteBlock or WriteAll), the RAM block state is set to VALID/UNCHANGED. This means a subsequent WriteAll will skip this block (unless SetRamBlockStatus(TRUE) is called again).

Startup & Shutdown Flow

The NvM lifecycle is strictly managed by BswM. Understanding the exact sequence, who calls what, and what happens to each block during ReadAll is critical for system-level testing and debugging.

Complete ECU Lifecycle

1

Power-On: NvM_Init() called by BswM [Sync]

Resets all internal variables (queues, state machines, request flags) to initial values. Sets all Dataset block indices to 0. Does NOT read any NV data. Does NOT initialize underlying drivers. Signals "INIT DONE" internally to enable job processing.

Critical

NvM_Init(ConfigPtr) — ConfigPtr shall ALWAYS be NULL_PTR (unused for now). Called exclusively by BswM. RAM blocks are still INVALID/UNCHANGED after this step.

2

NvM_ReadAll() called by BswM [Async]

Initializes all configured permanent RAM blocks from NV. Processed in block ID order (lowest first). For each block, one of the following outcomes occurs:

ConditionActionResulting State
RAM block CRC valid (revalidation)Skip reading from NV — use existing RAMVALID/UNCHANGED
NV block reads successfully, CRC OKCopy NV → RAMVALID/UNCHANGED
NV fails but ROM/callback availableCopy default → RAMVALID/CHANGED
NV fails, no default dataLeave RAM invalidINVALID/UNCHANGED
NvMResistantToChangedSw=FALSE + config mismatchLoad defaults regardless of NV validityVALID/CHANGED

Note 1: To ensure DEM data is ready early, configure DEM-related NVRAM blocks with low block IDs — they are processed first.
Note 2: Single-block callbacks fire after each block completes → RTE can start SW-Cs individually without waiting for full ReadAll.
Note 3: Multi-block callback fires when the entire ReadAll finishes.

3

Normal Operation: SW-C reads/writes via NvM APIs

Applications use single-block APIs. Requests are queued (FCFS or priority-based). NvM processes requests in NvM_MainFunction() context. Application uses polling or callbacks for completion notification. RAM blocks may be modified by app between NvM operations.

  • App fills RAM → calls NvM_WriteBlock(blockId, ramPtr) → waits for NVM_REQ_OK
  • App calls NvM_ReadBlock(blockId, ramPtr) → waits → reads updated RAM
  • App can call NvM_SetRamBlockStatus(blockId, TRUE) to mark block as CHANGED
4

Shutdown: NvM_WriteAll() called by BswM [Async]

Writes all VALID/CHANGED RAM blocks to NV. Skips INVALID and VALID/UNCHANGED blocks. After each successful write → block becomes VALID/UNCHANGED. Generates a single-block callback per block, and multi-block callback at end.

During WriteAll: new write requests are queued and processed after WriteAll. New read requests are also queued. No SW-C should modify RAM blocks during shutdown — this is enforced by application design, not NvM itself.

5

Optional: NvM_CancelWriteAll() called by BswM

Aborts an ongoing WriteAll. The current block being written completes fully, but no further blocks are written. Only BswM may call this. After CancelWriteAll, the multi-block result is NVM_REQ_OK if the cancellation itself was clean (not a write failure).

Dynamic Configuration (Software Update)

Config ParameterValueBehavior on ReadAll
NvMDynamicConfiguration FALSE (default) Ignore stored configuration ID. Normal runtime preparation: check RAM CRC → read NV → load defaults on failure.
NvMDynamicConfiguration TRUE Check configuration ID against stored ID. If mismatch detected → extended runtime preparation for blocks with NvMResistantToChangedSw=FALSE: load defaults regardless of NV validity.
NvMResistantToChangedSw TRUE Block is NOT re-initialized on software update. Existing NV data is used even after config ID mismatch.
NvMResistantToChangedSw FALSE Block IS re-initialized on software update (config mismatch). Defaults loaded unconditionally.
🎯 Testing Scenario

This is the mechanism that handles OTA software updates. After a new software image is flashed, blocks with NvMResistantToChangedSw=FALSE will get fresh default values — preventing old calibration data from being used with a new SW version that expects different data layout.

Status Polling

Single Block Status

  • NvM_GetErrorStatus(blockId, &result)
  • NVM_REQ_PENDING — job queued or in progress
  • NVM_REQ_OK — last operation succeeded
  • NVM_REQ_NOT_OK — last operation failed
  • NVM_REQ_INTEGRITY_FAILED — CRC mismatch
  • NVM_REQ_BLOCK_SKIPPED — block skipped (WriteAll)
  • NVM_REQ_NV_INVALIDATED — block was invalidated
  • NVM_REQ_RESTORED_FROM_ROM — default data used

Multi-Block Status

  • NvM_GetErrorStatus(0, &result) — Block ID 0 = reserved for global status
  • NVM_REQ_PENDING — ReadAll/WriteAll in progress
  • NVM_REQ_OK — all blocks processed successfully
  • NVM_REQ_NOT_OK — at least one block had an error
  • Individual block status still readable via block-specific block ID

API Reference

NvM provides 4 types of API functions. Understanding each function's behavior, error conditions, and synchronization requirements is essential for both development and testing.

NvM_Init Type 4 Sync Non-Reentrant Service ID: 0x00
void NvM_Init(const NvM_ConfigType* ConfigPtr); // ConfigPtr SHALL always be NULL_PTR

Resets all internal variables, queues, state machines. Does NOT read NV data. Does NOT initialize drivers. Called exclusively by BswM at startup.

DET errors

No DET errors (it's the initializer). But all other APIs report NVM_E_UNINIT if called before NvM_Init.

NvM_ReadAll Type 3 Async Non-Reentrant Service ID: 0x0C
void NvM_ReadAll(void);

Initializes all permanent RAM blocks from NV. Called exclusively by BswM. Not interrupted by single-block requests (except immediate-priority writes which preempt, then resume). Queues read/write requests received during processing.

Writes all VALID/CHANGED blocks to NV. Called exclusively by BswM at shutdown. Queues read/write requests received during processing. Can be aborted with NvM_CancelWriteAll().

Exam Tip

Application MUST call NvM_SetRamBlockStatus(id, TRUE) after modifying RAM data for WriteAll to pick it up — IF NvMSetRamBlockStatusApi is enabled.

NvM_SetDataIndex Type 1 Sync Reentrant Service ID: 0x01
Std_ReturnType NvM_SetDataIndex(NvM_BlockIdType BlockId, uint8 DataIndex);

Selects which dataset (NV or ROM copy) to access for a DATASET block. Index is stored in the Administrative block. On init: all dataset indices = 0. Does NOT modify RAM contents.

Returns E_NOT_OK (silently in production mode) for non-DATASET blocks. NOT permitted while a pending job exists for this block.

NvM_InvalidateNvBlock Type 2 Async Reentrant Service ID: 0x19
Std_ReturnType NvM_InvalidateNvBlock(NvM_BlockIdType BlockId); // Marks NV block as invalid in underlying memory layer // Result: block becomes INVALID/UNCHANGED in RAM state

Tells lower memory layers that this block's data is no longer valid. On next ReadAll, the block will be treated as unreadable → defaults will be loaded if available. Use to prevent lower layers from delivering old/stale data.

NvM_RestoreBlockDefaults Type 2 Async Reentrant Service ID: 0x18
Std_ReturnType NvM_RestoreBlockDefaults(NvM_BlockIdType BlockId, void* NvM_DstPtr); // Copies ROM default data → RAM block. Does NOT modify NV.

Explicit recovery. Copies ROM block data to RAM block. Does NOT modify the NV block content. Result → block becomes VALID/CHANGED. DET error if no ROM block and no NvMInitBlockCallback configured.

NvM_CancelJobs Type 2 Sync Reentrant Service ID: 0x10
Std_ReturnType NvM_CancelJobs(NvM_BlockIdType BlockId);

Removes all unprocessed requests for a specific block from the queue. Cannot cancel a job already in progress.

NvM_ValidateAll Type 3 Async Service ID: 0x19
void NvM_ValidateAll(void);

Marks all configured permanent RAM blocks as VALID/CHANGED (without reading from NV). Used for forced initialization of all blocks. All blocks will be written by subsequent WriteAll.

Job Queue & Priority

NvM processes all requests asynchronously and serially. Understanding the queuing mechanism and priority system is crucial for timing analysis and debugging "why didn't my write happen?" issues.

Queue Architecture

Immediate Write Queue

For crash data — safety-critical writes that must preempt ongoing operations.

  • Only write jobs with immediate priority
  • Preempts any currently running job
  • Preempted job is resumed/restarted after completion
  • Only available if NvMJobPrioritization = TRUE

SWS_NvM_00182: Immediate-priority write preempts even ReadAll and WriteAll jobs. The preempted job resumes from where it was interrupted.

Standard Queue

For all other single-block requests (read, write, erase, invalidate, restore defaults, etc.)

  • Processed in priority order (if enabled) or FCFS
  • Multiple concurrent requests allowed until queue full
  • Same block cannot be queued twice simultaneously
  • On queue overflow → returns E_NOT_OK + DET NVM_E_QUEUE_FULL

Multi-Block Request Queue

For ReadAll, WriteAll, ValidateAll, FirstInitAll. Queue length = 1 (only one multi-block operation can be queued at a time).

Priority-Based Job Processing

SettingBehaviorQueues Active
NvMJobPrioritization = TRUE Priority-based. Higher priority jobs processed first. Immediate writes preempt. Same block same priority → FCFS within priority level. Immediate + Standard + Multi-block
NvMJobPrioritization = FALSE Pure FCFS. No immediate writes. All single-block jobs processed in order received. Standard (FCFS) + Multi-block

Request Queuing Rules

  • NvM sets block status to NVM_REQ_PENDING immediately on successful enqueueing
  • Same block ID cannot be queued again while already queued or in progress → DET NVM_E_BLOCK_PENDING
  • Obsolete queued requests may be dequeued if the block completes by other means
  • CancelJobs() removes all pending requests for a block (cannot cancel in-progress job)

ReadAll / WriteAll Interaction with Single-Block Requests

Ongoing OperationRead Request ReceivedWrite Request ReceivedImmediate Write Received
NvM_ReadAll() Queued → executed after ReadAll Queued → executed after ReadAll Preempts ReadAll! Resumes after.
NvM_WriteAll() Queued → executed after WriteAll Queued → executed after WriteAll Preempts WriteAll! Resumes after.
🎯 Exam Critical — Preemption

An immediate-priority write (e.g., crash data from an accelerometer) CAN interrupt ReadAll/WriteAll. This is the ONLY exception to the rule that ReadAll/WriteAll cannot be interrupted. After the immediate write completes, the interrupted multi-block job RESUMES from the block it was processing. SWS_NvM_00182, SWS_NvM_00570.

CRC Processing

CRC calculation is done in multiple steps if the block length exceeds NvMCrcNumOfBytes bytes. This prevents long NvM_MainFunction() execution times for large blocks. NvM uses the initial values published by the CRC module.

Performance Note

For large blocks, CRC is computed incrementally across multiple MainFunction calls. This is why NvM jobs can take many scheduler cycles to complete — the CRC chunking spreads the computational load.

DET: Parameter errors, uninit calls, pending block calls, configuration errors → development builds only.

Key distinction: NVM_E_HARDWARE fires on MemIf-level failure. NVM_E_INTEGRITY_FAILED fires when MemIf succeeds but the CRC check on the data fails.

Advanced Features

These features are critical for real-world automotive applications: security (encryption), reliability (CRC, redundancy, static IDs), write optimization, and synchronization patterns.

CRC-Based Data Integrity

NV Block CRC (NvMBlockUseCrc)

  • Appended to NV block: [User Data | CRC]
  • Recalculated on every read to verify NV integrity
  • 8, 16, or 32-bit depending on NvMBlockCrcType
  • Failure → NVM_E_INTEGRITY_FAILED to DEM
  • Failure → read retry and recovery mechanisms triggered

RAM Block CRC (NvMCalcRamBlockCrc)

  • CRC stored alongside RAM data in RAM itself
  • Used during NvM_ReadAll() for RAM revalidation
  • If RAM CRC is valid → skip reading from NV (optimization!)
  • Provides protection against RAM corruption across resets
  • Failure → read error recovery triggered
  • Must be same CRC type as NV block CRC
🎯 CRC Interaction Rules

For blocks with NvMWriteBlockOnce=TRUE: enable CRC (NvMBlockUseCrc=TRUE) but disable static write protection (NvMBlockWriteProt=FALSE). This allows the application to write the block initially (on blank device) but CRC verification prevents accidental overwrites. CRC is computed in multiple steps if block size > NvMCrcNumOfBytes.

After a reset, the write protection flag of a NvMWriteBlockOnce=TRUE block is CLEARED (Admin block is RAM-based). To reactivate: the block must be READ (single or as part of ReadAll) before any write/erase/invalidate. This read sets the write protection if the block is valid and consistent.

Static Block ID Check

When NvMStaticBlockIDCheck is enabled, NvM stores a Block ID field in the NV Block Header. On every read, NvM compares the stored Block ID against the requested Block ID.

  • Purpose: Detect hardware failures that cause a wrong physical block to be read (address line fault, wear-leveling error)
  • On mismatch: Report NVM_E_WRONG_BLOCK_ID to DEM. Trigger read error recovery (read retries → load defaults).
  • NV Block Header layout: [Block ID Header | User Data | CRC]
  • Requirement: All Static Block IDs must be unique (verified at configuration time)
Automotive Context

Critical for Flash-based storage where FEE wear-leveling could, in case of corruption, return data from a different logical block. The Static Block ID acts as a sanity check independent of the address-based access.

Read Retry & Write Verification

Read Retry

On read failure (MemIf fails or CRC error): up to NVM_MAX_NUM_OF_READ_RETRIES additional read attempts before giving up and trying the redundant copy or ROM defaults.

Also applies to Static Block ID check failures on REDUNDANT blocks. After all retries: proceed to read the second NV copy for REDUNDANT blocks, or ROM block for all types.

Write Verification

When NvMWriteVerification = TRUE: after writing, NvM reads the written data back from NV and compares it to the original RAM content.

  • Comparison in steps: max NVM_WRITE_VERIFICATION_DATA_SIZE bytes at a time
  • Mismatch → NVM_E_VERIFY_FAILED to DEM + write retry
  • Read-back failure → no read retries (different from normal read retry)
  • All retries fail → NVM_REQ_NOT_OK result

Implicit vs Explicit Synchronization

Implicit Synchronization (default)

NvM accesses the application's RAM block directly. Application is responsible for not modifying the RAM block while a job is pending.

Write sequence:

  1. App fills RAM block with data
  2. App calls NvM_WriteBlock()
  3. App must NOT modify RAM until job completes
  4. App polls GetErrorStatus() or waits for callback
  5. After completion, RAM is freely modifiable again

Explicit Synchronization (NvMBlockUseSyncMechanism=TRUE)

NvM maintains an internal RAM mirror. Data exchanged via callbacks.

  • NvM allocates ONE shared mirror for all sync blocks (sized to largest)
  • NvMWriteRamBlockToNvCallback: app copies data TO mirror
  • NvMReadRamBlockFromNvCallback: app copies data FROM mirror
  • App can modify RAM even while job is processing (until callback)
  • Callback returns E_NOT_OK → NvM retries NvMRepeatMirrorOperations times
  • Explicit sync block CANNOT have a permanent RAM image
Use case

Shared NVRAM blocks accessed by multiple applications. The "owner" app manages the callback and ensures consistent data snapshot.

CRC-Based Compare Mechanism

Enabled per block via NvMBlockUseCRCCompMechanism=TRUE (requires NvMBlockUseCrc=TRUE). Before writing, NvM computes the RAM block CRC and compares it to the stored NV CRC. If they match → write is skipped (saves NV write cycles!).

Risk: Different data may produce the same CRC (collision). Only use this for blocks where occasional skipped writes are tolerable. Not suitable for safety-critical data.

Block Ciphering (Security)

When NvMBlockCipheringRef is configured, NvM uses the CSM (Crypto Service Manager) module for transparent encryption/decryption. Application always works with plain data.

Write flow

Plain data → Csm_Encrypt() → CRC over encrypted data → write ciphered data to NV

Read flow

Read from NV → CRC check over encrypted data → Csm_Decrypt() → plain data to app. If CRC fails → abort, no decryption.

  • Supports symmetric algorithms (e.g., AES128, 16-byte aligned)
  • CRYPTO_E_BUSY: retry up to NvMCsmRetryCounter times
  • Any other error → abort with NVM_REQ_NOT_OK
  • Length mismatch → DET NVM_E_BLOCK_CHIPHER_LENGTH_MISSMATCH
Automotive Context

Used for storing secret keys, security tokens, or sensitive calibration data that must not be readable from raw NV dumps. Integrator must ensure CSM is initialized before NvM uses it (timing dependency).

Block Compression

When NvMBlockUseCompression=TRUE: NvM compresses block data before writing to NV. Compression algorithm is vendor-specific (block split, delta, lossless compression).

Use case: large blocks where only small portions change per drive cycle (e.g., drive-cycle logging, usage counters) — reduces total NV write cycles.

Drawback: Higher runtime for read/write operations due to compression/decompression overhead.

PRAM API Variants

APIs ending in PRAM (e.g., NvM_ReadPRAMBlock, NvM_WritePRAMBlock, NvM_RestorePRAMBlockDefaults) work only with permanently assigned RAM blocks — blocks that have a static RAM address configured at tool time. They don't accept a RAM pointer parameter since the address is already known.

Difference

NvM_ReadBlock(id, ramPtr) allows a temporary RAM address. NvM_ReadPRAMBlock(id) uses the permanent RAM address configured in the block descriptor. Both trigger the same underlying job — only the pointer source differs.
