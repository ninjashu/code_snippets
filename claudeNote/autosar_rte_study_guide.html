R24-11 · Document ID 84 · Classic Platform · For exam, integration, interface testing & DaVinci configuration

What is the AUTOSAR RTE?

The Run-Time Environment (RTE) is the central middleware of an AUTOSAR Classic Platform ECU. It is the physical realization of the Virtual Function Bus (VFB) — the abstract communication bus defined at system design level. For every ECU in the system, one unique RTE is generated (not hand-coded) by an RTE generator tool.

Primary roles

• Communication infrastructure between SWCs
• Scheduling SWC runnable entities via OS tasks
• Abstracting SWCs from OS, COM, hardware

Key property

Generated per-ECU from ARXML configuration. SWCs are location-independent — the same SWC code can run on different ECUs with no changes.

Scope

R24-11 covers: SWC API (Ch.5), BSW Scheduler API (Ch.6), Functional spec (Ch.4), Generation process (Ch.3), ECU config (Ch.8).

The VFB → RTE mapping

During system design, SWCs communicate over the abstract VFB. When deployed to a specific ECU, the RTE generator reads the ECU Extract (ARXML) and produces concrete C code — the RTE — which physically routes all those VFB communications. The same SWC code compiles unchanged; only the generated RTE glue differs per ECU.

💡 Key exam point: The RTE provides static communication only [SWS_Rte_06026]. All communication paths are determined at code-generation time. There is no dynamic routing at runtime — this is a deliberate design decision for determinism and efficiency in embedded systems.

Two-phase generation model

1. Contract phase

Uses only the SWC's port interface description. Generates the application header file (Rte_<SWC>.h) — the API "contract" between the SWC and the RTE. The SWC can be compiled against this header without full system knowledge.

Output: Rte_SWCName.h with all API macros/prototypes

2. Generation phase

Uses full ECU configuration: all SWCs, their port connections, task mappings, exclusive area configs. Generates the complete RTE implementation files.

Output: Rte.c, Rte.h, SchM.c, SchM.h

AUTOSAR ECU layered architecture

Key architectural rules (exam critical)

SWC isolation

An SWC cannot directly call OS services, COM functions, or other BSW APIs. All access is exclusively through the RTE API (Rte_* functions).

Port-based comms

All SWC communication happens through typed ports. A port is either a P-Port (provides) or R-Port (requires). Port interfaces define the data/operations exchanged.

Generated, not handwritten

The RTE is entirely generated from ARXML. DaVinci Developer/Configurator produce this ARXML; the RTE generator (built into DaVinci Configurator Pro) produces the C code.

Static connections only

Every sender→receiver and client→server connection is resolved at generation time. No dynamic routing. This enables deterministic worst-case execution time analysis.

Communication paradigms

The RTE supports four fundamental communication paradigms. Each maps to specific API calls, port interface types, and connector types in ARXML.

Sender-Receiver (S/R) communication

Signal/data passing between SWCs. One SWC sends data; one or more SWCs receive it. The port interface type is SenderReceiverInterface containing one or more VariableDataPrototype elements.

Explicit mode

The SWC explicitly calls Rte APIs to send/receive. Data is transferred at the moment of the API call.

/* Sender */
Rte_Write_PortName_DataName(&value);
Rte_Send_PortName_DataName(&value); /* queued */

/* Receiver */
Rte_Read_PortName_DataName(&value);
Rte_Receive_PortName_DataName(&value); /* queued */
Use Rte_Write/Rte_Read for non-queued (last-is-best). Use Rte_Send/Rte_Receive for queued (FIFO) data.

Implicit mode

RTE automatically copies data in/out of a local buffer before/after the runnable executes. The runnable accesses local variables — no explicit API call needed for data exchange.

/* Receiver — RTE copies before runnable */
Rte_IRead_RunName_PortName_DataName();

/* Sender — RTE copies after runnable */
Rte_IWrite_RunName_PortName_DataName(val);
Rte_IWriteRef_RunName_PortName_DataName();
Implicit is used when runnable needs a consistent snapshot of data for its entire execution — no mid-run updates. Configured via DataReadAccess/DataWriteAccess in ARXML.

Client-Server (C/S) communication

Function call paradigm. A client SWC calls an operation provided by a server SWC/BSW module. Port interface type: ClientServerInterface with ClientServerOperation elements.

Synchronous call

Client blocks until server returns result. Server runnable runs in client's context or a different task (with OS event wait).

Std_ReturnType ret;
ret = Rte_Call_PortName_OperationName(
        in_arg1, &out_arg2);
/* client continues after server returns */
Restriction: synchronous C/S across task boundaries requires a WaitPoint and an extended task. Category 1 runnables cannot use synchronous C/S.

Asynchronous call

Client fires the call and returns immediately. Later, a second runnable (triggered by AsynchronousServerCallReturnsEvent) collects the result.

/* Fire-and-forget */
ret = Rte_Call_PortName_OperationName(args);

/* In ASCR runnable */
ret = Rte_Result_PortName_OperationName(
        &out_arg);  /* collect result */

Multiplicity rules for C/S

C/S supports n:1 only — many clients, one server. The RTE serializes concurrent calls. 1:n (one client, multiple servers) is NOT supported. Server runnables are triggered by OperationInvokedEvent.

Port-Defined Argument Values

The RTE can supply constant argument values from the port configuration — the SWC doesn't have to pass them. This allows port-level specialization of operations.

Direct function call optimization

When client and server runnables are in the same OS task and partition, the RTE may implement C/S as a direct C function call (no task activation, no OS event) — a key performance optimization visible in the generated code.

Mode switch communication

Used to broadcast system modes (e.g., ECU state: STARTUP, RUN, SLEEP). Port interface type: ModeSwitchInterface with a ModeDeclarationGroup.

Mode manager (P-Port side)

/* Switch to a new mode */
ret = Rte_Switch_PortName_ModeGroupName(
        RTE_MODE_ModeGroup_NEW_MODE);

The SWC providing the mode switch port is the mode manager. Typically the EcuM service component manages the main ECU mode.

Mode user (R-Port side)

/* Query current mode */
mode = Rte_Mode_PortName_ModeGroupName();

/* Optionally acknowledge mode switch */
ret = Rte_SwitchAck_PortName_ModeGroupName();

Mode-dependent runnable activation

Runnables can be disabled in certain modes. A SwcModeSwitchEvent triggers runnables on entry, exit, or transition between modes. Three kinds of event:

Activation kindWhen triggered
entryWhen entering the specified mode — "on-entry ExecutableEntity"
exitWhen leaving the specified mode — "on-exit ExecutableEntity"
transitionOn a specific A→B mode transition — "on-transition ExecutableEntity"
The RTE maintains a mode machine instance per ModeDeclarationGroupPrototype. Since a mode switch is not instantaneous (all on-exit runnables must complete first), the mode machine instance tracks the in-progress state. For every mode manager, both the RTE and SchM share the same mode machine instance.

NvBlock (Non-Volatile) communication

A special SWC type — NvBlockSwComponentType — manages persistent NvM data blocks. The RTE provides a dedicated API for NvM interactions.

Writing strategies

StrategyBehavior
Write on finalData written to NvM only when SWC explicitly requests it
Write on changeRTE detects data change and triggers NvM write automatically
Write on shutdownData committed to NvM during ECU shutdown sequence

NvBlock API

/* Trigger read from NvM */
Rte_Call_NvMService_ReadBlock(&block_data);

/* Trigger write to NvM */
Rte_Call_NvMService_WriteBlock(&block_data);

/* Callbacks (generated for NvBlock SWC) */
Rte_NvMNotifyJobFinished_BlockName(ServiceId, JobResult);
Rte_NvMNotifyInitBlock_BlockName();

Mode machine state diagram

Mode machines are a core RTE concept for ECU lifecycle and feature management. Understanding the state machine is critical for both debugging and DaVinci configuration.

Mode machine instance state flow

Each ModeDeclarationGroupPrototype has one mode machine instance. The instance tracks not just the current mode but the in-progress transition state, since on-exit runnables must complete before the new mode takes effect.

Initialization of mode machine instances

At Rte_Init(), mode machine instances are initialized to the initial mode defined in the ModeDeclarationGroup. All mode users start in this initial mode — no entry runnable fires for the initial mode unless explicitly configured.

Mode-dependent scheduling

Runnables can be mode-disabled: a SwcModeSwitchEvent can have a disabledInMode reference. When the system is in that mode, the runnable is not executed even if its trigger fires. This is how you implement behavior like "do not run this cyclic function during ECU SLEEP mode."

/* In ARXML: disabledInMode reference on RTEEvent */
<DISABLED-MODE-IREF>
  <CONTEXT-PORT-REF>/EcuM_Mode_Port</CONTEXT-PORT-REF>
  <CONTEXT-MODE-DECLARATION-REF>/SLEEP</CONTEXT-MODE-DECLARATION-REF>
</DISABLED-MODE-IREF>

Activation vs wakeup

ACT — Activate

The runnable is scheduled to run. The RTE activates the OS task that contains this runnable. Most events can ACT a runnable. No explicit RTE API call needed in the runnable to set up the activation.

WUP — Wake up at WaitPoint

A category 2 runnable has called a blocking RTE API (e.g. Rte_Receive()) and is waiting. The event wakes it up. Only a subset of events can WUP (see table in spec §4.2.2.4).

Inter-Runnable Variables (IRV)

Private shared data between runnables of the same SWC instance. Not visible outside the SWC. Accessed via:

/* Read IRV */
val = Rte_IrvRead_RunName_IrvName();

/* Write IRV */
Rte_IrvWrite_RunName_IrvName(value);
IRVs are in the SWC's PerInstanceMemory. Multiple instantiated SWCs get separate IRV copies. For data consistency across runnables of different tasks, protect with an ExclusiveArea.

Multiple triggers on one runnable

One runnable can be triggered by multiple RTEEvents (e.g., both a TimingEvent AND a DataReceivedEvent). This is common for runnables that need to react to either periodic timeouts or data arrival. The runnable code typically uses Rte_Mode() or checks data status to determine which event fired.

Partition lifecycle

In multi-partition ECUs (OS-Applications), each partition can have its own RTE lifecycle. The RTE supports partition restart (via OS) and handles cross-partition communication consistency. IOC (Inter-OS-Application Communication) is generated by the OS generator based on RTE configuration inputs.

Critical: cross-partition C/S calls require trusted function call (OS CallTrustedFunction()) which has significant latency. Design SWC partitioning carefully to minimize cross-partition synchronous server calls in timing-critical paths.

Vector DaVinci configuration — step by step

DaVinci Developer is used for SWC design (ARXML authoring). DaVinci Configurator Pro performs ECU configuration and RTE generation. Here is the complete workflow for configuring and generating the RTE for a project.

DaVinci Developer — SWC design

1
Create SWC type: In DaVinci Developer, define ApplicationSwComponentType. Add ports (P-Port/R-Port) and define their PortInterface (SenderReceiverInterface or ClientServerInterface).
2
Define SwcInternalBehavior: Add runnables under SwcInternalBehavior. For each runnable, define its RTEEvent (TimingEvent with period, DataReceivedEvent, InitEvent, etc.).
3
Configure data access: For each runnable, define DataSendPoint / DataReceivePoint (explicit) or DataReadAccess / DataWriteAccess (implicit). For C/S: define ServerCallPoint (sync/async).
4
Configure ComSpec: On each port, set InitValue, AliveTimeout (for receiver), QueueLength (for queued S/R), HandleInvalidOption (keep/replace/default).
5
ExclusiveAreas: Define ExclusiveArea elements in SwcInternalBehavior. Assign to runnables that share data. In DaVinci Developer: right-click runnable → Add ExclusiveArea reference.
6
PerInstanceMemory: Add PerInstanceMemory elements to the SwcInternalBehavior for SWC-local RAM. Each SWC instance gets a separate block.
7
Export ARXML: File → Export → AUTOSAR XML. This produces the SWC description file used in the system-level design.

System-level design (DaVinci Developer or System Designer)

8
Create system composition: Place SWC prototypes (instances of SWC types) in a CompositionSwComponentType. Connect ports with AssemblySwConnector (SWC-to-SWC) or DelegationSwConnector (composition boundary).
9
Map SWCs to ECUs: Assign each SWC prototype to an ECU instance. This generates the ECU Extract — the per-ECU view of the system configuration.

DaVinci Configurator Pro — ECU configuration & RTE generation

10
Import ECU Extract: File → Import → ECU Extract ARXML. DaVinci Configurator Pro loads all SWC descriptions, port connections, and inter-ECU signal mappings.
11
OS configuration: Configure AUTOSAR OS (tasks, alarms, schedule tables, events, OS applications). The RTE generator reads this to build task bodies. Critical: task names and OS objects must match what the RTE generator expects.
12
RTE event-to-task mapping: In the RTE configuration (RteEventToTaskMapping container), map each RTEEvent to a specific OS task. Example: map the 10ms TimingEvent to Task_10ms, the DataReceivedEvent to Task_Event.
13
ExclusiveArea implementation: Configure each ExclusiveArea's implementation mechanism: OS Resource (for cross-task protection), SuspendAllInterrupts, SuspendOSInterrupts, or NONE (if only accessed from one task).
14
COM configuration: Map SWC port data elements to COM signals/signal groups. Inter-ECU data is transmitted via COM. Configure signal PDU mappings, I-PDU composition.
15
Run RTE Contract Phase: In DaVinci Configurator Pro: RTE → Generate → Contract Phase. Produces Rte_<SWCName>.h headers. SWC code can now be compiled and checked against the contract.
16
Run RTE Generation Phase: RTE → Generate → All. Produces Rte.c, Rte.h, SchM.c, SchM.h, and supporting files. Check the generation report for errors.

Common DaVinci configuration mistakes

🔴 Missing event-to-task mapping: Every RTEEvent must be mapped to an OS task, or the RTE generator will fail with an error. Check: RteEventToTaskMapping completeness in the RTE container.
🔴 OS task not configured: Referenced OS tasks must exist in the OS configuration. The RTE generator validates this and rejects inconsistencies [SWS_Rte_08417].
⚠️ ExclusiveArea mechanism mismatch: If an ExclusiveArea is used by runnables in multiple tasks, the implementation must be OS Resource (not SuspendInterrupts). Incorrect mechanism = data corruption risk.
⚠️ Wrong ComSpec InitValue: If the receiver has AliveTimeout but the sender SWC's init phase is slow, the receiver may report RTE_E_MAX_AGE_EXCEEDED before real data arrives. Set appropriate timeout or handle this in the application.
⚠️ Queued length too small: If QueueLength on a receiver port is too small for the production rate, RTE_E_LOST_DATA will occur. Monitor with VFB tracing.
💡 Contract phase first: Always run contract phase first and compile SWC code against it before full generation. This catches API mismatches early without needing a complete ECU configuration.

Integration, interface testing & debugging

VFB tracing

The RTE has a built-in VFB tracing mechanism [SRS_Rte_00045]. When enabled, it hooks into all RTE API calls and sends trace events to a trace tool (e.g. AUTOSAR Trace Logger, Vector CANalyzer with AUTOSAR plugin).

Common integration bugs and how to diagnose

Bug: Rte_Read returns RTE_E_NEVER_RECEIVED

Root causes: (1) sender SWC not initialized yet, (2) COM signal not received from another ECU, (3) port not connected in ARXML (missing AssemblySwConnector).

Debug steps: VFB trace → check if Rte_Write on sender side ever called → check COM signal reception → check ARXML port connections in DaVinci.

Bug: Rte_Read returns RTE_E_MAX_AGE_EXCEEDED

Root cause: AliveTimeout configured on the receiver port, and the sender is not writing frequently enough. Can also happen during system startup before sender is running.

Debug: check TimingEvent period of sender vs AliveTimeout value. Also check if sender task is running (check OS task activation with trace tool).

Bug: Rte_Receive returns RTE_E_LOST_DATA

Root cause: queued port queue overflowed — receiver not reading fast enough or sender writing too fast. QueueLength too small.

Debug: check production vs consumption rate. Increase QueueLength in ComSpec or slow down the sender. VFB trace will show the exact moment of overflow.

Bug: Mode switch never completes (mode stuck in TRANSITION)

Root causes: (1) on-exit runnable not completing (infinite loop, blocking call), (2) ModeSwitchedAckEvent runnable not running (task mapping error), (3) multiple Rte_Switch calls without waiting for completion.

Debug: VFB mode switch trace events → check if on-exit runnable fired and completed → check OS task scheduling.

Bug: Synchronous C/S call hangs (timeout)

Root cause: server task has lower priority than client task, causing priority inversion. Or server runnable never activated due to task mapping error.

Debug: Check RTE_E_TIMEOUT return value. Verify server task priority in OS config. Check OperationInvokedEvent is mapped to a running task.

Bug: Data corruption / race condition

Root cause: Two runnables in different tasks accessing shared PerInstanceMemory or IRV without ExclusiveArea protection.

Debug: Enable OS stack monitoring and data trace. Review ExclusiveArea configuration in DaVinci — each shared variable access must be protected.

RTE implementation plug-ins (RIPS) for testing

The RTE supports RTE Implementation Plug-Ins — extension points where external code (e.g. XCP-based bypass tools) can intercept and optionally modify communication data. This is used for:

  • Measurement: reading data values via XCP/CAN calibration protocols (A2L file-based)
  • Calibration: writing calibration parameter values during runtime (XCP stimulation)
  • Bypass: replacing SWC output data with externally generated values (RP bypass)

MISRA C violations in RTE code

The RTE specification (Appendix C) explicitly acknowledges that generated RTE code may violate certain MISRA-C rules. These violations are formally justified and documented. In a project audit, you should be aware of and able to explain these justified MISRA deviations — they are not bugs.
