What is SecOC? — "The security guard at the PDU gateway"

SecOC (Secure Onboard Communication) adds a Message Authentication Code (MAC) and a Freshness Value to every outgoing PDU, and verifies them on every incoming PDU. This protects against two attacks: data manipulation (changed payload → MAC fails) and replay attacks (old recorded PDU → freshness check fails).

Mental model: Every CAN/FlexRay/Ethernet message that passes through SecOC gets a unique cryptographic stamp (MAC) that includes a freshness counter. The receiver checks the stamp. If the stamp is wrong (tampered message) or the counter is stale (replayed old message) → message is dropped. Same concept as HTTPS certificate + timestamp, but for vehicle bus PDUs.

SecOC sits between PduR layers. Upper PDU = Authentic I-PDU (plain data). Lower PDU = Secured I-PDU (data + FV + MAC). SecOC is invisible to COM and to the bus driver.

Three Core Actors

SecOC ModuleOrchestrates TX (authenticate) and RX (verify). Calls CSM for MAC operations and FVM for freshness. Runs in MainFunctionTx and MainFunctionRx cycles.
CSM (Crypto Service Manager)Executes the actual MAC-Generate and MAC-Verify using a symmetric key (AES-128 CMAC typically). Can be sync or async. SecOC provides callback for async CSM jobs.
Freshness Value Manager (FVM)Provides and tracks freshness values per PDU. Can be SW-C (via RTE) or C-function callouts (CFUNC). Manages counters/timestamps. Implements anti-replay logic.

Key Constraints (Exam must-know)

1. Big Endian only — all SecOC data (FV, MAC, DataId) encoded in Big Endian. [SWS_SecOC_00011]

2. FV aligned to MSB — freshness byte array: FV MSB is in bit 7 of byte 0. Unused bits set to 0. [SWS_SecOC_00220]

3. DataToAuthenticator = DataId | AuthenticIPDU | CompleteFV — exact input to MAC algorithm. [SWS_SecOC_00034, 00046]

4. TX: authenticate in MainFunctionTx — SecOC_IfTransmit just copies buffer. MAC built in next main function call.

5. RX: verify in MainFunctionRx — SecOC_RxIndication copies buffer. Verification in next main function call.

6. No production errors — Section 7.9.3 states none. No extended production errors either.

7. MAC truncation = MSB — truncated MAC always takes MOST SIGNIFICANT bits. [SWS_SecOC_00036]

8. FV truncation = LSB — truncated FV takes LEAST SIGNIFICANT bits of the FV. [SWS_SecOC_00094]

9. SomeIpTp constraint — SecOC must be before SomeIpTp (SecOC→PduR→SomeIpTp). Reverse not supported. [Section 4.2]

10. NOT for MOST and LIN — SecOC not specified for MOST and LIN networks. [Section 4.1]

11. Asymmetric: no truncation — for signatures (RSA/ECDSA), SecOCAuthInfoTruncLength MUST equal full signature length. Cannot truncate signatures. [Section 7.1.4]

Secured PDU Collection (split PDU mode) [SWS_SecOC_00201]

When SecOCTxSecuredPduCollection (TX) or SecOCRxSecuredPduCollection (RX) is configured: the Authentic I-PDU and Authentication Information are transmitted as SEPARATE I-PDUs on the bus.

Authentic I-PDU (separate)Original payload data, sent as one message.
Cryptographic I-PDU (separate)Authentication data | Message Linker. Sent in same main function cycle as Authentic I-PDU. [SWS_SecOC_00202]

Message Linker [SWS_SecOC_00209, 00210]: Part of Authentic I-PDU repeated inside Cryptographic I-PDU to link them together. Configured by SecOCMessageLinkPos and SecOCMessageLinkLen.

Use case: CAN frames where adding MAC would exceed 8 bytes. Send payload in one CAN frame and MAC in second CAN frame. SecOC reassembles on receiver before verification.

Secured Area — partial PDU security [SWS_SecOC_00311, 00312]

Not the whole Authentic I-PDU needs to be protected. Configure offset+length to protect only a portion:

SecOCSecuredTxPduOffset / SecOCSecuredRxPduOffset — byte offset where security starts.

SecOCSecuredTxPduLength / SecOCSecuredRxPduLength — length of secured area in bytes.

If offset + length > received PDU length → discard PDU. [SWS_SecOC_00313, 00314]

TX Authentication & RX Verification — Step by Step

TX: Creating a Secured I-PDU [SWS_SecOC_00031]

Six Steps of Authentication

Step 1: Prepare Secured I-PDU — allocate internal buffers for Authentic + Secured I-PDU + intermediate data. [SWS_SecOC_00033, 00057]
Step 2: Construct DataToAuthenticator = DataId(16-bit BigEndian) | SecuredPart(AuthenticIPDU) | CompleteFreshnessValue [SWS_SecOC_00034]
Step 3: Generate Authenticator — call CSM_MacGenerate(DataToAuthenticator) → full MAC output. [SWS_SecOC_00035]
Step 4: Truncate MAC to SecOCAuthInfoTruncLength (MSBits). Truncate FV to SecOCFreshnessValueTruncLength (LSBits). [SWS_SecOC_00036, 00094]
Step 5: Construct Secured I-PDU: [Header] | AuthenticIPDU | TruncFV | TruncMAC. [SWS_SecOC_00037]
Step 6: Broadcast via PduR_SecOCTransmit. Notify FVM via SPduTxConfirmation. [SWS_SecOC_00062, 00232]

TX Transmission types

Type Trigger API How Secured PDU is delivered to bus SWS
Direct (CanIf style) SecOC_IfTransmit() SecOC calls PduR_SecOCTransmit() in MainFunctionTx. Bus IF picks it up immediately. SWS_SecOC_00060-00064
Triggered (FrIf style) SecOC_IfTransmit() SecOC calls PduR_SecOCTransmit(). Bus IF calls SecOC_TriggerTransmit() to pull data. SecOC copies Secured I-PDU. [SWS_SecOC_00068] SWS_SecOC_00065-00069
TP (CanTp/FrTp style) SecOC_TpTransmit() SecOC calls PduR_SecOCTransmit(). TP calls SecOC_CopyTxData() iteratively to get NPDUs. [SWS_SecOC_00073] SWS_SecOC_00070-00075
Triggered TX note: If SecOC_TriggerTransmit called before any SecOC_IfTransmit → E_NOT_OK returned (no Secured I-PDU available). [SWS_SecOC_00150]
Re-authentication (Triggered): If SecOCReAuthenticateAfterTriggerTransmit=TRUE → after TxConfirmation, preserved Authentic I-PDU is re-authenticated in next MainFunctionTx. [SWS_SecOC_00069]

RX: Verifying a Secured I-PDU [SWS_SecOC_00040]

Six Steps of Verification

Step 1: Parse — extract AuthenticIPDU, FV (truncated), Authenticator (truncated) from Secured I-PDU. [SWS_SecOC_00042]
Step 2: Get Freshness — call FVM with truncated FV and verify attempt count → FVM returns complete FV candidate. [SWS_SecOC_00247/00250]
Step 3: Construct DataToAuthenticator = DataId | AuthenticIPDU | CompleteFV (from FVM). [SWS_SecOC_00046]
Step 4: Verify Authenticator — call CSM_MacVerify(DataToAuthenticator, ReceivedMAC, TruncLength). [SWS_SecOC_00047]
Step 5: Notify FVM of result via VerificationStatusCallout/VerificationStatus. FVM uses result to advance counter. [SWS_SecOC_00243]
Step 6: Pass Authentic I-PDU to PduR via PduR_SecOCIfRxIndication if success. Drop if failed (unless IgnoreVerificationResult=TRUE). [SWS_SecOC_00080, 00081]

Gateway (Re-authentication) [Section 7.6]

Gateway ECU receives Secured I-PDU from Bus A (verify) → routes Authentic I-PDU → authenticates again for Bus B (auth). Each direction configured separately. Supported use cases:

Verification of incoming I-PDU from Bus A
Re-authentication for Bus B (new MAC with potentially different key/FV)
"Gatewaying-on-the-fly" NOT supported — SecOC cannot re-authenticate while routing without buffering the complete Authentic I-PDU first.

Freshness Values & Retry Logic — The anti-replay mechanism

Freshness Value Architecture

Each Secured I-PDU has a unique FreshnessValueId. The FVM tracks per-PDU counters or timestamps and provides them to SecOC on request.

SecOCQueryFreshnessValue = CFUNCSecOC calls C-function callouts directly: SecOC_GetTxFreshness(), SecOC_GetRxFreshness(), SecOC_SPduTxConfirmation(). Integrator writes these functions. Fastest approach.
SecOCQueryFreshnessValue = RTESecOC calls RTE service operations: FreshnessManagement.GetTxFreshness(), FreshnessManagement.GetRxFreshness(), FreshnessManagement.SPduTxConfirmation(). SW-C Freshness Value Manager.

TX freshness query variants

SecOCProvideTxTruncatedFreshnessValue CFUNC query RTE query SWS
TRUE — FVM also provides truncated FV SecOC_GetTxFreshnessTruncData() FreshnessManagement.GetTxFreshnessTruncData SWS_SecOC_00221, 00223
FALSE — SecOC does its own truncation SecOC_GetTxFreshness() FreshnessManagement.GetTxFreshness SWS_SecOC_00222, 00224

RX freshness query variants

SecOCUseAuthDataFreshness CFUNC query RTE query SWS
TRUE — FV extracted from Authentic I-PDU data SecOC_GetRxFreshnessAuthData() FreshnessManagement.GetRxFreshnessAuthData SWS_SecOC_00246, 00244
FALSE — FV from Secured I-PDU FV field SecOC_GetRxFreshness() FreshnessManagement.GetRxFreshness SWS_SecOC_00247, 00245

TX Retry Logic [SWS_SecOC_00225–00229]

Authentication Build Counter (TX)

1. New TX request → AuthenticationBuildCounter = 0. [SWS_SecOC_00226]
2. If GetTxFreshness() returns E_BUSY OR CSM_MacGenerate returns E_BUSY/QUEUE_FULL → increment counter. [SWS_SecOC_00227]
3. If counter < SecOCAuthenticationBuildAttempts → retry in next MainFunctionTx. [SWS_SecOC_00228]
4a. Counter reached limit OR GetTxFreshness returns E_NOT_OK OR CSM returns KEY_FAILURE:
→ If SecOC_SendDefaultAuthenticationInformation enabled: send default pattern (0xA5A5...). [SWS_SecOC_00229]
→ Otherwise: DROP Authentic I-PDU, cancel TX. Report SECOC_E_CRYPTO_FAILURE to DET. [SWS_SecOC_00155]
E_NOT_OK is not recoverable — only E_BUSY/QUEUE_FULL trigger retry. E_NOT_OK from CSM or FVM triggers failure. [SWS_SecOC_00227 Note]

RX Retry Logic [SWS_SecOC_00234–00243, 00256]

Two counters for RX: Build Counter + Verify Attempt Counter

1. New RX → AuthBuildCounter = 0, AuthVerifyAttemptCounter = 0. [SWS_SecOC_00235]
2. If GetRxFreshness() returns E_BUSY → increment BuildCounter. NO MAC verification attempted. [SWS_SecOC_00236]
3. If CSM_MacVerify() returns E_BUSY/QUEUE_FULL → increment BuildCounter. [SWS_SecOC_00237]
4. If BuildCounter < SecOCAuthenticationBuildAttempts → retry in next MainFunctionRx. [SWS_SecOC_00238]
5. If MAC verify was executed but FAILED → increment VerifyAttemptCounter, reset BuildCounter to 0. [SWS_SecOC_00239]
Note: Reset of BuildCounter prevents premature drop even when verify attempts remain.
6a. BuildCounter reaches limit → DROP PDU. VerificationResult = SECOC_AUTHENTICATIONBUILDFAILURE. [SWS_SecOC_00240]
6b. VerifyAttemptCounter reaches SecOCAuthenticationVerifyAttempts OR CSM returns E_NOT_OK/KEY_FAILURE → DROP. VerificationResult = SECOC_VERIFICATIONFAILURE. [SWS_SecOC_00241]
6c. GetRxFreshness() returns E_NOT_OK → DROP. VerificationResult = SECOC_FRESHNESSFAILURE. Report SECOC_E_FRESHNESS_FAILURE to DET. [SWS_SecOC_00256, 00248]
7. MAC verify success → VerificationResult = SECOC_VERIFICATIONSUCCESS. Pass to PduR. [SWS_SecOC_00242, 00050]

Verification Status Override [SWS_SecOC_00122]

SecOC_VerifyStatusOverride() allows overriding normal verification behavior — e.g., during EOL testing or ECU startup phase before FVM is synchronized.

Override Status Behavior
SECOC_OVERRIDE_DROP_UNTIL_NOTICE (0x00) No MAC verification. PDU dropped. VerResult = NO_VERIFICATION. Forever until CANCEL.
SECOC_OVERRIDE_DROP_UNTIL_LIMIT (0x01) No MAC verification. PDU dropped. For N messages (numberOfMessagesToOverride).
SECOC_OVERRIDE_CANCEL (0x02) Cancel any active override.
SECOC_OVERRIDE_PASS_UNTIL_NOTICE (0x40) MAC verified but PDU passed regardless of result. VerResult = VERIFICATIONFAILURE_OVERWRITTEN on fail. Requires SecOCEnableForcedPassOverride=TRUE.
SECOC_OVERRIDE_SKIP_UNTIL_LIMIT (0x41) No MAC verification. PDU passed. VerResult = NO_VERIFICATION. For N messages.
SECOC_OVERRIDE_PASS_UNTIL_LIMIT (0x42) MAC verified. PDU passed regardless. For N messages.
SECOC_OVERRIDE_SKIP_UNTIL_NOTICE (0x43) No MAC verification. PDU passed. Forever until CANCEL.

Security Profiles [Section 7.10]

Profile Algorithm FV in PDU MAC in PDU Use case
Profile 1
24Bit-CMAC-8Bit-FV
[SWS_SecOC_00192]		 AES-128 CMAC 8 bits (LSB of FV) 24 bits (MSB of MAC) Standard CAN: compact MAC + small FV
Profile 2
24Bit-CMAC-No-FV
[SWS_SecOC_00193]		 AES-128 CMAC 0 bits (none) 24 bits (MSB of MAC) No synchronized FV. Minimal overhead. Lower security (no replay protection).
Profile 3
JASPAR
[SWS_SecOC_00194]		 AES-128 CMAC 4 bits (LSB of FV) 28 bits (MSB of MAC) CAN only. JasPar counter-based FV with master-slave sync. FV=64bits complete.
Profile selection rule: Profile 2 only if no synchronized FV is established. Profile 3 for JASPAR-compatible ECUs. Profile 1 is the most balanced standard choice.
Asymmetric approach (profiles 1-3 don't apply): For signatures (RSA, ECDSA): SecOCAuthInfoTruncLength = full signature length. Cannot truncate. Different verify API (Csm_SignatureVerify). [Section 7.1.4]

Multicore Distribution [Section 7.7, SWS_SecOC_00276]

SecOC supports multiple partitions. Each partition can have its own MainFunctionRx/MainFunctionTx instance. Each MainFunction has its own period setting. Inter-partition PduR communication managed by PduR.

Naming: SecOC_MainFunctionRx_<shortName>() and SecOC_MainFunctionTx_<shortName>() — shortName is the container name from ECU configuration.

Sequence Diagrams [Section 9]

Sequence 1: Direct IF Transmission (CAN typical path)
COM → PduR_ComTransmit() → SecOC_IfTransmit(TxPduId, PduInfoPtr)
SecOC: copy Authentic I-PDU to internal buffer. Return E_OK to PduR.
Next MainFunctionTx call:
SecOC: GetTxFreshness() → get complete FV from FVM
SecOC: build DataToAuthenticator = DataId | AuthenticIPDU | CompleteFV
SecOC: Csm_MacGenerate(DataToAuth) → full MAC
SecOC: truncate MAC (MSB), truncate FV (LSB)
SecOC: build Secured I-PDU = [Header] | AuthenticIPDU | TruncFV | TruncMAC
SecOC → PduR_SecOCTransmit(SecuredIPDU)
SecOC → FVM via SPduTxConfirmation(FreshnessValueID)
PduR → CanIf → CAN HW (frame transmitted)
CanIf → PduR → SecOC_TxConfirmation(TxPduId, E_OK)
SecOC → PduR_SecOCIfTxConfirmation(TxPduId, E_OK) [Authentic I-PDU confirmed to COM]
SecOC: free Secured I-PDU buffer
Sequence 2: Triggered TX (FlexRay typical path)
COM → SecOC_IfTransmit() → copy Authentic I-PDU, return
MainFunctionTx: build Secured I-PDU → PduR_SecOCTransmit()
FrIf scheduled transmission → PduR_FrIfTriggerTransmit() → SecOC_TriggerTransmit()
SecOC copies complete Secured I-PDU to FrIf buffer. Returns E_OK.
FrIf transmits frame. Confirms via SecOC_TxConfirmation()
If SecOCReAuthenticateAfterTriggerTransmit=TRUE: Preserve Authentic I-PDU → re-auth in next MainFunctionTx
If FALSE: free Authentication Information buffer only
Sequence 3: Direct IF Reception (CAN typical path)
CAN HW → CanIf → PduR → SecOC_RxIndication(RxPduId, PduInfoPtr)
SecOC: copy Secured I-PDU to internal buffer (up to min of received/configured length)
If received length < configured (static PDU): silently discard
Next MainFunctionRx call:
Parse: extract AuthenticIPDU, TruncFV, TruncMAC
Call GetRxFreshness(FVID, TruncFV, TruncLen, VerifyAttempts) → complete FV candidate
Build DataToAuthenticator = DataId | AuthenticIPDU | CompleteFV
Csm_MacVerify(DataToAuth, ReceivedMAC, TruncLength) → result
If SUCCESS: VerResult=VERIFICATIONSUCCESS. Notify FVM via VerificationStatusCallout.
→ PduR_SecOCIfRxIndication(Authentic I-PDU). Free buffers.
If FAILED: Increment VerifyAttemptCounter. If limit reached → drop + VerResult=VERIFICATIONFAILURE.
Notify FVM. Do NOT call PduR. Free buffers.
Sequence 4: Freshness Handling — FVM retry on E_BUSY
MainFunctionRx: GetRxFreshness() returns E_BUSY (FVM not ready yet)
BuildCounter++. No MAC verification this cycle.
Next MainFunctionRx call (PDU still buffered):
GetRxFreshness() returns E_OK → complete FV obtained
Proceed with MAC verification: Csm_MacVerify() → E_OK (MAC matches)
VerResult = VERIFICATIONSUCCESS. Pass to PduR.
If BuildCounter reaches SecOCAuthenticationBuildAttempts:
DROP PDU. VerResult = AUTHENTICATIONBUILDFAILURE. Free buffers.
Sequence 5: PDU Collection (split Authentic + Cryptographic PDU)
TX side:
SecOC_IfTransmit() → copy Authentic I-PDU
MainFunctionTx: build MAC, create Cryptographic I-PDU = Auth Info | Message Linker
Send Authentic I-PDU via PduR. Send Cryptographic I-PDU via PduR. Both in same main function cycle. [SWS_SecOC_00202]
Wait for TxConfirmation for BOTH PDUs. Forward E_OK only if both OK. [SWS_SecOC_00270]
RX side:
Receive Authentic I-PDU → buffer. Order doesn't matter.
Receive Cryptographic I-PDU → buffer.
Check Message Linker values match. [SWS_SecOC_00211]
When BOTH received with matching linker → reconstruct Secured I-PDU → verify in MainFunctionRx. [SWS_SecOC_00203]

Error Handling & Debugging Guide

Problem: MAC verification fails repeatedly (SECOC_VERIFICATIONFAILURE)
Root cause: MAC mismatch — either key mismatch, DataId mismatch, payload mismatch, or wrong FV.
1. Check DataId matches between TX and RX config: SecOCDataId in SecOCTxPduProcessing vs SecOCRxPduProcessing. Even 1-bit difference = total MAC failure.
2. Check CSM key is the SAME on both sides. Key provisioning error is common — verify key ID references in CSM config.
3. Check SecOCAuthInfoTruncLength identical TX and RX. TX truncates to N bits, RX must verify same N bits.
4. Check byte order: DataId must be Big Endian. If implementation sends Little Endian DataId → MAC always wrong.
5. Check if Secured Area (offset/length) configured consistently on both sides. Partial PDU security: must cover same bytes.
6. Check FV complete value used for MAC — not truncated FV. Common bug: implementer passes truncated FV to MAC instead of complete FV.
Problem: SECOC_FRESHNESSFAILURE — freshness not available
FVM returned E_NOT_OK (not E_BUSY) → permanent failure for this PDU.
1. FVM not initialized. Check FVM init order: FVM must init before SecOC starts receiving PDUs.
2. FreshnessValueID not matching — SecOCFreshnessValueId in SecOC config must match FVM's configured ID for same PDU.
3. FVM desynchronized: RX side counter too far behind TX counter. Need sync message from FVM master.
4. RX: check SecOCFreshnessValueTruncLength matches TX. If TX sends 8-bit truncated FV but RX expects 4-bit → FVM gets wrong input.
5. SECOC_E_FRESHNESS_FAILURE reported to DET — trace the FVID in DET to identify which PDU is failing.
Problem: SECOC_AUTHENTICATIONBUILDFAILURE — TX auth never completes
CSM or FVM E_BUSY exceeded SecOCAuthenticationBuildAttempts times.
1. CSM queue full — too many concurrent CSM jobs. Increase CSM queue or reduce SecOC PDU rate.
2. MainFunctionTx period too long — if SecOCAuthenticationBuildAttempts = 3 and MainFunctionTx period = 20ms → 60ms before failure. Check if this is acceptable.
3. FVM E_BUSY during startup synchronization — use SecOC_SendDefaultAuthenticationInformation() to send 0xA5 pattern during startup phase until FVM syncs.
4. CSM key not valid — Csm_MacGenerate returns KEY_FAILURE which counts as non-recoverable. Check key provisioning.
Problem: SecOC_TriggerTransmit returns E_NOT_OK
No Secured I-PDU available when FrIf/CanIf calls TriggerTransmit. [SWS_SecOC_00150]
Check: was SecOC_IfTransmit called BEFORE the first TriggerTransmit? If not, SecOC has no data to provide.
Check: was Secured I-PDU consumed by previous TriggerTransmit and MainFunctionTx has not yet created new one?
Check: is CSM authentication taking too long? If MainFunctionTx runs and CSM is still E_BUSY, SecOC cannot provide Secured I-PDU this cycle.
Problem: PDU collection — verification never triggers
SecOC waits for both Authentic I-PDU and Cryptographic I-PDU with matching Message Linker. [SWS_SecOC_00211]
Check Message Linker config: SecOCMessageLinkPos and SecOCMessageLinkLen must extract same bits from Authentic I-PDU as what TX embeds.
If Message Linker = 0 bits (SecOCUseMessageLink not configured): any Authentic + Crypto pair accepted as matching. [SWS_SecOC_00211 Note]
Timing: Authentic and Crypto I-PDU must arrive in succession. If gap too large → FVM may reject or buffer overflow.

Common Integration Mistakes

1. DataId mismatch TX vs RX — Most common. Even if 0x0001 vs 0x0100 (byte swap). MAC always fails. Double-check both sides in DaVinci config.
2. Full FV instead of truncated FV sent to FVM's GetRxFreshness — FVM receives wrong input → wrong complete FV → MAC mismatch. SecOC must pass only the truncated bits extracted from Secured I-PDU, not the full field.
3. CSM key not provisioned at SecOC_Init time — Csm_MacGenerate returns KEY_FAILURE → SECOC_E_CRYPTO_FAILURE to DET. Check key provisioning order in EcuM startup.
4. FVM not synced at startup → all RX PDUs dropped — Use VerifyStatusOverride(SKIP_UNTIL_NOTICE) during startup, cancel when FVM confirms sync.
5. SecOCAuthenticationBuildAttempts too low — If CSM is async and queue is shared, E_BUSY is expected for 2-3 cycles. Set BuildAttempts ≥ 5 to tolerate CSM latency.
6. MainFunctionTx/Rx period mismatch with OS task — SecOC processes PDUs at configured period. If OS task slower → effective period longer → freshness counter advances too fast relative to processing time.
