AUTOSAR CAN Transceiver Driver (CanTrcv) complete study guide — R24-11

What is CanTrcv?

The CAN Transceiver Driver (CanTrcv) is the AUTOSAR MCAL module that directly controls the physical CAN transceiver hardware chip on the ECU board. It adapts the digital signal levels from the microcontroller to the differential CAN bus (CANH/CANL) and back. It is the lowest layer in the CAN software stack — immediately above the DIO/SPI hardware abstraction. One CanTrcv module instance serves all transceivers of the same hardware type.

Key responsibilities

Mode control

Switch transceiver between NORMAL (full comms), STANDBY (wakeup capable, no comms), and SLEEP (low-power, wakeup capable) via DIO/SPI.

Wakeup detection

Detect wakeup events (bus, pin, POR, SYSERR) in MainFunction polling loop or during Init. Report to EcuM via EcuM_SetWakeupEvent().

Selective wakeup (PN)

When CanTrcvHwPnSupport=TRUE: configure WUF filter (CanId + mask + data mask). Clear WUF flag, read timeout/silence flags for PN shutdown sequence.

Two hardware access modes (exam critical)

DIO-controlled transceivers (e.g. TJA1050, TJA1042)

Simple transceivers with STB (standby), EN/INH pins. CanTrcv calls Dio_WriteChannel() to set pin levels for mode switching. NORMAL = STB low + EN high. STANDBY = STB high. SLEEP = depends on hardware. DIO is synchronous — instant feedback.

SPI-controlled transceivers / SBCs (e.g. TJA1145, UJA116x)

Advanced transceivers with register interface over SPI. CanTrcv calls Spi_SyncTransmit() (blocking) or Spi_AsyncTransmit() (callback-based). SPI has higher latency and may fail — CANTRCV_E_NO_TRCV_CONTROL if communication fails. Supports detailed diagnostics (BUSERR, WUF, timeout, silence flags).

Instance concept — important for multi-transceiver ECUs

One CanTrcv module instance serves ALL transceivers of the same hardware type. Each transceiver is a channel identified by CanTrcvChannelId (uint8). For different hardware types (e.g. TJA1050 + TJA1145 on same ECU), two separate CanTrcv module instances are used. Each has its own CanTrcvIndex. All APIs take a Transceiver (uint8) parameter to address the correct channel.

Physical wakeup scenarios (know all 3 — exam favourite)

Scenario 1 — Cold Start (NOT wakeup)

MCU off, transceiver in SLEEP, bus activity powers up MCU. AUTOSAR calls this a cold start, not a wakeup. No CanTrcv involvement until Init.

Scenario 2 — MCU wakeup from low power

MCU in low-power, transceiver in STANDBY. Bus activity causes hardware interrupt → MCU wakes up → EcuM chain → CanTrcv_CheckWakeup → EcuM_SetWakeupEvent.

Scenario 3 — CAN channel wakeup (MCU running)

MCU running, transceiver in STANDBY. Bus activity detected → polled in CanTrcv_MainFunction or interrupt → EcuM_SetWakeupEvent. MCU stays on.

State descriptions (know these cold)

NORMAL ⭐ — Full communication

CAN controller can transmit and receive. Transceiver drives CANH/CANL differentially. If transceiver controls ECU power: ECU fully powered. CanTrcv does NOT detect additional wakeup info in this state — WUF flag cleared/disabled. ICU notification disabled in NORMAL.

STANDBY — No comms, wakeup capable

No CAN communication possible. ECU still powered if transceiver controls power. Bus activity (WUP pattern) can trigger wakeup. ICU notification enabled. This is the gateway to SLEEP — you MUST pass through STANDBY to reach SLEEP.

SLEEP — Low power, wakeup capable

ECU may be unpowered depending on hardware. Bus activity can trigger wakeup. Selective wakeup (PN) only works if transceiver supports it. Only reachable FROM STANDBY. To exit: go to STANDBY first.

NOT_ACTIVE — Pre-init state

All channels in NOT_ACTIVE between power-on and CanTrcv_Init(). Hardware state depends on SPAL (DIO/Port) driver configuration. After DeInit() → all channels return to NOT_ACTIVE. CanTrcv_DeInit requires all transceivers to be in STANDBY or SLEEP.

Runtime Errors (DET_ReportRuntimeError)

0x26CANTRCV_E_NO_TRCV_CONTROLNo or incorrect communication to transceiver. SPI: Spi_SyncTransmit returned error, response data invalid/NACK, or SPI timeout exceeded. DIO: Port pins in wrong configuration. Check SPI wiring, SPI sequence config, transceiver power supply, CanTrcvSPICommRetries/Timeout settings.

Extended Production Error (DEM)

CANTRCV_E_BUS_ERROR: Physical bus failure detected by transceiver hardware.
FAILED: when BUSERR flag is set in transceiver registers → Dem_SetEventStatus(CANTRCV_E_BUS_ERROR, DEM_EVENT_STATUS_FAILED)
PASSED: when BUSERR flag cleared → DEM_EVENT_STATUS_PASSED
Only exists if CanTrcvBusErrFlag=TRUE and hardware supports it.
Checked cyclically by CanTrcv_MainFunctionDiagnostics().

Debug: BUSERR indicates physical bus issues (short, dominant stuck, ground offset). Check CANH/CANL wiring, termination resistors (120Ω each end), ECU ground connections.

Debugging checklist — integration and interface testing

1. SetOpMode(NORMAL) returns E_NOT_OK
— Is CanTrcv_Init() called? (CANTRCV_E_UNINIT) · Is TransceiverId valid? · Is SPI communication working? (CANTRCV_E_NO_TRCV_CONTROL) · Is hardware powered?

2. SetOpMode(SLEEP) raises CANTRCV_E_TRCV_NOT_STANDBY
— Transceiver is in NORMAL, not STANDBY. Must call SetOpMode(STANDBY) first, wait for CanIf_TrcvModeIndication(STANDBY), then call SetOpMode(SLEEP).

3. Wakeup not detected after STANDBY
— Is CanTrcvWakeUpSupport=CANTRCV_WAKEUP_BY_POLLING? Is CanTrcv_MainFunction scheduled at correct period? Is CanTrcvWakeupByBusUsed=TRUE? Is CanTrcvWakeupSourceRef correctly linked to EcuM wakeup source?

4. PN selective wakeup not working
— CanTrcvHwPnSupport=TRUE? CanTrcvPartialNetwork container configured? CanTrcvPnFrameCanId + Mask + DataMask match the expected WUF frame? Was CanTrcv_SetPNActivationState(PN_ENABLED) called? Was WUF flag cleared (ClearTrcvWufFlag) before entering STANDBY?

5. SPI communication failures (CANTRCV_E_NO_TRCV_CONTROL in Init)
— Spi_Init() called before CanTrcv_Init()? SPI sequence configured correctly (CanTrcvSpiSequence → SpiSequence ref)? SPI clock speed within transceiver limits? CanTrcvSPICommTimeout set? Check SPI CS/CLK/MOSI/MISO wiring.

6. CanIf_TrcvModeIndication not received after mode change
— CanTrcv must call CanIf_TrcvModeIndication even for same-mode requests. If using SPI async path: callback fires after Spi_AsyncTransmit completes (in ISR context). Ensure ISR priorities configured correctly.

7. DIO-controlled transceiver wrong state
— Check DIO pin mapping in CanTrcvDioChannelAccess. Check pin logic level (some transceivers use active-low STB). Verify Port driver initializes pins to expected default state. Use DIO_WriteChannel debugging to verify pin outputs.

DaVinci Configurator — CanTrcv setup workflow

Navigate to BSW → CanTrcv (vendor-specific instance, e.g. CanTrcv_TJA1145 or CanTrcv_Tja1050). Container hierarchy: CanTrcv → CanTrcvGeneral + CanTrcvConfigSet → CanTrcvChannel(s).

CanTrcvGeneral — global pre-compile settings

CanTrcvDevErrorDetectTRUE in development, FALSE in production. Enables DET error codes.
CanTrcvIndexModule instance ID. 0 if only one CanTrcv instance. Increment for multiple transceiver hardware types.
CanTrcvWakeUpSupportCANTRCV_WAKEUP_BY_POLLING: wakeup via periodic CanTrcv_MainFunction polling. CANTRCV_WAKEUP_NOT_SUPPORTED: no wakeup capability. Determines whether MainFunction must be scheduled.
CanTrcvMainFunctionPeriodPeriod (seconds) for CanTrcv_MainFunction call. Must match OS task period. Only used when WakeUpSupport=POLLING.
CanTrcvMainFunctionDiagnosticsPeriodPeriod (seconds) for CanTrcv_MainFunctionDiagnostics. Only if CanTrcvBusErrFlag=TRUE. Typical: 0.01–0.1s.
CanTrcvVersionInfoApiTRUE to enable CanTrcv_GetVersionInfo() API.
CanTrcvWaitTimeWait time (seconds, max 0.000255s = 255µs) for transceiver state changes. Uses Tm_BusyWait1us16bit. Set based on transceiver datasheet timing requirements.
CanTrcvTimerTypeNone or Timer_1us16bit. Set Timer_1us16bit if CanTrcvWaitTime > 0.

CanTrcvConfigSet — SPI global parameters

CanTrcvSPICommRetries0–255. Max SPI retries on failure/NACK. 0 = no retry. Only for SPI-accessed transceivers. Common: 3.
CanTrcvSPICommTimeout0–100ms. SPI communication timeout. 0 = no specific timeout (HW capacity). Set based on transceiver response time.

CanTrcvChannel — one per physical transceiver chip

CanTrcvChannelIdUnique uint8 ID (0..255). withAuto=true → DaVinci auto-assigns. This ID is what all API Transceiver parameter values correspond to. Must match CanIf's CanSMTransceiverId reference.
CanTrcvChannelUsedTRUE = this channel is active. FALSE = skipped. Allows disabling unused channels.
CanTrcvControlsPowerSupplyTRUE if this transceiver chip controls ECU power supply (SBC with voltage regulator). FALSE for standalone transceivers. Affects interpretation of SLEEP state power behavior.
CanTrcvHwPnSupportTRUE = hardware supports selective wakeup (PN). Enables CanTrcvPartialNetwork container, and PN-specific APIs (GetTrcvSystemData, ClearTrcvWufFlag, ReadTrcvTimeoutFlag, etc.).
CanTrcvInitStateCANTRCV_OP_MODE_STANDBY or CANTRCV_OP_MODE_SLEEP. Determines which state CanTrcv_Init puts this channel into. Choose STANDBY for most applications. SLEEP if hardware starts in sleep mode.
CanTrcvMaxBaudrateMaximum CAN baud rate supported by this transceiver hardware (kbit/s). Used for validation.
CanTrcvWakeupByBusUsedTRUE = CAN bus wakeup detection enabled for this channel. FALSE = wakeup by bus not used. Controls whether CanTrcvWakeupSourceRef is evaluated.
CanTrcvWakeupSourceRefReference to EcuMWakeupSource for standard bus wakeup. Used in EcuM_SetWakeupEvent() call. Bitmask bit position.
CanTrcvPorWakeupSourceRefReference to EcuMWakeupSource for power-on-reset wakeup. Separate source from bus wakeup. PN only.
CanTrcvSyserrWakeupSourceRefReference to EcuMWakeupSource for system error wakeup. PN only. Reported when SYSERR flag is set in transceiver.
CanTrcvIcuChannelRefReference to ICU channel for edge detection on wakeup pin. When configured: CanTrcv calls Icu_EnableNotification on STANDBY transition, Icu_DisableNotification on NORMAL transition.

CanTrcvAccess — hardware interface selection

Choice container: select one of CanTrcvDioAccess (GPIO-controlled) or CanTrcvSpiAccess (SPI-controlled).

CanTrcvDioAccessFor TJA1050, TJA1042 etc. Contains CanTrcvDioChannelAccess sub-containers, each with CanTrcvDioChannelId → DioChannelRef and CanTrcvDioChannelLevel (required logic level per state).
CanTrcvSpiAccessFor TJA1145, UJA116x etc. Contains CanTrcvSpiSequence sub-containers with SpiSequenceRef → SpiSequence reference from SPI driver config. Multiple sequences for different operations.

CanTrcvPartialNetwork — selective wakeup (only if CanTrcvHwPnSupport=TRUE)

CanTrcvPnEnabledTRUE = PN functionality enabled in this transceiver at init time. CanSM checks this to determine DeinitPnSupported vs DeinitPnNotSupported path.
CanTrcvBusErrFlagTRUE if transceiver can detect and report bus errors. Enables CanTrcv_MainFunctionDiagnostics and CANTRCV_E_BUS_ERROR DEM event.
CanTrcvPnFrameCanIdCAN ID of the selective wakeup frame (WUF). Transceiver wakes only on this ID (when PN active).
CanTrcvPnFrameCanIdMaskMask for WUF CAN ID. Allows waking on a range of IDs. 1=care bit, 0=don't care (check vendor docs — mask polarity varies!).
CanTrcvPnFrameDlcExpected DLC of the WUF frame. Transceiver checks DLC matches before reporting wakeup.
CanTrcvPnFrameDataMaskSpecSub-container list: byte-level data mask for WUF payload. One entry per byte. CanTrcvPnFrameDataMask (byte value) + CanTrcvPnFrameDataMaskIndex (byte position). All masked bytes must match for wakeup.

Step-by-step DaVinci configuration workflow

Step 1: CanTrcvGeneral — CanTrcvDevErrorDetect=TRUE (dev), WakeUpSupport, MainFunctionPeriod, WaitTime (from transceiver datasheet — e.g. TJA1050 needs ~7µs for NORMAL→STANDBY).
Step 2: CanTrcvConfigSet — If SPI: set SPICommRetries and SPICommTimeout.
Step 3: CanTrcvChannel — One per chip. Set CanTrcvChannelId (auto), CanTrcvInitState, CanTrcvMaxBaudrate, CanTrcvWakeupByBusUsed=TRUE, CanTrcvWakeupSourceRef → link to EcuMWakeupSource.
Step 4: CanTrcvAccess — Choose DioAccess or SpiAccess. For DIO: add CanTrcvDioChannelAccess entries for STB, EN, ERR pins. For SPI: add CanTrcvSpiSequence entries linking to SPI sequences.
Step 5: CanTrcvHwPnSupport — If PN hardware: set TRUE, configure CanTrcvPartialNetwork (PnFrameCanId, Mask, Dlc, DataMaskSpec). Set CanTrcvPorWakeupSourceRef and CanTrcvSyserrWakeupSourceRef.
Step 6: OS tasks — Add CanTrcv_MainFunction (if POLLING) and CanTrcv_MainFunctionDiagnostics (if BusErrFlag) to OS tasks with correct periods.
Step 7: Cross-reference validation — CanTrcvChannelId ↔ CanIf transceiver mapping ↔ CanSM CanSMTransceiverId. EcuMWakeupSource refs must be consistent across CanTrcv+CanIf+EcuM.

Common DaVinci mistakes

• CanTrcvWaitTime=0 for a transceiver that needs timing → mode switch completes before hardware is ready → unreliable behavior
• DIO pin logic level inverted (active-low STB on TJA1050 vs active-high on some SBCs) → transceiver always in wrong state
• CanTrcvWakeupSourceRef not linked → EcuM_SetWakeupEvent called with 0 → wakeup not detected
• CanTrcvSpiSequence references wrong SPI sequence → CANTRCV_E_NO_TRCV_CONTROL on every call
• CanTrcvHwPnSupport=TRUE but CanTrcvPartialNetwork not configured → compile-time error or undefined behavior
• CanTrcvWakeUpSupport=CANTRCV_WAKEUP_BY_POLLING but MainFunction not scheduled → wakeup events never detected
• CanTrcvInitState=SLEEP but hardware starts in reset state requiring NORMAL first → Init sequence fails
• Multiple CanTrcvChannel entries with same CanTrcvChannelId → parameter validation error in DaVinci
