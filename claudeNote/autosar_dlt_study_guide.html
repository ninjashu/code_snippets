DLT = the observability backbone of AUTOSAR CP. It collects log/trace data from SW-Cs, RTE (VFB traces), DEM events, and DET errors, then transmits them via communication buses to an external logging tool (e.g. DLT Viewer). Sits between RTE and PduR.

What DLT does

  • Collects log messages from SW-Cs, BSW (via Dlt_SendLogMessage)
  • Collects trace messages (VFB events via RTE hook functions)
  • Forwards DET errors as log messages (Dlt_DetForwardErrorTrace)
  • Forwards DEM events as log messages (via CDD/SW-C)
  • Filters messages by log level threshold per AppID/ContextID tuple
  • Routes messages to one or more LogChannels (physical buses)
  • Buffers messages in LogChannel buffers, sends via PduR
  • Receives Dlt commands from external tool (SetLogLevel, etc.)
  • Stores filter config persistently via NvM (optional)
  • Supports traffic shaping — bandwidth limit per LogChannel
  • Sends BufferOverflowNotification when buffers full

Architecture position

External DLT tool (PC)
    ↕ Ethernet/CAN/etc.
PduR ← DLT → PduR
    ↑           ↑
RTE (VFB traces)   BSW
    ↑           ↑
SW-Cs     DET / DEM

DLT sits above PduR (uses PduR_DltTransmit to send) and below RTE (RTE calls hook functions). It is NOT MCAL — it is a BSW module in the Services layer.

What DLT does NOT do

  • Does NOT store logs on ECU permanently (only buffers temporarily)
  • No standardized DEM↔DLT direct interface
  • VFB trace: only non-verbose mode supported
  • Does NOT support arbitrarily nested complex data types in verbose mode
  • Does NOT generate ASAM Fibex descriptions
  • MCU clock setup = MCU driver's job (like I2C)

Two message modes

Verbose mode: Arguments with type info included in payload. Human-readable but larger messages. Requires FIBEX/ARXML description for decoding.

Non-verbose mode: Only a Message ID (4-byte uint32) in payload, no type info. Smaller messages. Decoder needs FIBEX file mapping ID→description. VFB trace always uses non-verbose.

Important note on spec status

Many DLT SWS items are marked DRAFT. This is normal for R24-11 — DLT was heavily revised. Know which items are DRAFT for exam questions about spec maturity. The overall module is published and used in production automotive projects.

Master these 6 concepts and DLT becomes intuitive: ApplicationID, ContextID, SessionID, LogLevel, LogChannel, and the filtering chain.

ApplicationID (APID)

  • 4-byte ASCII string identifying an application/SW-C (e.g. "ABS0", "COMB")
  • Type: Dlt_ApplicationIDType = uint32 (4 ASCII chars packed)
  • 0x00000000 = wildcard (matches any AppID in filter rules)
  • Reserved: "DEM" (DEM events), "DET" (DET errors), "VFBT" (VFB traces)
  • Configured statically or registered dynamically via Dlt_RegisterContext

ContextID (CTID)

  • 4-byte ASCII string grouping log messages within an AppID (e.g. "STD0", "INIT")
  • Type: Dlt_ContextIDType = uint32
  • 0x00000000 = wildcard
  • Must be unique within one AppID
  • Source identified by tuple: (AppID, ContextID)
  • Each ContextID has its own LogLevel threshold + trace status
  • VFB trace: each hook function gets its own ContextID. AppID = "VFBT" fixed.

SessionID

  • Distinguishes multiple instances of same SW-C using same AppID/ContextID
  • For SW-Cs: port-defined argument value — not visible in SendLogMessage call
  • For BSW modules: module ID is used as SessionID
  • Range: starts at 0x1000 for SW-Cs. Each port gets unique, continuous SessionID.
  • BSW module ID = direct module number (e.g. DET module ID)
  • Configured as DltSwcSessionId per DltSwc container in DaVinci

LogChannel

  • A physical communication bus used to transmit DLT messages (e.g. Ethernet, CAN)
  • Multiple LogChannels can be configured — each with own buffer, threshold, bandwidth
  • One AppID/ContextID tuple can be assigned to multiple LogChannels
  • If tuple not assigned to any channel → default LogChannel used (DltDefaultLogChannelRef)
  • Each LogChannel has: name (4 chars), buffer, max message length, max retries, traffic shaping bandwidth, overflow timer
  • Identified in DaVinci as DltLogChannel container
Log levels — complete table (exam critical)
DLT_LOG_OFF0x00Turn off all logging. No messages pass.
DLT_LOG_FATAL0x01Fatal system error — ECU cannot continue
DLT_LOG_ERROR0x02Errors with impact on correct functionality
DLT_LOG_WARN0x03Incorrect behavior cannot be ensured
DLT_LOG_INFO0x04General information about internal behavior
DLT_LOG_DEBUG0x05Useful only for debugging
DLT_LOG_VERBOSE0x06Highest level — all states, everything logged

Filtering rule (exam trap): A message is DISCARDED if its log level is numerically HIGHER than the threshold. So if threshold = DLT_LOG_WARN (0x03), then INFO (0x04), DEBUG (0x05), and VERBOSE (0x06) are discarded. OFF (0x00) = highest threshold → nothing passes.

Message filtering algorithm (3-step lookup)

For each incoming message with (AppID, ContextID) tuple, DLT finds the filter threshold via:

1
Exact match: find mapping where AppID AND ContextID both match exactly → use that threshold
2
Partial match: find mapping where AppID matches AND ContextID = wildcard (0x00000000) → use that threshold
3
Default: use DltDefaultLogLevel (for log messages) or DltDefaultTraceStatus (for trace messages)

First step that matches wins. This algorithm applies identically for LogChannel selection (to find which channels to route to) and for log level filtering.

RegisterContext / UnregisterContext

  • SW-Cs call Dlt_RegisterContext(sessionId, appId, contextId, appDesc, appDescLen, contextDesc, contextDescLen) at startup
  • DLT remembers the (AppID, ContextID) ↔ port interface mapping
  • If DltGeneralRegisterContextNotification=TRUE → DLT sends GetLogInfo control message to external tool on every register/unregister
  • Unregister on SW-C stop — DLT deletes tuple from registry, stops sending LogLevel change notifications to that SW-C
  • SW-Cs can receive LogLevel/TraceStatus change notifications via LogTraceSessionControl R-Port (if DltSwcSupportLogLevelAndTraceStatusChangeNotification=TRUE)
DLT has 2 module states (UNINIT/INIT) plus a complex Tx data path with 7 inline steps + 3 deferred steps. The data path is the most exam-tested functional content.
Module states

UNINIT → INIT transition

After reset → UNINIT. After Dlt_Init() → INIT. There is no explicit SLEEP state.

Critical startup rule: DLT CAN buffer messages that arrive before Dlt_Init() is called (SWS_Dlt_00003). This is unique — other drivers drop messages in UNINIT state.

Startup delay: If DltGeneralStartUpDelayTimer configured, DLT waits after Init before processing buffered messages. All buffered data processed after timer expires.

Init order: Dlt_Init should be called AFTER NvM is initialized (if NvRAM support enabled). NvM_ReadBlock reads stored config during init.

Tx data path — 7 inline steps (called in SendLogMessage/SendTraceMessage context)
1
Generate timestamp — only if DltHeaderUseTimestamp=TRUE. Source: GPT (Gpt_GetTimeElapsed) OR StbM (StbM_GetCurrentTime). If timestamp fetch fails → set 0x00000000.
2
Filter message — 3-step AppID/ContextID lookup → find log level threshold. If message level > threshold → discard, return E_OK. For trace messages: check trace activation state.
3
Select target LogChannel(s) — same 3-step lookup for LogChannel assignment. Result: list of output LogChannels. If empty → use default LogChannel.
4
Check message length — if message + headers exceeds DltLogChannelMaxMessageLength for ALL assigned channels → discard, return DLT_E_MSG_TOO_LARGE. If fits at least one channel → continue for channels that fit.
5
Apply LogChannel threshold — per-channel filter. If log level > channel threshold → discard for that channel (not for others). Trace: if DltTraceStatus=FALSE for channel → skip that channel.
6
Copy to LogChannel buffer — copy message to all passing channels. If no space → DLT_E_NO_BUFFER, set buffer overflow flag, discard. Increase message counter. If copied to at least one → DLT_OK.
7
Apply message attributes — if attributes non-NULL and DltProtocolVersion ≥ 2 → add privacy level (WPVL/PRLV), message tags (WTGS). If protocol v1 and attributes non-NULL → DLT_E_NOT_SUPPORTED.
3 deferred steps (called from Dlt_TxFunction — decoupled from SW-C context)
A
Send via PduR — call PduR_DltTransmit. Decoupled to shorten runtime of SW-C calls, enable traffic shaping and retry logic. If DltGeneralTrafficShapingSupport=TRUE → respect DltLogChannelTrafficShapingBandwidth per channel. If FALSE → send all buffered messages at once.
B
Create Dlt header — assemble standard header (VERS, WEID, WSID, WTMS, MSBF, UEH, MCNT, LEN, optional ECU ID, Session ID, Timestamp) and extended header (VERB, MSTP, MTIN, NOAR, APID, CTID). Header bits set based on DltHeader* config params.
C
Remove from buffer — remove message after: PduR returns E_NOT_OK, OR positive Tx confirmation received, OR negative Tx confirmation + transmit counter > DltLogChannelMaxNumOfRetries.

Buffer overflow handling

  • When buffer full: set overflow flag, discard message, DLT_E_NO_BUFFER returned
  • Dlt_TxFunction sends BufferOverflowNotification cyclically (per DltLogChannelBufferOverflowTimer) while buffer still full
  • After timer: clear overflow flag (allows next overflow notification cycle)
  • BufferOverflowNotification includes counter of lost messages since last notification
  • Per-channel: each LogChannel has its own overflow flag and timer

24 functions + 8 callbacks + 1 scheduled function. DLT has the most complex API of any standard AUTOSAR module. Group them mentally: Init/DeInit, Send, Register, Control, Channel management.
Group 1: Lifecycle

Dlt_Init(config) — Service ID: 0x01

Sync, Non-reentrant, void return

Initializes DLT module. Loads NvM config (if DltGeneralNvRAMSupport=TRUE) or static config. Starts GPT timer for timestamps. Starts startup delay timer if configured. Must be called AFTER NvM_Init if NvRAM support enabled.

Dlt_GetVersionInfo(versioninfo) — Service ID: 0x02

Sync, Reentrant, void return | Only if DltGeneralVersionInfoApi=TRUE

Group 2: Message sending (core use cases)

Dlt_SendLogMessage(sessionId, logInfo, logData, logDataLength) — Service ID: 0x03

Sync, Reentrant, returns Std_ReturnType

Primary API for SW-Cs and BSW to send log messages. logInfo = Dlt_MessageLogInfoType (appId, contextId, logLevel, options, argCount). Returns: DLT_OK / DLT_E_MSG_TOO_LARGE / DLT_E_NO_BUFFER / DLT_E_UNKNOWN_SESSION_ID.

Dlt_SendTraceMessage(sessionId, traceInfo, traceData, traceDataLength) — Service ID: 0x04

Sync, Reentrant, returns Std_ReturnType

For VFB traces and other trace events. traceInfo = Dlt_MessageTraceInfoType (appId, contextId, traceInfo [VARIABLE/FUNCTION_IN/FUNCTION_OUT/STATE/VFB], options).

Dlt_SendLogMessageWithAttributes(…, attributes) — Service ID: 0x81

Sync, Reentrant | Same as SendLogMessage + Dlt_MessageAttributesType* parameter. If attributes=NULL → equivalent to SendLogMessage. If protocol v1 and attributes non-NULL → DLT_E_NOT_SUPPORTED.

Dlt_SendTraceMessageWithAttributes(…, attributes) — Service ID: 0x82

Sync, Reentrant | Same as SendTraceMessage + attributes. Same NULL/v1 rules apply.

Group 3: Context registration

Dlt_RegisterContext(sessionId, appId, contextId, appDesc, appDescLen, contextDesc, contextDescLen) — Service ID: 0x05

Sync, Reentrant | Call when SW module wants to use DLT for a specific context. Returns E_OK / DLT_E_CONTEXT_ALREADY_REG / DLT_E_UNKNOWN_SESSION_ID. For already-registered AppID, appDescription can be NULL and len=0.

Dlt_UnregisterContext(sessionId, appId, contextId) — Service ID: 0x16

Sync, Reentrant | Call when SW module is being stopped. Returns E_OK / DLT_E_CONTEXT_NOT_YET_REG / DLT_E_UNKNOWN_SESSION_ID.

Dlt_DetForwardErrorTrace(moduleId, instanceId, apiId, errorId) — Service ID: 0x07

Sync, Non-reentrant, void return | Called by DET fan-out. All DET parameters forwarded. Sends as DLT log message with AppID="DET", ContextID="STD", MessageID=0x00000002, LogLevel=Error. Available via Dlt_Det.h (not Dlt.h).

Scheduled function + configurable interface

Dlt_TxFunction(void) — Service ID: 0x80 [Scheduled]

Called cyclically by BSW scheduler (via RTE). Available via SchM_Dlt.h. Sends buffered messages via PduR, handles traffic shaping, checks buffer overflow flags, sends BufferOverflowNotification, retries failed transmissions (up to DltLogChannelMaxNumOfRetries). This is the deferred send engine.

Dlt_InjectCall_<SESSION>(appId, contextId, serviceId, dataLength, data) [Configurable]

Callback to inject a function call into SW-C. Configured per DltSwc (if DltGeneralInjectionSupport=TRUE). Asynchronous, Non-reentrant. Service ID and data interpretation are user-defined. Enables remote diagnostic triggers on specific SW-Cs.

4 key sequence diagrams. Know all actors: EcuM, DLT, NvM, GPT, Dlt User (SW-C/DET/RTE), SchM, PduR.

Seq 9.1 — Dlt initialization

Actors: EcuM, DLT, NvM, Gpt

1
EcuM calls Dlt_Init(config)
2
DLT calls Load configuration() — internal setup
3
[Optional - if DltGeneralTimeStampSource=UPTIME] DLT calls Gpt_StartTimer(Channel, Value) to start uptime counter
4
[Alt: DltGeneralNvRAMSupport=TRUE] DLT calls NvM_ReadBlock(BlockId, NvM_DstPtr) to restore persisted config (log levels, channel assignments)
5
[Alt: DltGeneralNvRAMSupport=FALSE] DLT loads local/static configuration
6
[Optional - if DltGeneralStartUpDelayTimer configured] DLT starts startup delay timer — buffered messages not processed until timer expires

Key insight: messages arriving BEFORE Dlt_Init are buffered (SWS_Dlt_00003). Once init + delay timer expire, all buffered messages are processed.

Seq 9.2 — Dlt message transmission on one LogChannel

Actors: Dlt User (DET/RTE/SWC), SchM, DLT, Gpt, PduR

1
[Alt: DET user] Dlt_DetForwardErrorTrace(moduleId, instanceId, apiId, errorId)
2
[Alt: RTE user] Dlt_SendTraceMessage(sessionId, traceInfo, traceData, length)
3
[Alt: SWC user] Dlt_SendLogMessage(sessionId, logInfo, logData, length)
4
DLT → Gpt_GetPredefTimerValue() — get timestamp (if enabled)
5
DLT internal: Apply message filtering → Select LogChannel → Check length → Apply LogChannel threshold → Increment message counter → Add to LogChannel buffer
6
SchM calls Dlt_TxFunction() cyclically
7
DLT: Compose Dlt header → call PduR_DltTransmit(PduId)
8
[Optional retry loop] If Tx fails: retry up to DltLogChannelMaxNumOfRetries
9
Remove message from LogChannel buffer after Tx confirm or max retries reached

Seq 9.3 — SetLogLevelFilter (runtime log level change)

Actors: SWC, DLT, PduR

1
[Optional loop: SWC registration] SWC calls Dlt_RegisterContext(sessionId, appId, contextId, descriptions) → DLT stores registered SWC
2
External tool sends SetLogLevel command → PduR delivers to DLT via Dlt_RxIndication(PduId)
3
DLT: Update LogLevelFilter for the addressed AppID/ContextID tuple
4
[Optional: if DltSwcSupportLogLevelAndTraceStatusChangeNotification=TRUE] DLT looks up registered SWC for this AppID/ContextID
5
DLT calls LogLevelChangedNotification(appId, contextId, newLevel) on SWC's R-Port (LogTraceSessionControl interface)
6
DLT sends SetLogLevel response via PduR_DltTransmit(PduId)

SW-Cs use this notification to SKIP generating log messages at the source — more efficient than generating then filtering at DLT level.

Seq 9.4 — Buffer overflow indication

Actors: SWC, Gpt, DLT, PduR

1
SWC calls Dlt_SendLogMessage → DLT processes: filter → channel → check length → check threshold
2
Buffer full → discard message, set BufferOverflowFlag, start BufferOverflowTimer
3
Dlt_TxFunction called cyclically by SchM
4
[If BufferOverflowFlag=1 AND overflow message not yet sent] DLT sends BufferOverflowNotification control message via PduR. Sets MessageSent=TRUE.
5
[If BufferOverflowFlag=1 AND overflow message already sent] Wait for DltLogChannelBufferOverflowTimer to expire → reset flag AND MessageSent=FALSE → next occurrence can send new notification
6
[If BufferOverflowFlag=0] Normal operation — no overflow notification sent

DaVinci step-by-step workflow
1
DltGeneral setup: Enable DltGeneralDevErrorDetect=TRUE (dev phase). Set DltGeneralNvRAMSupport based on need for persistent logging. Set DltGeneralRxDataPathSupport=TRUE if external DLT tool should control log levels over bus. Set DltProtocolVersion (1 or 2). Set DltMaxNumberOfChannels to number of LogChannels you plan to configure.
2
Timestamp configuration: If DltGeneralTimeStampSupport=TRUE, configure EITHER DltGeneralGptChannelRef (reference a GPT channel) OR DltGeneralStbMTimeBaseRef (reference a StbM time base) — NOT both. GPT = uptime counter, StbM = synchronized time.
3
NvRAM (if needed): Set DltGeneralNvRAMSupport=TRUE. Add DltGeneralNvRamRef pointing to your NvM block. This block will store LogLevel settings, LogChannel assignments, and thresholds persistently across ECU resets.
4
DltSwc per SW-C: Add one DltSwc container per SW-C that uses DLT. Set DltSwcSessionId (unique across ECU, ≥0x1000). Set DltSwcSupportLogLevelAndTraceStatusChangeNotification=TRUE if SW-C should be notified of level changes. Set max message lengths. Add DltSwcContext with AppID/ContextID pairs.
5
DltProtocol (header bits): Configure DltHeaderUseEcuId (include ECU ID in header), DltHeaderUseSessionID, DltHeaderUseTimestamp, DltUseVerboseMode (verbose vs non-verbose), DltUseExtHeaderInNonVerbMode.
6
DltLogChannel per physical bus: Add one DltLogChannel per bus (e.g. "ETH1", "CAN0"). Set name (4 chars), LogChannelThreshold (initial log level), TraceStatus, MaxMessageLength, MaxRetries, traffic shaping bandwidth, overflow timer. Add DltTxPdu reference for PduR routing.
7
Log level and channel assignments: Configure DltLogLevelThreshold for specific AppID/ContextID tuples. Configure DltLogChannelAssignment to route tuples to specific channels. Configure DltDefaultLogChannelRef (mandatory fallback).
8
DltRxPdu (if Rx enabled): Add DltRxPdu containers with PduR references for receiving control messages from external DLT tool.
9
DET integration (optional): In DET module config, enable DLT as DET fan-out target. DET calls Dlt_DetForwardErrorTrace via fan-out mechanism.
10
Generate code: DaVinci generates Dlt_Cfg.h, Dlt_Cfg.c, Dlt_PBcfg.c. SW-Cs call DLT via RTE port DltSwcMessageService_{SW-C}.
DLT integration testing focuses on 4 areas: message delivery, log level filtering, buffer management, and command reception from external DLT tool.

Integration testing — startup sequence

  • Verify Dlt_Init called after NvM_Init (if NvRAM support enabled)
  • Verify messages sent before Dlt_Init are buffered and then delivered after init + startup delay
  • Confirm no messages lost if SW-C starts logging before DLT is up
  • Check NvM_ReadBlock called during init when DltGeneralNvRAMSupport=TRUE
  • Verify startup delay timer fires and triggers buffered message processing
  • Confirm all sequence results initialized to correct state after init

Message delivery testing (bus-level with DLT Viewer)

  • Connect DLT Viewer (or similar tool) to configured LogChannel (Ethernet/CAN)
  • Verify ECU ID in DLT header matches DltEcuIdValue config
  • Verify Session ID matches DltSwcSessionId for each SW-C
  • Verify timestamp increments correctly (GPT/StbM based)
  • Verify MSBF bit matches ECU endianness
  • Verify AppID and ContextID correctly appear per configured tuples
  • Test verbose vs non-verbose mode: verbose includes typed arguments, non-verbose has MessageID only
  • Send DLT_LOG_VERBOSE message while threshold = DLT_LOG_INFO → verify message does NOT appear on bus
  • Send DLT_LOG_ERROR message → verify appears regardless of threshold (unless DLT_LOG_OFF)

Log level filtering testing

  • Test 3-step lookup: configure exact match, wildcard match, and default → verify each fires correctly
  • Change log level via DLT Viewer SetLogLevel command → verify new level effective immediately
  • Verify SW-C receives LogLevelChangedNotification (if configured) after SetLogLevel
  • Test SetMessageFiltering(FALSE) → verify ALL messages pass regardless of level
  • Verify SetDefaultLogLevel affects only unmatched contexts
  • Test StoreConfiguration → reset ECU → verify levels persist (NvRAM test)
  • Test ResetToFactoryDefault → verify static config restored

Buffer overflow testing

  • Flood DLT with high-rate messages → fill LogChannel buffer → verify DLT_E_NO_BUFFER returned
  • Verify BufferOverflowNotification sent to external tool when buffer full
  • Verify overflowCounter in notification contains count of lost messages
  • Verify notification not sent again until DltLogChannelBufferOverflowTimer expires
  • Verify no more than one BufferOverflowNotification per timer cycle
  • Reduce message rate → verify normal operation resumes as buffer drains

Industry best practice: Set DltGeneralDevErrorDetect=TRUE in development. Use DLT_LOG_DEBUG/VERBOSE during integration testing, switch to DLT_LOG_WARN or DLT_LOG_ERROR in production builds to reduce bus load. Use StoreConfiguration to persist tuned log levels discovered during vehicle testing.
DLT is fundamentally different from PORT and I2C — it is a BSW Services module, not MCAL. It handles software observability, not hardware I/O.

Exam trap summary — all 3 modules

  • PORT only: must be initialized before DIO functions; no runtime errors; direction refresh skips changeable pins; atomic access required
  • I2C only: DeInit required before re-Init; has NACK/bus error runtime errors; async FIFO queuing; StartListening for Target mode; no queuing for SyncTransmit
  • DLT only: buffers messages before init (unique!); Dlt_TxFunction is mandatory scheduler call; 3-step AppID/ContextID filter lookup; 7 inline + 3 deferred Tx steps; PduR is mandatory interface; VFBT AppID is reserved; FilterMessageFiltering(FALSE) = let ALL messages pass
  • All three: none have production errors; none have mandatory interfaces from modules other than their own stack dependencies; GetVersionInfo conditional on config flag
