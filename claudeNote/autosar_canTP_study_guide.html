What is CanTp? — "The postal service of CAN"

CanTp (CAN Transport Layer) segments large messages into CAN frames and reassembles them on the other side — exactly like a postal service breaks a big package into smaller boxes, numbers them, and reassembles at destination. Based on ISO 15765-2. Main use: UDS diagnostics (DCM layer uses CanTp to send/receive diagnostic messages that are too large for a single CAN frame).

Mental model: DCM says "send 500 bytes of diagnostic data". CanTp chops it into 7-byte chunks, sends them in sequence with flow control handshaking, and the receiver CanTp reassembles back to 500 bytes before giving to DCM.

Core Capabilities

Segmentation (TX)Breaks N-SDU into SF/FF/CF frames. CAN 2.0: up to 7 bytes per CF. CAN FD: up to 62 bytes per CF. Sends FF first, then CFs after FC(CTS) received.
Reassembly (RX)Receives SF/FF/CF frames. Copies data to upper layer buffer via PduR_CanTpCopyRxData(). Manages block size and flow control timing.
Flow ControlReceiver controls sender speed using FC frames: FC(CTS)=continue, FC(WAIT)=pause, FC(OVFLW)=overflow. Block size (BS) limits how many CFs before next FC.
Multiple connectionsHandles simultaneous TX + RX sessions. Each N-SDU linked to a connection channel with its own state machine, timers, buffer.
Addressing formats5 modes: Normal, Extended, Mixed 11-bit, Normal Fixed, Mixed 29-bit. Determines how N_TA and N_AE are placed in frame bytes.
No internal bufferingCanTp has NO internal data buffer. Copies data directly from upper layer (PduR_CanTpCopyTxData) or to upper layer (PduR_CanTpCopyRxData). Zero-copy design.

Key Constraints (Exam: always asked)

1. Event-triggered only — no polling mode [SWS_CanTp_00027 note, section 4.1]

2. Functional addressing = SF only — multi-frame with functional addressing is error [SWS_CanTp_00092, 00093]

3. No data buffering — upper layer must keep buffer locked from request to confirmation

4. One LSduR, one PduR — CanTp has single interface each way

5. CAN only — not FlexRay, not LIN

6. CANTP_OFF after power up — all services blocked until CanTp_Init() [SWS_CanTp_00168, 00031]

Frame Data Length Rules (CAN FD)

ISO 11898-1:2015 defines non-linear DLC values for CAN FD: 0-8, 12, 16, 20, 24, 32, 48, 64 bytes only. If CanTp data doesn't match these lengths → pad to next valid DLC with CanTpPaddingByte. [SWS_CanTp_00351]

RX_DL derivation [SWS_CanTp_00350]: If CAN_DL ≤ 8 → RX_DL = 8. If CAN_DL > 8 → RX_DL = CAN_DL.

Timer Relationship Rules (ISO 15765-2 performance requirements)

N_Br + N_Ar < 0.9 × N_BsReceiver must send FC(WAIT) or FC(CTS) fast enough that sender's N_Bs doesn't expire. N_Br is the buffer wait time. N_Ar is the FC TX time. Both together must be < 90% of sender's N_Bs timer.
N_Cs + N_As < 0.9 × N_CrSender must fetch data and transmit CF fast enough that receiver's N_Cr doesn't expire. N_Cs = data copy time from PduR. N_As = CAN frame TX time. Both must be < 90% of receiver's N_Cr timer.
Config tip: Set N_Bs > (N_Br + N_Ar) / 0.9 and N_Cr > (N_Cs + N_As) / 0.9. Typical values: N_As=N_Ar=25ms, N_Bs=75ms, N_Br=20ms, N_Cs=25ms, N_Cr=150ms for diagnostic on automotive CAN.

N_Br + WFTmax (Flow Control Wait) behavior

When buffer insufficient after FF or CF block → CanTp starts N_Br timer and polls PduR with zero-length calls each MainFunction [SWS_CanTp_00222]. When N_Br expires → send FC(WAIT) [SWS_CanTp_00341]. Max consecutive FC(WAIT) = CanTpRxWftMax. If WFTmax exceeded → abort RX with E_NOT_OK [SWS_CanTp_00223].

MainFunction Role in Timer Handling

CanTp_MainFunction() handles: N_As, N_Bs, N_Cs, N_Ar, N_Br, N_Cr timers and STmin inter-CF delay. [SWS_CanTp_00164]

Timer resolution = CanTpMainFunctionPeriod. Set this to match OS task period. Too large → timing accuracy loss → may violate ISO 15765-2 performance requirements.

N_Br polling (zero-length CopyRxData) happens each MainFunction cycle while N_Br active. If MainFunction period is large → FC(WAIT) may be sent later than optimal → N_Bs may expire on sender side.

Sequence 1: Single Frame Reception (SUCCESS)

CanTp → PduR_CanTpStartOfReception(id, TpSduInfoPtr, SF_DL, &bufferSize)

CanTp → PduR_CanTpCopyRxData(id, &data[SF_DL], &remainingBuffer)
CanTp → PduR_CanTpRxIndication(id, E_OK)
Case B: BUFREQ_E_NOT_OK

Sequence 2: Multi-Frame Reception (FF + CFs)
1. FF received → CanTp_RxIndication()
Parse FF. Extract total length (FF_DL).

If BUFREQ_E_OVFL → Send FC(OVFLW), abort. [SWS_CanTp_00318]
If BUFREQ_E_NOT_OK → abort silently. [SWS_CanTp_00081]
If BUFREQ_OK, bufSize ≥ next block size:
Send FC(CTS) via LSduR. Start N_Ar. Start N_Cr. State = expecting CFs.
If BUFREQ_OK, bufSize < next block size:

2. Each CF received → CanTp_RxIndication()

3. Block complete (BS CFs received) → send next FC(CTS)

PduR → CanTp_Transmit(TxPduId, PduInfoPtr)

Sequence 4: Multi-Frame Transmission (FF + CFs)
PduR → CanTp_Transmit() → E_OK (SduLength > max SF)
Copy first 6 bytes via PduR_CopyTxData. Build FF (N_PCI: type=1, FF_DL). Start N_As.
LSduR_CanTpTransmit(FF) → CanTp_TxConfirmation(E_OK) → Stop N_As.
Start N_Bs timer (waiting for FC from receiver).
Receiver → FC(CTS) received → CanTp_RxIndication(FC)
Stop N_Bs. Read BS (block size) and STmin from FC.
For each CF in block (BS count):
Wait STmin. Start N_Cs. Call PduR_CopyTxData (7 bytes). Build CF (SN++). Start N_As.
LSduR_CanTpTransmit(CF) → TxConfirmation(E_OK) → Stop N_As.
After last CF in block → restart N_Bs. Wait next FC.
FC(CTS) → continue next block. Last byte sent → PduR_TxConfirmation(E_OK).
Error scenarios:
FC(WAIT) received → stop N_Bs, restart N_Bs. [SWS_CanTp_00315]
FC(OVFLW) → abort, PduR_TxConfirmation(E_NOT_OK). [SWS_CanTp_00309]
N_Bs timeout → abort, PduR_TxConfirmation(E_NOT_OK). [SWS_CanTp_00316]
PduR_CopyTxData → BUFREQ_E_BUSY → retry in next MainFunction (N_Cs still running). [SWS_CanTp_00184]
N_Cs timeout → abort, PduR_TxConfirmation(E_NOT_OK). [SWS_CanTp_00280]

Addressing Formats — Deep Dive

Normal (Standard) Addressing — CANTP_STANDARD

CAN ID alone identifies the N-SDU connection. No address bytes consumed in data field. Maximum payload per frame.

CAN ID → identifies N_SA + N_TA + N_TAtype
|
Byte 0: N_PCI
Bytes 1-7: Data (SF: up to 7)

No extra config needed (no CanTpNTa, CanTpNSa containers). Pure CAN ID mapping.

Extended Addressing — CANTP_EXTENDED

CAN ID identifies N_SA + N_TAtype. First data byte = N_TA (target address). One data byte lost per frame.

Byte 0: N_TA
Byte 1: N_PCI
Bytes 2-7: Data (SF: up to 6)

Config: CanTpNTa container with N_TA value. Also CanTpNSa for physical RxNSDU.

Same CAN ID can serve multiple ECUs — distinguished by N_TA byte. [SWS_CanTp_00095]

Mixed 11-bit Addressing — CANTP_MIXED

CAN ID (11-bit) identifies N_SA + N_TA + N_TAtype. First data byte = N_AE (address extension). Used for sub-addressing.

Byte 0: N_AE
Byte 1: N_PCI
Bytes 2-7: Data (SF: up to 6)

Config: CanTpNAe container with N_AE value.

Normal Fixed Addressing — CANTP_NORMALFIXED

Uses CAN 29-bit ID. N_SA and N_TA encoded directly in CAN ID bytes. No data byte consumed.

29-bit CAN ID encodes N_SA + N_TA
|
Byte 0: N_PCI
Bytes 1-7: Data (SF: up to 7)

Config: CanTpNTa + CanTpNSa containers. Used in OBD applications (ISO 15765-4). MetaData carries N_SA, N_TA at runtime for generic connections.

Mixed 29-bit Addressing — CANTP_MIXED29BIT

29-bit CAN ID encodes N_SA + N_TA. First data byte = N_AE. Combines normal fixed with address extension.

29-bit CAN ID: N_SA + N_TA
Byte 0: N_AE
Byte 1: N_PCI
Bytes 2-7: Data (SF: up to 6)

Config: CanTpNTa + CanTpNSa + CanTpNAe.

Generic Connections (MetaData)

Enabled by CanTpGenericConnectionSupport=TRUE (requires CanTpDynIdSupport=TRUE). Allows one N-SDU to handle multiple peers dynamically at runtime — addressing info comes from MetaData of N-PDU.

Debugging Checklist

Problem: N_Bs timeout — sender side (TX hangs waiting for FC)
Root cause: Receiver not sending FC within N_Bs time after FF or last CF of block.
1. Check receiver buffer situation: if PduR_StartOfReception returned small buffer → receiver sends FC(WAIT). If WFTmax exceeded → receiver aborts RX → no more FCs → N_Bs expires on sender.
2. Check N_Br + N_Ar < 0.9 × N_Bs config constraint. If N_Bs too small → legitimate FC(WAIT) processing time causes timeout.
3. Check receiver N_Ar: if FC was requested but not confirmed by CAN driver → N_Ar expires on receiver → receiver aborts → no FC to sender → N_Bs expires.
4. CANoe: capture trace. Measure time from last FF or CF to FC. Compare to configured N_Bs.
5. Check if CanTp_MainFunction is called at configured period. Jitter can cause timer granularity issues.
Problem: N_Cr timeout — receiver side (expected CF not arriving)
Root cause: Sender stopped sending CFs. Could be sender-side N_Cs or N_As timeout.
1. Check sender's PduR_CopyTxData — if BUFREQ_E_BUSY returned repeatedly → N_Cs timer counts down → expires → sender aborts → no more CFs → receiver times out with N_Cr.
2. Check sender CAN bus — bus-off or buffer full → N_As timeout → sender aborts → no CF.
3. Check STmin from FC: sender must wait STmin between CFs. If STmin very large + N_Cr small → N_Cr expires during legal STmin wait. (N_Cr must be > STmin + N_As).
4. Check SN continuity on CANoe trace — wrong SN → receiver aborts immediately (not N_Cr timeout but SN error).
Problem: Wrong SN received (dropped/swapped CFs)
CanTp checks SN on every CF. Expected = (last_SN + 1) mod 16. [SWS_CanTp_00314]
1. SN = 0 after FF, increments 1-2-3...15-0-1... If sequence breaks → abort RX.
2. Common cause: CAN frame loss on bus → CF missed → next CF has SN+2 instead of SN+1 → wrong SN. DEM: DROPPED_CONSECUTIVE_FRAMES_DETECTED. Security: SEV_CAN_ERROR_WRONG_SN.
3. Capture CAN trace. Verify SN byte (low nibble of CF byte 0) increments correctly.
4. Check bus error rate — electrical issues causing frame drops increase SN errors.
Problem: FC(OVFLW) sent or received
Receiver sends FC(OVFLW): PduR_StartOfReception returned BUFREQ_E_OVFL for FF. DEM: CANTP_E_FC_OVERFLOW_TRANSMITTED.
Sender receives FC(OVFLW): abort TX, PduR_TxConfirmation(E_NOT_OK). DEM: CANTP_E_FC_OVERFLOW_RECEIVED.
Fix: Check DCM/upper layer buffer allocation. DCM might be busy with another request. Check CanTpBs and buffer sizes. Ensure upper layer can allocate FF_DL bytes.
Problem: CanTp_Transmit() returns E_NOT_OK
1. Channel already in TX_PROCESSING — another TX on same channel not finished. [SWS_CanTp_00123]
2. Module in CANTP_OFF — CanTp_Init() not called yet. DET: CANTP_E_UNINIT.
3. Invalid TxPduId. DET: CANTP_E_INVALID_TX_ID.
4. Functional addressing + multi-frame attempt. DET: CANTP_E_INVALID_TATYPE. [SWS_CanTp_00093]

CANoe / CANalyzer Integration Tips

CANoe CAPL Scripting

  • Use CANoe's UDS feature to send multi-frame diagnostic requests — generates TP frames automatically
  • Monitor FC frames: verify BS byte = configured CanTpBs, STmin byte = configured CanTpSTmin (hex value)
  • Inject wrong SN by modifying CF byte 0 low nibble — verify ECU sends E_NOT_OK and DEM DTC set
  • Simulate N_Bs timeout: send FF, don't send FC — verify sender aborts after N_Bs
  • Check padding: verify all frames = 8 bytes when OBD/padding mode (byte 7 = CanTpPaddingByte)
  • Trigger FC(OVFLW) by making receiver buffer full — check DTC CANTP_E_FC_OVERFLOW_TRANSMITTED

TRACE32 Debug Tips

  • Watch CanTp_ChannelState[] — verify transitions from WAIT to PROCESSING and back
  • Breakpoint on PduR_CanTpRxIndication() — check result parameter (E_OK vs E_NOT_OK)
  • Watch CanTp_Timer_N_Bs[] — decrement every MainFunction call. If stuck at max → FC not received
  • Watch CanTp_ExpectedSN[] — verify expected SN on each CF
  • Monitor CanTp_WftCounter[] → if approaching WFTmax → buffer starvation issue in upper layer
  • Check CanTp_DataIndex[] to verify progress of multi-frame session byte count

Common Integration Mistakes

1. LSduR_CanTpTransmit not linked (R24-11 migration) — Previously CanIf_Transmit was called. After R24-11, must use LSduR_CanTpTransmit. Check linker errors or check header includes for LSduR_CanTp.h.
2. CanTpMainFunctionPeriod doesn't match OS task — N_Bs configured as 75ms but MainFunction runs every 10ms instead of 5ms → effective N_Bs becomes 150ms → double the timeout → sender may tolerate 15 extra WFT frames.
3. Missing CanTpRxFcNPdu on TxNSDU for multi-frame — FC frames from remote receiver can't be routed to CanTp → CanTp never receives FC → N_Bs timeout on every multi-frame TX.
4. CanTpBs=0 (unlimited) with slow upper layer — Sender floods with CFs. Receiver buffer may overflow mid-transfer. Set CanTpBs=10 and WFTmax=3 to pace flow and allow buffer management.
5. Functional address N-SDU receiving multi-frame — Protocol violation. CanTp reports CANTP_E_INVALID_TATYPE and security event SEV_CAN_INVALID_TATYPE. Check OEM diagnostic spec — functional always must be 1-frame (SF).
