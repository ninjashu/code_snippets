AUTOSAR CAN State Manager (CanSM) complete study guide — R24-11

What is CanSM?

CanSM (CAN State Manager) is the AUTOSAR BSW module that provides network-level communication mode control for CAN buses. It sits in the Communication Service Layer between ComM (above) and CanIf (below), and implements a state machine per CAN network. ComM says "I want FULL_COM" — CanSM does all the hardware coordination to make it happen through CanIf.

CanSM's 3 core responsibilities

Mode Orchestration

Sequences controller + transceiver mode changes in correct order to achieve requested ComM mode. Handles asynchronous mode indications via callbacks.

BusOff Recovery

Two-level (L1/L2) timed recovery. Counts consecutive BusOff events, escalates from fast (L1) to slow (L2) recovery wait. Reports to DEM.

PDU Mode Control

CanSM is the ONLY module that calls CanIf_SetPduMode(CANIF_ONLINE/TX_OFFLINE/TX_OFFLINE_ACTIVE). CANIF_OFFLINE is set automatically by CanIf, not by CanSM.

Key concepts (examinable)

One state machine per CAN network (CANSM_BSM): CANSM_BSM runs independently for each configured CanSMManagerNetwork. One network = one state machine instance.

All state machine effects run in CanSM_MainFunction context: Triggers from callbacks (BusOff, ModeIndication) set flags; the actual CanIf API calls happen in the MainFunction task. This avoids long operations in ISR context.

CanSM does NOT notify ComM after init unless explicitly requested: After CanSM_Init, CanSM is silent until ComM sends its first CanSM_RequestComMode.

Mode request repetition: If CanIf API calls don't produce a mode indication within CanSMModeRequestRepetitionTime, CanSM repeats the request. If repeated more than CanSMModeRequestRepetitionMax times → runtime error CANSM_E_MODE_REQUEST_TIMEOUT + attempt to go back to NO_COM.

Transceiver bypass: If no CanSMTransceiverId configured for a network, ALL transceiver-related steps are bypassed and immediately treated as "already indicated".

Click any state below for details. Transitions shown in bold are the most exam-critical paths.

Top-level state descriptions

NOCOM

Stable NO_COMMUNICATION state. Controller and transceiver are in low-power (SLEEP/STANDBY). PDU channel: CANIF_OFFLINE (automatic). ComM notified COMM_NO_COMMUNICATION.

FULLCOM ⭐

Normal operation. Controller STARTED, transceiver NORMAL. PDU channel CANIF_ONLINE (or TX_OFFLINE_ACTIVE in ECU passive mode). ComM notified COMM_FULL_COMMUNICATION.

SILENTCOM

Listen-only mode. Controller STARTED, transceiver NORMAL. PDU channel TX_OFFLINE (Rx only). ComM notified COMM_SILENT_COMMUNICATION. Typical use: CanNm Prepare-Bus-Sleep.

SILENTCOM_BOR

Bus-off in SILENT mode: controller being restarted. PDU stays TX_OFFLINE. If recovery succeeds → back to FULLCOM. Unique to SILENT state BusOff handling.

PRE_FULLCOM

Transient startup sub-machine: Trcv→NORMAL, Ctrl→STOPPED, Ctrl→STARTED. All three steps must succeed before entering FULLCOM.

PRE_NOCOM

Transient shutdown sub-machine: PN-specific or generic teardown. Ctrl→STOPPED, Ctrl→SLEEP, Trcv→NORMAL, Trcv→STANDBY. Complex when PN is active.

WUVALIDATION

Wakeup validation sub-machine: started by EcuM. Same sequence as PRE_FULLCOM (Trcv NORMAL, Ctrl STOPPED, STARTED) to put network in listen-only state for wakeup validation.

CHANGE_BAUDRATE

Runtime baud rate change. Try direct CanIf_SetBaudrate first. If that fails → STOP, change baud rate on stopped controller, START again. Returns to FULL or SILENT or NO_COM depending on what was requested.

BusOff Recovery Algorithm — S_FULLCOM sub-machine detail

CanSM is the ONLY module responsible for CAN BusOff recovery. The algorithm has two levels (L1 = fast, L2 = slow). The key parameters are configured per network in CanSMManagerNetwork.

BOR two-level algorithm — step by step

BOR timing parameters

CanSMBorTimeL1 (fast recovery)

Wait time after BusOff before re-enabling Tx. Used for the first N BusOff events (N = CanSMBorCounterL1ToL2). Typically short (e.g. 100ms). During this wait, controller is STARTED but PDU mode is TX_OFFLINE.

CanSMBorTimeL2 (slow recovery)

Wait time after BusOff when count ≥ CanSMBorCounterL1ToL2. Longer delay (e.g. 1000ms). Indicates repeated failures — bus may be permanently disrupted. OEM-defined escalation policy.

CanSMBorCounterL1ToL2

Number of consecutive BusOff events before switching from L1 to L2 recovery time. If G_BUS_OFF_PASSIVE is achieved (TX confirmation received) → reset counter to 0, back to L1.

CanSMBorTimeTxEnsured

When CanSMBorTxConfirmationPolling=FALSE: time after E_TX_ON before G_BUS_OFF_PASSIVE is evaluated. Ensures a frame was actually sent successfully. Used to confirm bus is truly healthy.

TX Confirmation Polling vs Timer (G_BUS_OFF_PASSIVE)

CanSMBorTxConfirmationPolling=FALSE: G_BUS_OFF_PASSIVE passes after CanSMBorTimeTxEnsured has elapsed since E_TX_ON. Time-based — simpler but may not guarantee actual Tx occurred.

CanSMBorTxConfirmationPolling=TRUE: G_BUS_OFF_PASSIVE passes when CanIf_GetTxConfirmationState() returns CANIF_TX_RX_NOTIFICATION for ALL controllers of the network. Ensures at least one frame was actually acknowledged — more robust but requires polling.

When G_BUS_OFF_PASSIVE passes: E_BUS_OFF_PASSIVE effect → Dem_SetEventStatus(CANSM_E_BUS_OFF, DEM_EVENT_STATUS_PASSED). BusOff counter reset. Network confirmed healthy.

ECU passive mode during BOR: If CanSM_SetEcuPassive(TRUE) was called, E_TX_ON still sets PDU mode to CANIF_TX_OFFLINE_ACTIVE (not CANIF_ONLINE). Frames are simulated, not transmitted.

TxTimeout Exception (Partial Networking)

CanNm detects a Tx timeout for a partial network and calls CanSM_TxTimeoutException(Channel). This triggers the S_TX_TIMEOUT_EXCEPTION sub-machine inside FULLCOM:
1. Stop all controllers → wait for STOPPED indication
2. Restart all controllers → wait for STARTED indication
3. Re-enable PDU mode (CANIF_ONLINE or TX_OFFLINE_ACTIVE)

This is different from BusOff recovery — it's a soft reset of the CAN controller without a full BusOff escalation. Used specifically for PN Tx timeout scenarios to unblock the network.

BusOff and DEM reporting

On BusOff event (E_BUS_OFF): Dem_SetEventStatus(CANSM_E_BUS_OFF, DEM_EVENT_STATUS_PRE_FAILED) → PRE_FAILED immediately
On recovery confirmed (E_BUS_OFF_PASSIVE): Dem_SetEventStatus(CANSM_E_BUS_OFF, DEM_EVENT_STATUS_PASSED)
On BusOff in SILENTCOM_BOR also: E_BUS_OFF effect → Dem_SetEventStatus PRE_FAILED

DEM debouncing is OEM-configured. PASSED only comes after TX confirmation. DTC is FAILED after DEM debounce configuration (typically requires N consecutive PRE_FAILED without PASSED). MIL illumination: DEM-assigned.

Extended Production Error (reported to DEM)

CANSM_E_BUS_OFF: Bus-off detection event per CAN network.
PRE_FAILED: immediately when CanSM_ControllerBusOff() triggers T_BUS_OFF / E_BUS_OFF
PASSED: after successful Tx frame confirmed (G_BUS_OFF_PASSIVE / E_BUS_OFF_PASSIVE)
FAILED: after DEM debounce (OEM-configured). MIL: DEM-assigned.

Debug: If CANSM_E_BUS_OFF keeps going FAILED without PASSED → BOR is not converging. Check CanSMBorTimeL1/L2, CanSMBorTimeTxEnsured, check if bus is physically disrupted (wiring, termination, grounding).

Debugging checklist — integration testing

1. Network never reaches FULLCOM — Is CanSM_Init called? Is ComM requesting FULL_COM? Is CanSMModeRequestRepetitionTime adequate? Is CanSM_MainFunction scheduled? Is CanIf_SetControllerMode being called and do mode indications arrive (CanSM_ControllerModeIndication callback)?

2. CANSM_E_MODE_REQUEST_TIMEOUT fires repeatedly — CanDrv not responding to mode requests. Check Can_SetControllerMode returns E_OK. Check CanDrv ISR for mode indications. Increase CanSMModeRequestRepetitionTime or check CanMainFunctionModePeriod in CanDrv.

3. BusOff not recovered — Check CanSMBorTimeL1 matches physical bus recovery time (ISO 11898: 128×11 bit times). Check CanSMBorTxConfirmationPolling setting. If TRUE: is Tx working at all after recovery? Check CanIf_GetTxConfirmationState.

4. ComM_BusSM_ModeIndication not called on mode change — Effects run only in CanSM_MainFunction context. Verify SchM calls CanSM_MainFunction at configured period (CanSMMainFunctionTimePeriod).

5. Network stuck in PRE_NOCOM or PRE_FULLCOM — Sub-machine waiting for mode indication that never comes. Add DET tracing. Check if CANSM_E_MODE_REQUEST_TIMEOUT is fired. Check CanSMModeRequestRepetitionMax is not 0.

6. CanSM_DeInit raises CANSM_E_NOT_IN_NO_COM — ComM must request COMM_NO_COMMUNICATION for all channels before DeInit. Verify EcuM shutdown sequence.

7. ECU passive mode not working — Is CanSMTxOfflineActiveSupport=TRUE? Was CanSM_SetEcuPassive(TRUE) called? PDU mode should be TX_OFFLINE_ACTIVE when in FULLCOM with ECU passive. Check CanIfTxOfflineActiveSupport=TRUE in CanIf.

DaVinci Configurator — CanSM setup workflow

Navigate BSW → CanSM. Configure: CanSMGeneral → CanSMConfiguration → CanSMManagerNetwork (one per CAN bus).

CanSMGeneral — global pre-compile settings

CanSMDevErrorDetectTRUE in development builds (enables all DET error reporting), FALSE in production. Essential for debugging integration issues.
CanSMMainFunctionTimePeriodCycle time (seconds) for CanSM_MainFunction. Must match OS task period. All state machine effects execute at this rate — too slow = slow mode transitions.
CanSMSetBaudrateApiTRUE to enable CanSM_SetBaudrate() API. Only enable if runtime baud rate switching is needed (e.g. vehicle speed-dependent CAN rate).
CanSMTxOfflineActiveSupportTRUE to enable CanSM_SetEcuPassive() API. Required for ECU passive (listen-only diagnostic) mode support. Must also enable CanIfTxOfflineActiveSupport in CanIf.
CanSMPncSupportTRUE to enable Partial Networking support (PN). Enables DeinitPnSupported sub-machine path in PRE_NOCOM. Only if ComMPncSupport is also enabled in ComM.
CanSMGetBusOffDelayFunctionOptional: function name of callout providing additional random jitter for BOR. Used to spread BOR recovery timing across multiple ECUs to avoid simultaneous bus access. Requires CanSMEnableBusOffDelay=TRUE per network.
CanSMVersionInfoApiTRUE to enable CanSM_GetVersionInfo() API.

CanSMConfiguration — global timing

CanSMModeRequestRepetitionMaxMax times to retry a mode request (SetControllerMode, SetTrcvMode etc.) before reporting CANSM_E_MODE_REQUEST_TIMEOUT. Range: 0–255. Typical: 3–10. Too low = spurious timeouts; too high = long recovery time.
CanSMModeRequestRepetitionTimeInterval (seconds) between mode request retries. Must be longer than CanMainFunctionModePeriod in CanDrv to allow mode indication to arrive. Typical: 0.01–0.1s.

CanSMManagerNetwork — one per CAN network (most important!)

CanSMComMNetworkHandleRefReference to the ComM channel this network corresponds to. This is the NetworkHandleType used in CanSM_RequestComMode(). Must match exactly — wrong reference = CANSM_E_INVALID_NETWORK_HANDLE.
CanSMTransceiverIdOptional reference to CanIfTrcvCfg. If not configured: all transceiver steps are bypassed (immediately treated as done). Configure for networks that have a physical transceiver chip.
CanSMBorTimeL1Fast BOR wait time (seconds). Applied for first CanSMBorCounterL1ToL2 BusOff events. Typical: 0.1–0.2s (ISO 11898 requires min ~1.4ms for 128×11 bits at 500kbit/s).
CanSMBorTimeL2Slow BOR wait time (seconds). Applied when BusOff count ≥ CanSMBorCounterL1ToL2. Typical: 1–5s. OEM-specified — longer delay assumes serious bus fault.
CanSMBorCounterL1ToL2Number of L1-level BusOff events before switching to L2. E.g. value=3: first 3 BusOffs use L1 timing, 4th and beyond use L2. Counter resets when G_BUS_OFF_PASSIVE is reached.
CanSMBorTimeTxEnsuredTime (seconds) after E_TX_ON before G_BUS_OFF_PASSIVE evaluates (when polling=FALSE). Ensure at least one Tx cycle completes. Typical: 0.01–0.2s. Only used when CanSMBorTxConfirmationPolling=FALSE.
CanSMBorTxConfirmationPollingFALSE: time-based G_BUS_OFF_PASSIVE (wait CanSMBorTimeTxEnsured). TRUE: poll CanIf_GetTxConfirmationState — more robust. Set TRUE for safety-critical networks.
CanSMEnableBusOffDelayTRUE to add random delay from CanSMGetBusOffDelayFunction to L1/L2 times. Prevents multiple ECUs from recovering simultaneously and creating BusOff cascade. OEM feature.

CanSMController — one per CAN controller per network

CanSMControllerIdSymbolic name reference to CanIfCtrlCfg. Links this network's controller to the CanIf controller handle. Multiple controllers per network: add multiple CanSMController sub-containers.

CanSMDemEventParameterRefs

CANSM_E_BUS_OFF (EventId ref)Reference to DEM event configuration for bus-off production error. DEM handles debouncing and DTC storage. Required if bus-off detection needs to be reported as a DTC.

Step-by-step DaVinci configuration workflow

Step 1: CanSMGeneral — Set CanSMDevErrorDetect=TRUE, CanSMMainFunctionTimePeriod (match OS task), enable optional APIs (SetBaudrateApi, TxOfflineActiveSupport, PncSupport).
Step 2: CanSMConfiguration — Set CanSMModeRequestRepetitionMax (e.g. 5) and CanSMModeRequestRepetitionTime (e.g. 0.05s).
Step 3: Add CanSMManagerNetwork — one per physical CAN bus (e.g. HS-CAN, MS-CAN, LIN is different module). Set CanSMComMNetworkHandleRef → must point to correct ComM channel.
Step 4: BOR parameters per network — Set BorTimeL1, BorTimeL2, BorCounterL1ToL2, BorTimeTxEnsured, BorTxConfirmationPolling. Consult OEM AUTOSAR spec for values.
Step 5: CanSMController per network — Add CanSMController sub-container for each physical CAN controller on this network. Set CanSMControllerId → reference to CanIfCtrlCfg symbolic name.
Step 6: CanSMTransceiverId (if applicable) — Reference CanIfTrcvCfg if a transceiver is present. Leave unset if no transceiver (standalone CAN controller).
Step 7: DEM event reference — Set CANSM_E_BUS_OFF reference if DTC reporting needed.
Step 8: OS task — Add CanSM_MainFunction to a BSW task running at CanSMMainFunctionTimePeriod. Verify SchM exclusive areas configured for CanSM.
Step 9: Validate cross-references — CanSMComMNetworkHandleRef ↔ ComM channel, CanSMControllerId ↔ CanIf controller, CanSMTransceiverId ↔ CanIf transceiver, CanSMEnableBusOffDelay ↔ callout function exists.

Common DaVinci integration mistakes

• CanSMComMNetworkHandleRef pointing to wrong ComM channel → all CanSM_RequestComMode calls return CANSM_E_INVALID_NETWORK_HANDLE
• CanSMControllerId referencing wrong CanIf controller → BusOff callback received but not matched to any network → CANSM_E_PARAM_CONTROLLER
• CanSMModeRequestRepetitionTime shorter than CanMainFunctionModePeriod → mode indication never arrives in time → repeated timeouts
• CanSMBorTimeL1 too short → BOR recovery declared successful before physical bus is stable → immediate re-BusOff loop
• Forgetting to add CanSM_MainFunction to OS task → no state machine effects ever execute → network stuck in transition state
• CanSMTxOfflineActiveSupport=TRUE but CanIfTxOfflineActiveSupport=FALSE in CanIf → CanIf_SetPduMode(TX_OFFLINE_ACTIVE) silently ignored
• CanSMPncSupport=TRUE but CanSMTransceiverId not set → PN sub-machine entered but transceiver operations bypassed → undefined behavior in DeinitPnSupported path
