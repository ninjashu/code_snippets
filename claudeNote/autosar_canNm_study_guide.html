What is CanNm? — "The sleep coordinator of the CAN bus"

CanNm = decentralized, direct NM strategy. Each node self-sufficient — watches only NM PDUs on bus. No master required. Goal: coordinate ALL nodes entering bus-sleep at same time when no one needs the bus.

Mental model: Last person leaving office turns off lights. Each ECU sends periodic heartbeats ("I'm awake"). When ECU wants to sleep, stops sending. If no heartbeats received for NM-Timeout → everyone sleeps. Any heartbeat = bus stays alive.

Three Operational Modes

Network ModeBus active. 3 sub-states inside: Repeat Message, Normal Operation, Ready Sleep. NM-Timeout Timer running. NM PDUs transmitted.
Prepare Bus-Sleep ModeAll nodes stopped TX. Bus calming down (TX buffers draining). Wait CanNmWaitBusSleepTime. Then → Bus-Sleep.
Bus-Sleep ModeNo bus activity. CAN controller in sleep. Minimum power. Wakeup source active. Any RxIndication → Nm_NetworkStartIndication() to upper layer.

Key Protocol Concepts

Network State (orthogonal to mode)

Parallel state machine — exists inside any mode:

StateMeaningAPI call
RequestedECU needs bus communicationCanNm_NetworkRequest()
ReleasedECU can go to sleepCanNm_NetworkRelease()

Network state drives transitions: Requested→Normal Op, Released→Ready Sleep.

Decentralized Algorithm

Node transmits NM PDU periodically → resets all other nodes' NM-Timeout Timers. No RxIndication for CanNmTimeoutTime → bus released by all → Prepare Bus-Sleep. No central controller, no token ring. Pure broadcast heartbeat.

CanNmTimeoutTime MUST be identical on all cluster nodes. Same for CanNmWaitBusSleepTime. Mismatch = unsynchronized sleep = some nodes stuck awake.

Limitations (Exam: always asked)

1. One CanNm channel ↔ one NM cluster ↔ one network. No sharing.

2. One CanNm channel per network per ECU.

3. CAN only (CAN 2.0, CAN FD, TT-CAN). Not LIN, FlexRay.

4. Passive mode: ECU only listens, never transmits NM PDUs. Must be same for ALL networks in ECU.

Remote Sleep Indication

ECU in Normal Op detects ALL other nodes ready to sleep (no NM PDU received for CanNmRemoteSleepIndTime). Calls Nm_RemoteSleepIndication(). ECU itself can then safely release bus. Only for gateway nodes (needs CanNmRemoteSleepIndEnabled=TRUE).

CheckRemoteSleepIndication() returns E_NOT_OK in Bus-Sleep, PBS, Repeat Msg State — only valid in Normal Op and Ready Sleep. [SWS_CanNm_00154]

User Data

Remaining bytes after system bytes and PNC bit vector. Two access methods:

Method A: CanNm API (CanNmUserDataEnabled=TRUE)Set: CanNm_SetUserData(). Get: CanNm_GetUserData(). CanNm_SetUserData NOT available if CanNmComUserDataSupport=TRUE.
Method B: COM user data (CanNmComUserDataSupport=TRUE)CanNm calls PduR_CanNmTriggerTransmit() to fetch data from COM I-PDU each TX. CanNm_SetUserData() not available in this mode.

Init: All user data bytes set to 0xFF. [SWS_CanNm_00025]

User data sent in Repeat Msg State always. In Normal Op: only if bus load reduction disabled (= node is transmitting). In Ready Sleep: NOT sent (TX stopped).

PNC Bit Vector (Partial Networking)

Position: NmPncBitVectorOffset (byte start) + NmPncBitVectorLength (byte count). Configured in Nm channel, not CanNm directly.

TX: CanNm fetches PNC vector via Nm_PncBitVectorTxIndication() each TX. RX: CanNm passes received PNC vector to Nm_PncBitVectorRxIndication().

PNC byte vector initialized to 0x00. [SWS_CanNm_00500]

Partial Networking (PN) — Complete Guide

PN concept: Not all ECUs need to wake up for all requests. PNC bit vector = which Partial Network Clusters need to be awake. Each bit = one PNC. ECU only wakes if its PNC bits are set in received NM PDU. Enables selective wakeup of only needed subsystems.

Key Terms

PNC (Partial Network Cluster)Logical group of ECUs that can independently wake/sleep. Each PNC has a bit in PNC bit vector.
ERA (External Request Array)PNC requests received from outside (other ECUs). Aggregated from received NM PDUs.
EIRA (External + Internal Request Array)ERA + internal PNC requests from ComM. Full picture of which PNCs need to be active.
PNI bit (Partial Network Information)CBV bit 6. Set=1 if PnEnabled=TRUE. Signals "this NM PDU contains PNC bit vector". [SWS_CanNm_00413]
PNSR bit (PN Shutdown Request)CBV bit 1. Set by coordinator to signal synchronized PNC shutdown across topology. [SWS_CanNm_00519]
PNL bit (PN Learning)CBV bit 5. Dynamic PNC-to-channel mapping learning phase trigger. [SWS_CanNm_00380]

TX Handling — PN bit insertion

Every NM PDU TX: CanNm calls Nm_PncBitVectorTxIndication(ch, &pncBuf) → gets aggregated internal PNC requests. Copies to PNC bit vector bytes in NM PDU. [SWS_CanNm_00518]

TxConfirmation(E_OK) → Nm_PncBitVectorTxConfirmation(ch, pncVec, E_OK)

TxConfirmation(E_NOT_OK) OR TX request rejected → Nm_PncBitVectorTxConfirmation(ch, pncVec, E_NOT_OK)

Synchronized PNC Shutdown

Used by top-level PNC coordinator (ECU with ComMPncGatewayType=ACTIVE) to trigger synchronized shutdown across entire PN topology.

Nm detects all PNC requestors released
→
CanNm_ActivateTxPnShutdownMsg(ch)
→ PNSR=1 in CBV →
Transmit PN shutdown message NM PDU

PN shutdown message transmission has HIGHER priority than cyclic NM PDUs — cyclic TX is postponed one main function cycle when PNSR shutdown is active. [SWS_CanNm_00513]

If TX fails during PN shutdown: retransmit in next main function. [SWS_CanNm_00514] PN shutdown message must be assumed transmitted even if actual CanIf TX pending.

Each TX request → START Msg TX Timeout Timer (CanNmMsgTimeoutTime)
TxConfirmation received → STOP Msg TX Timeout Timer
Timer expires without TxConfirmation → Nm_TxTimeoutException() + CanSM_TxTimeoutException()

CanSM_TxTimeoutException() → CanSM stops+restarts CAN controller → recovers from stuck TX (e.g., bus-off while PN transceiver used).

CanNmGlobalPnSupport=TRUE → CanNm STOPS NM-Timeout Timer at init [SWS_CanNm_00061]. Timer only started after PN availability confirmed via CanNm_ConfirmPnAvailability().

CanNm_ConfirmPnAvailability() — Critical flow

Called by CanSM when PN-capable transceiver confirmed ready. ENABLES the PN filter on this channel.

Before ConfirmPnAvailability called: ALL NM PDUs considered relevant (safety — if PN HW malfunctions, ECU stays synchronized with network). [SWS_CanNm_00503 note]

After ConfirmPnAvailability called: only NM PDUs with relevant PNC bits wake up this ECU.

Debugging Checklist

ECU not going to Bus-Sleep (network stuck awake)
1. Check CanNmTimeoutTime IDENTICAL on ALL cluster nodes. Even 1ms difference = one node never times out.
2. Check NM PDU received from unexpected node (check SNI). Some node still active.
3. Check CanNm_NetworkRelease() actually called by ComM. Inspect network state variable.
4. Check DisableCommunication was called — if so, NM-Timeout Timer stopped → ECU never times out.
5. Check PN enabled: PN-capable transceiver may suppress NM PDUs from non-PN nodes. Use AllNmMessagesKeepAwake=TRUE on gateway.
6. CANoe: monitor NM PDU rate. If decreasing correctly → problem is WaitBusSleepTime or CanSM not transitioning.
CANNM_E_NETWORK_TIMEOUT in DET — frequently
Timer expired in Repeat Msg or Normal Op = TX not happening + no RX from others.
1. Check bus-off condition: CanSM reports CANSM_E_BUS_OFF → bus-off prevents TX → timer expires.
2. Check CanNmMsgCycleTime < CanNmTimeoutTime (required). If equal: single missed TX = timeout.
3. Check DisableCommunication called but NM-Timeout Timer not stopped properly (check SWS_CanNm_00174).
4. Check CanNmImmediateTxconfEnabled — if TRUE, CanNm doesn't wait for actual TxConfirmation → timer not restarted by TX conf.
CANNM_E_NETWORK_TIMEOUT is expected/harmless in Repeat Msg + Normal Op in some configurations — timer restart is spec'd, not a fatal error. Only concerning if repeated without reason.
PN filtering drops all NM PDUs (ECU stays in Bus-Sleep)
1. Check CanNm_ConfirmPnAvailability() was called by CanSM. Before this, ALL PDUs treated as relevant.
2. Check PNI bit=1 in received NM PDUs. If PNI=0 and AllNmMessagesKeepAwake=FALSE → PDU dropped.
3. Check NmPncBitVectorOffset and NmPncBitVectorLength match between TX and RX nodes.
4. Check PNC bit vector filter: relevant PNC bit must be set in received PDU AND match ECU's configured PNC.
5. Gateway ECU: AllNmMessagesKeepAwake MUST be TRUE, otherwise gateway drops non-PN NM PDUs and loses cluster synchronization.
CanSM_TxTimeoutException() called unexpectedly
CanNm TX timeout timer (CanNmMsgTimeoutTime) expired without TxConfirmation(E_OK).
1. CanNmMsgTimeoutTime must be < CanNmMsgCycleTime. Check value.
2. Check CanIf TX queue full — LSduR_CanNmTransmit returned E_NOT_OK. CanNm retransmits next main function (SWS_CanNm_00514).
3. Bus-off during PN operation → TX fails → timeout → CanSM_TxTimeoutException() → CanSM restarts driver. This is INTENDED behavior for PN robustness.
4. Check CanNmImmediateTxconfEnabled — if TRUE, [SWS_CanNm_00073] disables TX error handling entirely. No timer, no exception.

CANoe / Diagnostics Integration Tips

CANoe CAPL Scripts

  • Monitor NM PDU ID on CAN trace. Verify SNI byte matches expected NodeId.
  • Check CBV byte for Repeat Msg bit (bit 0) during startup burst.
  • Inject bus-off → verify Nm_TxTimeoutException + CanSM restart sequence.
  • Stop all NM PDUs from ECU under test → measure time until it enters PBS (= CanNmTimeoutTime). Compare to config.
  • PN test: send NM PDU without PNI bit → ECU with AllNmMessages=FALSE should not restart timer.

TRACE32 Debug Tips

  • Watch CanNm_NetworkModeState[] — transitions should match triggers.
  • Break on Nm_BusSleepMode() call — check timing from last NM PDU RX.
  • Break on Nm_TxTimeoutException() — check CanNmMsgTimeoutTime < MsgCycleTime.
  • Watch CanNm_NmTimeoutTimer[] — decrement rate = MainFunctionPeriod.
  • Inspect CanNm_RxBuffer[] immediately after RxIndication — verify PDU copied correctly.

Common Integration Mistakes (Industry level)

1. CanNmTimeoutTime not identical across cluster → some nodes stay awake forever. Always verify OEM network spec has one master timing table.
2. LSduR_CanNm.h not included (R24-11 change) → linker error or missing interface. Previously was PduR_CanNm.h. Check R24-11 migration notes.
3. CanNm_ConfirmPnAvailability() never called → PN filter never enabled → all NM PDUs treated as relevant → ECU never goes to Bus-Sleep in PN network.
4. CanNmMsgCycleOffset = 0 for all nodes → all nodes TX simultaneously at startup → CAN bus collision burst → arbitration chaos → some TX fails → network_timeout errors.
5. NM PDU ID conflict with regular CAN frames → LSduR routing wrong → CanNm_RxIndication called for wrong frames. Check CanNmRxPduRef PDU ID matches CanIf RX PDU config.
6. ImmediateNmTransmissions > 0 but ImmediateRestartEnabled also TRUE → conflict. ImmediateNmTransmissions must be 0 if ImmediateRestartEnabled=TRUE. [ECUC dependency]
