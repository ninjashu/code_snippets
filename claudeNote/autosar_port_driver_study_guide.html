Core purpose: PORT driver initializes the ENTIRE port structure of the MCU. Every pin's function, direction, and level is set here before any other MCAL driver uses it.

What PORT driver does

  • Initializes ALL on-chip ports and port pins
  • Assigns pin functionality (GPIO, ADC, SPI, CAN, LIN, PWM…)
  • Sets direction (IN / OUT) per pin
  • Sets initial output level (HIGH / LOW)
  • Enables runtime direction / mode change (if configured)
  • Sets unused pins to a defined safe state
  • Provides symbolic pin names via Port.h

What PORT driver does NOT do

  • Read/write pin data values → that is DIO driver
  • Configure pin-level inversion → IO HW Abstraction does that
  • Provide callback notifications → ICU or complex driver handles that
  • Have scheduled functions (no cyclic tasks)
  • Have mandatory interface dependencies (only optional DET)

Critical initialization rule

PORT_Init() MUST be called before any DIO function. If PORT is not initialized, DIO functions exhibit undefined behaviour. In AUTOSAR startup, PORT init comes immediately after MCU driver init.

Startup order: MCU_Init → PORT_Init → other MCAL inits → OS start → application

Supported pin functionalities (modes)

General purpose I/O (DIO) ADC SPI SCI PWM CAN LIN FlexRay ICU MEM (external memory) DIO_GPT (timer) DIO_WDG (watchdog)

Configuration and mode of port pins is microcontroller and ECU dependent.

Key constraint (exam critical)

It is the user's responsibility to ensure the same Port/Port pin is not accessed in parallel by different entities. No built-in protection against: two tasks configuring same port, same pin, or different pins on same port simultaneously.

PORT driver has 2 states: UNINIT and INIT. Simple compared to other MCAL drivers. No SLEEP state.
PORT_UNINIT
After reset / before Port_Init()
PORT_INIT
After Port_Init() succeeds

State transitions

UNINIT → Port_Init(ConfigPtr) → INIT
INIT → Port_Init(ConfigPtr) again → INIT (re-initialization is allowed)
UNINIT → Any other API call → DET error: PORT_E_UNINIT (0x0F)

Note: There is NO explicit de-initialization function. Calling Port_Init() again re-initializes to a new configuration state.

State diagram exam points

  • Power-on reset → MCU comes up in UNINIT state
  • Port_Init() called ONCE by EcuM (ECU Manager) during startup
  • After init, state stays INIT for entire ECU operation
  • Re-initialization (Port_Init again) allowed but environment must not call it during running operation if multiple callers exist
  • Error detection: if DET enabled and any API called in UNINIT → DET gets PORT_E_UNINIT, function skips execution (no side effect)
  • No MCU reset triggered by PORT driver state errors

[SWS_Port_00001] Initialization

PORT driver shall initialize the WHOLE port structure of the MCU. Defining the order in which ports and port pins are configured is the task of the configuration tool.

7.1.2.1 Configuration of Port Pin Properties

[SWS_Port_00004] Functionality

PORT driver shall allow configuration of different functionality per port/pin: ADC, SPI, DIO etc. Configuration is microcontroller dependent.

[SWS_Port_00079] Mandatory configs

  • Pin direction (input/output)
  • Pin level initial value
  • Pin direction changeable at runtime (yes/no)
  • Port mode changeable at runtime (yes/no)

[SWS_Port_00081] Optional configs (if HW supports)

Slew rate control Internal pull-up activation Input thresholds Pin driven mode (push-pull / open drain) Readback type (pin level or output register)
[SWS_Port_00082] PORT driver shall NOT configure pin level inversion. Default value (not inverted) is set. Level inversion is done by IO Hardware Abstraction layer.
7.1.2.2 Switch port pin direction (runtime)

[SWS_Port_00137] Runtime direction change

For pins configured as "direction changeable" in the configuration tool, PORT driver allows user to change direction at runtime via Port_SetPinDirection().

[SWS_Port_00138] Output latch preservation

If MCU has an output latch for the pin, switching direction shall NOT alter the level already set in the output latch. Critical for safety — prevents glitches.

7.1.2.3 Refresh port direction

[SWS_Port_00066] Refresh function purpose

Port_RefreshPortDirection() allows user to refresh direction of those port pins whose direction is set by configuration and cannot be changed dynamically. Used to restore direction registers if corrupted (e.g. by EMI, software bug).

[SWS_Port_00061] Pins marked as "direction changeable at runtime" are EXCLUDED from this refresh. Only static-direction pins are refreshed.

7.1.2.4 Unused ports and pins

[SWS_Port_00005] Unused pin safety

PORT driver shall configure ALL unused ports and pins (neither GPIO nor special purpose IO) to a defined state. Floating unconnected pins are a safety and EMC risk — this requirement ensures no pin is left undefined.

7.1.2.5 Symbolic names

[SWS_Port_00006] [SWS_Port_00207]

User configures symbolic names for port pins (e.g. PORT_A_PIN_0). These are defined in the configuration tool (DaVinci).

[SWS_Port_00208] Publication

PORT driver implementer publishes symbolic names via Port.h. Application uses these symbolic names (not raw numbers) for portability.

7.1.2.6 Atomicity

[SWS_Port_00075] Atomic port access

PORT driver shall provide atomic access to all ports and port pins. Atomic = non-interruptible access via atomic instructions OR exclusive area (interrupt disable by BSW scheduler). Prevents race conditions on shared port registers.

7.1.2 — Port_Init special rules

[SWS_Port_00055] Output latch on init

Port_Init() shall set output latch to configured default level BEFORE setting direction to output. Prevents pin glitch at startup (brief wrong-level output).

[SWS_Port_00043] No glitches

Port_Init() shall avoid glitches and spikes on affected port pins. Implementer must choose register write order carefully.

Port_Init (ConfigPtr) — Service ID: 0x00

Syntax: void Port_Init(const Port_ConfigType* ConfigPtr)

Reentrancy: Non-reentrant | Return: void

What it does: Initializes ALL ports and port pins per ConfigPtr. Sets all variables to initial state. Avoids glitches. Sets output latch before direction.

ConfigPtr = NULL: Only for VARIANT-PRE-COMPILE. NULL pointer passed — configuration is in ROM, pointer not needed.

Key rule: Must be called FIRST before any other PORT/DIO API. Can be called again to re-initialize. Do NOT call during running operation if multiple callers exist.

DET errors: PORT_E_INIT_FAILED if wrong parameter passed.

Port_SetPinDirection (Pin, Direction) — Service ID: 0x01

Syntax: void Port_SetPinDirection(Port_PinType Pin, Port_PinDirectionType Direction)

Reentrancy: Reentrant (different pins only) | Return: void

What it does: Sets direction of specified pin at runtime. Only works for pins with PortPinDirectionChangeable = TRUE.

Availability: Only compiled in if pre-compile parameter PortSetPinDirectionApi = TRUE. If FALSE → function does not exist.

Latch preservation: Does not alter output latch value when switching direction.

DET errors: PORT_E_UNINIT, PORT_E_PARAM_PIN, PORT_E_DIRECTION_UNCHANGEABLE

Port_RefreshPortDirection (void) — Service ID: 0x02

Syntax: void Port_RefreshPortDirection(void)

Reentrancy: Non-reentrant | Return: void

What it does: Refreshes direction of all statically-configured pins back to configured value. SKIPS pins marked as direction-changeable at runtime.

Use case: Safety mechanism — if EMI or a software bug corrupted direction registers, this restores them. Typically called cyclically by BSW scheduler.

DET errors: PORT_E_UNINIT

Port_GetVersionInfo (versioninfo) — Service ID: 0x03

Syntax: void Port_GetVersionInfo(Std_VersionInfoType* versioninfo)

Reentrancy: Reentrant | Return: void (output via pointer)

What it does: Returns module version info (vendor ID, module ID, SW major/minor/patch version) into caller-provided struct.

Availability: Only if PortVersionInfoApi = TRUE.

DET errors: PORT_E_PARAM_POINTER if NULL pointer passed (when DET enabled)

Note: No PORT_E_UNINIT check — can be called in UNINIT state.

Port_SetPinMode (Pin, Mode) — Service ID: 0x04

Syntax: void Port_SetPinMode(Port_PinType Pin, Port_PinModeType Mode)

Reentrancy: Reentrant (different pins only) | Return: void

What it does: Changes operational mode of pin at runtime (e.g. from DIO to SPI). Only for pins with PortPinModeChangeable = TRUE.

Availability: Only if PortSetPinModeApi = TRUE.

DET errors: PORT_E_UNINIT, PORT_E_PARAM_PIN, PORT_E_PARAM_INVALID_MODE, PORT_E_MODE_UNCHANGEABLE

[SWS_Port_00077] Parameter checking order (exam trap!)

When DET is enabled, parameters are checked in the order passed. If first check fails, further parameter checking is SKIPPED.

Example for Port_SetPinDirection(Pin, Direction):

  1. Check if module is initialized → if fail: report PORT_E_UNINIT, return
  2. Check Pin ID validity → if fail: report PORT_E_PARAM_PIN, return
  3. Check pin is direction changeable → if fail: report PORT_E_DIRECTION_UNCHANGEABLE, return
  4. Execute the function

[SWS_Port_00087] Behavior on error detection

If DET enabled and error detected: desired functionality is SKIPPED. Service returns without any action. No partial execution.

DET module used: Det_ReportError() via Det.h (optional interface — only if DET enabled).

Enable DET: set PortDevErrorDetect = TRUE in PortGeneral container.

Version check (compatibility guard)

PORT module prevents integration of incompatible files via preprocessor checks. .c file checks header version: SW version numbers of .c and .h must match. If mismatch → compile error. Refer to SWS BSW General section 5.1.8.

1

2
User calls Port_SetPinDirection(Port_PinType, Port_PinDirectionType)
3

4

5

Seq 9.3 — Refresh all port directions

1
Precondition: Port_Init() called
2
User (or BSW scheduler) calls Port_RefreshPortDirection(void)
3
PORT loops through ALL configured pins
4
For pins with PortPinDirectionChangeable=FALSE: restore direction register to configured value
5
Pins with PortPinDirectionChangeable=TRUE: SKIP (do not touch)
6
Returns void. State: "port pin direction is refreshed"

Seq 9.4 — Change pin mode (runtime)

1
!
Precondition: Port_Init() called. Pin must have PortPinModeChangeable=TRUE. PortSetPinModeApi=TRUE
2
User calls Port_SetPinMode(Port_PinType, Port_PinModeType)
3

4

5
Returns void. State: "port pin mode is set"

Sequence diagram exam tips

  • All diagrams show User as initiator, PORT module as responder — no PORT-initiated sequences
  • All functions are synchronous — there is NO asynchronous completion notification
  • No callback from PORT → user after operations complete
  • DET error check happens INSIDE Port module, before register access
  • Port_RefreshPortDirection is the only function with a loop internally across all pins
  • Reentrant functions (SetPinDirection, SetPinMode): two concurrent calls to different pins are safe; same pin from two tasks = hazard

DaVinci step-by-step for a project

Project configuration workflow

1
Add Port module to your project in DaVinci. It appears in the BSW module list under MCAL.
2
Configure PortGeneral: Set PortDevErrorDetect=TRUE (development phase). Enable PortSetPinDirectionApi and PortSetPinModeApi based on requirements. Enable PortVersionInfoApi for integration testing.
3
Create PortContainers: Add one per HW port (e.g. PortA, PortB). Set PortNumberOfPortPins to count of pins you configure in that port.
4
Add PortPin entries: For each physical pin that needs specific config: set PortPinId (auto-generated from short name), PortPinDirection, PortPinInitialMode, PortPinLevelValue. Give short name like "LED_STATUS" — this becomes your symbolic constant in Port.h.
5
Configure unused pins: Either add them explicitly with a safe state or configure PortContainer to cover all pins including unused ones set to defined state per SWS_Port_00005.
6
Multicore (if applicable): Set PortEcucPartitionRef in PortGeneral for each core/partition. Then set PortPinEcucPartitionRef per pin. Constraint: pin partitions must be subset of module partitions.
7
Generate code: DaVinci generates Port_Cfg.h, Port_Cfg.c, Port_PBcfg.c (for POST-BUILD variant). Port.h gets symbolic pin names. Integrate generated files into your project.
8
Validate: Check for DaVinci validation errors — mismatched direction vs mode, missing partition refs, duplicate PortPinIds.

Multicore constraint (exam important)

[SWS_Port_CONSTR_00233]: ECUC partitions in PortPinEcucPartitionRef must be a subset of partitions in PortEcucPartitionRef.

[SWS_Port_CONSTR_00234]: If PortEcucPartitionRef references one or more partitions, PortPinEcucPartitionRef MUST have multiplicity > 0 and reference one or more of those partitions.

[SWS_Port_00232]: PORT module shall reject configurations with partition mappings not supported by implementation.

Integration testing, interface testing, and debugging — industry-level checklist.
Integration testing checklist

Boot sequence verification

  • Verify Port_Init() called BEFORE any DIO_ReadChannel, DIO_WriteChannel, SPI_Init etc.
  • Check startup timing: oscilloscope on critical output pins to verify correct initial level appears without glitch
  • Measure: time from reset to Port_Init return — should be deterministic
  • Verify no DET errors during normal startup (read DET error log or set breakpoint in Det_ReportError)
  • Confirm Port_Init(NULL_PTR) used for PRE-COMPILE, valid pointer for POST-BUILD

Pin-level interface testing (oscilloscope / logic analyzer)

  • Output pins: verify correct initial level immediately after Port_Init (no glitch)
  • Input pins: verify correct pull resistor state (pull-up / pull-down / floating) per config
  • After Port_SetPinDirection(PIN, OUTPUT): verify direction changed, output latch preserved
  • After Port_SetPinMode(): verify alternate function active (e.g. SPI clock appears on pin)
  • After Port_RefreshPortDirection(): force-corrupt direction register via debugger first, then call refresh, verify register restored
  • Unused pins: probe with oscilloscope — must NOT be floating. Should show defined HIGH or LOW.

DET testing (development error detection)

  • Test all DET paths systematically — call each API with wrong parameters
  • Call any API before Port_Init → expect PORT_E_UNINIT (0x0F)
  • Call Port_SetPinDirection with invalid Pin ID → expect PORT_E_PARAM_PIN (0x0A)
  • Call Port_SetPinDirection on non-changeable pin → expect PORT_E_DIRECTION_UNCHANGEABLE (0x0B)
  • Call Port_SetPinMode with unsupported mode → expect PORT_E_PARAM_INVALID_MODE (0x0D)
  • Call Port_SetPinMode on non-mode-changeable pin → expect PORT_E_MODE_UNCHANGEABLE (0x0E)
  • Call Port_GetVersionInfo with NULL → expect PORT_E_PARAM_POINTER (0x10)
  • Verify function returns WITHOUT side effect on error (no register write occurs)

Reentrancy / concurrency testing

  • Verify Port_SetPinDirection called simultaneously on DIFFERENT pins from two tasks — no corruption
  • Verify Port_SetPinMode called simultaneously on different pins — no corruption
  • Verify Port_RefreshPortDirection NOT called from two tasks simultaneously (non-reentrant)
  • Verify Port_Init NOT called while other PORT functions running (environment responsibility — but test it)
  • Check atomic access: scope port data register write — no partial writes visible

Debugging techniques

  • Wrong pin behavior after init: Check PortPinInitialMode, PortPinDirection, PortPinLevelValue in DaVinci config. Regenerate and rebuild.
  • DIO reads wrong value: Check PORT was initialized before DIO call. Check pin direction = IN. Check pin mode = DIO.
  • Port_SetPinDirection silently fails: Check PortDevErrorDetect=TRUE, check PortPinDirectionChangeable=TRUE for that pin, check PortSetPinDirectionApi=TRUE in PortGeneral.
  • Pin floats after direction change: Check output latch (PORT hardware register OLATx) — should retain previous value per SWS_Port_00138.
  • SPI / CAN not working: Check PORT configured correct mode for those pins (PortPinInitialMode = PORT_PIN_MODE_SPI / PORT_PIN_MODE_CAN). Wrong mode = pin stuck as GPIO.
  • Multicore crash accessing port: Check PortPinEcucPartitionRef configured for correct core. Check Port_Init called on correct core.
