AUTOSAR Classic Platform — SPI Handler/Driver (SWS, R24-11) · Complete exam study guide

What is the SPI Handler/Driver?

A monolithic AUTOSAR Basic Software (BSW) module that sits in the MCAL layer (Microcontroller Abstraction Layer) and the ECU Abstraction layer. It provides a unified interface for all SPI bus communication within an ECU — to devices like EEPROMs, Watchdog ICs, and I/O ASICs.

Master mode only Full-duplex only 1 to N SPI buses Static pre-compile config

SPI Physical Signals

  • MOSI — Master Out Slave In (data from MCU)
  • MISO — Master In Slave Out (data to MCU)
  • CS — Chip Select (active per Job)
  • CLK — Clock (MCU drives it)

Clock config, PLL, prescalers are done by MCU Driver, not SPI driver.

Key Constraint

  • SWS_Spi_00040: Master mode only
  • SWS_Spi_00050: Full-duplex only
  • SWS_Spi_00244: MCU Driver manages clock/PLL
  • SWS_Spi_00342: Port pins shared with SPI may need Port Driver
  • CS can be HW peripheral-managed or software DIO-managed

Core Abstraction Model: Channel → Job → Sequence

This 3-tier hierarchy is the entire foundation of AUTOSAR SPI. Everything is built on top of it.

Channel

A logical data pipe with same config params. Identified by Spi_ChannelType (uint8). Contains N data elements. Can be reused across multiple Jobs.

SpiDataWidth: 1–64 bits. Access casting: uint8/16/32.

Job

One or more Channels on the same Chip Select (= one slave device). Identified by Spi_JobType (uint16). Has a priority (0–3). Atomic — cannot be interrupted.

CS asserted at start, deasserted at end of Job.

Sequence

Ordered list of Jobs. Identified by Spi_SequenceType (uint8). Can be Interruptible or Non-Interruptible. Jobs sorted highest→lowest priority.

First Job = highest (or equal) priority in Sequence.

Buffer Types

IB (Internal Buffer) — driver owns the buffer. Fixed size (1 entry deep per channel). Use: small transfers (<10 bytes), daisy-chain. Access via Spi_WriteIB / Spi_ReadIB.

EB (External Buffer) — application owns the buffer. Size variable (up to config max). Use: large streams, EEPROM, complex ASICs. Set via Spi_SetupEB before transmit.

SpiChannelBuffersAllowed: 0=IB only, 1=EB only, 2=Both

Data Width Handling

SpiDataWidth: 1–64 bits. Software access type (casting):

  • uint8 when SpiDataWidth < 9
  • uint16 when 9 ≤ SpiDataWidth < 17
  • uint32 when 17 ≤ SpiDataWidth

If data width < access type (e.g. 12-bit width, 16-bit access): lower bits sent, upper bits ignored on TX. On RX, lower bits received, upper bits zero-extended.

Not SPI's job: clock, PLL, port pin direction config

CS (Chip Select) Handling

  • CS is tied to the Job definition, not Channel or Sequence
  • CS activated at Job start, deactivated at Job end
  • SpiCsSelection: CS_VIA_PERIPHERAL_ENGINE or CS_VIA_GPIO
  • SpiCsBehavior: CS_KEEP_ASSERTED (all channels) or release per frame
  • If CS_VIA_GPIO: SPI driver drives CS line as DIO explicitly

Header File

All public types and API functions are in Spi.h. Callback notification definitions are in Spi_Externals.h. Scheduled function (Spi_MainFunction_Handling) is in SchM_Spi.h.

3 Scalability Levels — Configured at Pre-compile Time

Parameter: SpiLevelDelivered (0, 1, or 2). Cannot be changed at runtime.

Level 0Simple Sync
  • Synchronous only (Spi_SyncTransmit)
  • Caller is blocked during TX
  • No priorities needed
  • No notifications called
  • Concurrent sync TX allowed only if SpiSupportConcurrentSyncTransmit=TRUE

Use case: Simple ECU, high-speed slaves

Level 1Basic Async
  • Asynchronous only (Spi_AsyncTransmit)
  • Caller NOT blocked
  • Job priority scheduling (0–3)
  • End notifications (Job + Seq)
  • Polling OR interrupt mode (switchable at runtime)
  • Interruptible Sequences supported

Use case: Multi-priority SPI access, low-speed slaves

Level 2Enhanced
  • Both Sync AND Async
  • All Level 0 rules apply to sync
  • All Level 1 rules apply to async
  • Default async mode: SPI_POLLING_MODE
  • Spi_SetAsyncMode switches polling/interrupt

Use case: Complex ECU with mixed-speed slaves

Async Mode (Level 1 & 2)

Spi_AsyncModeType:

  • SPI_POLLING_MODE — interrupts disabled; Spi_MainFunction_Handling drives state machine (called by BSW Scheduler)
  • SPI_INTERRUPT_MODE — interrupts enabled; ISR drives state machine

Cannot switch mode if async TX in progress (returns E_NOT_OK). CAN switch during sync TX.

Interruptible Sequences (Level 1 & 2)

SpiInterruptibleSeqAllowed: ON or OFF

  • OFF (Non-Interruptible): Once a Sequence starts, it runs to completion. No preemption between Jobs.
  • ON (Mixed): Each Sequence marked SpiInterruptibleSequence=TRUE can be preempted between Jobs by a higher-priority Job from another Sequence. Rescheduling happens at each Job boundary.

Warning: Interruptible sequences sharing channels can overwrite each other's data!

Job Priority Rules

  • Priority values: 0 (lowest) to 3 (highest)
  • Jobs in a Sequence must have same or decreasing priority (first Job = highest)
  • Higher value = higher priority (SWS_Spi_00093)
  • Scheduler always sends highest-priority pending Job first
  • In Level 1/2: if new Sequence is requested, SPI evaluates lead Job priority to accept or reject

Spi_DeInit() succeeds (only if IDLE!) → SPI_UNINIT

DeInit is rejected if state is SPI_BUSY. Returns E_NOT_OK.

Job Result State (Spi_JobResultType)

  • SPI_JOB_OK (0x00) — default after init; last TX succeeded
  • SPI_JOB_PENDING (0x01) — TX in progress
  • SPI_JOB_FAILED (0x02) — TX failed (e.g. HW error)
  • SPI_JOB_QUEUED (0x03) — accepted but not yet started (Async only)

Flow: QUEUED → PENDING → OK or FAILED. Job end notification called regardless of OK/FAILED (to avoid deadlocks).

Sequence Result State (Spi_SeqResultType)

  • SPI_SEQ_OK (0x00) — default; last TX succeeded
  • SPI_SEQ_PENDING (0x01) — TX in progress
  • SPI_SEQ_FAILED (0x02) — TX failed
  • SPI_SEQ_CANCELED (0x03) — canceled by Spi_Cancel()

Seq end notification called for FAILED, OK, and CANCELED (to avoid deadlocks).

Hardware Unit State (Spi_GetHWUnitStatus)

Returns Spi_StatusType: SPI_IDLE or SPI_BUSY. Controlled by SpiHwStatusApi config. Used to check if a specific SPI peripheral (HW unit) is free — important when Jobs on different buses could run concurrently.

Complete Async Transmission State Flow

Spi_Init() → Driver: IDLE, Jobs: OK, Seqs: OK
Spi_WriteIB() / Spi_SetupEB() → No state change
Spi_AsyncTransmit(Seq0) → Driver: BUSY, Seq0: PENDING, Jobs: QUEUED
Job starts executing → Jobs: PENDING
Job done OK → Jobs: OK → JobEndNotification() called
All Jobs done → Seq0: OK → SeqEndNotification() called → Driver: IDLE
HW error → Jobs: FAILED, Seq: FAILED → DEM reported → notifications called

IB Channel Flow (AsyncTransmit)

1
Spi_Init(ConfigPtr) — initialize all channels, jobs, sequences
2
Spi_WriteIB(ChannelID, &TxBuf) — copy TX data into internal buffer
3
Spi_AsyncTransmit(SequenceID) — enqueue sequence, set BUSY+PENDING
4
Driver processes Jobs (highest priority first), asserts CS, clocks data
5
Spi_JobEndNotification() called for each Job (if configured)
6
Spi_SeqEndNotification() called when Sequence complete
7
Spi_ReadIB(ChannelID, &RxBuf) — read received data from internal buffer

1
Spi_Init(ConfigPtr)

Initialization / De-initialization Sequence

1
System: MCU_Init() — sets up clock, PLL, prescalers
2
System: Port_Init() — configures SPI pin directions (MOSI/MISO/CLK/CS as alt function)

Sequence Cancellation Flow

1
Spi_AsyncTransmit(Seq0) — sequence running (PENDING)
2
Current Job executes to completion (Jobs are ATOMIC — cannot abort mid-Job)
3
Spi_Cancel(Seq0) — sets cancel flag between Jobs
4
At Job boundary: Seq0 set to SPI_SEQ_CANCELED, next Job NOT started
5
Spi_SeqEndNotification() called (required even on cancel)

External devices may be in undefined state after cancellation. Application responsibility!

Rescheduling (Interruptible Sequence)

1
Seq0 (interruptible) running — Job1 (priority 2) executing
2
User calls Spi_AsyncTransmit(Seq1) — Seq1 has Job0 with priority 3 (higher)
3
Seq1 accepted, queued. Seq0/Job1 completes (Job is atomic!)
4
At Job boundary: rescheduling occurs → elect next Job (Job0 from Seq1, priority 3 wins)
5
Order: Scheduling first, THEN Job1 end notification (SWS_Spi_00270)
6
Seq1.Job0 executes → Seq1.SeqEndNotification
7
Seq0.Job2 (priority 1) resumes → Seq0.SeqEndNotification

Automotive Testing & Debug Checklist

Initialization Issues

!
Call Spi_GetStatus() immediately after init — must return SPI_IDLE, never SPI_UNINIT
!
Check that MCU, Port drivers are initialized BEFORE SPI_Init — clock/pin config must come first
✗
Calling Spi_Init twice without DeInit → SPI_E_ALREADY_INITIALIZED DET error
✗
Calling any API before Init → SPI_E_UNINIT DET error; return value undefined

Transmission Issues

AsyncTransmit returns E_NOT_OK → Check if Seq is already PENDING (SPI_E_SEQ_PENDING), or if Seq shares Jobs with a PENDING Seq
SyncTransmit returns E_NOT_OK → Check if same bus is busy and SpiSupportConcurrentSyncTransmit=FALSE (SPI_E_SEQ_IN_PROCESS)
Data corruption in IB mode → Ensure WriteIB is NOT called during active transmission on that channel (no consistency guarantee)
EB mode NULL data → NULL SrcPtr = default TX value is transmitted. NULL DstPtr = received data discarded
HW error (DEM event) → Check DEM for SPI_E_HARDWARE_ERROR. Read SPI status register error flags. Check clock, voltage, slave response.

Notification Not Called?

  • Level 0: Job/Seq notifications are NOT called (by spec)
  • Check SpiJobEndNotification / SpiSeqEndNotification configured (not NULL) in SpiJob/SpiSequence config
  • In polling mode: ensure Spi_MainFunction_Handling() is called periodically by BSW scheduler
  • Notification called from interrupt context (Level 1/2 interrupt mode) — check stack size

Async Mode Switching Issues

  • Spi_SetAsyncMode(SPI_INTERRUPT_MODE) during async TX → returns E_NOT_OK, mode not changed
  • Mode switch during sync TX → allowed, returns E_OK
  • After mode switch to POLLING: must call Spi_MainFunction_Handling regularly or TX will never complete
  • After mode switch to INTERRUPT: ensure ISR priority is correct

Priority / Scheduling Debug

Jobs within a Sequence must be highest→lowest priority. If Job2 has higher priority than Job1 in same Seq → configuration error
In interruptible mode: if Seq0 and Seq1 share Channels, whichever Job runs last wins the channel data. Avoid shared channels across interruptible sequences.
Use Spi_GetJobResult(JobID) to poll whether a specific Job completed OK or FAILED during debugging
Use Spi_GetHWUnitStatus(UnitID) to check individual bus availability — useful when Jobs on different buses belong to same Sequence

Q1. What is the difference between a Channel, a Job, and a Sequence?tap to reveal

Channel: Logical data pipe with uniform config (data width, buffer type). Can be reused across Jobs.
Job: One or more Channels sharing the same Chip Select (one slave device). Atomic — cannot be interrupted mid-Job. Has a priority (0–3).
Sequence: Ordered list of Jobs. Can be interruptible (preempted at Job boundaries by higher-priority Jobs) or non-interruptible. First Job must have highest priority in the Sequence.

Q2. What happens when Spi_DeInit() is called while SPI is BUSY?tap to reveal

DeInit is rejected. The function returns E_NOT_OK and no action is taken. The driver remains in SPI_BUSY state. Application must poll Spi_GetStatus() until SPI_IDLE before calling DeInit.

Q3. Can Spi_Cancel() stop a Job that is currently being transmitted?tap to reveal

No. Jobs are atomic. Spi_Cancel() sets a cancel flag that is checked at Job boundaries. The currently transmitting Job runs to completion. The Sequence is then set to SPI_SEQ_CANCELED and the next Job is not started. The Seq end notification IS called even on cancellation.

Q4. Which error is reported to DEM and which to DET? Why?tap to reveal

DET: Development Errors (wrong params, uninit, already-init) and Runtime Errors (SEQ_PENDING, SEQ_IN_PROCESS) — these are software usage errors caught in development or handled at runtime.
DEM: SPI_E_HARDWARE_ERROR — this is a production/field error (HW failure during actual operation) that needs to be stored for field diagnostics and OBD/diagnostic services. DEM persists fault info in NVM, DET just logs/traps.

Q5. What is the order of operations in polling mode async TX? Why does Spi_MainFunction_Handling matter?tap to reveal

In SPI_POLLING_MODE, SPI interrupts are disabled. The state machine advances only when Spi_MainFunction_Handling() is called — it polls the HW status register. If the BSW scheduler does not call this function periodically, the TX will never complete and notifications will never fire. In SPI_INTERRUPT_MODE, the ISR does this automatically when the HW fires an interrupt.

Q6. What is the rule for Job priorities within a Sequence?tap to reveal

Jobs within a Sequence must have same or decreasing priority. The first Job must have the highest (or equal) priority. Values: 0=lowest, 3=highest. Scheduler sends highest priority Job first across all sequences. If an interruptible Sequence's current Job finishes and a higher-priority Job from another Sequence is pending, rescheduling occurs.

Q7. What are the three driver-level statuses, and how do they transition?tap to reveal

SPI_UNINIT (default after reset) → SPI_IDLE (after successful Init) → SPI_BUSY (transmission started) → SPI_IDLE (transmission complete/failed). DeInit: IDLE→UNINIT. Cannot DeInit from BUSY.

Q8. What is the data type casting rule for SpiDataWidth?tap to reveal

uint8 when SpiDataWidth < 9 bits. uint16 when 9 ≤ SpiDataWidth < 17 bits. uint32 when SpiDataWidth ≥ 17 bits. If width < access type (e.g. 12-bit width with 16-bit access): lower 12 bits transmitted on TX; on RX, lower 12 bits received and upper bits zero-extended.

Q9. When does Spi_SetAsyncMode return E_NOT_OK?tap to reveal

When an asynchronous TX is in progress (driver is SPI_BUSY due to async). If a synchronous TX is in progress, SetAsyncMode is still accepted (returns E_OK) because the sync path doesn't use the async mechanism.

Q10. Which APIs are allowed to be called from inside a Job or Sequence end notification callback?tap to reveal

Only: Spi_ReadIB, Spi_WriteIB, Spi_SetupEB, Spi_GetJobResult, Spi_GetSequenceResult, Spi_GetHWUnitStatus, Spi_Cancel. All other APIs (AsyncTransmit, SyncTransmit, Init, DeInit, SetAsyncMode) are NOT permitted — they could cause re-entrancy issues or deadlocks.

Q11. What is the difference between IB and EB channels?tap to reveal

IB (Internal Buffer): Buffer allocated inside the SPI driver. Fixed size (1 entry deep per channel). Written with WriteIB, read with ReadIB. Driver cannot prevent overwrite during TX. Best for small transfers (<10 bytes), daisy-chain.
EB (External Buffer): Buffer in application memory. Variable size (up to SpiEbMaxLength). Set up with SetupEB (providing src/dst pointers + length). Application owns consistency. Best for large streams, EEPROM, complex ASICs.

Q12. What is SPI_JOB_QUEUED and when does it appear?tap to reveal

SPI_JOB_QUEUED appears only in async Level 1/2. When Spi_AsyncTransmit(Seq) is called, all Jobs in that Sequence are immediately set to QUEUED (accepted but not yet executing). When actual transmission starts for a Job, it transitions to PENDING. This allows monitoring which Jobs are in the queue vs. actively running.

Q13. In interruptible sequences, what is the order of rescheduling and notifications?tap to reveal

Per SWS_Spi_00270: Rescheduling first, then notification. At the end of a Job: (1) the scheduler elects the next Job to run, (2) then the Job end notification is called. This ensures the HW is not idle while the notification callback executes.
