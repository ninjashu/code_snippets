What is the Crypto Driver? — "The security engine of the ECU"

The Crypto Driver is the lowest software layer in the AUTOSAR crypto stack, sitting directly above the hardware (HSM, AES accelerator, SW library). It abstracts crypto hardware capabilities into a uniform job-based interface consumed by the Crypto Interface (CRYIF) above it.

Mental model: The Crypto Driver is like a factory floor with specialized machines (Crypto Driver Objects). Each machine (e.g., AES accelerator, RSA engine, SW implementation) can do specific operations. The factory foreman (CRYIF) assigns jobs to machines based on capability. Jobs arrive with all data needed — key, plaintext, output buffer — and the machine does the work.

Key Architectural Concepts

Crypto Driver Object (CDO)An independent crypto "device" (HW or SW). E.g., AES-HSM accelerator, RSA-SW library. Each CDO has its own job queue and workspace. One CDO processes one job at a time. [SWS_Crypto_00014]
JobA configured CSM job referencing: a crypto primitive (service+algorithm), a key, input/output buffers, and an operation mode (START/UPDATE/FINISH). Jobs are routed through channels: CSM → CRYIF → CDO.
PrimitiveA specific algorithm implementation: Service (e.g. ENCRYPT) + Family (e.g. AES) + Mode (e.g. CBC) + SecondaryFamily. One CDO references multiple primitives to declare its capabilities.
KeyContainer with one or more key elements. Each key has a state (valid/invalid). Key material always in element ID=1. Key types define which elements exist (template).
Job QueueOptional per CDO. Priority-ordered. CryptoQueueSize=0 disables. Higher priority number = higher priority. Queue holds only START-mode jobs. [SWS_Crypto_00029, 00030]
ChannelPath from CSM queue through CRYIF to a specific CDO. Jobs are routed on channels. Channel width determines how many parallel operations are possible.

Operation Modes — The streaming model

Every crypto service uses START/UPDATE/FINISH operation modes, which can be combined in a single call:

ModeBit valueMeaning
CRYPTO_OPERATIONMODE_START0x01New request. Cancels any previous request for this job. Switches to Active state.
CRYPTO_OPERATIONMODE_UPDATE0x02Feed input data to the primitive. Can be called multiple times (streaming).
CRYPTO_OPERATIONMODE_FINISH0x04All data fed. Finalize and produce output. Returns to Idle state.
CRYPTO_OPERATIONMODE_SINGLECALL0x07 (OR of all)START+UPDATE+FINISH in one call. Most efficient for small data.
CRYPTO_OPERATIONMODE_SAVE_CONTEXT-Save intermediate workspace state (snapshot). Security risk — use only if needed.
CRYPTO_OPERATIONMODE_RESTORE_CONTEXT-Restore a previously saved workspace state.
Only one interface: Crypto_ProcessJob(). The operation mode embedded in the job struct tells the driver what to do. START always resets any previous state for that job. [SWS_Crypto_00017, 00020]

Key Constraints (Exam must-know)

1. One CDO = one job at a time [SWS_Crypto_00014]. n CDOs = n parallel jobs [SWS_Crypto_00117].

2. configPtr always NULL — Crypto_Init(NULL). [SWS_Crypto_00215]

3. No sequence diagrams — Section 9 explicitly says "n/a".

4. CDO 0 = default RNG — If any CDO needs random numbers but has no own RNG, uses CDO 0. [SWS_Crypto_00224, 00225]

5. Key material = element ID 1 — Always. Consistency across key types.

6. NVM write only on KeySetValid — Not on KeyElementSet. [SWS_Crypto_00242]

7. Post-build NOT supported — VARIANT-PRE-COMPILE only. [ECUC_Crypto_00001]

8. Multiple driver instances — Use naming convention Crypto_<vi>_<ai> to avoid collisions. [SWS_Crypto_00012]

9. HASH and RANDOMGENERATE — Only services that don't require a key. [SWS_Crypto_00072]

10. Custom service dispatchId formula: (ServiceInfoType<<24) | (AlgoFamilyCustomId<<16) | (AlgoMode<<8) | (ServiceId). [SWS_Crypto_00259]

Synchronous vs Asynchronous Processing

Synchronous [SWS_Crypto_00026]Result computed within the function call context. Function returns result directly. Configured in Crypto_JobType. NOT queued (returns CRYPTO_E_BUSY if CDO busy). Appropriate for SW crypto libs and fast operations.
Asynchronous [SWS_Crypto_00027, 00028]Function hands off job and returns immediately (E_OK = accepted). Actual computation done in Crypto_MainFunction(). On completion: CryIf_CallbackNotification() called with job + result. Queue used for async jobs.
Synchronous priority interrupt: If sync job priority > highest queue priority, Crypto Driver disables queue processing until the sync job completes and the next MainFunction cycle runs. [SWS_Crypto_00199]

CryptoKey

Runtime key instance. Has a unique CryptoKeyId. Has state: VALID or INVALID. References one CryptoKeyType (template). May reference an NvBlock for persistence.

Key state rules [SWS_Crypto_00038, 00039]:

Key INVALID → all crypto services using it return CRYPTO_E_KEY_NOT_VALID
Crypto_KeyElementSet() called → key state set to INVALID immediately (unknown state during write) [SWS_Crypto_00038 Note]
Crypto_KeySetValid() called → key state set to VALID (triggers NVM write if persist configured)
Crypto_KeySetInvalid() called → key state set to INVALID explicitly
After init: if all required key elements have a value → key state = VALID [SWS_Crypto_00251]

CryptoKeyElement — The data slot

Actual storage container. Identified by CryptoKeyElementId. Elements across different keys have separate memory areas (RAM + NV).

Access Control Hierarchy (Critical exam topic)

Access levels are ranked by value — lower value = higher privilege:

Read Access:
CRYPTO_RA_ALLOWED = 0x00 → can be read as plaintext
CRYPTO_RA_ENCRYPTED = 0x01 → can be read encrypted only
CRYPTO_RA_INTERNAL_COPY = 0x02 → can only be copied within same driver
CRYPTO_RA_DENIED = 0x03 → cannot be read at all
Write Access:
CRYPTO_WA_ALLOWED = 0x00 → can be written as plaintext
CRYPTO_WA_ENCRYPTED = 0x01 → can be written encrypted only
CRYPTO_WA_INTERNAL_COPY = 0x02 → can only be filled from internal copy
CRYPTO_WA_DENIED = 0x03 → cannot be written at all

Hierarchy rule: If element has RA_INTERNAL_COPY (0x02), it ALSO inherits permissions of RA_ENCRYPTED (0x01) and RA_ALLOWED (0x00) — i.e., element can also be read encrypted or as plaintext. This is by integer comparison. [SWS_Crypto_00219]

Key copy rule [SWS_Crypto_00220]: Source key element must have ≥ rights (≤ value) than destination. Source must have at least CRYPTO_RA_INTERNAL_COPY or lower (≤ 0x02) to be copied.

SHE Key Handling

For SHE (Security Hardware Extension) hardware keys: [SWS_Crypto_00255]

Set CryptoKeyElementFormat = CRYPTO_KE_FORMAT_BIN_SHEKEYS
Call Crypto_KeyElementSet() with SHE key in M1M2M3 format, keyElementId=1
Call Crypto_KeySetValid() → triggers actual load into SHE hardware
SHE returns M4M5 proof. Read with Crypto_KeyElementGet(keyElementId=2) = CRYPTO_KE_MAC_PROOF or CRYPTO_KE_CIPHER_PROOF

Sequence Diagrams (constructed from functional spec)

Sequence 1: Single-Call MAC Generation (AES-CMAC)
Precondition: Key is VALID state. CDO has MACGENERATE/AES/CMAC primitive configured.
CSM → CRYIF → Crypto_ProcessJob(objectId, job)
job.jobPrimitiveInputOutput.mode = CRYPTO_OPERATIONMODE_SINGLECALL
job.jobPrimitiveInfo.primitiveInfo.service = CRYPTO_MACGENERATE
job.cryptoKeyId = [key with AES material]
job.inputPtr = plaintext, job.inputLength = len
job.outputPtr = MAC buffer, job.outputLengthPtr = &buf_size
Crypto Driver checks:
Module initialized? objectId valid? job != NULL? Service supported? Algorithm supported? Key valid?
Buffer (outputPtr) != NULL? outputLengthPtr != zero?
Sync processing:
State: Idle → Active (START). Feed input (UPDATE). Finalize MAC (FINISH). State: Active → Idle.
*outputLengthPtr = actual MAC length written.
Return E_OK. CSM calls application callback.
If outputLengthPtr < full MAC size: Truncated output (MSB placed in buffer). Still E_OK. [SWS_Crypto_00065]
Sequence 2: Streaming AES-CBC Encryption (multi-call)
Call 1: Crypto_ProcessJob(..., mode=START)
Job: Idle → Active. Initialize AES-CBC workspace, load key, load IV from key element.
Return E_OK (sync) or callback started (async).
Call 2..N: Crypto_ProcessJob(..., mode=UPDATE)
Feed plaintext chunks. Each UPDATE may produce partial ciphertext output.
Job stays Active. Driver accumulates state.
Call N+1: Crypto_ProcessJob(..., mode=FINISH)
Feed final data (may be empty). Complete final block. Produce remaining ciphertext.
Job: Active → Idle. outputLengthPtr = total output.
If UPDATE called from Idle (no START first): Return E_NOT_OK. [SWS_Crypto_00118]
If new START during Active: Previous state reset, new job starts. [SWS_Crypto_00020]
Sequence 3: Async Job Processing
CSM → Crypto_ProcessJob(objectId, asyncJob)
Job processing = CRYPTO_PROCESSING_ASYNC (in job struct)
CDO busy OR queue has space → job queued. Return E_OK immediately.
CDO idle → job starts immediately. Return E_OK.
SchM calls Crypto_MainFunction() cyclically:
MainFunction processes next step of async job.
On completion (or error): Crypto_MainFunction calls CryIf_CallbackNotification(job, result).
CRYIF → CSM → application callback.
On error in async job [SWS_Crypto_00119]:
Job state → Idle. Data discarded. CryIf_CallbackNotification called with error code.
Sequence 4: Key Provisioning (full workflow)
Step 1: Write key material
App/KeyM → Crypto_KeyElementSet(keyId, 1, aesKeyData, 16)
Crypto Driver: key state → INVALID. Element ID=1 updated in RAM. Return E_OK.
Step 2: (Optional) Write IV or other elements
Crypto_KeyElementSet(keyId, CRYPTO_KE_CIPHER_IV, ivData, 16)
Step 3: Validate key
Crypto_KeySetValid(keyId)
Crypto Driver: check all element consistency. key state → VALID.
If CryptoKeyElementPersist=TRUE: call NvM_SetRamBlockStatus(). If IMMEDIATE: also NvM_WriteBlock().
NvM writes to non-volatile storage asynchronously.
Step 4: Check key status during NVM write
Crypto_KeyGetStatus(keyId, &status) → CRYPTO_KEYSTATUS_UPDATE_IN_PROGRESS
Step 5: NvM callback fires when write complete
Crypto_NvBlock_Callback_<NvBlock>(NVM_WRITE_BLOCK, NVM_REQ_OK) → status updated.
Sequence 5: Key Exchange (ECDH)
Step 1: Own private key already in key element ID=private of keyId.
Step 2: Calculate own public value
Crypto_KeyExchangeCalcPubVal(keyId, pubValBuf, &pubValLen)
Crypto Driver: uses private key from keyId. May use CDO 0 for random (ECDH needs random for ephemeral key).
Result: own public value in pubValBuf. Also stored internally in key element.
Step 3: Exchange public values with peer (out of band)
Send own pubVal to partner. Receive partner's pubVal.
Step 4: Calculate shared secret
Crypto_KeyExchangeCalcSecret(keyId, partnerPubValPtr, partnerPubValLen)
Crypto Driver: computes shared secret (ECDH). Stores in key element of keyId.
Application can now use shared secret for encryption key derivation.

Key Storage in NVM — Complete Guide

When keys are written to NVM

Keys are ONLY written to NVM when Crypto_KeySetValid() is called (or CRYPTO_KEYSETVALID primitive via ProcessJob). NOT on Crypto_KeyElementSet(). [SWS_Crypto_00242]

Rationale: Key elements may have interdependencies. Writing element by element could create an inconsistent key in NVM. Single-point write (KeySetValid) ensures all elements are consistent before NVM write.

NVM Write Modes

DEFERRED (CRYPTO_NV_BLOCK_DEFERRED)On KeySetValid: call NvM_SetRamBlockStatus() only. NvM will write to non-volatile memory later (on NvM_WriteAll or next scheduled write). Recommended for most cases.
IMMEDIATE (CRYPTO_NV_BLOCK_IMMEDIATE)On KeySetValid: call NvM_SetRamBlockStatus() AND NvM_WriteBlock(). Forces immediate write to non-volatile. Use for critical keys that cannot wait.

Key Reading at Startup

NvBlock state at startupActionSWS
Block available + consistency check OKLoad all key elements from NvBlock. Key set to VALID if all elements have values. [SWS_Crypto_00248, 00251]SWS_Crypto_00248, 00251
Block available but CORRUPTED (CRC/consistency fail)All related keys → INVALID. Persisted element lengths → 0. Init values NOT applied (security: prevent rollback to init state). [SWS_Crypto_00249]SWS_Crypto_00249
Block EMPTY (never written)Apply CryptoKeyElementInitValue for elements that have one. Key may become VALID if all required elements have init values. [SWS_Crypto_00250]SWS_Crypto_00250
NvBlock read fails (key load error)Key state → INVALID. Application should check key states at startup and handle. [SWS_Crypto_00198]SWS_Crypto_00198
Security rule [SWS_Crypto_00249 Rationale]: Corrupted NvBlock → init values NOT loaded (even if InitValue is configured). This prevents an attacker from corrupting NvBlock to force rollback to a known initial state (e.g., default test key).

NVM Retry on Failure

If NvM_SetRamBlockStatus() or NvM_WriteBlock() returns E_NOT_OK, OR Callback reports write failure: [SWS_Crypto_00244]

Report CRYPTO_E_RE_NVM_ACCESS_FAILED to DET (runtime error)
Retry on next Crypto_MainFunction() call
Maximum retries configured in CryptoNvBlockFailedRetries (1..65535)

Key Status During NVM Update

While key has been validated but NVM write is not yet confirmed: Crypto_KeyGetStatus() returns CRYPTO_KEYSTATUS_UPDATE_IN_PROGRESS. [SWS_Crypto_00245] This allows application to know the key is valid in RAM but not yet persisted.

Debugging Checklist

Problem: Crypto_ProcessJob returns CRYPTO_E_BUSY
CDO is processing another job (Active state) OR queue is full.
1. Check CryptoQueueSize — if 0, queue disabled. Sync jobs never queue. Async needs space.
2. Check if Crypto_MainFunction() is being called — if not, async jobs never complete, queue fills.
3. Check job priority — lower priority async jobs may be waiting. High-priority sync job skips queue.
4. Check if SINGLECALL used for async jobs — recommended for throughput [spec note after SWS_Crypto_00032].
Problem: Key operations return CRYPTO_E_KEY_NOT_VALID
Key is in INVALID state. Crypto services check this before processing. [SWS_Crypto_00039]
1. Was Crypto_KeySetValid() called after Crypto_KeyElementSet()? KeyElementSet makes key INVALID.
2. Check if NVM load failed at startup → Crypto_KeyGetStatus() to inspect.
3. Check if Crypto_KeySetInvalid() was called explicitly.
4. Check if key elements are all present — init checks for valid elements post-startup [SWS_Crypto_00251].
Problem: CRYPTO_E_KEY_EMPTY returned
Key element has never been written (no data, no init value). Different from INVALID state.
1. Key was provisioned? Check if Crypto_KeyElementSet() was ever called for element ID=1.
2. Check CryptoKeyElementInitValue — if absent AND element never written → EMPTY.
3. For SHE: ERC_KEY_EMPTY equivalent. SHE hardware key slot was never loaded.
Problem: NVM key loss after ECU reset
Key set valid but NVM write failed or was DEFERRED and not completed.
1. Check Crypto_NvBlock_Callback — was it called with NVM_REQ_OK?
2. Check DET for CRYPTO_E_RE_NVM_ACCESS_FAILED — indicates write failure.
3. If DEFERRED mode: was NvM_WriteAll() called before ECU shutdown? Or was NvM configured for write-on-exit?
4. Check NvMSelectBlockForWriteAll=TRUE for key NV blocks (recommended).
5. Consider IMMEDIATE mode for critical keys.
Problem: Signature verification fails unexpectedly
1. Check key element format — public key must match expected format (e.g., BIN_OCTET for ECC, BIN_RSA_PUBLICKEY for RSA).
2. Check key element size — RSA key size must match configured algorithm key length.
3. Verify ECC coordinate byte order — must be little-endian per spec [SWS_Crypto_00190].
4. Check if hash algorithm in CSM primitive matches signature algorithm (family + secondary family combination).
5. Check input redirection — if signature data comes from key element, access rights must be CRYPTO_RA_INTERNAL_COPY or lower. [SWS_Crypto_00221]

Common Integration Mistakes

1. Not calling KeySetValid after KeyElementSet — Key stays INVALID. All crypto ops fail with KEY_NOT_VALID. Most common provisioning bug.
2. CryptoPrimitive has invalid Family+Mode combination — e.g., declaring RSA+CBC together. Both CDO capability and CSM job must use valid combinations. DET at runtime or no matching primitive found.
3. Crypto_MainFunction() not scheduled — Async jobs accepted (E_OK) but never processed. Queue fills up. CRYPTO_E_BUSY for all subsequent requests.
4. NvM not configured with Crypto callbacks — NvM reads/writes but Crypto never notified. Keys lost after reset. NvMInitBlockCallback, NvMReadRamBlockFromNvCallback, NvMWriteRamBlockToNvCallback, NvmSingleBlockCallbackFnc must all be set.
5. CDO IDs not starting from 0 or not consecutive — Spec requires consecutive gapless IDs starting from 0. CDO 0 must be configured as default RNG provider.
6. Custom algorithm IDs outside 128..254 range — Custom IDs must be in this reserved range [SWS_Crypto_00240]. Values 0..127 are standard enums.
