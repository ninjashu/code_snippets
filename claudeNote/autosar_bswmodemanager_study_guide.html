What BswM does — one sentence

Receives mode requests/indications from SW-Cs and BSW modules → evaluates Boolean rules → executes ordered action lists to switch ECU modes.

Two core tasks

Mode Arbitration

Evaluate Boolean rules on incoming requests. Decide what mode to switch to.

Mode Control

Execute Action Lists. Call other BSW APIs. Notify SW-Cs via RTE switch.

Constraints (exam trap)

Max one BswM instance per partition. Multi-core = separate config per partition.

Action list nesting max 7 hierarchic levels — enforced by generator tool.

Arbitration pipeline (memorise this order)

1
Input arrives — mode request (SW-C or BSW) or event request (SET/CLEAR)
2
Port updated — IMMEDIATE: updated just before arbitration. DEFERRED: updated on trigger, arbitrated in next MainFunction.
3
Rule evaluated — Boolean expression: AND/OR/XOR/NOT/NAND on ModeConditions
4
Action list selected — True list or False list, based on result
5
Actions executed — ordered by BswMActionListItemIndex (lowest first)
6
RTE switch called — mode switch notification sent to subscribed SW-Cs

IMMEDIATE vs DEFERRED — exam comparison

IMMEDIATE

Arbitration runs in caller's context (can be ISR)

Port updated just before arbitration fires

Only rules using that IMMEDIATE port are re-evaluated

Nested IMMEDIATE postponed until current finishes

DEFERRED

Arbitration waits for BswM_MainFunction cycle

Port updated when triggered

All rules with ≥1 DEFERRED port evaluated each cycle

Safer for complex multi-condition arbitration

Timer always DEFERRED — BswMRequestProcessing config ignored for BswMTimer; always arbitrated in MainFunction.

Event request port states (BSWM_EVENT_IS_SET / CLEARED)

Port = SET when

BswM_ComM_InitiateReset (or any event API) called

Port = CLEAR when

BswMClearEventRequest action executed

All EventRequestPorts initialised to CLEAR on BswM_Init.

Action list execution modes

BSWM_TRIGGER — runs only when rule changes (False→True fires True list; True→False fires False list)

BSWM_CONDITION — runs every evaluation that matches (True every cycle = True list every cycle)

If multiple rules trigger same top-level action list → only one execution per arbitration round [SWS_BswM_00223].

Init behaviour — BswMRuleInitState × BswMActionListExecution

Key: CONDITION always runs. TRIGGER only runs on change — so TRUE+TRIGGER won't run True list again on first TRUE eval.

DEFERRED processing sequence (Fig 9.1)

1
SW-C calls Rte_Write_<p>_<o>() — stores mode request in RTE
2
OS task fires BswM_MainFunction() cyclically via SchM
3
Rte_Read_<p>_<o>() reads buffered request into BswM
4
ModeArbitration() — evaluates all DEFERRED rules
5
BswActions() — executes triggered action lists
6
Rte_Switch_<p>_<o>() + Rte_Feedback() — notify subscribers

IMMEDIATE processing sequence (Fig 9.2)

1
SW-C calls Rte_Write_<p>_<o>() — RTE triggers data-receive event in BswM
2
ModeArbitrationRunnable() fires in SW-C caller context
3
ModeArbitration() — only rules using this IMMEDIATE port evaluated
4
BSW_Actions() executed immediately (still in caller context)
5
Rte_Switch_<p>_<o>() — mode switch notified immediately (no MainFunction wait)

BswM_Init() — what happens inside

1
Requires only OS + SchM to be init'd first [SWS_BswM_00118]
2
If BswMDevErrorDetect enabled → validate ConfigPtr. Error = BSWM_E_PARAM_CONFIG → abort init
3
Init all module global variables
4
All BswMEventRequestPorts → CLEAR state
5
All BswMTimers → STOPPED state
6
BswMModeInitValues applied to ports. Undefined ports skipped in arbitration until first update.

Postponement rules during busy state

Request during MainFunction execution: IMMEDIATE postponed → processed before MainFunction exits. DEFERRED postponed → next MainFunction cycle.

Request during IMMEDIATE execution: new IMMEDIATE processed after current IMMEDIATE finishes. DEFERRED → next MainFunction.

Core API quick reference

BswM_Init(ConfigPtr)ServiceID 0x00. Initialises module. Void return. Called once after OS+SchM init.
BswM_Deinit()De-initialises module. Stops all activity.
BswM_MainFunction()ServiceID 0x03. Scheduled by SchM. Evaluates all DEFERRED rules. Non-reentrant. No params, void return.
BswM_RequestMode(user, mode)SW-C or BSW requests a named mode. BswM_UserType + BswM_ModeType (both uint16).
BswM_CanSM_CurrentState(network, state)CanSM indication. Triggers arbitration per IMMEDIATE/DEFERRED config.
BswM_ComM_CurrentMode(channel, mode)ComM indicates comm mode change.
BswM_EcuM_CurrentState(state)EcuM indicates current state (e.g. RUN, POST_RUN). BswM uses for arbitration.
BswM_ComM_InitiateReset()Event-type request. Sets EventRequestPort to SET state. No mode value carried.
BswM_GetVersionInfo(info)Returns module version. Standard AUTOSAR BSW version info pattern.

Available action types

1. Predefined BSW actions — BswMAvailableActions: ComM mode, PDU group switch, NvM read/write, LinSM schedule, Rte mode switch, timer control, firewall state, etc.

2. User callouts — any custom function. Params defined at config time via BswMUserCallout.

3. Action list reference — link to another action list (max 7 levels). Nested action list executes before parent continues.

Action list ordering rules

Items ordered by BswMActionListItemIndex — lowest first. Gaps in index OK. Duplicate index values → config rejected.

Multiple top-level lists ready → executed highest BswMActionListPriority first (0 = lowest if unset).

Action returns E_NOT_OK + BswMAbortOnFail=true → action list aborted. Det runtime error reported.

Runtime errors

Action returned E_NOT_OK BSWM_E_ACTION_FAILED 0x80–0xFF

ErrorId value = BswMReportFailRuntimeErrorId config param. ApiId is implementation-specific (depends on IMMEDIATE/DEFERRED context).

No production errors and no extended production errors in BswM.

Testing & debugging checklist (automotive domain)

T
Enable BswMDevErrorDetect — activates all parameter range checks and UNINIT guards during development/test builds
T
Hook Det callback — all errors reported via Det_ReportError/Det_ReportRuntimeError. Log moduleId=BswM, serviceId to identify failing API
T
Undefined mode condition — if ModeInitValue absent, condition is UNDEFINED; rule won't arbitrate until first update. Common bug: rule never fires at startup.
T
Action list abort tracing — configure BswMReportFailRuntimeErrorId per action item. BSWM_E_ACTION_FAILED in Det log tells you which action failed
T
Timing: MainFunction period — BswMMainFunctionPeriod config param. DEFERRED accuracy = worst case = 1 period. Verify against communication timing budget.
T
Exclusive area / reentrancy — BswM can use OS exclusive areas to protect action list from interruption. Check integration: IMMEDIATE requests from ISR context + action list must not call non-ISR-safe functions
T
Multi-partition — each partition has separate BswM instance. Config mistakes show as UNINIT or PARAM_CONFIG errors in wrong partition.
T
Generator tool validates: max 7 nesting levels, no duplicate ActionListItemIndex, BswMReportFailRuntimeErrorId in 0x80–0xFF. Fail at generation = config error, not runtime.

7 hierarchic levels [SWS_BswM_00037]. The limit exists to make testing of BswM implementations and generator tools feasible. Without a limit, arbitration/execution chains become untestable. Enforced at generation time, not runtime.
Timer mode becomes BSWM_TIMER_EXPIRED. It is arbitrated AGAIN in the SAME cycle it expired. It does NOT auto-reset to STOPPED. Every subsequent MainFunction will continue to arbitrate it as EXPIRED until a BswMTimerControl action explicitly sets it to STOPPED or STARTED again.
